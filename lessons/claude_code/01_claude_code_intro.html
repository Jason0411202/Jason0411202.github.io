🔥 必學指令
每天都會用到，不會就是在浪費 token 和時間
#1 /init 原生 +

分析 codebase，建立 CLAUDE.md 配置文件 — 任何新專案的第一步

是什麼
/init 會掃描你的 repo，讀取 package.json、Cargo.toml、go.mod、CI 配置、README 等關鍵文件，然後和你對話確認細節，最後生成 CLAUDE.md 和 CLAUDE.local.md。這兩個文件會在每次 session 開始時自動載入，讓 Claude 知道你的專案慣例，不需要每次重複說明。
使用範例
/init 完整初始化，Claude 會訪談你確認細節
💡 CLAUDE.md 已存在時再跑 /init，Claude 只會提出「修改建議」不會覆蓋。可以隨時重跑。
生成的文件
CLAUDE.md 團隊共享（commit 進 git）—— build/test/lint 指令、架構說明、命名慣例
CLAUDE.local.md 個人私有（gitignore）—— 你自己的偏好設定，不影響團隊
何時使用
clone 新 repo 後 加入新專案第一天 專案架構大改後 想更新 Claude 對專案的認知
#2 /clear alias: /reset /new 原生 +

完全清除對話記錄，開啟全新 session（比退出重啟更快）

是什麼
清空所有對話歷史，釋放 context window，從同一個工作目錄重新開始。不是 /compact，不會做摘要，就是直接清掉。CLAUDE.md 裡的記憶依然保留。
使用範例
/clear 完全重置，開始新任務
🔥 高頻使用：每次開始不相關的新任務就清。舊歷史只會消耗 token、讓 Claude 被無關資訊干擾。不要捨不得清。
vs /compact
/compact = 保留摘要，繼續同一個任務  |  /clear = 完全重置，換新任務
#3 /permissions alias: /allowed-tools 原生 +

管理工具允許/禁止清單 — 減少 Claude 每次問你「可以嗎」

是什麼
Claude 預設每次執行 bash 指令、修改檔案都會問你。用 /permissions 預先設定「這些操作不用問」，讓工作流更順暢。設定存在 .claude/settings.json。
常用允許規則
Bash(git:*) 所有 git 指令不用問
Bash(npm:*) 所有 npm 指令不用問
Edit(src/**/*.ts) src 下所有 TS 檔可自由編輯
Bash(pnpm test:*) 跑測試不用問
⚡ 把常用操作預先允許，可以讓 Claude 做到你去喝咖啡回來已經完成，而不是一直停下來等你點確認。
#4 /help 原生 +

列出當前 session 所有可用指令（含 plugin 和 skill 新增的）

是什麼
最權威的指令清單來源。不同 session 可能有不同的 skills 和 plugins，/help 永遠顯示「當前環境」實際可用的所有指令。
何時使用
忘記有什麼指令 裝了新 plugin 確認是否生效 第一次用新環境

📦 Context 管理
Context window 是最珍貴的資源，要主動管理
#1 @filepath 原生 +

在對話中引用特定檔案，精準控制 context 內容

使用範例
@src/auth/login.ts 把這個檔案加入 context
@README.md 幫我改進這份說明 引用 + 下指令
@src/components/ 有哪些元件？ 引用整個目錄
💡 比「幫我看這個專案的 auth 邏輯」更精準。讓 Claude 只讀你要它讀的，不浪費 context 掃整個 codebase。
#2 # text  （對話中快速存入記憶） 原生 +

用 # 開頭快速把規則存入 CLAUDE.md，不中斷工作流程

使用範例
# 永遠用 pnpm 不用 npm 存入，下次 session 自動記住
# 用 zod 做所有 API validation 存入技術規範
# 回覆永遠用繁體中文 存入語言偏好
#3 !command  （直接執行 bash） 原生 +

不離開 Claude Code 就能執行 bash 指令，結果自動加入 context

使用範例
!ls src/ 查看目錄結構
!git log --oneline -10 查看最近 commit
!cat .env.example 查看環境變數範本
💡 執行結果會直接出現在對話中，Claude 可以看到並分析。適合查環境狀態、確認指令輸出。
03
🔧 工作流程指令
提升開發品質、自動化重複工作
#1 /simplify [focus] 內建 Skill +

平行跑 3 個 review agent，自動清理剛寫完的程式碼

是什麼
實作完功能後的「自動清潔」。Claude 同時派出三個 agent，分別從 code reuse（有沒有重複邏輯）、code quality（可讀性、命名）、efficiency（效能）三個維度 review 你剛改的檔案，匯總問題後直接修正。
使用範例
/simplify 全面清理最近改動
/simplify focus on memory efficiency 重點看記憶體效率
/simplify focus on type safety 重點看型別安全
最佳時機
實作完一個 feature 後 修完一個 bug 後 提 PR 前清理
#2 /review 原生 +

Code review 當前改動，找 bug 和 security issue

使用時機
提 PR 前自我 review 完成一個 feature 確認 不確定改動是否安全
#3 /mcp [enable|disable server-name] 原生 +

管理 MCP server — 讓 Claude 連接資料庫、GitHub、外部 API

常用 MCP
github 直接開 PR、查 issue、review PR
postgres / sqlite 直接查詢資料庫
filesystem 擴充檔案系統存取權限
slack 讀寫 Slack 訊息
使用範例
/mcp enable github 啟用 GitHub MCP
/mcp 查看所有 MCP 狀態
#4 /hooks 原生 +

查看目前 hook 設定 — 存檔後自動跑 lint/format/typecheck

是什麼
Hooks 是在特定事件（如存檔後、bash 執行前）自動觸發的 shell 指令。例如：每次 Claude 修改 .ts 檔案後，自動跑 tsc --noEmit 確認型別正確。
常用 hook 情境
PostToolUse (Edit) 存檔後自動跑 prettier
PostToolUse (Edit *.ts) 改 TS 後自動 type check
PreToolUse (Bash) 執行 bash 前先驗證安全
💡 設定好 hooks 後，Claude 改完程式碼，lint/format/typecheck 自動跑，只有通過才接受。品質不用靠人盯。

// CORE CONCEPT
Instinct 系統：ECC 的大腦
ECC 的核心是「instinct」—— 帶信心分數（0.3–0.9）的原子化知識單元，透過 hooks 在 100% 的操作中自動捕捉。

Scope 規則：instinct 預設是 project-scoped（存在 ~/.claude/homunculus/projects/<git-hash>/）。React 習慣留在 React repo，Python 慣例留在 Python repo。只有「永遠用 pnpm」這類工具偏好，或同一規則在 3+ 個 repo 高信心出現，才自動升為 global。
project-scoped 預設
信心分數 0.3–0.9
hooks 100% 可靠捕捉
export 給團隊共享
/learn
🔀 scope 自動判斷 高頻使用

主動教 Claude 一條規則，存入 instinct 系統。比 # 規則 更智慧：有信心分數、會隨使用演化、可以 export 給團隊。系統自動判斷這條規則適合 project 還是 global scope。

語法 /learn <自然語言描述的規則>
# 永遠用 pnpm → 直接寫進 CLAUDE.md，每次 session 全部載入
/learn always use pnpm → 存進 instinct 系統，依情境觸發，有信心分數，會演化
範例 — 由簡單到複雜
Lv.1  工具偏好（自動 → global）
/learn always use pnpm, never npm or yarn
工具偏好適用所有 repo → 系統自動存為 global scope
Lv.2  技術規範（→ project scope）
/learn use zod for all input validation in this API; never trust raw request.body directly
這個 repo 特有的技術選型 → project scope，只在這裡觸發
Lv.3  架構決策（→ project scope）
/learn prefer Result<T, E> over throwing exceptions for business logic errors; only throw for truly unexpected system failures like DB connection drop
錯誤處理模式，Claude 寫函式時自動用 Result 型別
Lv.4  複合慣例（→ project scope）
/learn in this React app: - co-locate tests in __tests__/ next to the component - use React Testing Library not Enzyme - mock at network boundary with MSW, not at module level - test behavior not implementation details
整套測試慣例，Claude 新增測試時全部自動遵循
📂 Scope 自動判斷邏輯
Global：工具偏好、語言習慣、回覆格式 →「永遠用 pnpm」「回覆用繁體中文」
Project：技術選型、框架慣例、架構決策 →「用 zod」「React 用 RTL」
自動晉升：同一規則在 3+ repo 高信心出現 → /instinct-evolve 建議 promote to global
/evolve  alias: /instinct-evolve
project → global 中頻使用

讓 Claude 分析這次 session 的 observations，主動提取 pattern 存為 instinct，並把信心分數夠高的 instinct 晉升為 skill 或 command。ECC 持續學習的主動觸發器。

語法 /evolve
完整使用流程
步驟 1  正常工作（hooks 自動記錄）
# 正常和 Claude 工作 # ECC hooks 在背景把每次 prompt 和 tool use 寫入： # ~/.claude/homunculus/projects/<hash>/observations.jsonl
你完全不需要做任何事，hooks 100% 可靠，不會漏掉任何操作
步驟 2  完成某件事後執行
/evolve
Claude 分析 observations：找你糾正 Claude 的地方、重複出現的 pattern、成功解決問題的方法 → 生成 instinct
步驟 3  審閱並管理
/instinct-status # → 列出所有 instinct，含 confidence score 和 scope # 可以手動刪除不想要的
確認 Claude 學到的是你要的，不對的就刪掉
步驟 4  晉升高信心 instinct
/instinct-evolve # → 建議哪些 instinct 可以 promote 為 skill 或 command # confidence 接近 0.9 的會被建議晉升
晉升後整個 team 都能用（需要 /instinct-export 然後 commit）
📂 存放路徑
~/.claude/homunculus/
├── projects/
│   └── a1b2c3d4/                    ← git remote URL hash
│       ├── observations.jsonl        ← hooks 自動寫入
│       └── instincts/personal/       ← /evolve 生成在這
│           ├── prefer-zod.yaml
│           └── rtl-testing.yaml
└── instincts/personal/              ← global（promote 後到這）
    └── always-use-pnpm.yaml
/instinct-status  /export  /evolve  /prune
project / global 低頻維護

管理 instinct 系統的四個維護指令，定期整理確保系統乾淨有效。

四個指令詳解
/instinct-status  —  查看現狀
/instinct-status # 輸出範例： # [project] prefer-zod confidence: 0.82 trigger: "API validation" # [project] rtl-testing confidence: 0.71 trigger: "writing tests" # [global] always-pnpm confidence: 0.95 trigger: "package install" # [project] avoid-any-type confidence: 0.45 trigger: "TypeScript"
了解 Claude 學到什麼，信心分數多高，哪些已達晉升標準
/instinct-export  —  匯出給 team
/instinct-export # → 生成 .claude/instincts/team.yaml # commit 進 repo → 其他成員 git pull 後自動載入 # 可以 import 到其他 repo： /instinct-import .claude/instincts/team.yaml
知識共享：你學到的 pattern，整個 team 的 Claude 都能受益
/instinct-evolve  —  強制 evolution cycle
/instinct-evolve # 重新評估所有 instinct 的 confidence # 更新分數（用了且有效 → 升，用了但效果差 → 降） # 建議晉升：confidence >= 0.85 的 → 可以 promote 為 skill
每隔幾週跑一次，保持 instinct 系統的準確性
/instinct-prune  —  清理雜訊
/instinct-prune # 刪除 confidence <= 0.3 的 instinct # 這些是：從未被觸發過的、或被觸發後沒有幫助的 # 定期跑避免系統因雜訊累積而變慢
建議每個月跑一次，保持系統精簡
Confidence Score 說明
0.3剛建立，試探性，可能不準確，不會強力影響 Claude 行為
0.6已被使用多次且有正面效果，開始穩定影響行為
0.85+高度可信，建議晉升為 skill 或 command，正式加入工具箱
/checkpoint
project 中頻使用

在工作流程中建立具名存檔點，記錄 session context、進度、關鍵決策。之後可以回溯或從這個點分支。類似 git tag，但針對 Claude session 狀態。

語法 /checkpoint <name> /checkpoint   # 不給名稱用時間戳
範例 — 由簡單到複雜
Lv.1  完成某個功能後存檔
/checkpoint auth-feature-done # 之後告訴 Claude："回到 auth-feature-done 的狀態" # Claude 會讀取這個 checkpoint 的 context 繼續工作
簡單好習慣，完成 milestone 就存一個
Lv.2  高風險操作前
git stash # 先把 code 狀態也存起來 /checkpoint before-db-migration # 開始跑危險的 migration... ALTER TABLE users ADD COLUMN subscription_tier VARCHAR(20)... # 如果出問題： # "請回到 before-db-migration，我們重新規劃 migration 策略"
高風險操作前的雙重保險：git stash + checkpoint
Lv.3  多方案探索
# 探索方案 A /checkpoint exploring-approach-A # 實作方案 A... # 方案 A 不理想，換 B /checkpoint exploring-approach-B # 實作方案 B... # 方案 B 也不對，回到 A 繼續改進 # "回到 exploring-approach-A，繼續在那個方向優化"
架構設計時，平行探索不同方向，不用怕走錯路
💡 和 git 搭配用：checkpoint 記錄的是 Claude session context，git 記錄的是 code 狀態。兩個都要，才能完整回溯。
/sessions
global 中頻使用

查看、搜尋、匯出過去的 session 記錄。Session 存在 ~/.claude/session-data/，跨 repo、跨時間都可以搜尋。

語法（多個子指令） /sessions                 # 列出最近 sessions /sessions search <keyword>  # 搜尋包含關鍵字的 session /sessions branch <id>      # 從某個 session 分支出來 /sessions export <id>      # 匯出成 markdown 文件 /sessions metrics           # 使用量統計 /sessions compact <id>     # 壓縮舊 session 節省空間
常見使用情境
找回上次的解法
/sessions search "race condition" # → 找到 2 週前的 session，當時解決了 mutex 問題 /sessions branch sess_20260315_142301 # → 從那個 session 分支，帶著當時的 context 繼續
不用重新解釋問題背景，直接繼承當時的 context
把 session 變成設計文件
/sessions export sess_20260320_090000 # → 輸出 design-decision-2026-03-20.md # 包含：決策過程、討論的方案、最終選擇的理由 # 存進 docs/ADR/（Architecture Decision Records）
重要的架構討論不要讓它消失在 session history 裡
查成本統計
/sessions metrics # 輸出： # 本週 sessions: 23 個 # 總 token 使用: 2.4M input / 890K output # 最常用指令: /compact (18次), /review (12次) # 最長 session: 4.2hr on 2026-04-02
了解自己的使用模式，找出哪裡可以優化
/code-review
project 中頻使用

完整的 PR review pipeline：讀 PR 全文（非只看 diff hunk）、跑 CI 驗證、輸出帶決策的結構化報告，存到 .claude/PRPs/reviews/。

語法 /code-review      # review 當前 branch /code-review PR#123  # review 指定 PR
輸出範例
典型輸出結構
PR #47: Add payment processing module Decision: REQUEST_CHANGES Issues: 2 critical, 1 high, 3 medium, 0 low Validation: 4/6 checks passed Critical Issues: [security] API key hardcoded in src/payment/client.ts:23 → Move to environment variable, rotate immediately [logic] Race condition: concurrent payments can double-charge → Add distributed lock or idempotency key High Issues: [test] No error handling tests for network failures (timeout, 5xx) Validation Results: ✅ TypeScript typecheck — passed ✅ ESLint — 0 errors ❌ Test coverage — 62% (required: 80%) ✅ Build — succeeded ❌ Security scan — 1 hardcoded secret found ✅ No console.log — clean Report saved: .claude/PRPs/reviews/pr-47-review.md
APPROVE / REQUEST_CHANGES / BLOCK 三種決策。Critical issue → 自動 BLOCK
原生 /review → 快速，看 diff 給意見，不跑 CI，無正式報告
ECC /code-review → 讀全文脈、跑所有 CI、輸出結構化決策報告到檔案
/loop-start  &  /loop-status
project 中頻使用

啟動自動迭代迴圈：Claude 持續執行，直到你指定的條件達成。有明確完成條件的任務可以完全自動化，你可以去做別的事。

語法 /loop-start <完成條件描述> /loop-status    # 查看迴圈目前狀態
範例 — 由簡單到複雜
Lv.1  跑測試直到全通過
/loop-start run pnpm test until all tests pass # Claude 自動：跑測試 → 看失敗 → 修 code → 再跑 → ... # 直到全綠才停止
最常用。寫完 code 讓 Claude 自動跑到 green，你去泡茶
Lv.2  修 lint 直到乾淨
/loop-start fix all ESLint errors until `pnpm lint` exits with code 0 # 查看進度： /loop-status # → Round 3/? | Fixed 12 errors | 4 remaining | ETA ~2min
大量 lint 錯誤時很有用，不需要一個一個手動修
Lv.3  多重完成條件
/loop-start keep iterating until ALL of these pass: 1. pnpm typecheck exits 0 (no TypeScript errors) 2. pnpm test exits 0 with coverage >= 80% 3. pnpm lint exits 0 (no ESLint errors) 4. pnpm build exits 0 (production build succeeds)
CI pre-check：確保所有關卡通過後再 commit。適合跑完整個晚上
⚠️ 注意：沒有內建次數上限。Claude 陷入死循環時（修了 A 壞了 B，修了 B 壞了 A）要手動 Ctrl+C 中斷。建議在 git feature branch 上跑，保護 main branch。
/quality-gate
project 一次設定

定義這個 repo 的最低品質標準。設定後，Claude 每次完成工作都自動驗證，未通過會繼續修。一次設定，永久生效，存在 .claude/settings.json。

語法 /quality-gate              # 互動式設定 /quality-gate status        # 查看當前設定 /quality-gate set: <描述>   # 直接描述關卡
設定範例
互動式設定（推薦新手）
/quality-gate # Claude 引導你選擇： # ✓ TypeScript typecheck # ✓ ESLint (zero errors) # ✓ Unit tests pass # ○ Test coverage >= ? → 你輸入：80 # ✓ Build succeeds # ○ Security scan # → 設定完成，存入 .claude/settings.json
互動式，Claude 會建議常用關卡讓你選擇
進階：直接描述
/quality-gate set: - pnpm typecheck must exit 0 - pnpm lint must exit 0 with zero errors - pnpm test must pass with coverage >= 80% - pnpm build must succeed - no console.log in src/ (except debug files) - no hardcoded secrets (run git-secrets scan)
完整的 production-grade 品質標準
查看現況
/quality-gate status # 輸出： # Gate: typecheck ✅ last run: 2 min ago # Gate: lint ✅ last run: 2 min ago # Gate: test ✅ coverage: 84% # Gate: build ✅ last run: 2 min ago # Gate: secrets ❌ 1 issue found → details below
一眼看出哪個關卡沒過
/harness-audit
project + global 低頻診斷

全面審計 ECC 環境健康狀態，帶分數的診斷報告加具體建議。剛裝 ECC、感覺 Claude 行為怪怪的、懷疑設定沒生效時跑。

輸出範例
典型輸出（評分 74/100）

涵蓋範圍
