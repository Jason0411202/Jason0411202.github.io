01 — 全景

Azure DevOps 全景：從 Organization 到 5 個 Hub

Azure DevOps（簡稱 ADO）是微軟一站式 DevOps 平台。和 GitHub 不一樣的地方在於它把「專案管理」「版本控制」「CI/CD」「測試計畫」「製品儲存庫」包成同一個產品 — 也因此左側導覽會看到 5 個 hub：Boards / Repos / Pipelines / Test Plans / Artifacts。

圖 23.1 — Azure DevOps 階層：Organization → Project → 5 個 Hub

權限模型基礎

ADO 的存取控制以 Security Group 為核心，每個 Project 預設有四個內建群組：

Group能做什麼常見用途
Project Administrators所有 Project 設定，含建立 service connection、刪除 repoTech lead / DevOps engineer
Contributors建立 / 編輯 work item、push commit、開 PR、執行 pipeline一般開發者（預設群組）
Readers只讀，看不到 secret 內容稽核者、產品經理視需要
Stakeholders (免費 license)看 Boards、編輯 work item，看不到 Repos / Pipelines業務 / PM 不寫程式但要追進度
i
Stakeholder license 是 ADO 一個被低估的省錢方案 每位 Basic license 用戶大約 6 美金/月，但 Stakeholder 是完全免費。一個有 50 人的公司，如果有 30 個是 PM/業務只看 Boards，那一年省下接近 2,000 美金。在 Organization Settings → Users 可以改 access level。

常用導覽路徑

  • My Work hub（左下角頭像旁邊的圖示）：跨所有 project 列出指派給你的 work item、你開的 PR、你正在 review 的 PR — 開工第一個要看的頁面。
  • Project Settings（左下角齒輪）：service connection、agent pool、repository policy、pipeline 變數群組都在這裡。
  • Organization Settings：billing、parallel jobs 配額、跨 project 的擴充功能。
02 — 專案管理

Azure Boards：Process 模板、階層、Sprint

Boards 是 ADO 的 work item 系統，但建立 project 時選的 Process 模板會決定後面所有事情 — 模板選錯後面要改非常痛苦。

四種 Process 模板比較

模板頂層階層狀態流適合對象
BasicEpic → Issue → TaskTo Do / Doing / Done新團隊、輕量需求
Agile （推薦）Epic → Feature → User Story → TaskNew / Active / Resolved / Closed大多數軟體團隊（本課採用）
ScrumEpic → Feature → PBI → TaskNew / Approved / Committed / Done嚴格 Scrum 流程的團隊
CMMIEpic → Feature → Requirement → TaskProposed / Active / Resolved / Closed受規範產業（金融、醫療、航太）
i
選 Agile 不會錯 User Story 比 PBI 對非工程角色更直覺、欄位少；Bug 是獨立 work item type 而非 task；狀態流剛好夠用。Agile 可以後續客製，但 Process 一旦建立就不能改成另一個內建類型 — 只能複製成 Inherited Process 再修改。

Work Item 階層範例

以一個典型的 backend 重構需求為例：

work-item-tree 
# Epic — 半年級的策略目標
Epic      "Containerise the example backend"
├── Feature   "Build container image in CI"
│   ├── User Story  "As a developer, push to main builds an image"
│   │   ├── Task  "Write Dockerfile (multi-stage)"
│   │   ├── Task  "Add Docker@2 task to azure-pipelines.yml"
│   │   └── Task  "Tag image with commit SHA"
│   └── User Story  "Push image to container registry"
└── Feature   "Deploy via GitOps manifest update"
    └── User Story  "Pipeline updates values.yaml on tag push"

Sprint、Iteration、Capacity

在 Project Settings → Project configuration → Iterations 設定 sprint 週期。Boards 內 Sprint 頁面提供：

  • Backlog：drag 排優先順序
  • Taskboard：sprint 內以 user story 為列、status 為欄的看板
  • Capacity：每位成員設定可用工時 + 休假，自動算出 sprint 容量
  • Analytics：burndown、velocity 自動產生（搭配 Dashboards 使用）

Kanban Board 客製

Boards 頁面右上 ⚙️ 圖示 → Configure team settings 可以加 column、設 WIP limit、加 swimlane。

✗常見錯誤
預設 4 欄（New / Active / Resolved / Closed）就直接用，沒設 WIP limit，結果 Active 欄堆 30 張卡，沒人知道誰在做哪張。
✓實務做法
拆成 To Do / In Dev (WIP=3) / In Review (WIP=2) / QA / Done。Swimlane 拉出 Expedite（線上事故）讓緊急卡優先。

Queries、Delivery Plans、Wiki

  • Queries：用 WIQL 寫過濾條件儲成 Shared Query。例：「My team this sprint, state ≠ Done, sort by priority」。Dashboard widget 可以直接吃 query。
  • Delivery Plans：跨 team 的 Gantt 視圖，看哪個 feature 卡哪個 sprint。
  • Dashboards：每個 project 可以自訂 dashboard，常用 widget：Sprint burndown、Cumulative Flow Diagram (CFD)、Build history、Pull request 列表、Markdown（放團隊 SLO）。
  • Wiki：放 runbook、on-call 流程、架構決策記錄。Wiki 本身就是 Git repo，可以直接 clone 編輯。
WIQL 查詢範例：「sprint 內未完成的 user story」 進階
wiql 
SELECT [System.Id], [System.Title], [System.AssignedTo], [System.State]
FROM WorkItems
WHERE [System.TeamProject] = @project
  AND [System.WorkItemType] = 'User Story'
  AND [System.IterationPath] = @currentIteration
  AND [System.State] <> 'Closed'
ORDER BY [Microsoft.VSTS.Common.Priority]
03 — 版本控制

Azure Repos 與 Branch Policies

Azure Repos 是內建的 Git 服務（也支援 TFVC，但新 project 一律用 Git）。和 GitHub 比起來介面更傳統一點，但 branch policy 的細緻度其實比 GitHub 強。

Clone：HTTPS vs SSH

bash 
# HTTPS（推薦給多數開發者，搭配 Git Credential Manager 自動處理 PAT/MSAL）
$ git clone https://<org>@dev.azure.com/<org>/<project>/_git/<repo>

# 確認 GCM 已安裝（Windows 安裝 Git for Windows 預設就有）
$ git config --global credential.helper manager

# SSH（適合 CI agent / 不想常常彈登入視窗者）
# 1. 在 User Settings → SSH public keys 加上你的 ~/.ssh/id_ed25519.pub
$ git clone git@ssh.dev.azure.com:v3/<org>/<project>/<repo>
!
不要用 PAT 寫進 git remote URL 看到網路上有人寫 https://<PAT>@dev.azure.com/... — 這會把 token 寫進 .git/config，git log 與 hook 也可能洩漏。永遠用 Git Credential Manager（GCM）讓它走 OAuth，token rotate 也自動處理。

Branch Policies：把 main 鎖好

到 Project Settings → Repositories → <repo> → Policies → Branch policies → main 設定。建議啟用以下五項：

1
Require minimum number of reviewers
至少 1 位 approver。勾選 “Reset votes on push”，PR 一旦有新 commit 推上來，approve 自動重置 — 避免 reviewer 看完 A 版本，作者偷塞 B 版本進去。
2
Check for linked work items
PR 必須關聯至少一個 work item — 強制 traceability。要關掉時設 “Optional”，不要直接刪除。
3
Build validation
指定一個 pipeline，PR 開啟時自動觸發；build 沒過不能 complete merge。可以選 “Automatic” 或 “Manual” 觸發；測試覆蓋夠多時用 “When the source branch updates”。
4
Limit merge types
勾選 Squash merge only，禁用 Basic merge 與 Rebase。讓 main 線性、每個 user story 對應一筆 commit。
5
Automatically include reviewers
用 path filter 指定 codeowners — 例如 /infra/* 自動加 DevOps team；/security/* 自動加 Security team。比 GitHub CODEOWNERS 檔案更彈性。

PR 完整流程

bash 
# 1. 從 main 切 feature 分支（命名建議帶 work item id）
$ git checkout -b feat/1234-add-health-endpoint

# 2. 寫程式、commit 時必帶 AB#<id> 自動連結 work item
$ git commit -m "feat: add /healthz endpoint AB#1234"

$ git push -u origin feat/1234-add-health-endpoint

# 3. 開 PR（瀏覽器點 Repos → Pull requests → New PR）
#    或用 az cli（需要 azure-devops extension）
$ az repos pr create --auto-complete --squash --delete-source-branch true

PR Complete 視窗有幾個關鍵選項一定要勾：

  • Complete linked work items after merging：merge 後自動把關聯 work item 設為 Closed
  • Delete source branch after merging：避免分支墳場（永遠勾起來）
  • Squash changes when merging：Squash merge（如果 branch policy 已強制就不用每次手動）
i
AB#<id> 連結語法 在 commit message、PR 標題或內文寫 AB#1234 會自動連結到 work item 1234（雙向可見）。從 GitHub 連到 ADO 也是同一個語法，前提是已經透過 Azure Boards GitHub app 連結兩邊 repo。
04 — Pipelines 基礎

Azure Pipelines：YAML 結構與 agent 模型

Pipelines 是 ADO 最複雜也最核心的功能。新 project 一律用 YAML pipeline（檔案存在 repo 內）— 不要用 Classic Editor 那種點選式 pipeline，後者沒法 code review、沒法 diff、無法 rollback。

YAML pipeline 五層結構

yaml · azure-pipelines.yml 
# 觸發條件（哪個分支 push 會跑）
trigger:
  branches:
    include: [main, release/*]
  paths:
    exclude: ['docs/**', '**/*.md']

# PR 觸發（搭配 branch policy 的 build validation）
pr:
  branches:
    include: [main]

# 執行環境（agent）
pool:
  vmImage: ubuntu-latest

# 變數（純文字 + 引用 variable group）
variables:
  - group: app-shared-vars     # 從 Library → Variable groups 來
  - name: imageName
    value: example-app

# 階段
stages:
  - stage: CI
    jobs:
      - job: build
        steps:
          - checkout: self
          - script: echo "Hello, $(imageName)"
            displayName: 'Smoke step'

Agent 兩種模式

類型來源特性適合
Microsoft-hosted vmImage: ubuntu-latest / windows-latest / macos-latest 每次 job 全新 VM、用完即丟、不可登入；無持久 cache（要靠 Cache@2）；free tier 1 個 parallel job 絕大多數 build/test
Self-hosted 自己跑 agent service 在 VM/容器/k8s 上 持久磁碟、可裝特殊軟體、可進公司內網；維護成本由你扛 需要 GPU、私網存取、大型 build cache

第一個 Go pipeline 範例

yaml · azure-pipelines.yml 
trigger: [main]
pool:
  vmImage: ubuntu-latest

steps:
  - task: GoTool@0
    inputs:
      version: '1.22'

  - script: |
      go mod download
      go vet ./...
      go test -race -coverprofile=coverage.out ./...
    displayName: 'Test'

  - script: |
      go build -o bin/app ./cmd/server
    displayName: 'Build'

  - task: PublishPipelineArtifact@1
    inputs:
      targetPath: 'bin'
      artifact: 'app-bin'

05 — 完整 CI/CD 實作

Test → Docker Build → Push → 更新 manifest

本節組出一條真實可用的 pipeline：跑測試 → 建 Docker image → 推到 registry → 更新另一個 GitOps manifest repo 觸發部署。最後一步把 image 推給後續 lesson 25 的 Argo CD 接手。

步驟 1：建立 Service Connection（一次性）

到 Project Settings → Service connections → New service connection：

  • 選 Docker Registry（不論你用 GHCR / ACR / Docker Hub）
  • Registry type 選對應的服務（Docker Hub / Others）
  • 取個好認的名字，例如 ghcr-example-app
  • 勾 Grant access permission to all pipelines（或之後逐個 pipeline 授權，更安全）
!
最小權限原則：service connection 不要一次給 all pipelines 建立 service connection 時取消勾選 “Grant access permission to all pipelines”，每個需要它的 pipeline 第一次跑時手動授權一次。這樣新建 pipeline 的開發者沒辦法不經審核就把 image 推到 production registry。

步驟 2：完整 azure-pipelines.yml

yaml · azure-pipelines.yml 
trigger:
  branches:
    include: [main]
  tags:
    include: ['v*']

variables:
  - group: app-shared-vars            # 含 REGISTRY_HOST / IMAGE_REPO
  - group: app-prod-secrets          # 連 KV，含 MANIFEST_REPO_PAT
  - name: tagSha
    value: $(Build.SourceVersion)

stages:
  # ───────── CI：測試 ─────────
  - stage: test
    jobs:
      - job: unit
        pool: { vmImage: ubuntu-latest }
        steps:
          - task: GoTool@0
            inputs: { version: '1.22' }

          - task: Cache@2
            inputs:
              key: 'go | "$(Agent.OS)" | go.sum'
              path: '$(Pipeline.Workspace)/.cache/go-mod'
            displayName: 'Cache Go modules'

          - script: |
              export GOMODCACHE=$(Pipeline.Workspace)/.cache/go-mod
              go test -race -coverprofile=coverage.out ./...
            displayName: 'Run tests'

  # ───────── CI：建置 + 推送 ─────────
  - stage: build_image
    dependsOn: test
    jobs:
      - job: docker
        pool: { vmImage: ubuntu-latest }
        steps:
          - task: Docker@2
            displayName: 'Build & push image'
            inputs:
              command: buildAndPush
              repository: $(IMAGE_REPO)           # <org>/example-app
              containerRegistry: ghcr-example-app  # 上面建的 service connection 名字
              dockerfile: Dockerfile
              tags: |
                $(tagSha)
                latest

  # ───────── CD：更新 manifest 觸發 Argo CD ─────────
  - stage: update_manifest
    dependsOn: build_image
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: bump
        environment: prod                     # 啟用 Approvals 的關鍵
        pool: { vmImage: ubuntu-latest }
        strategy:
          runOnce:
            deploy:
              steps:
                - script: |
                    git clone https://x:$(MANIFEST_REPO_PAT)@dev.azure.com/<org>/<project>/_git/manifests
                    cd manifests
                    curl -sSL https://github.com/mikefarah/yq/releases/latest/download/yq_linux_amd64 -o /usr/local/bin/yq
                    chmod +x /usr/local/bin/yq
                    yq -i '.image.tag = strenv(TAG)' apps/example-app/values.yaml
                    git config user.email "ci@example.com"
                    git config user.name "ADO CI"
                    git commit -am "chore: bump example-app to $TAG"
                    git push
                  env:
                    TAG: $(tagSha)

標籤策略：commit SHA + semver + latest

Tag來源用途
<commit-sha>每次 build 自動產生不可變、可追溯，部署到 prod 一律用這個
v1.2.3git tag v* 觸發對外發版、CHANGELOG 對應
latestmain 分支自動覆蓋方便 dev 環境 demo，不可用於 production

Variable Group + Azure Key Vault 連動

把 secret 寫死在 YAML 是初級錯誤。ADO 的最佳做法是 KV-linked variable group：

  1. 建 Key Vault：在 Azure Portal 建 KV，加入需要的 secret（例如 MANIFEST-REPO-PAT，注意 KV secret 名稱不能有底線）。
  2. 授權 service principal：到 KV → Access policies，給 ADO 的 service principal Get + List 權限。
  3. 建 KV-linked variable group：到 ADO → Pipelines → Library → Variable groups → + Variable group，勾選 “Link secrets from an Azure key vault as variables”，選 KV，挑要載入的 secret。
  4. YAML 引用：variables: - group: app-prod-secrets，之後可以用 $(MANIFEST_REPO_PAT)。
i
為什麼這比直接寫 ADO secret 好 KV 是 secret 的 single source of truth — 同一份 KV 也給 K8s External Secrets Operator、Azure Functions、其他 pipeline 用。rotate 一次，到處都更新。直接寫在 ADO 的 variable group 內，rotate 要去翻每個專案的 settings。

Pipeline Templates 重用

三條 pipeline 都要跑同一組測試？把它抽成 template：

yaml · templates/go-test.yml 
parameters:
  - name: goVersion
    default: '1.22'

steps:
  - task: GoTool@0
    inputs: { version: '${{ parameters.goVersion }}' }
  - script: go test -race -coverprofile=coverage.out ./...
    displayName: 'go test'
yaml · azure-pipelines.yml（呼叫端） 
extends:
  template: templates/build-test-deploy.yml
  parameters:
    appName: example-app
    goVersion: '1.22'
06 — 品質閘門

測試結果、覆蓋率、Environments 批准

有 pipeline 還不夠 — 要讓 PR 沒過測試 / 覆蓋率太低 / 沒人批准就根本 merge / deploy 不下去。這三個閘門是企業 DevOps 與「會跑 build 的玩具」之間的差距。

發布測試結果（讓 ADO UI 看得到）

單純 go test 噴 stdout 沒法在 ADO 的 Tests tab 看到 pass/fail 統計。要先輸出 JUnit XML，再 publish：

yaml 
- script: |
    go install github.com/jstemmer/go-junit-report/v2@latest
    go test -v ./... 2>&1 | go-junit-report -set-exit-code > report.xml
  displayName: 'Test + JUnit XML'

- task: PublishTestResults@2
  condition: always()           # 失敗也要 publish，否則 UI 看不到失敗在哪
  inputs:
    testResultsFormat: JUnit
    testResultsFiles: '**/report.xml'
    failTaskOnFailedTests: true

發布測試覆蓋率

yaml 
- script: |
    go install github.com/boumenot/gocover-cobertura@latest
    gocover-cobertura < coverage.out > coverage.xml
  displayName: 'Convert to Cobertura'

- task: PublishCodeCoverageResults@2
  inputs:
    summaryFileLocation: '$(System.DefaultWorkingDirectory)/coverage.xml'

之後 pipeline run 頁面會多一個 Code Coverage tab，PR 頁面也會顯示「這次 PR 把覆蓋率從 78% 提到 81%」這種訊息。

Build validation 連動 PR（再強調一次）

Section 3 已經設定 branch policy 的 Build validation 指向這個 pipeline。組合起來的效果：

✗沒設 build validation
測試掛了但 reviewer 沒注意 → 直接 approve → merge 進 main → main 變成壞的 → 大家阻塞。
✓有 build validation
測試掛了 → PR 頁面顯示紅色 X，Complete merge 按鈕直接 disabled。reviewer 想 approve 也沒用，作者必須先把測試修綠。

Environments + Approvals

Pipeline 跑到 deploy 階段時，要不要先讓人手動 approve？這就是 Environment：

  1. 到 Pipelines → Environments → New environment，命名 prod。
  2. 進入 environment → 右上 ⋯ → Approvals and checks → 加 Approvals。
  3. 指定 approver（人或 group）+ 最少幾位 approve、approver 不能是 PR 作者本人、超時自動拒絕等。
  4. YAML 用 environment: prod 對 deployment job 套用 — 跑到這個 job 會卡住等 approve。
i
Approvals 不只是「人按按鈕」 除了人工批准，也可以設 Invoke REST API（呼叫 ServiceNow change ticket）、Business hours（只有上班時間能 deploy）、Required template（只有用指定 template 才放行）。把 change management 自動化。
07 — Operations 與排錯

讀 logs、找慢步驟、開 debug、self-hosted agent

Pipeline 跑掛是日常。會排錯比會寫 pipeline 更重要 — pipeline 寫一次，掛了你會修十次。

Logs UI 三層結構

Pipeline run 頁面結構是：Stage → Job → Step。失敗會顯示紅色 X，點進去會 deep-link 直接跳到失敗的那行 log。實務操作：

  • 右上 Download logs 把整個 run 的 log 抓成 zip — 排錯時拉到本地用 grep / less 找關鍵字比 UI 順手。
  • Timeline view（job 內右上）：把每個 step 的耗時排出來 — 看到某個 step 從平常 2 分鐘變 8 分鐘 → 通常是 cache miss 或網路慢。
  • Tests tab（要 publish 過才有）：直接看哪幾個測試失敗、失敗 trace。

Rerun：兩種模式差很多

動作行為什麼時候用
Rerun all jobs 整條 pipeline 從頭跑 程式有改、commit 推新版本
Rerun failed jobs 沿用前一輪成功的 stage artifact，只重跑失敗 stage 網路 flaky、image push 因 registry 503 失敗 — 不需要重新 build

開 diagnostic logs

遇到「task 失敗但訊息很模糊」這種狀況，加變數開 verbose：

bash 
# 方法 1：UI 點 Run pipeline → Variables → 加 system.debug = true

# 方法 2：用 az cli 觸發 run 時帶 variable
$ az pipelines run --name example-app-ci --variables system.debug=true

# 方法 3（不推薦）：寫死在 YAML — 永遠 verbose 會讓 log 很巨
variables:
  system.debug: true

平行度與 free tier 限制

方案Microsoft-hosted parallel jobs每月分鐘數
Free（public project）10無限
Free（private project）11,800 分鐘
Paid（每個 +$40/月）+1無限
Self-hosted不限不限（自家機器）

Free private project 只有 1 條平行線 — PR 跑 build validation 時，main 的 release pipeline 就排隊。10 人團隊基本上要付錢買 parallel job，或切去 self-hosted。

什麼時候該切去 Self-hosted Agent？

  • 需要存取公司內網資源（私網 DB、內部 image registry）— Microsoft-hosted 沒法 VPN 進去。
  • build cache 想跨 run 持久化（雖然 Cache@2 也行，但 self-hosted 直接用本機 disk）。
  • 要跑 K3d / Kind 起本地 K8s 跑 e2e — Microsoft-hosted ubuntu image 雖然也行，但 self-hosted 不用每次 pull image 比較快。
  • 合規要求 build 必須在自家機房內。

08 — 落地最佳實踐

把專案管理 + DevOps 串成一條 traceability 鏈

到這裡所有零件都有了。最後一節談的是怎麼把它們串成一條從業務需求到部署 artifact都能反追的鏈條 — 這是 ADO 比 GitHub 真正占優勢的地方。

Sprint Planning 標準流程

1
設 Capacity
Boards → Sprint → Capacity tab，每位成員填可用工時、休假。系統會自動算出 sprint 容量上限。
2
從 Backlog 拖卡到 Sprint
拖到接近 capacity 上限就停 — 不要塞滿，留 20% buffer 給線上事故。
3
每日 stand-up 看 Taskboard
焦點在 “In Review” / “Blocked” 的 task — Active 太久（> 2 天）的也要點名問。
4
Sprint 結束看 Velocity
Analytics 自動算出每個 sprint 完成的 story point。連續三個 sprint 取平均當下次 capacity 參考，不要看單一 sprint。
