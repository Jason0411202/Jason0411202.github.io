01 — 全景

Azure DevOps 全景：從 Organization 到 5 個 Hub

Azure DevOps（簡稱 ADO）為微軟的一站式 DevOps 平台。與 GitHub 的主要差異在於 ADO 將「專案管理」、「版本控制」、「CI/CD」、「測試計畫」、「製品儲存庫」整合於同一產品中——故左側導覽提供 5 個 hub：Boards、Repos、Pipelines、Test Plans 與 Artifacts。

圖 23.1 — Azure DevOps 階層：Organization → Project → 5 個 Hub

權限模型基礎

ADO 的存取控制以 Security Group 為核心，每個 Project 預設有四個內建群組：

Group能做什麼常見用途
Project Administrators所有 Project 設定，含建立 service connection、刪除 repoTech lead / DevOps engineer
Contributors建立 / 編輯 work item、push commit、開 PR、執行 pipeline一般開發者（預設群組）
Readers只讀，看不到 secret 內容稽核者、產品經理視需要
Stakeholders (免費 license)看 Boards、編輯 work item，看不到 Repos / Pipelines業務 / PM 不寫程式但要追進度
i
Stakeholder license 為 ADO 中常被忽略的成本優化選項 每位 Basic license 約 6 美金 / 月，而 Stakeholder 為完全免費。以 50 人規模的公司為例，若其中 30 位為 PM 或業務僅需檢視 Boards，每年可節省近 2,000 美金。可於 Organization Settings → Users 中調整 access level。

常用導覽路徑

  • My Work hub（左下角頭像旁的圖示）：跨所有 project 列出個人被指派的 work item、開啟的 PR 與待 review 的 PR——日常開工首要檢視的頁面。
  • Project Settings（左下角齒輪）：service connection、agent pool、repository policy 與 pipeline 變數群組皆於此設定。
  • Organization Settings：billing、parallel jobs 配額、跨 project 的擴充功能。
02 — 專案管理

Azure Boards：Process 模板、階層、Sprint

Boards 為 ADO 的 work item 系統，建立 project 時所選的 Process 模板將決定後續所有設定——模板選擇錯誤，日後修改成本極高。

四種 Process 模板比較

模板頂層階層狀態流適合對象
BasicEpic → Issue → TaskTo Do / Doing / Done新團隊、輕量需求
Agile （推薦）Epic → Feature → User Story → TaskNew / Active / Resolved / Closed大多數軟體團隊（本課採用）
ScrumEpic → Feature → PBI → TaskNew / Approved / Committed / Done嚴格 Scrum 流程的團隊
CMMIEpic → Feature → Requirement → TaskProposed / Active / Resolved / Closed受規範產業（金融、醫療、航太）
✓
採用 Agile 模板為穩妥的選擇 User Story 較 PBI 對非工程角色更為直覺、欄位較少；Bug 為獨立的 work item type 而非 task；狀態流的設計適切而完整。Agile 可後續客製，但 Process 一旦建立即無法改為其他內建類型——僅能複製為 Inherited Process 後再修改。

Work Item 階層範例

以一個典型的 backend 重構需求為例：

work-item-tree 
# Epic — 半年期的策略目標
Epic      "Containerise the example backend"
├── Feature   "Build container image in CI"
│   ├── User Story  "As a developer, push to main builds an image"
│   │   ├── Task  "Write Dockerfile (multi-stage)"
│   │   ├── Task  "Add Docker@2 task to azure-pipelines.yml"
│   │   └── Task  "Tag image with commit SHA"
│   └── User Story  "Push image to container registry"
└── Feature   "Deploy via GitOps manifest update"
    └── User Story  "Pipeline updates values.yaml on tag push"

Sprint、Iteration 與 Capacity

於 Project Settings → Project configuration → Iterations 設定 sprint 週期。Boards 內 Sprint 頁面提供：

  • Backlog：以拖曳方式排定優先順序
  • Taskboard：sprint 內以 user story 為列、status 為欄的看板
  • Capacity：每位成員設定可用工時與休假，系統自動計算 sprint 容量
  • Analytics：burndown 與 velocity 圖表自動產生（搭配 Dashboards 使用）

Kanban Board 客製化

Boards 頁面右上的 ⚙️ 圖示 → Configure team settings 可新增 column、設定 WIP limit、加入 swimlane。

✗常見錯誤
直接沿用預設的 4 欄（New / Active / Resolved / Closed），未設定 WIP limit，導致 Active 欄堆積 30 張卡，無從追蹤各成員負責的項目。
✓實務做法
拆解為 To Do、In Dev（WIP=3）、In Review（WIP=2）、QA、Done。透過 Swimlane 拉出 Expedite（線上事故）以優先處理緊急項目。

Queries、Delivery Plans、Wiki

  • Queries：以 WIQL 撰寫過濾條件並儲存為 Shared Query。例如：「My team this sprint, state ≠ Done, sort by priority」。Dashboard widget 可直接讀取 query 結果。
  • Delivery Plans：跨 team 的 Gantt 視圖，可掌握各 feature 對應的 sprint。
  • Dashboards：每個 project 可自訂 dashboard，常用 widget 包含 Sprint burndown、Cumulative Flow Diagram（CFD）、Build history、Pull request 列表、Markdown（放置團隊 SLO）。
  • Wiki：用於存放 runbook、on-call 流程、架構決策紀錄。Wiki 本身即為 Git repo，可直接 clone 進行編輯。
WIQL 查詢範例：「sprint 內未完成的 user story」 進階
wiql 
SELECT [System.Id], [System.Title], [System.AssignedTo], [System.State]
FROM WorkItems
WHERE [System.TeamProject] = @project
  AND [System.WorkItemType] = 'User Story'
  AND [System.IterationPath] = @currentIteration
  AND [System.State] <> 'Closed'
ORDER BY [Microsoft.VSTS.Common.Priority]
03 — 版本控制

Azure Repos 與 Branch Policies

Azure Repos 為內建的 Git 服務（亦支援 TFVC，但新 project 一律採用 Git）。介面風格較 GitHub 傳統，但 branch policy 的細緻度勝過 GitHub。

Clone：HTTPS vs SSH

bash 
# HTTPS（推薦給多數開發者，搭配 Git Credential Manager 自動處理 PAT/MSAL）
$ git clone https://<org>@dev.azure.com/<org>/<project>/_git/<repo>

# 確認 GCM 已安裝（Windows 安裝 Git for Windows 預設就有）
$ git config --global credential.helper manager

# SSH（適合 CI agent / 不想常常彈登入視窗者）
# 1. 在 User Settings → SSH public keys 加上你的 ~/.ssh/id_ed25519.pub
$ git clone git@ssh.dev.azure.com:v3/<org>/<project>/<repo>
!
不應將 PAT 寫入 git remote URL 網路上常見的 https://<PAT>@dev.azure.com/... 寫法會將 token 寫入 .git/config，git log 與 hook 亦可能因此洩漏。應始終使用 Git Credential Manager（GCM）以 OAuth 處理，並支援 token rotate 的自動化。

Branch Policies：鎖定 main 分支

於 Project Settings → Repositories → <repo> → Policies → Branch policies → main 進行設定。建議啟用以下五項：

1
Require minimum number of reviewers
至少需 1 位 approver。勾選 “Reset votes on push”——PR 有新 commit 推送時，approve 自動重置——避免 reviewer 已審視 A 版本後，作者後續推送 B 版本而未經審核。
2
Check for linked work items
PR 必須關聯至少一個 work item——強制建立 traceability。需暫停時設為 “Optional”，不應直接刪除規則。
3
Build validation
指定 pipeline，PR 開啟時自動觸發；build 未通過則無法 complete merge。可選 “Automatic” 或 “Manual” 觸發；測試覆蓋率充足時建議使用 “When the source branch updates”。
4
Limit merge types
僅勾選 Squash merge，停用 Basic merge 與 Rebase，使 main 保持線性，每個 user story 對應單筆 commit。
5
Automatically include reviewers
透過 path filter 指定 codeowners——例如 /infra/* 自動加入 DevOps team；/security/* 自動加入 Security team。較 GitHub CODEOWNERS 檔案更具彈性。

PR 完整流程

bash 
# 1. 從 main 切 feature 分支（命名建議帶 work item id）
$ git checkout -b feat/1234-add-health-endpoint

# 2. 寫程式、commit 時必帶 AB#<id> 自動連結 work item
$ git commit -m "feat: add /healthz endpoint AB#1234"

$ git push -u origin feat/1234-add-health-endpoint

# 3. 開 PR（瀏覽器點 Repos → Pull requests → New PR）
#    或用 az cli（需要 azure-devops extension）
$ az repos pr create --auto-complete --squash --delete-source-branch true

PR Complete 視窗中有幾個關鍵選項應勾選：

  • Complete linked work items after merging：merge 後自動將關聯的 work item 設為 Closed
  • Delete source branch after merging：避免累積過多無用分支（建議常駐勾選）
  • Squash changes when merging：Squash merge（若 branch policy 已強制要求，無須每次手動勾選）
i
AB#<id> 連結語法 於 commit message、PR 標題或內文中寫入 AB#1234，將自動連結至 work item 1234（雙向可見）。由 GitHub 連結至 ADO 亦使用相同語法，前提是已透過 Azure Boards GitHub app 連結兩端 repo。
04 — Pipelines 基礎

Azure Pipelines：YAML 結構與 agent 模型

Pipelines 為 ADO 最複雜亦最核心的功能。新 project 一律採用 YAML pipeline（檔案存於 repo 內）——不應使用 Classic Editor 的點選式 pipeline，後者無法進行 code review、diff 與 rollback。

YAML pipeline 五層結構

yaml · azure-pipelines.yml 
# 觸發條件（哪個分支 push 會跑）
trigger:
  branches:
    include: [main, release/*]
  paths:
    exclude: ['docs/**', '**/*.md']

# PR 觸發（搭配 branch policy 的 build validation）
pr:
  branches:
    include: [main]

# 執行環境（agent）
pool:
  vmImage: ubuntu-latest

# 變數（純文字 + 引用 variable group）
variables:
  - group: app-shared-vars     # 從 Library → Variable groups 來
  - name: imageName
    value: example-app

# 階段
stages:
  - stage: CI
    jobs:
      - job: build
        steps:
          - checkout: self
          - script: echo "Hello, $(imageName)"
            displayName: 'Smoke step'

Agent 的兩種模式

類型來源特性適用情境
Microsoft-hosted vmImage: ubuntu-latest / windows-latest / macos-latest 每次 job 使用全新 VM，執行後即銷毀、無法登入；無持久 cache（須透過 Cache@2）；free tier 僅 1 個 parallel job 絕大多數 build 與 test
Self-hosted 自行於 VM、容器或 k8s 上執行 agent service 具備持久磁碟、可安裝特殊軟體、可存取公司內網；維護成本由使用者承擔 需要 GPU、私網存取或大型 build cache

第一個 Go pipeline 範例

yaml · azure-pipelines.yml 
trigger: [main]
pool:
  vmImage: ubuntu-latest

steps:
  - task: GoTool@0
    inputs:
      version: '1.22'

  - script: |
      go mod download
      go vet ./...
      go test -race -coverprofile=coverage.out ./...
    displayName: 'Test'

  - script: |
      go build -o bin/app ./cmd/server
    displayName: 'Build'

  - task: PublishPipelineArtifact@1
    inputs:
      targetPath: 'bin'
      artifact: 'app-bin'

05 — 完整 CI/CD 實作

Test → Docker Build → Push → 更新 manifest

本節組裝一條可實際運作的 pipeline：執行測試 → 建立 Docker image → 推送至 registry → 更新另一個 GitOps manifest repo 以觸發部署。最後一步將 image 交予後續 Lesson 03 的 Argo CD 接手處理。

步驟 1：建立 Service Connection（一次性設定）

前往 Project Settings → Service connections → New service connection：

  • 選擇 Docker Registry（無論使用 GHCR、ACR 或 Docker Hub）
  • Registry type 選擇對應的服務（Docker Hub 或 Others）
  • 取一個易識別的名稱，例如 ghcr-example-app
  • 勾選 Grant access permission to all pipelines（或後續逐 pipeline 授權，安全性較高）
!
最小權限原則：service connection 不應一次授權予所有 pipelines 建立 service connection 時取消勾選 “Grant access permission to all pipelines”，每個需使用此連線的 pipeline 首次執行時須手動授權一次。如此可確保新建 pipeline 的開發者無法在未經審核的情況下將 image 推送至 production registry。

步驟 2：完整 azure-pipelines.yml

yaml · azure-pipelines.yml 
trigger:
  branches:
    include: [main]
  tags:
    include: ['v*']

variables:
  - group: app-shared-vars            # 含 REGISTRY_HOST / IMAGE_REPO
  - group: app-prod-secrets          # 連 KV，含 MANIFEST_REPO_PAT
  - name: tagSha
    value: $(Build.SourceVersion)

stages:
  # ───────── CI：測試 ─────────
  - stage: test
    jobs:
      - job: unit
        pool: { vmImage: ubuntu-latest }
        steps:
          - task: GoTool@0
            inputs: { version: '1.22' }

          - task: Cache@2
            inputs:
              key: 'go | "$(Agent.OS)" | go.sum'
              path: '$(Pipeline.Workspace)/.cache/go-mod'
            displayName: 'Cache Go modules'

          - script: |
              export GOMODCACHE=$(Pipeline.Workspace)/.cache/go-mod
              go test -race -coverprofile=coverage.out ./...
            displayName: 'Run tests'

  # ───────── CI：建置 + 推送 ─────────
  - stage: build_image
    dependsOn: test
    jobs:
      - job: docker
        pool: { vmImage: ubuntu-latest }
        steps:
          - task: Docker@2
            displayName: 'Build & push image'
            inputs:
              command: buildAndPush
              repository: $(IMAGE_REPO)           # <org>/example-app
              containerRegistry: ghcr-example-app  # 上面建的 service connection 名字
              dockerfile: Dockerfile
              tags: |
                $(tagSha)
                latest

  # ───────── CD：更新 manifest 觸發 Argo CD ─────────
  - stage: update_manifest
    dependsOn: build_image
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: bump
        environment: prod                     # 啟用 Approvals 的關鍵
        pool: { vmImage: ubuntu-latest }
        strategy:
          runOnce:
            deploy:
              steps:
                - script: |
                    git clone https://x:$(MANIFEST_REPO_PAT)@dev.azure.com/<org>/<project>/_git/manifests
                    cd manifests
                    curl -sSL https://github.com/mikefarah/yq/releases/latest/download/yq_linux_amd64 -o /usr/local/bin/yq
                    chmod +x /usr/local/bin/yq
                    yq -i '.image.tag = strenv(TAG)' apps/example-app/values.yaml
                    git config user.email "ci@example.com"
                    git config user.name "ADO CI"
                    git commit -am "chore: bump example-app to $TAG"
                    git push
                  env:
                    TAG: $(tagSha)

標籤策略：commit SHA、semver 與 latest

Tag來源用途
<commit-sha>每次 build 自動產生不可變、可追溯，部署至 prod 一律使用此標籤
v1.2.3由 git tag v* 觸發對外發版，與 CHANGELOG 對應
latest由 main 分支自動覆蓋適用於 dev 環境 demo，不應用於 production

Variable Group 與 Azure Key Vault 的連動

將 secret 寫於 YAML 中為基本錯誤。ADO 的最佳實踐是採用 KV-linked variable group：

  1. 建立 Key Vault：於 Azure Portal 建立 KV，加入所需的 secret（例如 MANIFEST-REPO-PAT；注意 KV secret 名稱不可包含底線）。
  2. 授權 service principal：於 KV → Access policies，授予 ADO 的 service principal Get 與 List 權限。
  3. 建立 KV-linked variable group：於 ADO → Pipelines → Library → Variable groups → + Variable group，勾選 “Link secrets from an Azure key vault as variables”，選擇 KV 並指定要載入的 secret。
  4. YAML 引用：variables: - group: app-prod-secrets，後續即可使用 $(MANIFEST_REPO_PAT)。
i
採用此方式優於直接寫於 ADO secret 的原因 KV 為 secret 的 single source of truth——同一份 KV 亦可供 K8s External Secrets Operator、Azure Functions 與其他 pipeline 使用。單次 rotate 即可同步更新所有使用端。若直接寫於 ADO 的 variable group 內，rotate 時須逐專案修改 settings。

Pipeline Templates 的重用

多條 pipeline 需執行同一組測試時，可將其抽離為 template：

yaml · templates/go-test.yml 
parameters:
  - name: goVersion
    default: '1.22'

steps:
  - task: GoTool@0
    inputs: { version: '${{ parameters.goVersion }}' }
  - script: go test -race -coverprofile=coverage.out ./...
    displayName: 'go test'
yaml · azure-pipelines.yml（呼叫端） 
extends:
  template: templates/build-test-deploy.yml
  parameters:
    appName: example-app
    goVersion: '1.22'
06 — 品質閘門

測試結果、覆蓋率與 Environments 批准

具備 pipeline 仍嫌不足——須使 PR 在未通過測試、覆蓋率不足或未取得批准時無法 merge 或 deploy。此三項閘門是企業級 DevOps 與「僅可執行 build 的雛形」之間的差異所在。

發布測試結果（使 ADO UI 可見）

單純執行 go test 將結果輸出至 stdout，無法於 ADO 的 Tests tab 顯示 pass/fail 統計。須先輸出 JUnit XML 再進行 publish：

yaml 
- script: |
    go install github.com/jstemmer/go-junit-report/v2@latest
    go test -v ./... 2>&1 | go-junit-report -set-exit-code > report.xml
  displayName: 'Test + JUnit XML'

- task: PublishTestResults@2
  condition: always()           # 失敗也要 publish，否則 UI 看不到失敗在哪
  inputs:
    testResultsFormat: JUnit
    testResultsFiles: '**/report.xml'
    failTaskOnFailedTests: true

發布測試覆蓋率

yaml 
- script: |
    go install github.com/boumenot/gocover-cobertura@latest
    gocover-cobertura < coverage.out > coverage.xml
  displayName: 'Convert to Cobertura'

- task: PublishCodeCoverageResults@2
  inputs:
    summaryFileLocation: '$(System.DefaultWorkingDirectory)/coverage.xml'

後續 pipeline run 頁面將新增 Code Coverage tab，PR 頁面亦會顯示「此次 PR 將覆蓋率自 78% 提升至 81%」等資訊。

Build validation 與 PR 的連動（補充說明）

第 3 節已設定 branch policy 的 Build validation 指向此 pipeline。組合運作的效果：

✗未設定 build validation
測試失敗但 reviewer 未察覺 → 直接 approve → merge 進入 main → main 處於不穩定狀態 → 影響整個團隊。
✓已設定 build validation
測試失敗 → PR 頁面顯示紅色 X，Complete merge 按鈕直接停用。reviewer 即使 approve 亦無法合併，作者必須先修補測試使其通過。

Environments 與 Approvals

Pipeline 進入 deploy 階段時是否需人工批准？此即 Environment 的功能：

  1. 前往 Pipelines → Environments → New environment，命名為 prod。
  2. 進入 environment → 右上 ⋯ → Approvals and checks → 新增 Approvals。
  3. 指定 approver（人員或 group）、所需的最少批准人數、approver 不可為 PR 作者本人、超時自動拒絕等規則。
  4. 於 YAML 中以 environment: prod 套用至 deployment job——執行至此 job 時將暫停並等待批准。
i
Approvals 不僅限於人工點擊 除人工批准外，亦可設定 Invoke REST API（呼叫 ServiceNow change ticket）、Business hours（僅工作時間允許 deploy）、Required template（僅使用指定 template 方能通過）。可將 change management 全面自動化。
07 — Operations 與排錯

解讀 logs、定位慢速 step、啟用 debug 與 self-hosted agent

Pipeline 失敗為日常事件。具備排錯能力較撰寫 pipeline 更為重要——pipeline 撰寫一次，後續修補可能高達數十次。

Logs UI 的三層結構

Pipeline run 頁面結構為：Stage → Job → Step。失敗將顯示紅色 X，點選後將以 deep-link 直接跳至失敗的 log 行。實務操作：

  • 右上的 Download logs 可將整個 run 的 log 下載為 zip——排錯時於本地以 grep 或 less 搜尋關鍵字較 UI 更為便利。
  • Timeline view（job 內右上）：列出每個 step 的耗時——若某 step 由平時的 2 分鐘延長至 8 分鐘，通常為 cache miss 或網路延遲所致。
  • Tests tab（須先 publish）：可直接查看失敗的測試項目與失敗 trace。

Rerun：兩種模式的差異

動作行為適用情境
Rerun all jobs 自頭重新執行整條 pipeline 程式碼有變動或 commit 推送新版本
Rerun failed jobs 沿用前一輪成功的 stage artifact，僅重新執行失敗的 stage 網路 flaky、image push 因 registry 503 失敗——無須重新 build

啟用 diagnostic logs

遇到「task 失敗但訊息不夠明確」的情況時，可加上變數啟用 verbose 輸出：

bash 
# 方法 1：UI 點選 Run pipeline → Variables → 新增 system.debug = true

# 方法 2：以 az cli 觸發 run 時附帶 variable
$ az pipelines run --name example-app-ci --variables system.debug=true

# 方法 3（不建議）：寫於 YAML——常駐 verbose 將使 log 體積過大
variables:
  system.debug: true

Free private project 僅有 1 條平行線——PR 執行 build validation 時，main 的 release pipeline 將進入排隊。10 人規模的團隊通常須付費購買 parallel job，或切換至 self-hosted。

切換至 Self-hosted Agent 的時機

  • 須存取公司內網資源（私網 DB、內部 image registry）——Microsoft-hosted 無法透過 VPN 連入。
  • build cache 需跨 run 持久化（雖然 Cache@2 亦可達成，但 self-hosted 可直接使用本機 disk）。
  • 需執行 K3d 或 Kind 啟動本地 K8s 進行 e2e——Microsoft-hosted ubuntu image 雖然可行，但 self-hosted 無須每次重新 pull image，速度較佳。
  • 合規要求 build 必須於公司機房內執行。

$ ./config.sh                              # 互動式輸入 ADO URL + PAT + pool 名
$ sudo ./svc.sh install                   # 註冊成 systemd service
$ sudo ./svc.sh start

# YAML 端改用

  name: my-self-hosted-pool

將專案管理與 DevOps 串接為一條 traceability 鏈

至此所有元件皆已備齊。最後一節說明如何將其串接為一條由業務需求至部署 artifact皆可反向追溯的鏈條——此為 ADO 較 GitHub 真正占優勢的領域。

Sprint Planning 標準流程

1
設定 Capacity
Boards → Sprint → Capacity tab，由各成員填寫可用工時與休假。系統將自動計算 sprint 的容量上限。
2
由 Backlog 拖曳卡片至 Sprint
拖至接近 capacity 上限即停止——不應全數填滿，建議保留 20% buffer 以應對線上事故。
3
每日 stand-up 檢視 Taskboard
聚焦於 “In Review” 與 “Blocked” 的 task；Active 狀態超過 2 天者亦應主動關注。
4
Sprint 結束時檢視 Velocity
Analytics 自動計算每個 sprint 完成的 story point。應取連續三個 sprint 的平均值作為下次 capacity 的參考，不應僅看單一 sprint。
