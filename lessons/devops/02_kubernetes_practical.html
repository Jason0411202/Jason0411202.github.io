01 — kubectl 心智模型

三層概念：context、namespace、kubeconfig

很多 K8s 災難都是在錯的 cluster 上下了正確的指令。在按下 Enter 之前，你必須知道：「我這個指令會打到哪一座 cluster、用什麼身份、預設操作哪一個 namespace？」這三個問題的答案就在 kubeconfig 裡。

圖 24.1 — kubeconfig 三大區與 current-context 的關係

每天必背的三個切換指令

bash 
# 1. 看目前所有 contexts，星號 (*) 標記目前用的那個
$ kubectl config get-contexts
CURRENT   NAME          CLUSTER    AUTHINFO        NAMESPACE
*         prod-admin    prod       admin           default
          stg-dev       staging    dev-readonly    app
          dev           dev        admin           default

# 2. 看目前 context (簡短版)
$ kubectl config current-context
prod-admin

# 3. 切 context
$ kubectl config use-context stg-dev

# 4. 改變目前 context 的預設 namespace (不切 cluster)
$ kubectl config set-context --current --namespace=<your-ns>
!
每次操作前先確認 context 業界最常見的 K8s 災難是「以為自己在 dev，結果 kubectl delete 打在 prod」。把kubectl config current-context放進你的 shell prompt，或用 kubectx / k9s 隨時可見。永遠不要 muscle memory 直接打 kubectl delete。

多份 kubeconfig 怎麼合併

很多公司會發給你多份 config（每個 cluster 一份）。KUBECONFIG 環境變數可以用冒號（Linux/macOS）或分號（Windows）串接多個檔案，kubectl 會把它們合併在記憶體裡使用：

bash 
# 暫時合併三個 config (Linux / macOS)
$ export KUBECONFIG=~/.kube/config:~/.kube/prod.yaml:~/.kube/stg.yaml
$ kubectl config get-contexts        # 三份 config 的 context 都列出來

# 永久合併到 ~/.kube/config (做完記得備份原本的)
$ kubectl config view --flatten > ~/.kube/merged
$ mv ~/.kube/merged ~/.kube/config
i
kubectx + kubens 是必裝 把 kubectl config use-context 變成 kubectx prod-admin，把 set-context --current --namespace 變成 kubens <your-ns>。每個 K8s 老手電腦上都有，下節會教安裝。
02 — 每天會用的五個動詞

get / describe / logs / exec / port-forward

kubectl 有幾百個 subcommand，但 90% 的工作日只會用到下面五個。把它們玩熟，你就能應付絕大多數的線上排查。

kubectl get — 看資源狀態

get 是儀表板。常用旗標一次看完：

bash 
# 基本
$ kubectl get pods                              # 目前 namespace 的 pods
$ kubectl get pods -A                           # 所有 namespace
$ kubectl get pods -n <your-ns>                 # 指定 namespace

# -o wide 多兩欄 (Node、Pod IP)，排查網路必備
$ kubectl get pods -o wide

# -o yaml 看完整資源定義 (含 status)
$ kubectl get pod <pod-name> -o yaml

# Label selector：抓某個 release 的所有資源
$ kubectl get all -l app.kubernetes.io/instance=<release>

# Field selector：抓所有不在 Running 的 pod
$ kubectl get pods --field-selector=status.phase!=Running -A

# JSONPath：只抽取你要的欄位 (寫 script 必用)
$ kubectl get pods -o jsonpath='{.items[*].metadata.name}'
$ kubectl get pods -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.phase}{"\n"}{end}'

kubectl describe — 看 events 與細節

describe 跟 get -o yaml 差別在：它會幫你撈出對應的 Events，這是 K8s 的 syslog，所有調度、健康檢查、image pull 失敗都會出現在這裡。

bash 
$ kubectl describe pod <pod-name> -n <your-ns>

# 重點通常在最下面的 Events 區段，例如：
# Events:
#   Type    Reason     Age   From     Message
#   ----    ------     ----  ----     -------
#   Normal  Scheduled  2m    default-scheduler  Successfully assigned ...
#   Warning Failed     1m    kubelet  Failed to pull image "example-app:v9": ...

# 也能直接撈 namespace 內的所有 events，按時間排序
$ kubectl get events -n <your-ns> --sort-by=.lastTimestamp

kubectl logs — 看容器輸出

bash 
# 基本：看單一 pod 的 stdout/stderr
$ kubectl logs <pod-name>

# -f follow，像 tail -f
$ kubectl logs -f <pod-name>

# --previous：看上一輪 (crash 前) 的 stdout — 排查 CrashLoopBackOff 必用
$ kubectl logs <pod-name> --previous

# 多容器 pod：指定容器
$ kubectl logs <pod-name> -c <container>

# 只看最近 100 行 + 最近 10 分鐘
$ kubectl logs <pod-name> --tail=100 --since=10m

# 一次看一個 deployment 底下所有 pod 的 log (用 selector)
$ kubectl logs -l app.kubernetes.io/name=<app> --tail=50 -f

kubectl exec — 進 pod 內 shell

bash 
# 開互動式 shell
$ kubectl exec -it <pod-name> -- sh
$ kubectl exec -it <pod-name> -c <container> -- bash

# 跑單一指令
$ kubectl exec <pod-name> -- env
$ kubectl exec <pod-name> -- ls /etc/config

kubectl port-forward — debug 內部服務

當 Service 是 ClusterIP，從外面碰不到，port-forward 是你 debug 的瑞士刀：

bash 
# 把本機 8080 對到 pod 的 80
$ kubectl port-forward <pod-name> 8080:80

# 對 service (推薦：service 後面誰當 endpoint 由 K8s 決定)
$ kubectl port-forward svc/<svc-name> 8080:80

# 連 DB 來除錯資料 (注意：這條會把 prod DB 暴露在你筆電上！)
$ kubectl port-forward -n <your-ns> svc/postgres 5432:5432

順手好用：kubectl cp

從 pod 把檔案抓出來，或塞檔案進去：

bash 
# pod 的檔 → 本機
$ kubectl cp <your-ns>/<pod>:/var/log/app.log ./app.log

# 本機 → pod (只在 debug 用，正式環境檔案應由 image / ConfigMap 管)
$ kubectl cp ./fix.sh <your-ns>/<pod>:/tmp/fix.sh

動詞速查表

動詞用途什麼時候用
get列出資源狀態儀表板，先看狀態
describe看詳細 + Events狀態不對時下一步
logs看 stdout/stderr應用程式層問題
exec進 pod 內 shell內部檔案系統 / 環境變數查驗
port-forward本機連集群內服務沒 Ingress 時 debug 唯一解
cp檔案進出 pod抓 log / 塞臨時 patch
top看 CPU/Memory 使用率OOM / CPU throttle 排查
rollout看/控制部署版本Deployment 升級回滾
i
alias 一次設好 幾乎每個 K8s 老手都把 kubectl alias 成 k。在 ~/.bashrc 加上 alias k=kubectl，再加 complete -F __start_kubectl k（zsh 用對應語法）就有 tab 補全。一天可省幾百次按鍵。
03 — Pod 狀態排錯

五種「Pod 不健康」的故障流程

kubectl get pods 出來不是 Running 就是有事。下面這條 timeline 是經驗法則，五種最常見的不健康狀態各自要看的東西：

1
Pending — 還沒被排到 Node 上
kubectl describe pod 看 Events。常見三大原因：(1) 沒有 Node 有足夠 CPU/Memory；(2) PVC 還在 binding；(3) Node 上有 taint 而 pod 沒對應 toleration。
2
ImagePullBackOff — 抓不到 image
image tag 拼錯、private registry 沒設 imagePullSecrets、registry 暫時掛掉都會中。describe 的 Events 會直接寫 "manifest unknown" 或 "401 Unauthorized"。
3
CrashLoopBackOff — 開起來就死，重複重啟
kubectl logs <pod> --previous 看上一輪輸出。或 describe 看 lastState.terminated.reason / exitCode。常見：missing env var、依賴的 service 還沒起、health check 設太嚴。
4
OOMKilled — 記憶體爆掉被 kernel 殺掉
describe 會看到 Reason: OOMKilled、ExitCode: 137。對策：調高 resources.limits.memory，或先用 kubectl top pod + Prometheus 看真實尖峰用量再設值。
5
Error — 通用失敗
看 exitCode：1 是應用程式錯誤、137 是 OOM、139 是 segfault、143 是收到 SIGTERM 沒處理乾淨。配合 logs 看真實錯誤訊息。

狀態與下一步速查

狀態意義下一步指令
Pending還沒排上 Nodekubectl describe pod 看 Events
ContainerCreating正在拉 image / mount volume等 30 秒；超過用 describe
ImagePullBackOffimage 抓不到檢查 tag、imagePullSecrets
CrashLoopBackOff開起來就死kubectl logs --previous
OOMKilledRAM 爆掉調高 limits.memory
Error通用失敗看 exitCode + logs
Completed正常結束 (Job 用)不是錯誤，Job/CronJob 是正常
EvictedNode 資源壓力被趕走看 Node 容量，考慮加 PDB

kubectl debug — distroless image 的救星

越來越多 image 是 distroless / scratch（沒有 shell、沒有 curl），kubectl exec 進去什麼都做不了。kubectl debug 注入一個臨時的 ephemeral container，跟目標容器共享 PID / network namespace，等於開了一個 debug sidecar：

bash 
# 用 busybox 注入一個 ephemeral container 共享 target 的 namespace
$ kubectl debug -it <pod> --image=busybox:1.36 --target=<container>

# 進去後可以看 target 的 process / 網路
/ # ps aux               # 看到 target container 的 process
/ # nslookup db.svc.cluster.local
/ # wget -O- http://localhost:8080/health

# --copy-to：複製一個新 pod 並改 image / command (不影響原 pod)
$ kubectl debug <pod> --copy-to=debug-copy --image=busybox:1.36 -- sh
!直接 ephemeral container
共享 target pod 的 namespace，能直接看到 target process、它打開的 socket。但因為注入到 live pod，無法改 target 的 entrypoint / env，也不能在 production 上隨便試。
✓--copy-to 複製版
克隆一份新 pod，可以改 image、改 command、塞 sleep infinity。原本的 production pod 完全不動。線上排查 production pod 比較安全的做法。
!
ephemeral container 不能刪 kubectl debug 注入後，那個 ephemeral container 會留在 pod spec 裡直到 pod 被刪掉重建。在 production 對很多 pod 都用 debug 注入會把資源吃光，記得做完事 pod 砍掉重來。
04 — Workload 管理

Deployment、StatefulSet、Job 怎麼選

Deployment 的 rolling update 怎麼安全升級

Deployment 預設策略 RollingUpdate：一次取代一部份 pod，全部換完才算完成。兩個關鍵旗標：

欄位意思建議值
maxSurge升級時允許多開幾個 pod25% 或 1（小規模）
maxUnavailable升級時允許暫時不可用幾個25% 或 0（要無中斷）
yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: example-app
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1            # 升級時最多多開 1 個
      maxUnavailable: 0      # 不容忍任何 downtime
  selector:
    matchLabels:
      app: example-app
  template:
    metadata:
      labels:
        app: example-app
    spec:
      containers:
        - name: app
          image: example-app:v1.4.0
          readinessProbe:
            httpGet:
              path: /healthz
              port: 8080

rollout：升級監控與回滾

bash 
# 看升級進度 (會 block 直到全部完成或失敗)
$ kubectl rollout status deploy/<deploy-name>

# 看歷史 (CHANGE-CAUSE 來自 annotation)
$ kubectl rollout history deploy/<deploy-name>
REVISION  CHANGE-CAUSE
1         <none>
2         "upgrade to v1.4.0"
3         "hotfix v1.4.1"

# 回滾到上一版
$ kubectl rollout undo deploy/<deploy-name>

# 回滾到指定版本
$ kubectl rollout undo deploy/<deploy-name> --to-revision=1

# 手動觸發 rollout (例如改了 ConfigMap，pod 不會自動重啟)
$ kubectl rollout restart deploy/<deploy-name>

scale vs HPA — 一個經典陷阱

!
HPA 會吃掉你的 kubectl scale 如果這個 Deployment 已經被 HorizontalPodAutoscaler 管理，你下 kubectl scale --replicas=2 之後幾秒鐘 HPA 就會根據 CPU/Memory 指標把它調回 5。要先暫停 HPA（kubectl delete hpa 或改 minReplicas）才會生效。
bash 
# 手動 scale
$ kubectl scale deploy <deploy-name> --replicas=5

# 看有沒有 HPA 在管
$ kubectl get hpa
NAME           REFERENCE                 TARGETS         MINPODS   MAXPODS   REPLICAS
example-app    Deployment/example-app    cpu: 65%/70%    2         10        5

StatefulSet vs Deployment

選擇規則很簡單：應用程式需不需要「穩定的身份」？

需求DeploymentStatefulSet
Pod 名稱隨機（<app>-7d8f...）有序（<app>-0、<app>-1）
啟動順序同時併發嚴格從 0 開始順序起
持久存儲shared / 無每個 pod 自己的 PVC
網路身份共用 Service 後面每個 pod 獨立 DNS
典型用途無狀態 web、APIDB、訊息佇列、ZK、etcd

Job vs CronJob

Job 跑一次（DB schema migration、批次匯入），CronJob 是 cron 排程的 Job。

yaml 
# 一次性：跑一次 schema migration 就好
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate-v1.5.0
spec:
  backoffLimit: 3
  template:
    spec:
      restartPolicy: OnFailure
      containers:
        - name: migrate
          image: example-app:v1.5.0
          command: ["./migrate.sh"]
---
# 排程：每天凌晨 3 點備份
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 3 * * *"
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: example-app/backup:v1
              command: ["/usr/local/bin/backup.sh"]
05 — 網路與設定

Service、Ingress、ConfigMap、Secret

Service 三種類型，三種用途

類型對外可見性什麼時候用
ClusterIP (預設)只有集群內部能連內部微服務之間溝通；99% 的 service 都用這個
NodePort每個 Node 開一個 port (30000–32767)開發、本機測試；prod 不建議直接用
LoadBalancer雲端 LB 拉一個外部 IP有雲商支援時，公開單一服務
ExternalNameDNS CNAME把外部服務（例如 RDS）包成集群內可解析的 service
yaml 
apiVersion: v1
kind: Service
metadata:
  name: example-app
spec:
  type: ClusterIP
  selector:
    app: example-app
  ports:
    - name: http
      port: 80          # Service 對外 (給其他 pod 連) 的 port
      targetPort: 8080   # pod 內部容器實際在聽的 port

集群內 DNS：三種長度都能用

K8s 內建 CoreDNS 為每個 Service 配一個固定的 DNS 名字。同一 namespace 內，三種寫法都通：

長度寫法使用情境
完整 FQDN<svc>.<ns>.svc.cluster.local跨 namespace 最保險，也是內部實際解析的全名
簡短<svc>.<ns>跨 namespace 寫法，常用
最短<svc>同 namespace 內最常見

Ingress：path-based 與 host-based 路由

Ingress 是 L7 入口。實作通常是 nginx-ingress 或雲商的 ALB / GLB。一個 Ingress 可以同時做：

  • host-based：api.example.com 跟 web.example.com 走不同 service
  • path-based：/api/* 給 backend、/* 給前端 SPA
  • TLS termination：HTTPS 在 Ingress 解掉，後面 Service 跑 HTTP
yaml 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: example-app
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/proxy-body-size: "20m"
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts: [app.example.com]
      secretName: app-tls
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: example-app-api
                port:
                  number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: example-app-web
                port:
                  number: 80

nginx-ingress 常用 annotations

Annotation用途
nginx.ingress.kubernetes.io/rewrite-target路徑改寫，例如 /api/v1/x → /x
nginx.ingress.kubernetes.io/ssl-redirect強制 HTTP → HTTPS
nginx.ingress.kubernetes.io/proxy-body-size上傳大小上限（預設 1m）
nginx.ingress.kubernetes.io/proxy-read-timeout後端回應 timeout（秒）
nginx.ingress.kubernetes.io/cors-allow-originCORS 跨域
nginx.ingress.kubernetes.io/limit-rps每秒請求數限速

ConfigMap vs Secret

兩者結構幾乎一樣，差別是「定位」。

✓ConfigMap
放非機密的設定：feature flag、URL endpoint、log level、時區、Java 設定檔。明文存在 etcd。
!Secret
放機密的：API key、DB 密碼、TLS 私鑰。注意：Secret 不是加密！只是 base64 編碼。真正的加密要打開 EncryptionConfiguration，或改用 SealedSecrets / External Secrets / Vault。

三種注入方式

方式YAML 寫法適合
單一 key 為 env env.valueFrom.configMapKeyRef 挑幾個欄位變成環境變數
整批為 env envFrom.configMapRef 整個 ConfigMap 全部變環境變數
掛成檔案 volumes + volumeMounts 應用程式只接受設定檔（如 nginx.conf、application.yaml）
yaml 
spec:
  containers:
    - name: app
      image: example-app:v1
      # (1) 挑單一 key 變成環境變數
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: log_level
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secret
              key: db_password
      # (2) 整個 ConfigMap 全變環境變數
      envFrom:
        - configMapRef:
            name: app-config
        - secretRef:
            name: app-secret
      # (3) 掛成 /etc/app/config.yaml 檔案
      volumeMounts:
        - name: config
          mountPath: /etc/app
          readOnly: true
  volumes:
    - name: config
      configMap:
        name: app-config
ConfigMap 改了，pod 不會自動重啟 陷阱

這是新手最常踩的雷之一：

  • 用 env 注入：環境變數在 pod 啟動時就讀好，ConfigMap 改了 pod 完全不會察覺，要 kubectl rollout restart 才會生效。
  • 用 volume 掛載：檔案內容會被 kubelet 自動更新（約 1 分鐘延遲），但應用程式自己要重新讀檔，K8s 不會 SIGHUP 它。

實務上常用 stakater/Reloader 之類的 controller，看到 ConfigMap / Secret 變動就自動觸發 rollout。

06 — Helm chart 操作

Helm：K8s 的 package manager

把幾十個 YAML 模板化、參數化、版本化，一條指令裝完整套服務 — 這就是 Helm 在做的事。

Chart 的目錄長相

text 
example-chart/
├── Chart.yaml          # chart 的 metadata (name / version / appVersion)
├── Chart.lock          # dependency 鎖版本 (相當於 lockfile)
├── values.yaml         # 預設參數
├── values-prod.yaml    # 環境覆寫 (慣例命名)
├── charts/             # dependency 抓下來的 sub-chart
└── templates/
    ├── _helpers.tpl    # 共用的 named template (相當於 macro)
    ├── deployment.yaml
    ├── service.yaml
    ├── ingress.yaml
    └── configmap.yaml

三條最常用的指令

bash 
# (1) 本機渲染：不接觸集群，把 chart 渲染成最終的 manifest YAML
#     除錯時的第一步！能看出 template 有沒有寫對
$ helm template <release> ./example-chart -f values-prod.yaml

# (2) 安裝或升級 (冪等：沒裝過就 install，裝過就 upgrade)
$ helm upgrade --install <release> ./example-chart \
      -n <your-ns> --create-namespace \
      -f values.yaml -f values-prod.yaml \
      --set image.tag=v1.5.0

# (3) helm-diff plugin：upgrade 前先看會變什麼
$ helm plugin install https://github.com/databus23/helm-diff
$ helm diff upgrade <release> ./example-chart -f values-prod.yaml

# 其他常用
$ helm list -A                                  # 看所有 release
$ helm history <release> -n <your-ns>      # 看升級歷史
$ helm rollback <release> 3 -n <your-ns>   # 回滾到第 3 版
$ helm uninstall <release> -n <your-ns>    # 卸載
i
helm template 是除錯神器 當 helm upgrade 失敗只給你一行神祕錯誤，先在本機 helm template 渲染出最終 YAML，肉眼掃一遍就知道哪裡縮排錯、哪個 value 沒帶到。它完全不接觸集群，可以隨便試。

_helpers.tpl 與 named templates

每個 K8s 物件都要重複寫一堆 labels（app、version、release...），把這些抽到 _helpers.tpl 變成 named template，需要的地方 {{ include "..." . }}：

yaml 
# templates/_helpers.tpl
{{/* 共用 labels {{ include "myapp.labels" . }} */}}
{{- define "myapp.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end -}}
yaml 
# templates/deployment.yaml — 任何地方需要那組 labels 就 include 它
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}

面向HelmKustomize
核心概念模板 + 參數純 YAML overlay
學習曲線需學 Go template幾乎只是 YAML
跨環境-f values-prod.yamloverlays/prod/
第三方套件生態完整（artifacthub.io）沒有正式 registry

i
實務上常常兩個並用 公司內部自家應用程式用 Kustomize 管（diff 看得清楚），第三方依賴（PostgreSQL、Redis、cert-manager、ingress-nginx）就裝官方 Helm chart。下一堂 Argo CD 會看到這兩個怎麼跟 GitOps 整合。

krew 是 kubectl 的 brew，一條指令裝插件。下面這幾個近乎必裝：

$ kubectl krew install ctx ns tree neat who-can

# 切 context (取代 kubectl config use-context)

# 切 namespace (取代 set-context --current --namespace=...)

$ kubectl tree deploy <deploy-name>

# 把 kubectl get -o yaml 的輸出整理乾淨 (拿掉 status / managedFields)
$ kubectl get pod <pod> -o yaml | kubectl neat

# 看誰有權做某事 (RBAC 反查)
$ kubectl who-can delete pods -n <your-ns>

08 — 資源容量調校

requests / limits / QoS class

kubectl top — 即時用量

需要 cluster 裝 metrics-server（多數雲商 managed K8s 預設有）。沒裝的話 kubectl top 會直接報錯。

bash 
# 看每個 Node 的 CPU / 記憶體用量
$ kubectl top node
NAME           CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
node-1         420m         21%    3200Mi          40%
node-2         1100m        55%    5800Mi          72%

# 看每個 Pod 的用量 (按 CPU 排序)
$ kubectl top pod -A --sort-by=cpu

# 多容器 pod，看到容器層級
$ kubectl top pod <pod> --containers

從 Prometheus 看歷史趨勢

kubectl top 只看「現在」。要設合理的 requests / limits，要看一週 / 一個月的歷史尖峰。常用兩條 PromQL：

promql 
# 某 namespace 內每個 pod 的 working set memory (近 7 天 P99)
quantile_over_time(0.99,
  container_memory_working_set_bytes{namespace="<your-ns>", container!=""}[7d]
)

# 某 deployment 的 CPU 使用率 (近 1 小時)
sum(rate(
  container_cpu_usage_seconds_total{namespace="<your-ns>", pod=~"example-app-.*"}[5m]
)) by (pod)
i
下一堂課的伏筆 這兩個 metric 名稱會在 Lesson 04 (監控實務) 再見。先記住：working_set_bytes 是 K8s 用來判斷 OOM 的依據，比 RSS 更準。

requests vs limits — 兩組不同意義的數字

欄位調度器看kernel 強制建議寫法
requests ✓ 用來決定能塞到哪個 Node ✗ 填「平均使用量 + 一點 buffer」
limits ✗ ✓ 超過 CPU 會被 throttle、超過記憶體會被 OOMKill 填「P99 / 尖峰 + 安全係數」
yaml 
resources:
  requests:                # 排程器以這個算 Node 容量
    cpu: "100m"           # 100 millicores = 0.1 vCPU
    memory: "256Mi"
  limits:                  # 超過這條 kernel 出手
    cpu: "500m"           # CPU 超過 → throttle (變慢但不死)
    memory: "512Mi"       # Memory 超過 → OOMKilled (直接砍掉)

QoS class：三個等級

K8s 根據你怎麼設 requests / limits，自動把 pod 分到三種 QoS class。Node 資源不足時，從低等級先趕。

QoS class條件被 evict 順序適合
Guaranteed 每個容器 requests == limits（CPU 和 memory 都要設） 最後 核心服務、DB、必須穩的
Burstable 有設 requests 但 ≠ limits（最常見） 中 大多數 web app / API
BestEffort 完全沒設 requests 或 limits 最先（資源不足直接被踢） 跑批次工作、臨時 debug
✓
實務建議 所有正式服務最低也要 Burstable —— 至少把 requests 設好。核心 DB / 訊息佇列再考慮 Guaranteed。BestEffort 永遠不該出現在 production。
