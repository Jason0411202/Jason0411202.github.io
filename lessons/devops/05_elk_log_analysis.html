01 — 日誌管線

一條 log 的旅程：從 println 到 Kibana

「我在程式裡 print 了一行錯誤訊息，怎麼讓它出現在 Kibana 的搜尋欄裡？」這個過程比想像中經過更多手。先看完整的 5 步：

1
App 寫到 stdout / stderr
應用層用 fmt.Println / console.log / logger.info(...) 輸出，不要寫到本地檔案。container 的世界裡 stdout 才是合約。
2
Container runtime 落地成檔
containerd / cri-o 會把 stdout 寫到 host node 的 /var/log/containers/<pod>_<ns>_<container>.log，每行包成 JSON 並附上 timestamp / stream。
3
Fluent Bit DaemonSet 在每個 node 讀檔
tail 那些 log 檔、用 Kubernetes filter 補上 namespace / pod / container / labels metadata、用 JSON parser 拆出結構化欄位。
4
Push 到 Elasticsearch
寫進 <app>-YYYY.MM.DD 之類的 daily index。ES 自動幫你建 mapping，欄位變成可被搜尋的 inverted index。
5
Kibana Data View 查詢
Kibana 不直接讀 log 檔，它透過 Data View（pattern <app>-*）跟 ES 對話。@timestamp 是 time field，所有的 KQL / Lens / Dashboard 都建立在 Data View 之上。
圖 1.1 — 一條 log 的 5 步旅程：app → node → Fluent Bit → ES → Kibana

為什麼 namespace 過濾這麼重要

在一個動輒幾十個 namespace 的 cluster 裡，kube-system、ingress-nginx、cert-manager 這些系統 namespace 噴出來的 log 通常是業務 log 的數倍。在 Fluent Bit 設定中加上 namespace allowlist 或 denylist，能讓 ES index 大小直接降 5 - 10 倍：

yaml · fluent-bit configmap 
# 只收業務 namespace；用 grep filter 過濾
[FILTER]
    Name        kubernetes
    Match       kube.*
    Merge_Log   On
    Keep_Log    Off

[FILTER]
    Name        grep
    Match       kube.*
    # 只保留 namespace 是 <ns> / production / staging 的
    Regex       $kubernetes['namespace_name'] ^(<ns>|production|staging)$

[FILTER]
    Name        grep
    Match       kube.*
    # 排除健康檢查 noise
    Exclude     log /healthz

JSON 結構化 vs 純文字 log

同一個錯誤事件，應用程式輸出 JSON 還是純文字，會徹底改變 Kibana 裡能做的事。

✗純文字 log
2026-04-28 12:31:09 ERROR /api/orders 500 latency=842ms user=u_42 ES 只把它當成一坨 message 字串。想找「latency 超過 500ms 的 POST」？只能 wildcard latency=*，找不出範圍，更不能聚合算 p95。
✓JSON 結構化 log
{"ts":"2026-04-28T12:31:09Z","level":"error","path":"/api/orders","status":500,"latency_ms":842,"user_id":"u_42"} ES 自動把 status 變 int、latency_ms 變 float。能 range 查詢、能 aggregate、能畫 p95 line chart。
i
要讓 log 變結構化，應用程式本身就要先輸出 JSON Fluent Bit 的 JSON parser 只能解開「app 已經是 JSON」的 log。常見做法：Go 用 logrus 的 JSONFormatter 或 zap；Node.js 用 pino；Python 用 python-json-logger。不要還在 print 純文字然後寄望下游用 regex 拆 — 那條路又脆又慢。

Index 命名與 ILM 基礎

常見命名 pattern：

Pattern例子適用
<app>-YYYY.MM.DDexample-app-2026.04.28每天一個 index，舊資料容易整批刪
logstash-YYYY.MM.DDlogstash-2026.04.28遺留命名，ECK 預設模板還會用
<app>-<env>-*example-app-prod-000001搭配 rollover，靠 ILM 自動切新 index

ILM (Index Lifecycle Management) 把 index 的生命分成幾個階段，每個階段做不同的事：

階段典型動作用途
hotrollover (達到 50GB 或 1 天就切新 index)剛寫進來、被頻繁查詢的資料
warmforce merge、降 replica、移到便宜 node偶爾查詢、不再寫入
cold / frozen搬到 searchable snapshot合規保留、極少查詢
delete30/60/90 天後直接刪釋放磁碟，控成本
02 — Kibana 全景

Kibana 8.x 的 UI 地圖

第一次打開 Kibana 8 會有點迷路，因為左上角那個 hamburger menu 把功能塞得很滿。先記住這張地圖，後面 5 個 section 都在這上面操作。

準備工作：取密碼 + port-forward

如果你用 ECK (Elastic Cloud on Kubernetes) 安裝，初始 elastic 密碼會放在 secret 裡：

bash 
# 1. 抓 elastic super-user 的密碼
$ kubectl -n elastic-system get secret \
    <es>-es-elastic-user -o jsonpath='{.data.elastic}' | base64 -d; echo

# 2. port-forward Kibana service 到本機 5601
$ kubectl -n elastic-system port-forward svc/<kibana>-kb-http 5601

# 3. 瀏覽器開 https://localhost:5601
#    帳號 elastic、密碼貼上面那個
!
port-forward 是 dev 用，不是 prod 用 正式環境應該要把 Kibana 套到 Ingress + OIDC SSO 後面，並且用獨立的低權限 role 給工程師查 log，不要所有人共用 elastic super-user。

Hamburger menu 的兩大區塊

✓Analytics — 平常用
  • Discover — 看 log，下 KQL
  • Dashboards — 組好的儀表板
  • Visualize Library — 存好的圖表
  • Lens — 拖拉式新建圖表
  • Maps — 地理視覺化（IP / GeoIP 用得到）
✓Management — 偶爾用
  • Stack Management
    • Data Views — 定義要查哪些 index
    • Index Lifecycle Policies — ILM
    • Index Management — 看 index 大小
    • Connectors / Rules — 告警設定
  • Dev Tools — 直接打 ES API

整堂課我們會在這兩個區塊輪流跳。先記住路徑大概在哪，之後就不會覺得迷宮。

03 — Data View

第一個 Data View：把 index 變成 Kibana 看得懂的東西

Kibana 不是直接讀 ES index，它讀的是「Data View」（舊版叫 Index Pattern）。一個 Data View 包了三件事：哪些 index、用哪個欄位當時間、欄位怎麼解讀。

建立步驟

  1. 左上 hamburger → Stack Management
  2. 左側 Kibana → Data Views
  3. 右上 Create data view
  4. 填三個欄位：
    • Name：<app>-logs (給人看的名字)
    • Index pattern：<app>-* (對應 ES 那邊的 index)
    • Timestamp field：@timestamp (時間軸要靠它)
  5. 按 Save data view to Kibana
i
欄位列表是 snapshot，不會自動跟 當你的應用新增了一個欄位（例如多輸出 request_id），Kibana 不會自動知道。要回到 Data View 詳細頁，按右上角的 Refresh field list，新欄位才會出現在 Discover 的 sidebar。

典型 Data View 範例

用途NameIndex patternTime field
單一服務<app>-logs<app>-*@timestamp
整個 clustercluster-logs-alllogstash-*,fluentbit-*@timestamp
指標型 metric (跟 log 同樣思路)metricbeatmetricbeat-*@timestamp

同一個 Data View 可以對到多個 index pattern，逗號分隔。但不要把一個業務跟一個系統 namespace 混在同一個 Data View，會讓查詢條件變很長。

04 — Discover & KQL

Discover：log 工程師每天 90% 的時間都在這頁

Analytics → Discover。打開後左邊選剛才的 Data View，畫面會分成三塊：

  • 右上時間選擇器 — 時間範圍 Last 15 minutes / Today / absolute date。incident 排查時通常切成 Last 5 minutes，再開 auto-refresh 10s 變即時 tail。
  • 左側欄位 sidebar — Available fields vs Selected fields；上方搜尋框可以篩欄位名稱。
  • 右側主畫面 — 上方 histogram 顯示分佈，下方表格列出每筆 document。

用「加 column」把表格變得有意義

預設只看到 @timestamp 跟 _source（一坨 JSON），完全沒法快速掃。從左 sidebar 把這幾個欄位加成 column，畫面馬上有可讀性：

columns 
# 推薦的標準 5 欄
@timestamp
kubernetes.namespace_name
method
path
status
latency_ms

# 點任一 row 左邊的展開箭頭，可以看完整 document JSON
# 滑鼠 hover 在欄位值上會出現 + / - 按鈕，一鍵加成 filter

KQL 五式

KQL (Kibana Query Language) 是 Kibana 的查詢方言，比 Lucene 直觀。學會這五式就能應付 95% 的 log 排查：

類型語法例子
等於field:valuelevel:"error"
數字範圍field >= n / > / <=status >= 500
萬用字元field:value*path:/api/*
布林組合AND / OR / NOTlevel:"error" AND NOT path:"/healthz"
欄位存在field:*request_id:*

實戰查詢範例

kql 
# 找這個 namespace 的所有 5xx
kubernetes.namespace_name:"<ns>" AND status >= 500

# error level 但排除健康檢查雜訊
level:"error" AND NOT path:"/healthz"

# 慢的 POST 請求
latency_ms > 500 AND method:"POST"

# 同一個 request 的所有 log（如果你有埋 trace_id）
trace_id:"abc123"

# 多個 status 都要
status:(500 OR 502 OR 503 OR 504)
!
數字欄位不能用 wildcard status:5* 看似合理，但 status 是 int，wildcard 只對 keyword/text 有效。要寫 status >= 500 才正確。剛開始幾乎所有人都會踩這個雷。

KQL ↔ Lucene 切換

查詢框右邊的小 KQL 標籤可以切到 Lucene。Lucene 的優勢是支援 regex (field:/regex/) 跟 fuzzy (~)，KQL 沒有。日常用 KQL，只有少數需要 regex 的場合切過去。

Saved Search 與 CSV 匯出

按右上 Save 把當前 Discover 狀態（filter + columns + Data View + 時間）存成 Saved Search，下次直接打開、也可以拿去 Dashboard 當 panel。要把表格匯出給非工程師看，Share → Generate CSV report，最多 10000 筆。

KQL 進階技巧 cheat sheet 深入
kql 
# nested 欄位（k8s metadata 通常是 nested）
kubernetes.labels.app:"<app>"

# 多值欄位 array contain
tags:("slow" AND "retry")

# 否定多個值
NOT level:("debug" OR "trace")

# 文字含關鍵字（會走 analyzer，可能斷詞）
message:"connection refused"

# 完全比對（用 .keyword sub-field）
message.keyword:"connection refused"

# 範圍 inclusive / exclusive 寫法
latency_ms:[100 TO 500]      # inclusive
latency_ms:{100 TO 500}      # exclusive

# 判斷欄位「不存在」
NOT request_id:*
05 — Lens 視覺化

Lens：拖拉欄位就出圖

Analytics → Lens（或 Discover 右上「Visualize」也會帶你進來）。Lens 是 Kibana 8 主推的圖表編輯器：你拖欄位進 X / Y / Break down，它幫你猜圖表類型；不滿意可以右上角換。

三大常用視覺化類型

類型適合的問題範例
Bar (vertical / stacked)分類×時間的計數每分鐘 5xx vs 4xx vs 2xx 數量
Line連續指標隨時間變化p95 latency 趨勢線
Metric / Gauge單一數字 KPI過去 5 分鐘錯誤總筆數
Table / Top values誰最多Top 10 path、Top 10 error pod
Heat map兩維度交叉每小時 × namespace 的錯誤分佈

範例 1：log 量隨時間變化（stacked bar by status）

lens config 
# 操作步驟
1. Data View      = <app>-logs
2. 拖 @timestamp 到 Horizontal axis
3. 拖 status      到 Vertical axis     → Lens 自動變 Count of records
4. 拖 status      到 Break down by     → 變成 stacked bar
5. 圖表類型右上選 Bar vertical stacked
6. 命名: "<app> · log volume by status"

範例 2：Top 10 path

lens config 
1. 圖表類型      = Bar horizontal
2. Vertical axis = Top values of "path" (size 10)
3. Horizontal    = Count of records
4. 排序          = By Count of records descending
5. 命名: "Top 10 paths · last 24h"

範例 3：p95 latency line

lens config 
1. 圖表類型     = Line
2. Horizontal   = @timestamp (auto interval)
3. Vertical     = Percentile of "latency_ms" at "95"
4. (可選) Break down by "path" 看每條 endpoint 的 p95
5. 命名: "latency p95 by path"
i
多 layer 疊加：line + bar 同框 Lens 右側的 Layers 區可以加第二層。常見組合：bar 顯示請求數量、線疊加顯示 p95 latency。視覺上一眼看出「量大 + 慢」的時段是什麼時候。

圖表做好後按右上 Save，存到 Visualize Library。下一節 Dashboard 會直接從 Library 抓進來。

06 — Dashboard

把單個圖表組成有故事的 Dashboard

單獨的 Lens 是片段；Dashboard 是整套故事。一個好的服務 Dashboard 通常回答這三個問題：

  • 現在 — 過去 5 分鐘 RPS、錯誤率、p95 latency（top 大字 metric）
  • 趨勢 — log volume / error rate / latency 的 line chart
  • Top N — 哪個 path、哪個 pod、哪個 user 出問題最多

建立流程

  1. Analytics → Dashboards → Create dashboard
  2. 右上 Add from library 把剛存的 Lens 加進來
  3. 拖大小、排版（建議 metric 放最上，趨勢圖中段，Top N 放下方）
  4. 每個 panel 點右上齒輪可以加 panel-level filter（例如固定 kubernetes.namespace_name:"<ns>"）
  5. 整個 dashboard 加全域 filter / KQL，整片同步收斂
  6. 右上 Save

Controls：給觀看者的快速 filter

Dashboard 上方可以加 Controls，讓非工程師同事也能 self-service 切換條件：

controls config 
# 進入 dashboard edit 模式 → 上方 Controls → Add control

Control 1:
  type:    Options list
  field:   status                         # 下拉多選 200/404/500
  label:   "HTTP status"

Control 2:
  type:    Options list
  field:   kubernetes.pod_name            # 多選 pod
  label:   "Pod"

Control 3:
  type:    Range slider
  field:   latency_ms
  label:   "Latency (ms)"

# Save controls 後，這些選擇器會掛在 dashboard 最上方
# 觀看者點選後整片 dashboard 都會跟著更新

分享方式

方式內容適合場景
Permalink (Snapshot)當下狀態凍結成 URL，含時間範圍與 filterSlack 貼給同事看「這個時段的 incident」
Permalink (Saved object)連結指向 Saved Dashboard，內容會跟著更新放進 wiki / runbook
Embed codeiframe 嵌進內部入口內部 Status Page
PNG / PDF report排程產出靜態快照每週週報、合規存證
i
Dashboard 不是越炫越好 每個 panel 都要能回答一個明確問題。如果一張圖你自己也說不出「這在告訴我什麼」，那它就只是裝飾。把那種圖刪掉，整個 dashboard 會立刻好用 3 倍。
07 — Alerting

從「事後查 log」進化到「主動被通知」

到這一節為止，我們做的都是 pull：人去查 log、看 dashboard。Alerting 把它翻過來變 push：條件命中就主動敲你 Slack。Kibana 的 Alerting framework 由三個概念組成：

概念是什麼例子
Connector對外通訊的端點（一次性設好就重複用）Slack webhook、SMTP、PagerDuty、generic Webhook
Rule定期執行的條件（cron + query + threshold）每分鐘檢查最近 5 分鐘 5xx 是否 > 10
ActionRule 命中時要做什麼，必須綁一個 Connector呼叫 Slack connector 發訊息

Step 1：建一個 Slack Connector

  1. Stack Management → Alerts and Insights → Connectors
  2. Create connector → 選 Slack
  3. 填入 Slack Incoming Webhook URL（從 Slack App config 複製過來）
  4. 命名 slack-prod-alerts → Save

Step 2：建一條 Rule

路徑：Stack Management → Alerts and Insights → Rules → Create rule。下面以 Elasticsearch query 類型為例：

rule config 
Name:         "<app> 5xx surge"
Rule type:    Elasticsearch query
Index:        <app>-*
Time field:   @timestamp
Query (KQL):  status >= 500
Time window:  last 5 minutes

Threshold:
  WHEN: count of documents
  IS:   > 10
  OVER: last 5 minutes

Schedule:     every 1 minute
Notify:       Only on status change   # 避免 spam

# Action
Connector:    slack-prod-alerts
Run when:     Query matched
Message:
  :rotating_light: {{rule.name}} fired
  Hits: {{context.hits}} over last 5m
  <{{context.link}}|Open in Kibana>

常用的 mustache 變數：{{rule.name}}、{{context.hits}}、{{context.title}}、{{context.link}}、{{date}}。

Maintenance Windows：計畫部署時的消音閥

每次部署一定會有短暫 5xx，照原本 rule 一定會吵。Stack Management → Maintenance Windows → Create 設一個時段（也可以 RRULE 週期化），那段時間命中的 rule 不會發 action：

maintenance window 
Name:        "<app> release window"
Start:       2026-04-29 10:00 (Asia/Taipei)
Duration:    30 minutes
Filter:      kibana.alert.rule.tags:"<app>"   # 只壓這支服務的 rule
Repeat:      Weekly · 每週二 10:00 (如果你固定週二發版)
!
不要把 maintenance window 開太寬 見過團隊因為怕被吵，把 window 設成 24/7 結果服務真的掛了也沒人知道。Maintenance window 是「我預期會吵」，不是「我不想被吵」。前者是工程，後者是逃避。
08 — Production 維運

真實環境的疑難雜症處理

「日誌都沒出來」三步排錯

新手 95% 會遇到的問題：「我 deploy 了、應用程式有 print 啊，可是 Kibana 一片空。」依下面三步順序檢查，5 分鐘內能定位：

1
Fluent Bit pod 是不是 Running？
90% 的「沒 log」都死在這層。CrashLoopBackOff 通常是 ConfigMap YAML 有錯。
2
ES 那邊有 index 嗎？
Fluent Bit 健康，但 ES 裡沒 index，多半是 Fluent Bit 認證或網路出狀況。Dev Tools 直接 GET _cat/indices 確認。
3
Data View pattern + 時間範圍對嗎？
ES 有 index 但 Discover 空白？檢查 Data View pattern 是否真的 cover 那個 index，再把時間拉成 Last 24 hours，避免 5 分鐘內剛好沒事件。
bash · 三步指令 
# Step 1 · Fluent Bit 健康嗎
$ kubectl -n logging get pods -l app=fluent-bit
# 期望: 每個 node 一個 pod，全部 Running

$ kubectl -n logging logs -l app=fluent-bit --tail=100
# 期望: 看到 [info] [output] elasticsearch ... ok 之類

# Step 2 · 在 Kibana Dev Tools 確認 index 存在 (下節介紹)
GET _cat/indices/<app>-*?v&s=index

# Step 3 · 回 Discover，時間拉寬 + 確認 Data View 抓得到

Index 容量管理

Stack Management → Index Management 看每個 index 的 size。常見治理動作：

  1. 找出佔用最大的 top-N index，按 Storage size 倒序排
  2. 確認熱門 index 都掛了 ILM policy（Settings → Index lifecycle management policy）
  3. 沒 ILM 的就立刻補，否則磁碟很快爆

典型 ILM policy：

ilm policy 
Phase: hot
  rollover when:
    - max primary size: 50 GB
    - max age:          1 day

Phase: warm  (after 2 days)
  - replicas: 1   (從 hot 的 2 降下來)
  - force merge: max segments = 1
  - shrink (optional)

Phase: delete (after 30 days)
  - delete index

Dev Tools：直接打 ES API

Management → Dev Tools 是 Kibana 內建的 ES REST API console，左邊寫 request，Ctrl + Enter 送出，右邊看 JSON 回應：

dev tools · es api 
# 1. 集群健康（red 就準備痛苦）
GET _cluster/health

# 2. 看你關心的 index 列表 + 大小 + doc 數
GET _cat/indices/<app>-*?v&s=index

# 3. 直接搜尋 (跟 Discover 等價，但能直接看 raw response)
POST <app>-*/_search
{
  "size": 5,
  "query": {
    "match": { "status": 500 }
  },
  "sort": [{ "@timestamp": "desc" }]
}

# 4. 看單一 index 的 mapping（欄位型別對不對）
GET <app>-2026.04.28/_mapping

# 5. 緊急：手動刪一個爆掉的測試 index
DELETE <app>-test-2026.04.28
常用 ES API cheat sheet 深入
es-api 
# 看 node / shard 分佈
GET _cat/nodes?v
GET _cat/shards/<app>-*?v

# 看哪個 index 沒掛 ILM
GET */_settings/index.lifecycle.name?flat_settings=true

# 看 ILM 跑到哪一步、有沒有失敗
GET <app>-*/_ilm/explain

# aggregation 範例：每個 status 多少筆
POST <app>-*/_search
{
  "size": 0,
  "aggs": {
    "by_status": {
      "terms": { "field": "status", "size": 10 }
    }
  }
}

# 改一個 index 的 number_of_replicas (省成本)
PUT <app>-2026.04.27/_settings
{ "index.number_of_replicas": 0 }
