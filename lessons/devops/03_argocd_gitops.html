Argo CD GitOps 持續部署

把 Git 變成 cluster 的 single source of truth：CI 不再 kubectl apply，而是更新 manifest，剩下交給 cluster 內常駐的 Argo CD 自動拉取、比對、套用。從根本上改變部署的權責與安全模型。

01 — 心智模型

為什麼 Git 該是 single source of truth

傳統 CI/CD 的部署流是這樣的：CI build 完 image 之後，同一條 pipeline 直接拿著 cluster 的 kubeconfig 去 kubectl apply。這條路看起來很順，但有四個結構性問題：

  • 權限太大：CI runner 會有對 production cluster 的寫入權，CI 一旦被入侵就等於 cluster 被入侵。
  • 狀態不可知：cluster 現在跑的版本，跟 Git 上看到的版本是不是一致？沒人能精確回答。
  • 漂移無法察覺：有人手動 kubectl edit 改了 production 的 deployment，沒人知道、沒人攔得住。
  • Rollback 靠運氣：要回到上週版本，得手動回 CI 找對應 commit、重 build、重 apply。

GitOps 把這條路反轉過來：cluster 自己跑一個 controller，定期看 Git，發現有差異就自己同步。CI 永遠不碰 cluster，它只負責更新 manifest repo。

✗Push-based（傳統 CI）
  • CI 持有 cluster credential
  • CI 跑 kubectl apply 推到 cluster
  • 無漂移偵測，手動 edit 不會被反映出來
  • Rollback 要靠重跑 CI
  • 多 cluster 要管 N 份 credential
✓Pull-based（GitOps / Argo CD）
  • cluster 內 controller 主動 pull Git
  • CI 只 push commit，不碰 cluster
  • Self-heal 自動把手動改動拉回 Git 狀態
  • Rollback = git revert，歷史清楚
  • credential 統一在 Argo CD 管

Argo CD 在這張圖裡的角色

Argo CD 是一個跑在 Kubernetes 內的 controller。它做兩件事：(1) 定期 fetch manifest repo、(2) 把 Git 想要的 desired state 跟 cluster 現在的 live state 比對。如果不一致，依照 sync policy 決定要不要自動套用。

圖 1.1 — GitOps Pull-based 部署模型
i
code repo 與 manifest repo 通常分開 一個 repo 放應用程式原始碼，另一個 repo 放 Kubernetes manifests / Helm values。CI 跑完 build 後 commit 到 manifest repo，Argo CD 才 watch manifest repo。這樣 image build 失敗時不會把舊版的 manifest 弄髒。小團隊也可以放在同一個 repo 不同資料夾，重點是 Argo CD watch 的 path 要清楚。
02 — UI 全景

Argo CD UI 怎麼看

裝完 Argo CD 後，第一件事是登入 UI 確認狀態。預設 admin 密碼存在 secret 裡，要自己撈出來。

bash 
# 1. 取得初始 admin password (預設只用一次，第一次登入後請改掉)
$ kubectl -n argocd get secret argocd-initial-admin-secret \
    -o jsonpath='{.data.password}' | base64 -d; echo

# 2. port-forward 把 UI 暴出來 (production 請用 Ingress + TLS)
$ kubectl port-forward svc/argocd-server -n argocd 8080:443

# 3. 瀏覽器打開 https://localhost:8080，使用者 admin、密碼貼上

三個你會反覆回來的頁面

頁面看到什麼常用操作
Application 列表 所有 App 的 health（綠/黃/紅）與 sync 狀態（Synced / OutOfSync） filter by health/sync/project/cluster；切 tile 與 list view；星號 favorite
App detail resource tree：Deployment → ReplicaSet → Pod 一層層展開 點 resource 看 live diff vs Git desired manifest；看 events / logs
Settings repo connection、cluster、project、user、SSO 設定 新增 private repo（SSH key 或 token）、加新的 cluster

Resource tree 上每個節點的顏色代表 health：

  • 綠色：Healthy（Pod Running、Service 有 Endpoints）
  • 黃色：Progressing（rolling update 中、還在等 readiness）
  • 紅色：Degraded（CrashLoopBackOff、ImagePullBackOff）
  • 灰色：Missing / Unknown
i
關掉 admin / 改用 SSO 上線後請把 admin 帳號透過 ConfigMap argocd-cm 設成 admin.enabled: "false"，並接 OIDC（Google / GitHub / Auth0）。Production 集群不該長期留 shared admin 密碼。
03 — Sync 選項

Sync 對話框每個 checkbox 的真實後果

UI 上點 Sync 會跳出一堆 checkbox，每個都對應 cluster 的真實行為。不要無腦全勾，下面這張表是線上事故來源 Top 5。

選項實際做什麼什麼時候勾
Prune 刪掉「Git 已移除、但 cluster 還在」的資源 確定 Git 是 source of truth，重構 manifest 結構時要勾
Force 用 kubectl replace 取代 apply。會強制覆蓋整個物件，破壞 immutable 欄位也直接砍重建 幾乎不該勾。只有 spec.selector 之類 immutable 欄位真的要改才用
Dry Run 只跑 server-side dry-run，看 diff 不寫入 cluster 第一次 sync 一個複雜的 chart 時，先勾這個確認結果
Apply Out Of Sync Only 只 sync 偵測到 drift 的資源，其他 in-sync 的略過 App 很大、只想動有變化的部分時。可以加快 sync 速度
Sync Source Branch 暫時切換到別的 branch / tag / commit SHA sync 一次 要部署 hotfix branch 看效果，又不想動 Application 設定時
Replace 跟 Force 類似但不會刪 finalizer。仍然會破壞欄位 處理 Helm chart 從舊版本升級時 immutable 欄位卡住的情況

單一 resource sync vs 整個 App sync

UI 上點某個資源（例如某個 Deployment）右鍵也能選 Sync，這樣只會動那個資源、不會碰其他。修一個 ConfigMap 不希望順便 restart 鄰居 Deployment 時超有用。

!
Force + Prune 同時勾，可能在 30 秒內讓整個 namespace 消失 假設你不小心把 Application 的 spec.source.path 改成空資料夾（或新建空 directory 的 PR 被誤 merge），Prune 會把所有「Git 沒有但 cluster 有」的資源刪掉、Force 會繞過 finalizer 直接刪除。實務上強烈建議：自動 sync 不要開 prune；手動點 Force 前看三遍 diff。下面 Section 4 會講 allowEmpty: false 這條保命線。
04 — Application CRD

Application 完整 spec 解析

Argo CD 的所有概念最終都落在一份 Application CRD。學會手寫這份 YAML，等於掌握 Argo CD 的全部。

yaml 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: example-app
  namespace: argocd                # Application 物件本身存哪
  finalizers:
    - resources-finalizer.argocd.argoproj.io  # 刪 Application 時連 cluster 內的 resources 一起刪

spec:
  project: default                 # 對應 AppProject，限制 sourceRepos / destinations

  source:
    repoURL: https://github.com/<your-org>/<repo>.git
    targetRevision: HEAD          # 也可以是 branch / tag / commit SHA
    path: deploy/example-app       # Helm chart 或 kustomize 根目錄
    helm:                          # 自動偵測 Chart.yaml；想覆蓋 values 才寫
      valueFiles:
        - values.yaml
        - values-prod.yaml          # 多 values 後面覆蓋前面
      parameters:
        - name: image.tag
          value: v1.4.2

  destination:
    server: https://kubernetes.default.svc  # in-cluster；外部 cluster 要先 argocd cluster add
    namespace: example-prod        # resources 安裝到哪個 namespace

  syncPolicy:
    automated:                     # 不寫這段就是純手動 sync
      prune: true                 # Git 移除的資源也跟著刪
      selfHeal: true              # 任何 kubectl edit 都會被改回 Git
      allowEmpty: false           # 預設 false：保命，防止 Git path 變空時清光 namespace
    syncOptions:
      - CreateNamespace=true        # namespace 不存在時自動建
      - PrunePropagationPolicy=foreground
      - PruneLast=true              # prune 排到 sync 最後一步，避免依賴錯亂
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m

四個 automated 旗標一個一個拆

欄位預設意義
automated.enabled true（只要寫了 automated 區塊） 是否要自動 sync。設 false 等於暫時關閉自動，sync 改用手動觸發
prune false 自動清除 Git 移除的資源。沒開的話刪除 Git 檔案後 cluster 仍會留下舊資源
selfHeal false 手動 kubectl edit 後是否強制回 Git 狀態。跟 HPA 衝突的元兇（看 Section 6）
allowEmpty false 當 Git 偵測到 0 個 resource 時，是否允許 prune 把 cluster 也清空。預設 false 是保命機制
!
allowEmpty: false 救過很多人，不要關 想像一下：有人手滑把 deploy/example-app 整個目錄刪掉提了 PR，CI 通過、merge 進 main。Argo CD 看到 Git path 下 0 個 manifest，如果 allowEmpty: true 又開了 prune: true，就會把整個 production namespace 的 Deployment / Service / ConfigMap 全部 prune 掉。預設 false 會擋下這個動作、把 App 標成 Sync Failed，等人去看。非常多公司被這條救過命，不要為了「我覺得 Git 應該總是對」就把它打開。

Helm 與 Kustomize 自動偵測

Argo CD 看 spec.source.path 底下的檔案：

  • 有 Chart.yaml → 當 Helm chart，跑 helm template
  • 有 kustomization.yaml → 當 Kustomize，跑 kustomize build
  • 都沒有 → 純 directory of YAML，直接 apply

不用在 Application spec 裡明寫 helm: 或 kustomize: 區塊，只有要傳 values / parameters 才需要寫。

05 — Sync waves 與 Hooks

控制 sync 順序：sync waves & resource hooks

有些資源必須跑在前面、有些必須跑在後面。例如 DB schema migration 一定要在新版 Pod 滾出來前跑完，否則新 code 連到舊 schema 會炸。Argo CD 用兩種機制解這件事：sync waves 控制順序，hooks 控制時機。

Sync waves — annotation 排序

每個 resource 可以掛一個 argocd.argoproj.io/sync-wave annotation，數字小的先 apply、數字大的後 apply。沒掛的預設 wave 是 0。

yaml 
# wave -2：先建 Namespace 與 ConfigMap（基礎設施）
apiVersion: v1
kind: Namespace
metadata:
  name: example-prod
  annotations:
    argocd.argoproj.io/sync-wave: "-2"
---
# wave -1：跑 schema migration Job（必須在新 Pod 起來前完成）
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
---
# wave 0：預設，Deployment / Service 在這層
apiVersion: apps/v1
kind: Deployment
metadata:
  name: example-app

Resource hooks — 在 sync 生命週期的哪個點跑

Hook annotation 把一個 resource 從「正常 manifest」升級成「sync 流程的一部分」：

Hook什麼時候跑典型用途
PreSync 所有資源 apply 之前 DB schema migration、清快取
Sync 主要 sync 流程中（跟其他 resource 一起，但可控 wave） 少用，幾乎等同於普通 manifest
PostSync 所有資源 apply 且 healthy 之後 send Slack notification、跑 e2e smoke test
SyncFail sync 失敗時 send 告警、自動回滾

Hook delete policy — 跑完之後要不要留

Hook 通常是 Job，跑完想要自動清掉避免堆積。用 argocd.argoproj.io/hook-delete-policy annotation：

  • HookSucceeded：成功就刪
  • HookFailed：失敗就刪
  • BeforeHookCreation：下次跑之前刪舊的（最常用，方便看上次 log）
yaml 
# 完整範例：DB migration Job 放 PreSync wave -1
apiVersion: batch/v1
kind: Job
metadata:
  name: example-app-migrate
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
    argocd.argoproj.io/sync-wave: "-1"
spec:
  backoffLimit: 2
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: ghcr.io/<your-org>/example-app-migrator:v1.4.2
          command: ["./migrate", "up"]

關鍵：PreSync hook 失敗，後續 wave 全部不會跑。schema migration 沒成功 → 新版 Pod 不會被 apply 出去 → app 不會看到自己連到不對的 schema。這就是 GitOps 安全 deploy 的核心。

06 — ignoreDifferences

處理「Argo CD 跟其他 controller 打架」

Cluster 上很多東西不只 Argo CD 在改：HPA 動態調 replicas、cert-manager 換憑證內容、kube-controller 加 pod-template-hash label。Argo CD 看 Git 是 A、cluster 是 B → 就要 self-heal 改回 A → 然後其他 controller 又改回 B → 無限循環。

HPA 案例：最常見的衝突

1
Git 寫 spec.replicas: 2
這是團隊定的「最小起手 replica」
2
流量上來，HPA 動態擴到 8 個 Pod
HPA controller 把 spec.replicas 改成 8
3
Argo CD reconcile：發現 cluster 8 ≠ Git 2
selfHeal 把 replicas 改回 2，Pod 開始大量 terminate
4
HPA 看到負載還是高，又把 replicas 改回 8
兩個 controller 開始 ping-pong，使用者體驗瞬間爆炸

解法：ignoreDifferences 用 JSON pointer 排除特定欄位

yaml 
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: example-app
spec:
  # ... 其他欄位省略 ...
  ignoreDifferences:
    # 1. HPA 管的 replicas，不要追
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas

    # 2. cert-manager 動態填 secret data，不要追
    - group: ""
      kind: Secret
      name: example-app-tls
      jsonPointers:
        - /data

    # 3. 用 jqPathExpressions 處理 list 內的特定 entry
    - group: apps
      kind: Deployment
      jqPathExpressions:
        - '.spec.template.spec.containers[].image'  # 例如 image 由其他工具改
i
ignoreDifferences 不是逃避，是必要設計 很多新手第一次看到 OutOfSync 的反應是「為什麼 Argo CD 一直紅」，然後就把 selfHeal 關掉。這是錯的方向。正確作法是搞清楚是誰在改、把那欄位排除掉。selfHeal 仍然會盯其他欄位（image、env、resources、selector），保護真正的 drift。常見要排除的還有：kube-controller 加的 pod-template-hash、Linkerd / Istio 注入的 sidecar annotation、metrics-server 寫的 status 欄位。
進階：cluster-wide 預設 ignoreDifferences 深入

Argo CD 的 argocd-cm ConfigMap 可以設一份全 cluster 預設的 resource.customizations.ignoreDifferences.<group>_<kind>，這樣不用每個 Application 都重寫一次：

yaml 
resource.customizations.ignoreDifferences.apps_Deployment: |
  jsonPointers:
    - /spec/replicas
resource.customizations.ignoreDifferences.all: |
  jqPathExpressions:
    - '.metadata.annotations."deployment.kubernetes.io/revision"'

寫在 ConfigMap 裡時改完要 restart argocd-application-controller Pod 才會生效。

07 — 完整流程

從 commit 到 Pod 滾完的完整路徑

把前面所有概念串起來，一次 deploy 從開發者按下 git push 到新版 Pod 跑起來，會經過下面這些階段：

1
CI build image 並 push 到 registry
image tag 通常用 git commit SHA 或 semver，例如 v1.4.2 或 sha-a1b2c3d
2
CI 用 yq 改 manifest repo 的 values.yaml
yq -i '.image.tag = "v1.4.2"' deploy/example-app/values.yaml 然後 commit + push 回 manifest repo
3
Argo CD 偵測 Git 有新 commit
兩條路：(a) 預設 polling 每 3 分鐘；(b) 設 webhook 達到秒級
4
Refresh + reconcile
跑 helm template 算出 desired manifests，跟 cluster live state diff
5
Sync（自動或手動）
依 sync waves 排序、先跑 PreSync hook（migration）、再 apply Deployment
6
kubectl rollout 進行中
App health 變黃（Progressing），等 readinessProbe 通過後變綠
7
PostSync hook 跑完 → 通知
Notification controller 把 sync-succeeded 事件發到 Slack/Discord

從 3 分鐘到秒級：webhook 設定

預設 timeout.reconciliation: 180s（在 ConfigMap argocd-cm 裡）會讓所有 App 每 3 分鐘 polling 一次 Git。production 通常嫌慢，要設 webhook：

bash 
# GitHub webhook 設定步驟
# 1. 在 manifest repo: Settings → Webhooks → Add webhook
#    Payload URL: https://<argocd-server>/api/webhook
#    Content type: application/json
#    Secret: <random-string>  (記下來)
#    Events: Just the push event

# 2. 把 secret 寫進 argocd-secret
$ kubectl -n argocd patch secret argocd-secret \
    --type merge \
    -p '{"stringData":{"webhook.github.secret":"<random-string>"}}'

# 3. 從此 push commit → 幾秒內 Argo CD 就 refresh
#    polling 仍然在跑（fallback），但 webhook 是主要觸發路徑

四種 Application 狀態的解讀

狀態意思怎麼處理
OutOfSync Git desired 跟 cluster live 有差異，但 cluster 上的東西還活著 看 diff，大多時候 sync 一下就好。如果是 HPA 之類的合理 drift → 加進 ignoreDifferences
Degraded 套用了，但運行不健康（Pod CrashLoop、readinessProbe 一直失敗） 點進去看 Pod logs / events（接 Lesson 02 的 kubectl logs / describe 知識）
Missing Git 預期存在的資源 cluster 沒有 多半是新建 App 還沒 sync 過。手動 sync 即可
Unknown 比對失敗，連 desired state 都算不出來 看 argocd-application-controller Pod 的 logs。常見：helm template 渲染錯（values 缺東西）、targetRevision 指到不存在的 branch
08 — Rollback

Rollback 兩種做法 — 但只有一種是 durable

UI：Application → History → 選舊 revision

UI 點 History and Rollback，會列出最近 N 筆 sync 紀錄（含對應的 Git SHA）。選一筆按 Rollback，Argo CD 會用那筆 SHA 重新 sync。

CLI：精準腳本化

bash 
# 1. 看歷史，找回想要的版本 ID
$ argocd app history example-app
ID  DATE                  REVISION
3   2026-04-25 10:14:02   d4e5f6a  (latest)
2   2026-04-24 18:30:11   a1b2c3d
1   2026-04-23 09:05:48   8f9e0c1

# 2. 滾回 ID=2 (a1b2c3d)
$ argocd app rollback example-app 2
!
auto-sync + rollback 是個陷阱 假設 automated.enabled: true，你 UI rollback 到舊版 → cluster 暫時回到舊狀態 → 但 Git 還是新版 → 下一次 reconcile（最多 3 分鐘）就把它 sync 回新版了。UI rollback 在 auto-sync App 上是「臨時」的。真正想 durable rollback 要在 manifest repo 跑 git revert，讓 Git 也回到舊版。

真正正確的 rollback 流程

  1. 線上有問題 → UI 先按 rollback 把 cluster 拉回上一版（救火）
  2. 同時在 manifest repo git revert <bad-commit> 並 push（讓 Git 也回到正確狀態）
  3. 之後 Argo CD reconcile 看 Git 跟 cluster 已經一致，OutOfSync 消失
  4. 事後再去查 root cause、決定要不要 forward-fix

這樣 Git 仍然是 single source of truth，git log 看得到「我們在 X 點 revert 了 Y commit 因為 Z」。這條紀錄一年後翻還看得懂，是 GitOps 的真正價值。

09 — argocd CLI

argocd CLI：腳本化、CI gate、debug 利器

UI 適合人類偶爾看，但批量操作、CI 整合都要靠 CLI。

bash 
# 安裝
$ brew install argocd                  # macOS
$ scoop install argocd                  # Windows

# 登入 (--grpc-web 是穿過 ingress 必加)
$ argocd login argocd.example.com \
    --grpc-web \
    --username admin \
    --password <your-password>

# 也支援 SSO 流程
$ argocd login argocd.example.com --sso --grpc-web

七個你會天天用的指令

指令用途
argocd app list -p <project> 列出某 project 下所有 App 的健康/sync 狀態
argocd app get <app> 看單一 App 的細節：source、destination、最近 sync、resource tree
argocd app diff <app> 顯示 Git desired 跟 cluster live 的差異。有差異時 exit code = 1，超適合接 CI gate
argocd app sync <app> --prune --dry-run dry-run 整個 sync 流程，看會做什麼但不真的做
argocd app history <app> 列出歷史 sync 紀錄與對應 Git SHA
argocd app rollback <app> <id> rollback 到 history 裡的某一筆（注意 auto-sync 陷阱）
argocd app set <app> --sync-policy none 暫時關閉自動 sync（事故處理時很有用）
bash 
# CI gate 範例：PR 階段檢查這次的 manifest 改動會造成什麼 diff
$ argocd app diff example-app --revision "$GITHUB_SHA"
# exit 0 → 沒差異；exit 1 → 有差異，CI 印出 diff、人類審查

# 強制重新從 Git 抓最新 (跳過快取)
$ argocd app get example-app --refresh
$ argocd app get example-app --hard-refresh      # 連 Helm/Kustomize 渲染快取也丟
10 — 進階主題

ApplicationSet、AppProject、Notifications

ApplicationSet — 一份 template 產 N 個 App

當你要在 dev / staging / prod 三個 cluster 各佈一份同個 App，手寫三份 Application 很笨。ApplicationSet 用 generator 模板化：

yaml 
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: example-app-envs
  namespace: argocd
spec:
  generators:
    - list:
        elements:
          - env: dev
            cluster: https://dev-cluster.example.com
            valueFile: values-dev.yaml
          - env: prod
            cluster: https://prod-cluster.example.com
            valueFile: values-prod.yaml
  template:
    metadata:
      name: 'example-app-{{env}}'     # 變數插值
    spec:
      project: default
      source:
        repoURL: https://github.com/<your-org>/<repo>.git
        targetRevision: HEAD
        path: deploy/example-app
        helm:
          valueFiles: ['{{valueFile}}']
      destination:
        server: '{{cluster}}'
        namespace: 'example-{{env}}'
      syncPolicy:
        automated:
          selfHeal: true

除了 list 之外還有：

  • cluster generator — 自動依 Argo CD 註冊的所有 cluster 各產一份 App
  • git generator — 看 Git repo 裡有哪些子目錄就產對應的 App（適合「每個微服務一個資料夾」）
  • matrix / merge generator — 兩兩組合（cluster × env）
  • scm generator — 自動掃描 GitHub org / GitLab group 的所有 repo

AppProject — 多租戶權限隔離

AppProject 是 Argo CD 的安全邊界。設一個 project 給某個 team，限制他們能用哪些 repo、佈到哪些 cluster、創哪些 cluster-scoped 資源。

yaml 
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: team-platform
  namespace: argocd
spec:
  description: Platform team owns shared infra
  sourceRepos:
    - https://github.com/<your-org>/platform-manifests.git
  destinations:
    - namespace: 'platform-*'
      server: https://kubernetes.default.svc
  clusterResourceWhitelist:
    - group: ''
      kind: Namespace
    - group: rbac.authorization.k8s.io
      kind: ClusterRole
  namespaceResourceBlacklist:
    - group: ''
      kind: ResourceQuota

Notifications — sync 失敗自動發 Slack

Argo CD 有獨立的 notifications-controller。設好 service（Slack webhook、Discord、email、Teams）、寫一個 trigger（什麼條件觸發）、加 template（訊息內容），最後在 Application 的 annotation 訂閱。

yaml 
# 在 Application metadata.annotations 訂閱想收的事件
metadata:
  name: example-app
  annotations:
    notifications.argoproj.io/subscribe.on-sync-failed.slack: deploys
    notifications.argoproj.io/subscribe.on-health-degraded.slack: oncall
    notifications.argoproj.io/subscribe.on-deployed.slack: deploys

內建的 trigger 有 on-sync-running、on-sync-succeeded、on-sync-failed、on-sync-status-unknown、on-deployed、on-health-degraded 等。強烈建議至少訂 on-sync-failed 和 on-health-degraded 到 oncall channel。
