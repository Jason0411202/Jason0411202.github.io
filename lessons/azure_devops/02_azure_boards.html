Azure Boards 規劃與追蹤工作

上一課建立了 Azure DevOps 的全景。本課深入第一個服務 Boards——一套以網頁為介面的敏捷規劃與追蹤工具。從六個 hub 的分工、work item 階層,到看板、Backlog、Sprint 與查詢,逐步說明如何把需求拆解、排序並追蹤到上線。

01 — Boards 全景與六個 hub

一個服務,六個協作的 hub

Azure Boards 是一套以網頁為介面的服務,用於規劃、追蹤與討論整個開發生命週期的工作,並原生支援敏捷流程。它把「需求是什麼」「目前做到哪」「誰負責」「何時完成」這些問題,集中在同一處呈現。

進入 Boards 後,左側會展開六個 hub,各自負責一種檢視角度。理解這六者的分工,是熟悉整個服務的起點:

圖 2.1 — Azure Boards 的六個 hub (來源:Microsoft Learn)
Hub負責什麼
工作項目 (Work items)所有 work item 的篩選清單,可依指派對象、狀態、類型快速檢索
看板 (Boards)以卡片呈現 work item,拖放即更新狀態,實施 Kanban 流程管理
待辦清單 (Backlogs)規劃與排序的清單,涵蓋 product backlog 與 portfolio backlog
衝刺 (Sprints)管理 sprint/iteration,提供 task board 與 capacity 規劃
查詢 (Queries)儲存可重複使用的篩選條件,並可產出趨勢圖與 widget 資料
交付計劃 (Delivery Plans)以行事曆檢視跨團隊的交付與相依,觀察 Feature/Epic 的彙整進度
i
同一份資料,多種檢視 六個 hub 並非六套獨立的資料,而是同一批 work item 的不同呈現方式。同一個 User Story 既出現在 Backlog 的排序清單,也出現在看板的某一欄,還能被查詢檢索。在哪個 hub 修改,其他 hub 都會同步反映。
02 — Work Item 階層

Process 範本決定 work item 的類型與層次

建立 Project 時選擇的流程範本 (process),決定了可用的 work item 類型與其階層關係。Azure DevOps 內建三種範本:

  • Basic——最精簡,只有 Epic、Issue、Task 三層,適合初次接觸或小型團隊。
  • Agile——以 User Story 為需求單位,Bug 與 User Story 同層,適合一般敏捷團隊。
  • Scrum——以 Product Backlog Item 為需求單位,術語貼近 Scrum 框架。

以最常用的 Agile 範本為例,需求由大到小拆解為四層,另有 Bug 作為缺陷單位:

圖 2.2 — Agile Process 的四層 work item 階層
層級典型時間跨度描述的內容
Epic數個 sprint 至數季大型業務主題或目標,例如「使用者帳號模組」
Feature數個 sprintEpic 之下可交付的功能集,例如「註冊與登入」
User Story一個 sprint 內站在使用者角度的最小需求,可獨立驗收
Task數小時至一兩天實作 User Story 所需的具體執行步驟
Bug視嚴重度而定缺陷;在 Agile 範本中可與 User Story 同層或視為其子項
!
流程範本一旦選定,事後切換成本高 範本決定了團隊往後使用的 work item 類型與欄位,術語也隨之固定 (Agile 用 User Story、Scrum 用 Product Backlog Item)。建立 Project 時應先與團隊確認採用哪一套流程;雖然可後續變更,但已建立的 work item 對應與報表都需重新調整。
03 — 工作項目

work item:追蹤工作的最小單位

work item 是 Boards 中追蹤工作的基本單位。無論 Epic 或 Task,本質上都是一張結構化的表單,記錄一件工作的全部資訊。常見欄位如下:

欄位作用
標題 (Title)一句話描述這件工作
狀態 (State)目前進度,例如 New / Active / Resolved / Closed
指派對象 (Assigned To)負責這件工作的成員
Area Path / Iteration Path歸屬於哪個團隊區域 (誰)、排在哪個 iteration (何時)
描述 (Description)需求細節、驗收條件或重現步驟
連結 (Links)關聯父子 work item、commit、PR、build 等
討論 (Discussion)留言串,可 @mention 成員;所有變更皆留下歷史紀錄

建立一個 work item 的最直接路徑,是在「工作項目」hub 點選「新增工作項目」,選擇類型 (如 User Story),填入標題後即可儲存;其餘欄位可稍後補齊。以下為一個典型的 User Story:

  • 標題:使用者可用 email 與密碼註冊帳號
  • 狀態:Active
  • 指派對象:後端工程師
  • 描述:提交 email 與密碼後建立帳號,並寄出驗證信;email 重複時回傳明確錯誤
  • 子項 Task:設計資料表、實作註冊 API、串接寄信服務
i
協作與歷史內建於每一張 work item 每次欄位變更與留言都會記錄於「歷史」中,可回溯誰在何時改了什麼。work item 也支援附件 (每檔上限 60 MB、每張 work item 最多 100 個),讓設計稿、規格與截圖能與需求保存在一起,而非散落於外部工具。
04 — 看板

Kanban 看板:讓工作流動可視化

看板 (Kanban Board) 以卡片呈現 work item,每張卡片代表一個項目,每一欄對應一種狀態。將卡片由左欄拖放到右欄,即同步更新該 work item 的 state——進度的推進變成可視化的動作。

看板提供幾項常用的設定,協助管理工作的流動:

  • 欄位對應狀態 (Columns)——可自訂欄位名稱與對應的 state,例如 To Do / In Progress / Code Review / Done。
  • WIP limits (在製品上限)——為某一欄設定卡片數量上限,超過即顯示警示,避免同時開啟過多工作。
  • 泳道 (Swimlanes)——以水平列再次分組,例如把「加急」與「一般」分開檢視。
  • 卡片自訂 (Card customization)——調整卡片上顯示的欄位、顏色與標籤,讓重點一眼可辨。
✓
用 WIP limits 管理流動,而非塞滿待辦 為「進行中」欄位設定合理的 WIP limit,是看板最有價值的實踐之一。它迫使團隊先完成手上的工作再開新的,減少同時進行造成的切換成本與半成品堆積。當某欄頻繁觸頂,正是流程瓶頸浮現的訊號。
05 — 待辦清單與衝刺

Backlog 排序,Sprint 認領

待辦清單 (Backlog) 是一份依優先順序排列的 work item 清單。看板強調「流動」,Backlog 則強調「順序」——把最該先做的項目排在最上方,團隊由上而下認領工作。

兩個 path 欄位決定了 work item 的歸屬,是 Backlog 與 Sprint 規劃的基礎:

  • Iteration Path (何時)——把 work item 排入哪一個 iteration/sprint,代表預計在哪個時間區間完成。
  • Area Path (誰)——把 work item 歸屬於哪個團隊區域,決定它出現在哪個 Team 的 backlog 與看板。

規劃一個 Sprint 時,典型流程如下:

1
建立 Iteration
於 Project Settings 定義 sprint 的時間區間 (起訖日),作為可排入工作的容器。
2
Backlog 排序
於待辦清單拖放 work item,把優先度最高的項目排到最上方。
3
設定 Capacity
於 Sprint 的 Capacity 頁設定各成員每日可投入工時與休假,推算本 sprint 的總產能。
4
拖 work item 進 Sprint
將 Backlog 上方項目的 Iteration Path 指向本 sprint,認領為本期目標。
5
拆 Task
於 Sprint 的 task board 把 User Story 拆成可執行的 Task,並估計各自工時。
圖 2.3 — 待辦清單頁面 (來源:Microsoft Learn)
i
Backlog 與 Sprint 是同一批工作的兩種視角 Backlog 回答「接下來要做什麼、依什麼順序」,Sprint 回答「這段期間要完成哪些、由誰投入多少」。設定 Capacity 後,當 sprint 內排入的工作超過總產能,系統會以紅色提示過載,協助及早調整範圍。
06 — 查詢

Queries:儲存可重複使用的篩選清單

查詢 (Queries) 讓 work item 的篩選條件得以儲存並重複使用。與其每次手動翻找,不如把「指派給我且尚未關閉的所有 Bug」這類條件存成一個查詢,隨時點開即得最新結果。查詢分為三種類型:

查詢類型用途
Flat list (平面清單)單純列出符合條件的 work item,最常用
Work items and direct links (直接連結)顯示符合條件的項目及其直接關聯的項目
Tree of work items (樹狀)以父子階層呈現,觀察 Epic → Feature → Story 的展開

查詢依儲存位置分為 My Queries (僅自己可見) 與 Shared Queries (團隊共享)。儲存後的查詢還能餵給儀表板 widget,例如以查詢結果產出長條圖或趨勢圖,讓 Dashboard 上的數字自動更新。

查詢編輯器底層以 WIQL (Work Item Query Language) 描述條件,語法近似 SQL。日常使用以圖形化編輯器即可完成,進階情境才需要直接撰寫 WIQL:

SELECT [System.Id], [System.Title], [System.State]
FROM WorkItems
WHERE [System.WorkItemType] = 'Bug'
  AND [System.AssignedTo] = @Me
  AND [System.State] <> 'Closed'
ORDER BY [System.ChangedDate] DESC
i
把常用視角存成 Shared Query 將「本 sprint 進行中的工作」「逾期未完成」「待我審查」等常用條件存為 Shared Queries,團隊成員即可共用同一份檢視,並接上儀表板形成即時報表。查詢也支援 @Me、@CurrentIteration 等動態變數,使同一個查詢對每位成員與每個 sprint 都維持正確結果。
07 — 交付計劃與可追蹤性

跨團隊的行事曆與端對端追蹤

交付計劃 (Delivery Plans) 以行事曆檢視呈現跨多個團隊、跨多個 sprint 的交付排程。它把各團隊的 Feature 或 Epic 攤在時間軸上,協助觀察彼此的相依關係與時程衝突,適合用於跨團隊的版本規劃與里程碑對齊。

Boards 真正的價值,在於它與其他 Azure DevOps 服務串成的端對端可追蹤性。一個 work item 可以一路連結到實作與驗證它的每個產物:

  • 從 work item 直接建立分支,使分支自始即與需求關聯。
  • 開 Pull Request 時關聯 work item,合併後狀態可自動推進。
  • build 與發行連回 work item,得知某個需求由哪次 build 產出、部署到哪個環境。
  • 測試結果連回需求,構成「需求 ↔ branch/commit ↔ PR ↔ build ↔ test」的完整鏈。

Boards 也能連結 GitHub 儲存庫:在 commit message 或 PR 中以特定語法引用 work item 編號,即可把 GitHub 的 commit、PR 與 issue 連回 Azure Boards 的 work item,讓需求管理留在 Boards、程式碼留在 GitHub 的團隊同樣享有可追蹤性。

圖 2.4 — 由查詢與 work item 餵養的儀表板 (來源:Microsoft Learn)
✓
讓每件工作都能往回追溯來源 建立 work item 後養成「從需求建分支、開 PR 關聯需求」的習慣,整條鏈便會自動成形。日後檢視任一筆部署或一個 Bug,都能反查其需求、負責人與測試紀錄。下一課將進入 Repos,看這些分支與 Pull Request 如何在 Azure DevOps 中運作。
