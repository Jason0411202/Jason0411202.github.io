Azure Repos 原始碼托管與協作

在已具備 Git 與 Pull Request 觀念的前提下，本課聚焦 Azure Repos——Azure DevOps 內建的原始碼托管服務。除了標準 Git 的全部能力,Repos 還補上線上瀏覽、跨服務連結與分支原則 (branch policies),讓團隊的程式碼審查與分支保護有制度可循。

01 — Repos 是什麼

原始碼托管:Git 與 TFVC 兩種選擇

版本控制負責記錄程式碼的每一次變更、支援多人並行開發並保留完整歷史 (Git 的運作機制屬於本課的先備知識)。Azure Repos 即是 Azure DevOps 中負責這項工作的服務,提供兩種版本控制系統:

類型模型分支方式適用
Git分散式:每位開發者本機都有完整 repo 副本,可離線提交輕量分支,合併為常態操作新專案預設,絕大多數情境
TFVC集中式:單一版本工作區,歷史與檔案存於伺服器路徑式分支 (以資料夾路徑切分)既有 TFVC 專案、需集中控管的少數情境

Azure Repos 的 Git 是標準 Git——可搭配任何 Git 用戶端或 IDE 使用,例如 VS Code、Visual Studio、命令列、Xcode、Eclipse 與 IntelliJ。既有的所有 Git 指令在此完全適用,差別只在於遠端托管於 dev.azure.com。

圖 3.1 — Azure Repos Files 頁,瀏覽預設分支的檔案與 README (來源:Microsoft Learn)
i
本課聚焦 Git TFVC 屬於早期集中式版本控制的延續,新專案幾乎都選 Git。後續所有說明 (Files、Commits、Branches、Pull Requests、Branch Policies) 皆以 Git repo 為對象。Repos 左側子功能也依此順序排列:Files、Commits、Pushes、Branches、Tags、Pull requests。
02 — Files

把程式碼放上去的三種起點

Files 是 Repos 的入口頁,顯示目前所選分支的資料夾結構、檔案內容與 README。要讓程式碼進到這裡,依專案狀態有三種起點:

一、Clone 既有 repo。Repo 已存在於 Azure Repos,在本機取得一份副本後即可開始作業。複製頁面會提供 HTTPS 或 SSH 網址:

bash 
# 複製 Azure Repos 上的 Git 儲存庫
$ git clone https://dev.azure.com/<org>/<project>/_git/<repo>
$ cd <repo>

二、把既有本機專案推上一個空 repo。本機已有程式碼但尚未托管,先在 Repos 建立空 repo,再加上遠端並推送:

bash 
# 在既有專案目錄初始化並推上遠端
$ git init
$ git add .
$ git commit -m "chore: initial commit"
$ git remote add origin https://dev.azure.com/<org>/<project>/_git/<repo>
$ git push -u origin main

三、直接在 Repos 建立新 repo。於 Repos 頁面右上角的 repo 選單選擇「New repository」,設定名稱並可勾選自動建立 README 與 .gitignore,之後再 clone 下來。

檔案進到 repo 後,Files 頁可線上瀏覽任一資料夾、檢視檔案內容,README 會自動渲染於頁面下方;單一檔案也支援直接於瀏覽器內編輯並提交,適合修正錯字或調整設定這類小幅變更。

i
線上編輯仍是一次 commit 在瀏覽器內修改檔案,Azure Repos 會要求填寫 commit message,並可選擇直接提交到當前分支或另開新分支。本質上與本機 git commit 相同,只是介面不同。涉及多檔變更時,仍建議在本機開發以利測試與審查。
03 — Commits 與 Pushes

兩個檢視歷史的角度

Repos 將「歷史」拆為兩個子頁,觀看角度不同:

子頁呈現內容
Commits所選分支的提交歷史,逐筆列出 commit message、作者、時間與變更檔案;點入可看完整 diff
Pushes每一次 git push 的記錄,顯示誰、在何時、把哪些 commit 推到哪個分支

兩者的差異在於粒度:一次 push 可能包含多個 commit。Commits 頁回答「程式碼如何逐步演進」,Pushes 頁回答「變更何時、由誰送上遠端」。

Azure Repos 的關鍵加值在於 commit message 可直接關聯 work item。在訊息中加入 # 加 work item 編號,提交後即自動建立雙向連結:

bash 
# 在 commit message 內以 #123 關聯 Boards 上的 work item
$ git commit -m "fix(auth): 修正登入逾時 #123"

之後在該 work item 表單可看到此 commit,在 commit 詳情亦可反查對應需求。這正是 Lesson 01 所述「可追蹤性」在 Repos 層的具體展現。

04 — Branches

分支隔離與管理介面

分支讓不同功能、修補與實驗在各自的軌道上並行推進,彼此不互相干擾,完成後再合併回主線。這項 Git 的核心概念屬於本課的先備知識,此處說明 Repos 提供的管理介面。

於 Repos 的 Branches 子頁,可進行下列操作而無須切到命令列:

  • 建立 / 刪除分支——可指定以哪個分支或 commit 為基準建立新分支。
  • 設定預設分支——指定 clone 與 PR 預設指向的分支 (通常為 main)。
  • 比較分支——以「Compare」檢視兩分支間的差異與落後/領先的 commit 數。
  • Favorite (加星)——把常用分支標記為我的最愛,固定於清單頂端方便存取。

命令列端的操作與標準 Git 完全一致,例如建立並推送新分支:

bash 
$ git checkout -b feature/profile-page
$ git push -u origin feature/profile-page
i
分支命名與 work item 連動 Repos 支援從 Boards 的 work item 直接「建立分支」,系統會以該 work item 為名建立分支並自動連結。維持一致的命名 (如 feature/、bugfix/ 前綴) 有助於在 Branches 清單中快速辨識用途。
05 — Pull Requests

Repos 的核心協作流程

Pull Request (PR) 是 Azure Repos 進行 code review 的主要機制。它把「某分支的變更要合併進目標分支」這件事變成一個可審查、可討論、可投票的提案,而非直接覆蓋主線。既有的 PR 觀念在此完全適用,以下說明 Repos 介面的具體操作。

建立 PR 時須指定來源分支與目標分支 (通常為 main),並填寫標題與描述。建立後可進行:

  • 加入 Reviewers——指定必審或選審的審查者;搭配分支原則可設為必要核准。
  • 留 Comment 與投票——審查者於差異處逐行留言,並以 Approve / Approve with suggestions / Wait for author / Reject 表態。
  • 關聯 work item——將此 PR 連到對應需求,合併後可自動更新 work item 狀態。
  • 完成合併 (Complete)——通過審查與原則後合併,可選擇合併策略。

合併策略 (merge type) 決定歷史的呈現方式:

策略行為
Merge commit保留來源分支所有 commit,並新增一個合併 commit
Squash merge把來源分支的多個 commit 壓成單一 commit 併入,主線歷史最乾淨
Rebase將來源 commit 逐一接到目標分支頂端,維持線性歷史而不留合併 commit

一次完整的 PR 協作流程如下:

1
開分支
從 main 建立功能分支,於本機開發並提交。
2
Push 到 Repos
以 git push -u origin <branch> 將分支推上遠端。
3
建立 Pull Request
指定目標分支、填寫描述、加入 reviewers,並關聯對應 work item。
4
Review 與投票
審查者逐行留 comment、提出修改,作者回應並更新;審查者投票表態。
5
完成合併
通過原則後選擇合併策略完成 (Complete),可一併刪除來源分支。
i
Draft PR 與自動完成 尚未準備好審查時可先建為 Draft PR,僅分享進度而不觸發必要審查者通知。若希望原則一過即自動合併,可開啟「Set auto-complete」,系統會在所有分支原則滿足後自動完成,免去等待時手動操作。
06 — Branch Policies

保護重要分支:Repos 相較裸 Git 的關鍵加值

裸 Git 本身不限制誰能 push 到 main;團隊的審查紀律全靠人為約定。Azure Repos 的分支原則 (branch policies) 把這些約定變成系統強制執行的規則,套用在受保護分支 (通常是 main) 上。設定後,該分支只能透過符合原則的 PR 合併,任何直接 push 都會被拒絕。

圖 3.2 — 分支原則作為合併進主線前的關卡

常見可設定的原則如下:

原則作用
最少核准數合併前須有指定數量的 reviewer 投 Approve
要求關聯 work itemPR 必須連結至少一個 work item 方可合併,確保需求可追蹤
要求 comment 已解決所有審查留言須標記為 Resolved,避免意見被忽略
Build validation合併前須通過指定的 pipeline 建置與測試,失敗則無法合併
限制合併類型強制只能用某種合併策略 (如僅允許 squash),統一主線歷史風格

其中 build validation 把 Pipelines 與 Repos 串起來:每個 PR 都會先跑一次建置與測試,確認不會把壞掉的程式碼併入主線。這正是下一堂 Pipelines 與本堂 PR 流程的交會點。

!
不可對受保護分支強行繞過原則或 force push 以 --force 改寫受保護分支的歷史,或用權限繞過分支原則直接合併,會破壞團隊共用的歷史與審查紀律,且難以追究與還原。受保護分支的任何變更都應走正規 PR 流程;確有例外需求時,應透過正式的權限調整與紀錄,而非私下強推。
07 — Tags 與可追蹤性

標記版本點,串起整條鏈

Tag 用來在歷史上標記特定的版本點,例如發行版本 v1.0.0。它是某個 commit 的具名指標,常用於標示「這就是當時上線的版本」。Repos 的 Tags 子頁可瀏覽、建立與刪除標籤:

bash 
# 為當前 commit 建立含註解的標籤並推上遠端
$ git tag -a v1.0.0 -m "release: 第一個正式版本"
$ git push origin v1.0.0

除了標籤,Repos 還提供兩項提升可追蹤性的能力。PR status 讓外部服務以程式化方式 (成功 / 失敗) 簽署 PR,例如外部掃描或合規檢查可把結果掛在 PR 上,作為合併前的額外依據。跨服務連結則讓 PR、work item 與 build 彼此互通:

  • PR ↔ work item——PR 關聯需求,合併後可自動推進需求狀態。
  • PR ↔ build——build validation 把 Pipelines 的結果綁進 PR。
  • commit ↔ work item——commit message 內的 #編號 自動雙向連結。

這些連結串起來,即可從上線的某個 tag 反查:它對應哪次 build、哪個 PR、解決了哪些 work item——完整重現一個版本的來龍去脈。

✓
以 PR + 分支原則作為團隊預設 建議將「所有變更走 PR、main 套用分支原則」設為團隊的標準做法:最少核准數確保有人審過、build validation 確保不併入壞程式碼、關聯 work item 確保每筆變更可追溯需求。這套組合把 code review 的審查紀律,從人為約定升級為系統強制,是 Azure Repos 相較裸 Git 最值得採用的加值。
