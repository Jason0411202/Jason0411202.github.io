Lesson 11 · 大型專案實務篇

依賴管理 與供應鏈安全

現代專案 90% 的程式碼是別人寫的依賴。依賴 = 風險 — 從 left-pad 到 xz utils 後門，這堂教你怎麼防。

01 — 為什麼依賴是風險

你寫的程式碼只佔 10%，剩下 90% 是別人的

一個現代 Node.js 專案 npm install 後 node_modules 動輒幾百 MB、上千個套件。Python、Rust、Go 也類似。每一個依賴都帶著：

  • 未知的 bug — 影響你的 production
  • 未知的安全漏洞 — 攻擊者的入口
  • 未知的維護狀況 — 作者哪天棄坑你不知道
  • 傳遞性依賴 — 你直接裝 10 個，間接裝 800 個
!
新增一個依賴前先問三個問題 ① 這個功能我自己寫要多久？(很短就別裝) ② 這個套件最近一年活躍嗎？(看 GitHub 的 commit / issue 回應) ③ 它的 transitive 依賴是怎樣？(用 npm view <pkg> dependencies 看)
02 — 真實案例

三個讓全世界記住「依賴是風險」的事件

2016 — left-pad 事件

Azer Koçulu 因為 npm 跟另一家公司商標糾紛，把自己所有套件下架，包括 left-pad (一個 17 行的字串補空格函式)。結果半個 JavaScript 生態系炸了 — Babel、React、Node 全部建置失敗。

教訓：不要用「微小到只有十幾行」的依賴；npm 也因此引入「unpublish 限制」(72 小時內、無人下載才能撤回)。

2018 — event-stream 事件

有個受歡迎的 npm 套件 event-stream，原作者沒空維護，把權限給了一個熱心的「貢獻者」。新主人接手後加入了一個依賴 flatmap-stream，裡面藏了竊取 Bitcoin 錢包私鑰的後門，影響數百萬下載。

教訓：套件「換手」是供應鏈攻擊的高風險點。直接看 package.json 的依賴版本號鎖死、留意自動更新後的 PR diff。

2024 — xz utils 後門 (CVE-2024-3094)

近年最嚴重的供應鏈攻擊。 攻擊者花了 2 年偽裝成善意貢獻者，逐步取得 xz utils (Linux 普遍依賴的壓縮函式庫) 的維護權，然後在 5.6.0 / 5.6.1 加入後門，可以從 SSH 遠端執行任意指令。

幸好被微軟工程師 Andres Freund 在 SSH 慢了 0.5 秒時發現。如果發現晚了一兩週、xz 進入 Debian / Ubuntu 穩定版，全世界 Linux 伺服器都可能被入侵。

教訓：開源專案的單一維護者是攻擊者下手點。社區與政府開始重視「critical 開源專案要有多人維護」、推 SBOM、SLSA、artifact signing 等防護。

03 — 鎖檔

Lock 檔的意義

沒有鎖檔，install 出來的依賴版本可能每次都不同 — 你機器跑得好好的，CI 上炸了。鎖檔記錄每個套件的精確版本 + 整顆依賴樹。

生態鎖檔備註
npmpackage-lock.jsonnpm 5+ 預設
pnpmpnpm-lock.yaml記錄完整依賴樹 + 整合性 hash
yarnyarn.lockYarn 1 / Yarn Berry 格式不同
Python (pip + Pipenv)Pipfile.lock但 pip 本身不強制鎖
Python (Poetry)poetry.lock建議用 Poetry / uv 取代純 pip
RustCargo.lockLibrary 不該 commit、binary 應該 commit
Gogo.sum記錄 hash 確保整合性
✓
鎖檔必須 commit 進版本控制 這是 Lesson 01 提過的鐵律。沒有鎖檔等於團隊每個人 / 每個 CI run 都用不同版本的依賴，bug 重現困難。

CI 上要用「frozen」模式

所有套件管理器都有「不要解析、嚴格按 lock 檔裝」的旗標：

bash 
$ npm ci                                # 不是 npm install
$ pnpm install --frozen-lockfile
$ yarn install --immutable
$ poetry install --no-update
$ cargo build --locked
04 — 漏洞掃描

主動掃描已知漏洞

各語言生態都有官方漏洞資料庫 + CLI 工具：

生態指令資料來源
npmnpm auditGitHub Advisory Database
pnpmpnpm audit同上
Pythonpip-auditPython Packaging Authority + OSV
Rustcargo auditRustSec Advisory Database
GogovulncheckGo Vulnerability Database
跨語言OWASP Dependency-CheckNVD (美國國家漏洞資料庫)
output 
$ npm audit

axios  <1.6.0
Severity: high
SSRF in axios - https://github.com/advisories/GHSA-xxxx-xxxx
fix available via `npm audit fix`
node_modules/axios

3 vulnerabilities (1 moderate, 2 high)
i
把 audit 加進 CI npm audit --audit-level=high 在 CI 跑，high / critical 漏洞讓 build 失敗。低風險的 (low / moderate) 通常不擋 build 但留紀錄。
05 — 自動更新

Dependabot vs Renovate

手動追依賴更新太累，業界用機器人自動開 PR 升版：

面向DependabotRenovate
歸屬GitHub 內建 (2019 收購)Mend (前 WhiteSource) 開源 + SaaS
設定簡單 (一個 yml)強大但複雜
合 PR 策略每個依賴一個 PR可配置批次 / 群組更新
排程固定間隔 (daily / weekly)支援 cron / 時區 / quiet hours
支援生態主流語言都有更廣 (Docker base image / GitHub Actions / Terraform)
適合多數專案這個就夠大型 monorepo / 細緻控制

Dependabot 設定範例

yaml 
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    groups:
      eslint:
        patterns: ["eslint*"]
      testing:
        patterns: ["vitest*", "@testing-library/*"]
    open-pull-requests-limit: 5

  # GitHub Actions 也要更新
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
!
不要無腦合併 bot 開的 PR 就算 CI 過了，major version 升級必看 release notes — breaking change 可能 CI 抓不到 (例如新版改了某個 hook 的副作用)。Patch / minor 比較安全但仍要快速掃過 changelog。
06 — CVE / GHSA

怎麼看 CVE 與 GHSA 識別碼

系統格式誰發放
CVE CVE-2024-3094 MITRE / 美國 CISA — 跨平台通用識別碼
GHSA GHSA-xxxx-xxxx-xxxx GitHub Advisory — 開源套件專用
NPM advisory 數字 ID npm 早期格式 (現多遷移到 GHSA)

CVSS 評分

每個 CVE 會給 0-10 的 CVSS (Common Vulnerability Scoring System) 分數：

  • 9.0-10.0 Critical — 立刻處理
  • 7.0-8.9 High — 本週處理
  • 4.0-6.9 Medium — 排進近期 sprint
  • 0.1-3.9 Low — 看狀況
i
分數高不一定影響你 例如某個漏洞需要「使用者已被釣魚 + 開啟特定設定」才能利用，CVSS 可能還是給 7.0，但實務影響極小。看完整描述、評估 attack vector 是否在你的使用情境下可達，再決定優先級。
07 — SECURITY.md

SECURITY.md：負責任的揭露流程

有了專案後，必須讓人知道發現漏洞時該找誰、怎麼通報。否則安全研究員可能直接公開 issue (zero-day)，反而讓使用者陷入危險。

業界做法是放一個 SECURITY.md 在 repo 根目錄：

markdown 
# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 2.x     | :white_check_mark: |
| 1.x     | :white_check_mark: (until 2026-12) |
| < 1.0   | :x:                |

## Reporting a Vulnerability

請不要透過公開 issue 通報安全漏洞。

請改用以下任一方式：

1. GitHub Private Vulnerability Reporting (推薦)
   到 Security 分頁按「Report a vulnerability」
2. Email: security@example.com (PGP key: 0xABCD1234)

我們會在 48 小時內回覆，90 天內發補丁。
依照 [CVD policy](https://...) 處理 — 在補丁發布前不會公開細節。

## Hall of Fame
感謝以下發現過漏洞的研究員：
- @researcher1 (CVE-2024-1234)
- @researcher2 (CVE-2024-5678)
i
用 GitHub 的 Private Vulnerability Reporting GitHub 2022 推出的功能 — repo Settings → Security → Private vulnerability reporting → Enable。研究員可以在 web UI 私下 file 漏洞，maintainer 在 private fork 修補後再公開。比 email 流程順很多。
08 — SBOM / SLSA

SBOM 與 SLSA — 供應鏈安全的下一步

SBOM (Software Bill of Materials)

「軟體物料清單」 — 列出一個軟體用了哪些依賴、什麼版本、什麼授權。美國 2021 行政命令 14028 後成為政府採購軟體的必要文件。

主流 SBOM 格式：

  • SPDX — Linux Foundation 發起，ISO 標準
  • CycloneDX — OWASP 維護，較聚焦安全

產生 SBOM：syft <repo> (Anchore 開源工具) 或各 build tool 的內建支援。

SLSA (Supply-chain Levels for Software Artifacts)

Google 主推的供應鏈安全等級框架，分四級：

  • SLSA 1 — 有自動化建置 + 可追溯來源
  • SLSA 2 — 用 hosted CI (GitHub Actions / GCB) + 簽名
  • SLSA 3 — 不可竄改的 build (含 ephemeral runner)
  • SLSA 4 — 最高，含 two-party review、可重現建置

Sigstore — 開源簽章

Sigstore 是 Linux Foundation 的開源 artifact 簽章方案 (含 Cosign / Fulcio / Rekor)。npm、PyPI、GitHub Actions 都已支援。npm 從 2023 起對熱門套件強制要求 provenance — 證明這個套件是從哪個 CI / 哪個 commit 建置出來的。

private registry 與 internal mirror 深入

大企業會在內部架 mirror (Verdaccio、Artifactory、Nexus)，原因：

  • 下載速度快 (內網)
  • 核准過的套件才能用 (黑名單 / 白名單)
  • 留下審計紀錄
  • 離線環境可用
  • 避免上游下架 (left-pad 後遺症)
版本鎖定策略：caret vs tilde vs exact 細節
"axios": "1.4.5"完全鎖死
"axios": "~1.4.5"允許 1.4.x (只 patch)
"axios": "^1.4.5"允許 1.x.y (minor + patch，預設)
"axios": "*"任何版本，不要用

應用程式：^ 或 ~ + lock file 鎖定。Library：通常用 ^ 給使用者更大彈性。
