Azure Boards Work Item 與看板

Azure DevOps 的 Boards hub 是 work item 集散地——Epic / Feature / User Story / Task 四層階層、Backlog 與 Sprint 視圖、Kanban Board 與 WIQL Query 共構整個專案管理面。本課以 2026 年最新 UI 與 Agile process 為基準。

01 — Boards 全景

Boards 在 Azure DevOps 中的定位

登入 dev.azure.com/<org>/<project> 後，左側導覽列共有 Boards、Repos、Pipelines、Test Plans、Artifacts 五個 hub。Boards 為其中的「工作管理面」——所有工作以 work item 為單位，並依專案層級組織。

子頁面用途
Work items全專案 work item 的列表視圖，搭配 Recently updated、Following、Mentioned 等預設過濾
BoardsKanban 看板，狀態欄對應 work item 的 State 欄位
Backlogs階層列表視圖，可拖曳排序、批次設定 Iteration / Area
Sprints當前 Sprint 的 backlog、Taskboard 與 capacity 規劃
Queries儲存自訂查詢、共享於團隊；底層為 WIQL 語法
Delivery Plans跨多個 team 的時間軸視圖，2026 後納入內建功能
i
Boards 的權限模型隨 Project 而定 每個 Project 內預設四個 security group（Readers、Contributors、Project Administrators、Build Administrators）。Work item 的可見性、可編輯欄位皆依此分配。跨 Project 工作須額外配置 Cross-Project Sharing。
02 — Process Template

四種 Process Template 的差異

建立 Project 時須選擇一個 process template，選定後決定該 Project 可用的 work item 類型、狀態流（state workflow）與預設欄位。Microsoft 提供四個內建模板：

模板核心 work item 類型適用情境
BasicEpic / Issue / Task新建 Project 預設；最精簡，適合非 IT 團隊或入門使用
AgileEpic / Feature / User Story / Task / Bug多數企業導入時的預設選擇；對應 Scrum/Kanban 通用詞彙
ScrumEpic / Feature / Product Backlog Item / Task / Bug / Impediment嚴格遵循 Scrum 詞彙；PBI 取代 User Story
CMMIEpic / Feature / Requirement / Task / Bug / Change Request / Risk受監管產業、需 audit trail 與正式 change management
!
Process template 一旦選定即難以變更 雖然 ADO 提供「Migrate Project Process」功能，但不同 template 的欄位定義與 state 對應差異甚大，遷移後常須手動修正。新建 Project 前務必先確認團隊使用哪一套方法論。

自訂繼承式 Process

內建模板無法直接修改；若需新增欄位、修改 state 或調整表單版型，須先建立繼承（Inherited）Process：

Organization Settings → Process → 選一個內建模板 →「Create inherited process」
  ↓
新增自訂 Field（例:Severity、Customer Impact）
修改 State Workflow（例:New → Triaged → Active → Resolved → Closed）
調整 Layout（拖曳欄位於 work item form）
  ↓
回到 Project Settings → 切換該 Project 套用繼承 Process
03 — Work Item 階層

Epic → Feature → User Story → Task

Agile process 為最常見的選擇，其 work item 階層由大至小如下圖。每一層皆透過 Parent 連結指向上層，且階層關係驅動 Backlog 視圖的捲合（rollup）與進度計算：

圖 9.1 — Agile Process 的四層 work item 階層

各層級的設計指引

層級典型完成時間應描述
Epic多個 sprint，可能跨季商業目標、預期成效（OKR 對齊）
Feature1 至 3 個 sprint對使用者交付的具體功能模組
User Story單一 sprint 內使用者觀點的需求（As a... I want... So that...）
Task數小時至 2 天實作步驟，技術觀點；通常由實作者自行拆分
Bug不限缺陷修復；可掛於 User Story 之下，或獨立掛於 Feature
i
Bug 應掛在 User Story 的同層或 Task 層? ADO 預設將 Bug 視為與 User Story 同層的 backlog item（皆受 Sprint capacity 約束）；亦可於 Project Settings 切為 Task 層（視為 Story 的子項）。前者較貼近現實——bug 修復需估點且占用 sprint 容量；後者較精簡但會低估 bug 工作量。
04 — Backlog 與 Sprint

Backlog 視圖與 Sprint Planning

Backlog 為「依優先序排序的待辦清單」。ADO 的 Backlog 視圖支援三個層級切換（Epic / Feature / Story）——切換層級時，僅顯示該層及其下層 work item：

Boards → Backlogs → 右上角切換層級

層級 = Stories                    層級 = Features
─────────────────                ─────────────────
□ Story 101 [3 pt]                ▾ Feature A
□ Story 102 [5 pt]                    └ Story 101, 102, 103
□ Story 103 [2 pt]                ▾ Feature B
□ Story 104 [8 pt]                    └ Story 104

Iteration Path 與 Area Path

兩個欄位決定 work item「何時做」與「誰來做」：

  • Iteration Path——時間維度。例：MyProject\Sprint 2026.05。一個 work item 僅屬於一個 iteration。
  • Area Path——組織維度。例：MyProject\Frontend\Web。決定該 work item 由哪個 team 認領。

於 Project Settings → Project configuration 即可建立階層式 iteration 與 area path。設定完成後，可於 Team Settings 將特定 path 指派給特定 team——後續該 team 的 Backlog 視圖即自動篩出對應 work item。

Sprint Planning 工作流

1
建立 Iteration
Project Settings → Project configuration → Iterations，新增 Sprint 並設定起訖日。
2
Backlog 排序
於 Backlog 視圖拖曳調整優先序；最頂端的項目即為下個 sprint 候選。
3
設定 Sprint Capacity
Sprints → Capacity，輸入每位成員的可用工時，扣除休假與會議時間。
4
將 work item 拉進 Sprint
拖曳 Story 至左側 Sprint 區塊，系統自動將 Iteration Path 改為該 Sprint。
5
拆 Task 與認領
每個 Story 由實作者拆出 Task（含時間估算），於 Taskboard 認領並追蹤進度。
05 — Kanban Board

Kanban Board 設定與 WIP 限制

Boards → Boards 為 Kanban 視圖。每個 column 對應 work item 的一個 state；卡片由左至右流動即代表狀態進展。可於右上角齒輪設定：

設定用途
Columns新增 / 刪除 / 重新命名 column；可一個 state 對應多 column（例 Active 拆為 Dev / Code Review）
WIP Limits每個 column 設定最大卡片數；超過時 column 標頭轉紅，提醒團隊先消化再接新工作
Swimlanes橫向分區，常用於拆「Expedite」「Tech Debt」「Feature Work」等不同優先級流線
Card fields選擇卡片正面顯示哪些欄位（assignee、tags、effort 等）
Definition of Done每個 column 可寫一份檢查清單，提醒移卡前確認
✓
WIP Limit 並非懲罰機制，而是流量管理工具 當「In Progress」column 持續觸頂時，意味著瓶頸出現——應先合作清掉現有工作（例如多人 review 同一個 PR），而非繼續從 Backlog 拉新項目。流動效率（flow efficiency）優於個人忙碌度。
06 — Queries

Queries：用 WIQL 自訂查詢

固定報表外，可透過 Boards → Queries 建立自訂查詢。底層為 WIQL（Work Item Query Language）——語法近似 SQL 但專為 work item 設計：

wiql 
-- 列出我建立、尚未關閉的 Bug
SELECT [System.Id], [System.Title], [System.State], [System.AssignedTo]
FROM WorkItems
WHERE [System.WorkItemType] = 'Bug'
  AND [System.CreatedBy] = @Me
  AND [System.State] <> 'Closed'
ORDER BY [System.ChangedDate] DESC

三種查詢類型：

  • Flat list——單層列表，最常用
  • Work items and direct links——母項與其直接連結（適合追母 Story 與其子 Task）
  • Tree of work items——多層階層展開（適合 Epic → Feature → Story → Task 全鏈）

查詢可儲存至 My Queries（私人）或 Shared Queries（團隊共享）。共享查詢亦可作為 Dashboard widget 的資料源。

07 — 2026 UI 更新

近期值得關注的 Boards 更新

2026 年 Boards 經歷數項 UI 與功能改版：

  • Markdown 多行欄位的 preview / edit 模式分離——例如 Description、Repro Steps，預設以 preview 模式開啟，避免不慎進入編輯狀態誤動內容
  • Backlog 與 Kanban 額外欄位篩選——可將任何已顯示於卡片或 backlog column 的欄位加入篩選器
  • Boards UI 整體刷新——對齊 Azure DevOps 其他 hub 的間距、鍵盤導航、捲動行為與視覺一致性
  • Code Search 內建——不再需要額外安裝 Marketplace extension，可直接於頂部搜尋列跨 Repo 搜尋程式碼
i
Microsoft 對 Boards 的長期投資方向 自 2024 年起，Boards 的新功能著重於「降低點擊次數」與「跨 hub 一致性」。Pipelines 端的新 task 與 Repos 的進階功能多以 YAML 為唯一介面，但 Boards 仍保有完整 UI 工作流——因為 work item 操作多為非工程角色（PM、QA、Scrum Master）。
