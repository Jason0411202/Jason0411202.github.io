分支策略選擇 / Issue

01 — 三大分支策略

三大主流分支策略——Git Flow / GitHub Flow / GitLab Flow

開源與商業團隊最常見的三套分支模型。差異主要在「分支層數」與「環境抽象」：Git Flow 採用 5 條分支對應傳統長週期發布；GitHub Flow 簡化至僅保留主幹與短命 feature 分支；GitLab Flow 則在 GitHub Flow 基礎上補上環境分支。選擇時應依團隊現況評估，並無絕對最佳方案。

Git Flow（Vincent Driessen, 2010）

最早被廣泛採用的策略，適合發布週期長、需同時維護多版本的產品（例如桌面軟體、企業套裝軟體、需支援舊版本的 SDK）。

圖 3.1 — Git Flow 拓樸 (master / develop 雙主幹)
  • master：永遠對應 production；每次釋出皆打 tag（v1.0、v1.1 等）
  • develop：開發中的下一版本
  • feature/*：自 develop 切出進行功能開發，完成後合併回 develop
  • release/*：釋出時自 develop 拉出，最終合併回 master 與 develop。將待釋出的內容凍結為快照，QA 於此分支驗收、修補 bug、調整版本號；同時 develop 可繼續接收下一版 feature，使釋出準備與下一版開發得以並行。
  • hotfix/*：production 緊急修補。應自當前線上的 release tag 切出（避免從 master HEAD 切出而帶入尚未部署的 commit），僅進行最小幅度的修正與測試（不應同時夾帶 refactor），修補完成後合併回 master 並打 patch tag（v1.0 → v1.0.1），並同步合回 develop，否則下次發布時將出現 regression。

GitHub Flow——多數新創與 Web 團隊採用

將 Git Flow 簡化為兩個核心原則：main 永遠可 deploy；其餘皆為 feature branch。

圖 3.2 — GitHub Flow 拓樸 (一條主幹 + 多個短命 feature)

分支類型

  • main：永遠可 deploy 的主幹，合併後即可上線
  • 短命分支（自 main 切出）：不存在正式分類——無論是新功能、bug 修補或緊急修補，流程一致。社群慣用 feat/、fix/、hotfix/、chore/ 等前綴標示意圖（詳見後續「分支命名」章節），但此屬命名約定，並非 GitHub Flow 本身定義的分支類型。

規則

  1. main 上的任何 commit 皆須可 deploy
  2. 新工作一律自 main 切出分支（短命，建議 1-3 天內合併回主幹）
  3. 分支 push 後即開啟 PR——PR 是討論的單位，而非合併的單位
  4. PR 通過 review 與 CI 後合併至 main
  5. 合併至 main 後立即 deploy

GitLab Flow——加入環境分支

GitHub Flow 的延伸，補上「分支與環境的對應關係」。適合具備 staging / production 多環境、且 deploy 並非即時的場景。

圖 3.3 — GitLab Flow 拓樸 (feature → 環境分支鏈)
  • feature/*：與 GitHub Flow 相同，自 main 切出短命分支進行開發，PR 通過後合併回 main
  • main：「最新穩定版」——通過 CI 與 review 的 commit 皆位於此分支
  • pre-prod（或 staging）：「即將上線」——自 main merge 或 cherry-pick 過來，部署至 staging 環境進行驗收
  • prod（或 production）：「目前線上版本」——驗收通過後自 pre-prod 推送，對應正式環境

注意：不存在專屬的 release / hotfix 分支類型。版本發布的角色由「環境分支」（pre-prod / prod）承擔，取代 Git Flow 的 release/*；緊急修補則透過「自 main 切出短命 fix 分支 → 合回 main → cherry-pick 至 prod」的流程處理，不會另行建立 hotfix/* 分支。

02 — 分支命名

分支命名原則——feature/ 後續的命名規範

前述的 feature/login、hotfix/v1.0.1、release/1.5.0，斜線前的「類型」由策略決定，斜線後的「名稱」則由團隊自行約定。

建議格式

<type>/<issue-id>-<short-description>

  • type——標示該分支「所屬的工作類型」，使人能從分支名稱判讀意圖、優先級與 review 必要性。常用詞彙列於下表，通常與 commit message 共用同一組詞彙（Lesson 01「Commit Message 規範」已介紹 Conventional Commits 的 commit 端細節）。
  • issue-id——對應的 issue 或 ticket 編號，例如 #34、JIRA-456。將分支與工單綁定，便於搜尋、追蹤與自動關閉 PR。
  • short-description——採用 kebab-case（小寫加連字號），以 3 至 5 個英文單字描述工作內容，避免冗長。

常見的 type 一覽

type 用途 範例
feature（或 feat） 新功能、新增能力。對使用者可見的「新增某項能力」。 feature/#34-add-cart
fix 一般性 bug 修補——非緊急，依正常 release 排程上線。 fix/#56-login-redirect
hotfix 線上發生重大事故時的緊急修補，須插隊以最短路徑直送 production。詳細流程見下一節「Hotfix」。 hotfix/#101-csrf-bypass
refactor 重構——行為不變，僅調整程式結構（重新命名、抽出函式、替換實作），不新增功能也不修補 bug。 refactor/#78-extract-auth-service
chore 維運性改動——升版套件、調整 CI 設定、修改 lint 規則、整理 .gitignore 等與產品邏輯無關的工作。 chore/#42-upgrade-node-20
docs 僅修改文件——README、註解、API 文件、教學。未涉及任何會影響執行結果的程式碼。 docs/#15-update-readme-install
test 僅新增或修改測試，未調整實作。常見於補強回歸測試、新增 e2e 案例。 test/#88-add-checkout-e2e
perf 效能優化——行為與 refactor 同樣不變，目標為提升執行速度或降低資源消耗。 perf/#62-optimize-search-index
release Git Flow 專屬——自 develop 拉出以凍結內容，進行 QA 與版本號收尾的釋出分支，命名以版本號為主（例：release/1.5.0），不套用 issue-id 格式。GitHub Flow / GitLab Flow 不使用此類分支。 release/1.5.0
i
為何不包含 style、build、ci、revert？ 這幾項於 commit message 有獨立 type（見 Lesson 01「常用的 type」），但鮮少獨立開長期分支：style（純格式化）、build（調整建置設定）、ci（調整 workflow）在分支層通常併入 chore/*；revert（還原單一 commit）則通常直接於 main 上 revert，不另闢分支。
03 — Issue

Issue 兩大類與撰寫結構

無論使用 GitHub Issues、GitLab Issues、Jira 或 Linear，issue 通常可劃分為兩大類，各自具備不同目的與撰寫方式：

類型用途
Bug Report回報軟體的錯誤行為
Feature Request提議新功能或改善建議

Bug Report 區塊說明

區塊用途
Describe the bug以一兩句話簡述問題現象，使 reviewer 一眼掌握主題
Steps to Reproduce描述 bug 的重現步驟，條列式撰寫，maintainer 依序即可復現
Expected behavior依規格或常理應出現的行為
Actual behavior實際觀察到的行為，與 expected 形成對照
EnvironmentOS、瀏覽器、應用版本、Node / Python 等執行環境資訊
Screenshots / Video附上錄影、截圖或 log，協助 reviewer 直接看見現象
Additional context補充資訊：相關 issue、推測原因、首次發現時間等

Bug Report 範例

# [Bug] Logout doesn't clear React Query cache (v1.4.2)

## Describe the bug
登出後 token 未完全清除，下次登入其他帳號時會短暫看到上一位使用者的資料。

## Steps to Reproduce
1. 以 user A 登入
2. 點選右上角的「登出」
3. 以 user B 登入
4. 首頁顯示 user A 的 profile

## Expected behavior
登出後應完全清除所有 user 相關資料，登入新帳號時應顯示新使用者的資料。

## Actual behavior
頁面短暫閃現 user A 的 avatar 與 name，刷新後才恢復正常。

## Environment
- OS: macOS 14.5
- Browser: Chrome 124.0.6367
- App version: 1.4.2
- Node: 18.19.0

## Screenshots / Video
（附上錄影或截圖）

## Additional context
推測可能是 React Query cache 未清除。首次發現於 2026-04-25 部署後。
!
缺少重現步驟即無從處理 無法重現的 bug 即無法 debug。「有時候會壞」屬於品質最低的 bug report。提供明確的重現步驟、影片、log 與版本資訊，才能進入 triage 流程。

Feature Request 區塊說明

新功能請求應描述痛點而非解法。直接寫「請新增 X 功能」常會被拒絕，因 maintainer 無從判斷實際的使用情境。

區塊用途
Is your feature request related to a problem?描述當前痛點與使用情境，說明為何需要此功能
Describe the solution you'd like提議的解決方案，可附上 API 設計或介面草稿
Describe alternatives you've considered已嘗試或評估過的替代做法，以及為何不採用
Additional context補充資訊：類似專案的實作參考、相關討論、是否願意送出 PR

Feature Request 範例

# [Feature] Allow custom date format in <DatePicker>

## Is your feature request related to a problem?
應用需顯示中文日期格式（民國年），目前 DatePicker 僅支援
ISO 格式，必須於 component 外額外進行轉換。

## Describe the solution you'd like
新增 `format` prop，使用者可傳入 day.js 或 date-fns 的格式字串：
```tsx
<DatePicker format="民國 YYY 年 M 月 D 日" />
```

## Describe alternatives you've considered
- 以 wrapper component 自行轉換（目前做法，但 onChange 處理較為繁瑣）
- 整合 i18n library（過於厚重）

## Additional context
react-aria/Calendar 具備類似功能，可作為參考。
i
「願意親自實作」可顯著提升被接受的機率 在 issue 末尾附上一句「願意送出 PR」，可大幅降低 maintainer 評估時的心理成本——對於企業內部 backlog 亦同。
