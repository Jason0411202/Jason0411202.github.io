分支策略選擇

四種主流分支策略 — Git Flow、GitHub Flow、GitLab Flow、Trunk-Based。沒有最好，只有最適合你團隊的。

01 — 為什麼要策略

沒策略的後果

沒有共識的分支策略，團隊會出現這些症狀：

  • 每個人開分支命名都不一樣 — jason-fix、tmp、new、final-final-v2
  • 不知道哪個分支可以 deploy — main 上有 RC、staging 上有舊版
  • 合併衝突永遠存在 — 大家分支活很久，最後 merge 像災難
  • release 時找不到當時的 code — 沒打 tag、沒有 release branch

分支策略就是團隊共同遵守的「分支該怎麼開、什麼時候合進去、什麼時候 deploy」的契約。

02 — Git Flow

Git Flow (Vincent Driessen, 2010)

最早被廣泛採用的策略，適合發布週期長、需要同時維護多版本的產品 (例如桌面軟體、企業套裝軟體、需要支援舊版本的 SDK)。

圖 3.1 — Git Flow 拓樸 (master / develop 雙主幹)
  • master：永遠是 production；每次釋出打 tag (v1.0、v1.1...)
  • develop：開發中的下一版本
  • feature/*：從 develop 開出去做功能，做完合回 develop
  • release/*：要釋出時從 develop 拉出來，最後合進 master + develop
  • hotfix/*：production 緊急修補，從 master 拉、合回 master + develop
!
Git Flow 在 SaaS / Web 專案是過度設計 原作者 Vincent Driessen 在 2020 年自己跳出來說 —— 對「持續部署的 Web 應用」來說 Git Flow 太重，建議改用 GitHub Flow 或類似的簡化策略。原文。
03 — GitHub Flow

GitHub Flow — 大多數新創與 Web 團隊用這個

把 Git Flow 砍到只剩兩件事：main 永遠可 deploy、其他都是 feature branch。

圖 3.2 — GitHub Flow 拓樸 (一條主幹 + 多個短命 feature)

規則

  1. main 上的任何 commit 都必須能 deploy
  2. 要做事就從 main 開分支 (短命，最好 1-3 天內合回去)
  3. 分支 push 上去就開 PR — PR 是討論的單位，不是合併的單位
  4. PR 通過 review + CI 綠燈後合進 main
  5. 合進 main 後立刻 deploy
✓
適合大部分場景 80% 的 SaaS / Web 服務用 GitHub Flow 就夠了。它的優點是：簡單、好教新人、CI/CD 容易設定。
04 — GitLab Flow

GitLab Flow — 加上環境分支

GitHub Flow 的延伸，補上「哪個分支對應哪個環境」的概念。適合有 staging / production 多環境，且 deploy 不是即時的場景。

圖 3.3 — GitLab Flow 環境分支 (預備 → 正式)

main 是「最新穩定」、pre-prod 是「即將上線的」、prod 是「正在線上的」。每個環境對應一個分支，要升級環境就把 commit 從前一條 cherry-pick 或 merge 過去。

05 — Trunk-Based

Trunk-Based Development — Google / Meta 大規模採用

把分支極簡化：所有人都直接 commit 到 main (trunk)，feature branch 活不超過 24-48 小時。沒做完的功能用 feature flag 包起來放進 main，等準備好再開關開啟。

i
為什麼大公司這樣做？ 團隊上千人時，分支活越久衝突越可怕。Trunk-based 強迫所有人不停 sync 到最新，配合 feature flag 達成「程式碼 ship 跟功能 release 解耦」 — 程式碼天天上 main，但 feature 什麼時候開給使用者由 flag 控制。

Trunk-Based 的兩個核心配套

  • Feature flag — 半成品功能用 if (flagOn('newCheckout')) 包起來，在 main 上但對使用者隱形
  • 強大的自動化測試 + CI — main 隨時可 deploy 的前提是 CI 真的能擋下壞東西
!
沒有自動化測試底子的團隊不要硬上 Trunk-based 把品質責任全壓在 CI 上。CI 不可靠的話 = 線上每天爆炸。先把 Lesson 09 的 CI / 測試打好基礎再考慮這個策略。
06 — 怎麼選

怎麼選最適合自己團隊的策略？

條件 建議策略
需要長期維護多個歷史版本 (v1.x、v2.x) Git Flow
SaaS / Web 服務、團隊 5-50 人、CI 健全 GitHub Flow
有 staging → prod 階段、deploy 不是即時的 GitLab Flow
百人以上開發團隊、極致敏捷、有 feature flag 基建 Trunk-Based
小型開源 library、貢獻者來來去去 GitHub Flow (用 PR 過濾品質)
剛起步、還在找方向 GitHub Flow (簡單到不會錯)
i
策略可以演進 團隊 5 人時用 GitHub Flow，長到 200 人發現衝突太多時自然會走向 Trunk-Based + feature flag。不要過度設計起手就上 Git Flow，等真的需要再升級。
07 — Hotfix

Hotfix branch 何時該開

無論用哪種策略，線上爆炸時的「緊急修補分支」流程都類似：

1
從正在線上的 commit / tag 開分支
不是從 main 最新開 — 因為 main 上可能已經有還沒上線的東西。
2
命名 hotfix/<issue>
明確標記這是緊急修補，CI 會跑、reviewer 會優先看。
3
最小修改 + 測試
只改要修的，不要順便 refactor。線上回穩才是優先。
4
合進 production 並打 patch tag
v1.4.2 → v1.4.3。SemVer 規則之後 Lesson 06 會講。
5
把修補也合回 main / develop
否則下次 deploy 時這個 bug 會再回來 (regression)。
