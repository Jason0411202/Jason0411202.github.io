01 — 三大分支策略

三大主流分支策略 — Git Flow / GitHub Flow / GitLab Flow

開源與商業團隊最常見的三套分支模型。差異主要在「分支層數」與「環境抽象」：Git Flow 用 5 條分支對應傳統長週期發布；GitHub Flow 砍到只剩主幹 + 短命 feature；GitLab Flow 在 GitHub Flow 上再補回環境分支。沒有最好，只有最適合團隊的。

Git Flow (Vincent Driessen, 2010)

最早被廣泛採用的策略，適合發布週期長、需要同時維護多版本的產品 (例如桌面軟體、企業套裝軟體、需要支援舊版本的 SDK)。

圖 3.1 — Git Flow 拓樸 (master / develop 雙主幹)
  • master：永遠是 production；每次釋出打 tag (v1.0、v1.1...)
  • develop：開發中的下一版本
  • feature/*：從 develop 開出去做功能，做完合回 develop
  • release/*：要釋出時從 develop 拉出來，最後合進 master + develop。把要釋出的內容凍結成快照，QA 在這上面驗收、修 bug、更版本號，同時 develop 可以繼續收下一版 feature，釋出準備和下一版開發能平行進行。
  • hotfix/*：production 緊急修補。從正在線上的 release tag 開 (不要從 master HEAD，免得帶到還沒部署的 commit)，只做最小修改 + 測試 (別順便 refactor)，修完合回 master 並打 patch tag (v1.0 → v1.0.1)，再合回 develop，否則下次發布這個 bug 會 regression。

GitHub Flow — 大多數新創與 Web 團隊用這個

把 Git Flow 砍到只剩兩件事：main 永遠可 deploy、其他都是 feature branch。

圖 3.2 — GitHub Flow 拓樸 (一條主幹 + 多個短命 feature)

分支類型

  • main：永遠可 deploy 的主幹，合進來就能上線
  • 短命分支 (從 main 開出去)：沒有正式分類 — 不論是新功能、修 bug、緊急修補，流程完全一樣。社群慣用 feat/、fix/、hotfix/、chore/ 這類前綴標記意圖 (見後面「分支命名」章節)，但這些只是命名約定，不是 GitHub Flow 本身定義的分支類型。

規則

  1. main 上的任何 commit 都必須能 deploy
  2. 要做事就從 main 開分支 (短命，最好 1-3 天內合回去)
  3. 分支 push 上去就開 PR — PR 是討論的單位，不是合併的單位
  4. PR 通過 review + CI 綠燈後合進 main
  5. 合進 main 後立刻 deploy

GitLab Flow — 加上環境分支

GitHub Flow 的延伸，補上「哪個分支對應哪個環境」的概念。適合有 staging / production 多環境，且 deploy 不是即時的場景。

圖 3.3 — GitLab Flow 拓樸 (feature → 環境分支鏈)
  • feature/*：跟 GitHub Flow 一樣，從 main 開短命分支做事，PR 過了就合回 main
  • main：「最新穩定版」— 通過 CI 與 review 的 commit 都在這
  • pre-prod (或 staging)：「即將上線的」— 從 main merge / cherry-pick 過來，部到 staging 環境驗收
  • prod (或 production)：「正在線上的」— 驗收通過才從 pre-prod 推上來，對應正式環境

注意：沒有專屬的 release / hotfix 分支類型。版本發布的角色由「環境分支」(pre-prod / prod) 承擔，取代 Git Flow 的 release/*；緊急修補則是「從 main 開短命 fix 分支 → 合回 main → cherry-pick 到 prod」的流程，不會額外開一條 hotfix/*。

i
關鍵差異：GitLab Flow ≠ 砍掉 feature branch 它跟 GitHub Flow 上半段 (feature → main) 完全一樣，只是多疊了「環境分支」這層，讓 deploy 不是 main 一合就立刻上線，而是要走完 main → pre-prod → prod 的審核路徑。適合金融、醫療、政府等需要驗收窗口的場景。
02 — 分支命名

分支命名原則 — feature/ 後面該怎麼寫

前面看到的 feature/login、hotfix/v1.0.1、release/1.5.0，斜線前的「類型」由策略決定，斜線後的「名字」由團隊約定。沒有約定就會出現 jason-fix、tmp、final-final-v2 這種誰都看不懂的災難。

建議格式

<type>/<issue-id>-<short-description>

  • type — 標記這個分支「在做哪一類事」，讓人從分支名一眼判斷意圖、優先級、要不要 review。常用詞如下表，跟 commit message 通常共用同一組詞彙（Lesson 06 Conventional Commits 會再講 commit 那邊的細節）。
  • issue-id — 對應的 issue / ticket 編號，例如 123、JIRA-456。把分支跟工單綁起來，搜尋、追蹤、自動關閉 PR 都方便。
  • short-description — kebab-case（小寫 + 連字號），3 - 5 個字描述要做什麼，不要寫小說。

常見的 type 一覽

type 用途 範例
feature（或 feat） 新功能、新增能力。對使用者可見的「多了某件事」。 feature/123-add-cart
fix 修一般的 bug — 不緊急、跟著正常 release 排程上線。 fix/456-login-redirect
hotfix 線上爆炸的緊急修補，要插隊、最短路徑直送 production。流程見下一節「Hotfix」。 hotfix/v1.4.3-csrf-bypass
refactor 重構 — 行為不變，只動程式結構（改命名、抽函式、換實作）。沒有新功能、也沒有修 bug。 refactor/789-extract-auth-service
chore 雜務 — 升版套件、改 CI 設定、調 lint 規則、整理 .gitignore 這類「跟產品邏輯無關」的維運性改動。 chore/upgrade-node-20
docs 只動文件 — README、註解、API 文件、教學。沒碰到任何會影響執行結果的程式碼。 docs/update-readme-install
test 只加測試或改測試，沒改實作。常見於補強回歸測試、加 e2e 案例。 test/add-checkout-e2e
perf 效能優化 — 行為跟 refactor 一樣不變，但目的是讓它跑更快、用更少資源。 perf/optimize-search-index
release 專屬 Git Flow — 從 develop 拉出來凍結內容、做 QA 與版本號收尾的釋出分支。GitHub Flow / GitLab Flow 不會用到。 release/1.5.0

不用全部都用。多數團隊只會挑其中 4 - 6 個來用（常見組合：feature / fix / hotfix / chore / docs），挑完寫進 CONTRIBUTING.md 跟 CI lint 裡，新人才不會自己發明 improvement/、update/、misc/ 這種模糊字眼。

好範例 vs 壞範例

✓ 好範例 ✗ 壞範例 為什麼壞
feature/123-add-cart jason-cart 用人名 — 換人接手就斷線
fix/456-login-redirect tmp / new / test 無描述 — 看了等於沒看
hotfix/v1.4.3-csrf-bypass final-final-v2 用版本後綴對抗自己 — 語意零
chore/upgrade-node-20 Feature/AddCart 大寫 / camelCase — case-insensitive 系統會踩雷
refactor/789-extract-auth-service 修登入bug 中文 / 空格 — Git 雖然能用但 CI、URL 會壞

Git 本身的硬性限制

就算團隊不在意命名美感，下列字元 Git 直接拒絕，要先知道：

  • 不能含 ~、^、:、?、*、[、\、空白
  • 不能有連續的 ..，也不能以 / 或 .lock 結尾
  • 不能以 -、. 開頭
i
把規則寫進 CI，新人就會自動學會 很多團隊在 CI 加一條 lint：分支名不符合 ^(feature|fix|hotfix|chore|refactor|docs)/.+$ 就擋下 push 或 PR。比起在 review 時口頭糾正，自動化把規則內化得更快。
