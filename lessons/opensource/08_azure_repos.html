Azure Repos 分支治理

Azure Repos 是 Azure DevOps 的版本控制 hub——以 Git 為主、TFVC 為輔。本課著重 Branch Policies、Pull Request 工作流、Required / Automatic Reviewers 與 vote reset 行為——在 Azure 環境具體落地受保護分支治理。

01 — Repos 全景

Azure Repos 提供哪些功能

Azure Repos 為一站式版本控制 hub，支援兩種 VCS：

類型說明建議
Git業界標準，每個 Project 可建立多個 Git repo新建 Project 一律選 Git
TFVC (Team Foundation Version Control)中央式版本控制，舊有 TFS 沿用，每個 Project 僅能有一個 TFVC repo僅在維護既有 TFVC 專案時保留；新專案不建議

子頁面包括：

  • Files——瀏覽 repo 檔案、線上編輯、查看 history
  • Commits——commit 歷史、graph、search
  • Pushes——push 紀錄與時間軸
  • Branches——分支列表，可設定 compare、policies、permissions
  • Pull requests——PR 列表，可篩選 Active / Completed / Abandoned
  • Tags——Git tag 管理
02 — Clone 與認證

三種主流認證方式

Clone Azure Repo 時須通過認證。三種主流選擇：

方式適用備註
Git Credential Manager (GCM)個人開發機首次 clone 時瀏覽器彈出 Microsoft Entra ID 登入；token 自動快取於 OS keychain
Personal Access Token (PAT)CI 機器、自動化腳本於 User Settings → Personal access tokens 產生；可細分權限與有效期限
SSH Key跨平台、Mac/Linux 開發者偏好於 User Settings → SSH public keys 貼入公鑰；clone URL 改為 git@ssh.dev.azure.com:v3/<org>/<project>/<repo>
bash 
# HTTPS clone(GCM 或 PAT)
$ git clone https://dev.azure.com/<org>/<project>/_git/<repo>

# SSH clone
$ git clone git@ssh.dev.azure.com:v3/<org>/<project>/<repo>
!
PAT 須最小權限原則 PAT 若給予 Full access，泄漏後可操作整個 organization。建議建立 scoped PAT，僅勾選必要的權限（例如 CI 只需 Code Read、Build Read & Execute），並設定 30–90 天有效期限。
03 — Branch Policies

Branch Policies：受保護分支的核心

於 Repos → Branches → 對主分支按右側「…」→ Branch policies，可設定以下保護規則。一旦啟用，主分支即無法被直接 push，所有變更須透過 PR：

Policy效果
Require a minimum number of reviewers須至少 N 人 approve 才能 complete；可設「Allow requestors to approve their own changes」與「Prohibit the most recent pusher from approving their own changes」
Check for linked work itemsPR 必須關聯至少一個 work item，建立 work item → code 的追溯鏈
Check for comment resolution所有 PR comment 須標記為 Resolved 才能 complete
Limit merge types可限定 Squash / Rebase / Rebase and fast-forward / Merge no-fast-forward 中的哪幾種
Build Validation指定 CI build pipeline，未通過時無法 complete
Status Checks掛 Azure DevOps 以外的服務（例如 SonarCloud、Snyk）的 status；通過後才能 complete
Automatically included reviewers依檔案 path 自動加入必要 / 建議 reviewer（替代 GitHub 的 CODEOWNERS）
!
切勿給 admin「Bypass policies」權限 開啟此例外將使所有 policy 形同虛設。緊急熱修流程應另行設計（例：hotfix branch 走簡化 review），而非依賴 admin bypass——一旦習慣 bypass，policy 將迅速退化為「禮貌建議」。
04 — Required Reviewers

Required Reviewers 與 Vote Reset 行為

設定「最少 N 位 reviewer」時，須一併決定新 commit 進入後既有 approve 如何處理——這是平衡「review 嚴謹度」與「PR 流動性」的關鍵設定：

Vote reset 行為新 push 後適用
Reset all votes所有 approve 失效，須重新 review金融、醫療、公部門等高合規環境
Reset votes of approvers who approved older iterations僅清除原已 approve 者的票；尚未投票者不受影響多數團隊的合理預設
Do not reset votes一旦 approve，後續無論再 push 多少 commit 仍有效僅適合小型內部 prototype 專案，幾乎不建議用於 main

Vote 的五種狀態

  • Approve（+10）——無條件同意
  • Approve with suggestions（+5）——同意，但希望作者考慮建議
  • Wait for author（-5）——已 review，等待作者回應
  • Reject（-10）——明確反對；除非撤銷否則無法 complete
  • Reset feedback——清除自己的 vote，回到未投票狀態
i
「Approve with suggestions」是被低估的選項 作者已可推進但保留改進空間時，使用此選項可避免「Wait for author 來回三輪卻只是一個變數命名建議」的浪費。建議與 author 約定：approve with suggestions = 我相信你會處理，不必再回過頭找我。
05 — Automatic Reviewers

Automatic Reviewers:Azure 版的 CODEOWNERS

Azure Repos 沒有 CODEOWNERS 檔案——同樣需求由 Branch Policies 中的 Automatically included reviewers 達成。設定時指定一組 path filter 與一組 reviewer（個人或 group），當 PR 變更命中該 path 時系統自動加入指定 reviewer：

範例設定:

Path filter                       Reviewer              Required?
─────────────────────────────────────────────────────────────────
/src/payment/**                   payment-team           Required
/src/auth/**                      security-team          Required
/.azure-pipelines.yml             platform-team          Required
/docs/**                          tech-writers          Optional
/db/migrations/**                 dba-group              Required

Required 與 Optional 差異：

  • Required——加入後該 reviewer 必須 approve，否則 PR 無法 complete
  • Optional——加入後僅收到通知，不阻擋 complete
✓
用 Group 而非個人 Required reviewer 應指向 group（例如 payment-team），而非具名個人。否則該成員休假時所有相關 PR 都會卡住。Group 中任一成員 approve 即可滿足 policy。
06 — PR 工作流

Azure Repos 的 PR 完整工作流

1
從 work item 建立分支
於 work item 右側「Development」區塊點選「Create a branch」——系統自動以 work item ID 命名（例 feature/1234-login-page），且該分支與該 work item 預先關聯。
2
提交與 push
於本機開發、commit、push 至新分支。若 commit message 含 #1234 字串，亦會自動關聯該 work item。
3
建立 PR
Repos → Pull requests → New，選擇 source 與 target 分支。填寫 title、description、reviewers；模板可置於 .azuredevops/pull_request_template.md，將自動帶入 description。
4
CI Build 與 Reviewer 並行
建立 PR 即觸發 Build Validation pipeline；同時 reviewer 進行 code review。雙方完成才能 complete。
5
Set auto-complete
不想等的話，可點「Set auto-complete」——所有 policy 通過後即自動 complete，無須回頭點按鈕。
6
Complete:選 merge 策略
Complete 對話框可選 Squash / Rebase / Merge 等策略；可勾「Delete source branch after merging」與「Complete linked work items」（自動將關聯 work item 設為 Closed）。

Draft PR

未就緒的 PR 可建立為「Draft」——同樣可獲得 reviewer 早期回饋與 CI build 結果，但Required reviewer 不會被自動指派、policies 不會強制驗證，亦不會發送通知打擾 reviewer。準備好後點「Publish」即轉為正式 PR。

07 — Code Search

Code Search:跨 Repo 搜尋程式碼

頂部搜尋列輸入關鍵字後即可跨整個 Project 的所有 repo 搜尋。2026 年起 Code Search 已內建——新建 organization 無須安裝 Marketplace extension 即可使用。

常用搜尋過濾器

過濾器用法範例
ext:限定副檔名useAuth ext:tsx
file:限定檔名secret file:appsettings.json
proj:限定 ProjectUserDto proj:Backend
repo:限定 repoTODO repo:web-frontend
arg: / def: / ref:符號層級搜尋（限部分語言）def:calculatePrice
i
Code Search 與 Wiki Search、Work Item Search 共用入口 頂部搜尋列右側的下拉選單可切換搜尋目標（Code / Work Items / Wiki / Packages）。常用組合：先以「Work Items」搜舊 ticket 確認類似問題是否處理過，再以「Code」定位實作位置。
