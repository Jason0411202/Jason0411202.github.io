01 — 完整旅程

從零開始貢獻一個開源專案的全流程

圖 4.1 — Fork → Clone → push → PR → maintainer merge
02 — Fork & Clone

第一步：Fork 到自己帳號 + Clone 到本地

Fork 是 GitHub 的「複製整個 repo 到自己帳號下」按鈕。為什麼需要 fork？因為你對別人的 repo 沒有 push 權限，要先複製一份才能改。

bash 
# 在 GitHub 網頁按右上角的 Fork 按鈕
# 然後 clone 你 fork 過去那份
$ git clone git@github.com:<你的帳號>/repo.git
$ cd repo
i
用 SSH 還是 HTTPS？ 推薦 SSH (git@github.com:...) — 一次設好 key 後就不用每次輸密碼。HTTPS 雖然方便起步，長期會被 push 時要求 token 煩死。
03 — upstream

關鍵步驟：設定 upstream remote

Clone 完之後，你的本地預設只有一個 remote 叫 origin，指向你 fork 過的版本。但原始專案可能還在持續更新，你需要一個指向「上游 (原始) 專案」的 remote 來追蹤更新 — 慣例上叫 upstream。

bash 
# 加上 upstream remote 指向原始專案
$ git remote add upstream git@github.com:<原始作者>/repo.git

# 確認設好了
$ git remote -v

origin    git@github.com:你的帳號/repo.git (fetch)
origin    git@github.com:你的帳號/repo.git (push)
upstream  git@github.com:原始作者/repo.git (fetch)
upstream  git@github.com:原始作者/repo.git (push)
i
命名慣例 origin = 你 fork 的；upstream = 原始專案。這是業界通用慣例，不要自己亂改名字以免之後在文件 / Stack Overflow 看到指令時搞混。
04 — 同步上游

讓自己的 fork 跟上原始專案

原始專案可能每天有人 commit。如果你的 fork 跟它落差太多再開分支，等等 PR 會充滿衝突。每次開新工作前先同步：

bash 
# 1. 切回 main
$ git switch main

# 2. 從 upstream 拉最新
$ git fetch upstream

# 3. 把本地 main 跟上 upstream/main
$ git rebase upstream/main
# 或 git merge upstream/main，兩者都可

# 4. 推回自己的 fork (origin)，讓 GitHub 上的 fork 也是最新
$ git push origin main

# 5. 從最新的 main 開新分支做事
$ git switch -c feat/add-button
GitHub 網頁版的 Sync fork 按鈕 小撇步

GitHub 在 fork 頁面有「Sync fork」按鈕，可以一鍵把 GitHub 上的 fork 跟 upstream main 同步。但要記得做完按鈕後要回到本地 git pull 一次，否則本地還是舊的。

這個按鈕適合「完全沒在自己 fork 上加東西」的情況。如果你 fork 上已經有改動，還是用上面的指令流程比較安全。

05 — 開 PR

從 push 到開 PR

1
寫程式 + commit
commit message 用 Conventional Commits 格式 (Lesson 06 會講)。
2
push 到自己的 fork
git push -u origin feat/add-button
3
GitHub 顯示橫幅，按下 Compare & pull request
推完之後重整 GitHub 頁面，會看到「你的分支領先 main N commits, Compare & pull request」橫幅。
4
填好 PR 描述
下一節會詳細講格式。專案有 PR template 的話會自動帶出來。
5
如果還沒準備好被 review，開成 Draft PR
下拉選 Create draft pull request — 表示「請先看 CI 跑得如何，不要叫人 review」。
i
什麼時候用 Draft PR？ ① 想先看 CI 結果但程式還沒寫完；② 想公開展示進度讓相關人員預覽；③ 跨時區協作，先放上去早起的同事可以看。準備好了按 Ready for review 即可。
06 — PR 描述

業界通用的 PR 描述格式

大型開源專案 (Kubernetes、React、TypeScript) 的 PR 描述通常包含這幾個區塊：

PR 描述標準範本

## 背景 / 動機
為什麼要做這個改動？解決什麼問題？

Closes #234

## 改動內容
- 新增 `Button` component 支援 loading 狀態
- 修改 `Form.tsx` 整合新的 Button props
- 補上對應的 unit test

## 怎麼測試
1. `pnpm test src/components/Button.test.tsx`
2. `pnpm dev` 然後到 /demo/button 手動點擊測試

## 截圖 / 影片
| 改動前 | 改動後 |
|--|--|
| (圖) | (圖) |

## 是否 Breaking Change
否。Button 新增的 prop 都有 default 值。

## Checklist
- [x] 補了測試
- [x] 跑過 lint / type check
- [ ] 更新 docs (這個 PR 不需要，下個會補)
✗糟糕的 PR 描述
fix bug

沒有任何上下文，reviewer 完全不知道在 fix 什麼 bug、怎麼測、影響範圍。

✓好的 PR 描述
fix(auth): clear stale token on logout (Closes #234)

含背景 / 改動 / 測試方式 / 截圖 / 是否 breaking。Reviewer 可以快速進入狀況。

i
沒寫描述 = 拒絕被 review 很多 maintainer 看到空白 PR 描述會直接退回「請補描述」。對他們來說「不知道改什麼為什麼改」沒辦法 review。多花 5 分鐘寫描述能省掉好幾天的來回。
07 — 連結 issue

用關鍵字自動關聯 / 關閉 issue

在 PR 描述或 commit message 中寫這些關鍵字 + issue 編號，GitHub 會自動建立連結；當 PR 合併進主分支時，issue 會被自動關閉：

關鍵字會自動關閉 issue?用在哪
Closes #234✓ 會這個 PR 完整解決了 #234
Fixes #234✓ 會修 bug 時用
Resolves #234✓ 會解 feature request 時用
Refs #234✗ 不會「相關但只是部分解」
See #234✗ 不會單純引用，不負責關閉
i
跨 repo 也能用 Closes owner/repo#234 可以從另一個 repo 的 PR 關閉指定 repo 的 issue (需要寫權限)。
08 — 處理衝突

PR 出現衝突怎麼辦

你的 PR 在 GitHub 顯示 This branch has conflicts that must be resolved 時，不要按網頁的 Resolve conflicts 按鈕 — 那只能處理超簡單的衝突，且編輯體驗很糟。在本地解：

bash 
# 1. 切到自己的 PR 分支
$ git switch feat/add-button

# 2. 把 upstream/main 拉下來
$ git fetch upstream

# 3. rebase 到最新 upstream/main 上 (推薦) 或 merge
$ git rebase upstream/main

# 4. 解衝突 (用 IDE 或 git mergetool)
#    解完後
$ git add <files>
$ git rebase --continue

# 5. 因為 rebase 改寫了歷史，要 force push 更新 PR
$ git push --force-with-lease
!
用 --force-with-lease 不要用 --force 這條規則 Lesson 02 講過了 — 同事剛好在你 rebase 期間 push 了東西，--force 會直接覆蓋；--force-with-lease 會擋下來警告。
09 — 更新 PR

review 後改完，怎麼推上去

Reviewer 留了一些 comment 要你改，你照建議改完後：

狀況 A：純粹 push 新 commit

最簡單也最常用。新 commit 會自動加進這個 PR。

bash 
$ git add <files>
$ git commit -m "refactor(button): rename loading prop to isLoading"
$ git push

狀況 B：用 fixup 保留歷史乾淨

review 都過了，等 maintainer merge 前想把歷史整理乾淨：

bash 
# 提交 fixup 修小東西 (Lesson 02 學過的)
$ git commit --fixup=<原 commit SHA>

# 之後 squash 收乾淨
$ git rebase -i --autosquash upstream/main
$ git push --force-with-lease

狀況 C：合併方式由 maintainer 決定

不同專案有不同合併政策：

合併模式結果常見專案
Merge commit所有 commit 保留 + 一個 merge commitLinux Kernel、Git 本身
Squash and merge整個 PR 壓成一個 commit多數新創、Vercel、Next.js
Rebase and merge所有 commit 接在 main 後 (無 merge commit)偏好線性歷史的團隊
i
所以你 PR 內 commit 要不要整理乾淨？ 看專案政策。如果是 squash and merge，你 PR 內 100 個 commit 最後也只會變 main 上 1 個，commit message 不重要。如果是 merge commit / rebase，每個 commit 都會出現在 main 歷史，那就值得花時間整理。看專案的 CONTRIBUTING.md，或者觀察最近合併的 PR 是哪種模式。
