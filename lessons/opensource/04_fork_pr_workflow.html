01 — 完整旅程

從零開始貢獻一個開源專案的全流程

圖 4.1 — Fork → Clone → push → PR → maintainer merge
02 — Fork

第一步：Fork 至個人帳號

Fork 是 GitHub 提供的「將整個 repo 複製至個人帳號下」功能。之所以需要 fork，是因為對他人 repo 不具備 push 權限，必須先複製一份才能進行修改。

於 GitHub 網頁開啟目標 repo，點擊右上角的 Fork 按鈕並指定個人帳號為複製目的地。GitHub 將於 github.com/<個人帳號>/repo 建立一份個人副本，後續即可對此副本進行任意修改。

03 — Clone

第二步：Clone 至本機並設定 upstream

Clone 是將 GitHub 上的個人 fork 取回本機的動作，使其成為日常開發所操作的工作目錄。

bash 
# Clone 個人 fork 至本機
$ git clone https://github.com/<個人帳號>/repo.git
$ cd repo

設定 upstream remote

Clone 完成後，本地預設僅有一個名為 origin 的 remote，指向個人 fork 後的版本。然而原始專案可能仍持續更新，因此需建立一個指向「上游（原始）專案」的 remote 以追蹤更新——慣例上命名為 upstream。

bash 
# 新增 upstream remote 指向原始專案
$ git remote add upstream https://github.com/<原始作者>/repo.git

# 確認設定完成
$ git remote -v

origin    https://github.com/個人帳號/repo.git (fetch)
origin    https://github.com/個人帳號/repo.git (push)
upstream  https://github.com/原始作者/repo.git (fetch)
upstream  https://github.com/原始作者/repo.git (push)
i
命名慣例 origin 指向個人 fork；upstream 指向原始專案。此為業界通用慣例，不建議自行修改名稱，以免日後參照文件或 Stack Overflow 上的指令時產生混淆。

同步個人 fork 與原始專案

原始專案可能每日皆有 commit 進入。若 fork 與原始專案落差過大才切出分支，後續 PR 將容易產生大量衝突。建議每次開始新工作前先進行同步：

bash 
# 1. 切換回 main
$ git checkout main

# 2. 自 upstream 取得最新內容並整合至本地 main
$ git pull upstream main

# 3. 推送回個人 fork（origin），使 GitHub 上的 fork 同步至最新
$ git push origin main

# 4. 自最新的 main 切出新分支進行開發
$ git checkout -b feat/add-button
GitHub 網頁版的 Sync fork 按鈕 補充說明

GitHub 在 fork 頁面提供「Sync fork」按鈕，作用為在伺服器端將個人 fork（origin）的 main 推進至 upstream main 的最新 commit。具體而言等價於以下本地流程：

bash 
$ git fetch upstream
$ git checkout main
$ git merge --ff-only upstream/main
$ git push origin main

由於此按鈕僅更新 GitHub 上的 fork（origin），本地 clone 並不會自動跟上。使用後仍須回到本地執行 git pull 將本地 main 與 origin/main 同步。

此外，此按鈕僅適用於「個人 fork 的 main 能 fast-forward 至 upstream/main」的情況——亦即個人 fork 的 main 上未產生與 upstream 分歧的 commit。若已存在分歧改動，GitHub 會提示無法乾淨同步，建議改採前述指令流程處理。

04 — push

第三步：commit 改動並 push 至個人 fork

於 feature 分支上完成程式碼修改後，將 commit 推送至個人 fork（origin）對應的分支。Push 完成後 GitHub 會偵測到該分支領先於 upstream/main，並提示可開啟 PR。

bash 
# 1. 編輯檔案後將改動加入暫存區並 commit
$ git add <files>
$ git commit -m "feat(button): add loading state"

# 2. 推送至個人 fork（首次推送加 -u 設定 upstream tracking）
$ git push -u origin feat/add-button
i
commit message 格式 建議採用 Conventional Commits（Lesson 01「Commit Message 規範」已詳述）。Reviewer 從訊息標題即可辨識改動類型（feat / fix / refactor 等）與影響範圍，亦便於日後生成 changelog。
05 — PR

第四步：開 PR 並撰寫描述

從 push 完成到開啟 PR

1
GitHub 顯示橫幅後點選 Compare & pull request
推送完成後重新整理 GitHub 頁面，將出現「分支領先 main N commits, Compare & pull request」橫幅。
2
填寫 PR 描述
下一節將詳述格式。若專案配置了 PR template，將自動帶入。
3
尚未準備接受 review 時，建立 Draft PR
下拉選單選擇 Create draft pull request，表示「請先觀察 CI 執行結果，暫不需 review」。
i
Draft PR 的適用時機 ① 希望先觀察 CI 結果但程式碼尚未完成；② 希望公開展示進度供相關人員預覽；③ 跨時區協作，先行推送以便不同時區的成員檢視。準備就緒後按 Ready for review 即可。

PR 標題格式

PR 標題會出現在 PR 清單、通知信、commit 搜尋結果與 changelog 自動生成工具之中,是 reviewer 與後續維護者最先讀到的一行字。建議遵循 Conventional Commits 規範,使標題具備可解析的語意結構。

建議格式

<type>(<scope>): <subject>

  • type——標示該 PR 的改動類型。Conventional Commits 規格僅強制 feat 與 fix,其他 type 來自 Angular Commit Convention 的延伸(完整一覽見 Lesson 01「常用的 type」)。
  • scope——影響範圍,常見為模組名稱、套件名稱或功能區塊(例如 auth、button、api)。範圍單純或跨多個模組時可省略。
  • subject——簡潔描述,小寫開頭、不加句點、祈使句語氣,建議控制於 50 字內。

各 type 對應的 PR 標題範例

type 用途 PR 標題範例
feat 新功能 feat(button): add loading state prop
fix bug 修補 fix(auth): clear stale token on logout
docs 純文件改動 docs(readme): update installation steps for pnpm
style 格式、縮排、分號等不影響邏輯的調整 style(button): reformat with prettier defaults
refactor 重構(不新增功能、不修補 bug) refactor(form): extract validation logic into hook
perf 效能優化 perf(list): memoize row renderer
test 新增或修改測試 test(auth): add e2e for logout redirect
build 建置系統或依賴變動 build(deps): bump vite to 5.0
ci CI 設定變動 ci(github): cache pnpm store across jobs
chore 其他維運事項(升級 dep、調整 config) chore(eslint): migrate to flat config
revert 還原某筆 commit revert: feat(api)!: rename /v1/users to /v1/accounts
任意 type + ! Breaking change feat(api)!: rename /v1/users to /v1/accounts
i
PR 標題沿用 Conventional Commits 的語法,但單位不同 Commit 描述「這次提交改了什麼」,PR 標題則是整個 change set 的摘要——兩者格式一致,但 PR 標題不等於內部任一筆 commit 標題。
i
採用 squash and merge 政策時,PR 標題會成為 main 上的 commit 標題 此時 PR 標題的品質直接影響 main 的歷史。其他合併模式下 PR 標題不會進到 main(見下方「合併政策由 maintainer 決定」)。

業界通用的 PR 描述格式

大型開源專案(Kubernetes、React、TypeScript 等)的 PR 描述通常包含以下幾個區塊:

PR 描述標準範本

## 背景 / 動機
此次改動的目的為何？解決了什麼問題？

Closes #234

## 改動內容
- 新增 `Button` component 支援 loading 狀態
- 修改 `Form.tsx` 整合新的 Button props
- 補上對應的 unit test

## 測試方式
1. `pnpm test src/components/Button.test.tsx`
2. `pnpm dev` 後於 /demo/button 進行手動點擊測試

## 截圖 / 影片
| 改動前 | 改動後 |
|--|--|
| (圖) | (圖) |

## 是否為 Breaking Change
否。Button 新增的 prop 皆有 default 值。

## Checklist
- [x] 已補上測試
- [x] 已通過 lint / type check
- [ ] 更新 docs（本 PR 不涉及，將於下一 PR 補上）
i
缺少描述等同於拒絕被 review 多數 maintainer 看到空白的 PR 描述會直接退回並要求補上描述。對 reviewer 而言，「不清楚改了什麼、為何要改」便無法進行 review。撰寫描述所花的數分鐘，可避免後續多日的來回溝通。

透過關鍵字自動關聯與關閉 issue

於 PR 描述或 commit message 中以下列關鍵字搭配 issue 編號，GitHub 將自動建立連結；當 PR 合併進主分支時，對應的 issue 會被自動關閉：

關鍵字是否自動關閉 issue適用情境
Closes #234✓ 會本 PR 完整解決了 #234
Fixes #234✓ 會修補 bug 時使用
Resolves #234✓ 會處理 feature request 時使用
Refs #234✗ 不會「相關但僅為部分解決」
See #234✗ 不會單純引用，不負責關閉
i
支援跨 repo 引用 Closes owner/repo#234 可由另一個 repo 的 PR 關閉指定 repo 的 issue（需具備寫入權限）。
06 — maintainer merge

第五步：解決衝突、回應 review 並等待 maintainer merge

PR 發生衝突的處理方式

當 PR 在 GitHub 顯示 This branch has conflicts that must be resolved 時，不建議使用網頁的 Resolve conflicts 按鈕——其僅能處理極為單純的衝突，且編輯體驗欠佳。建議於本地解決：

bash 
# 1. 切換至個人 PR 分支
$ git checkout feat/add-button

# 2. 取得 upstream/main 最新內容
$ git fetch upstream

# 3. rebase 至最新 upstream/main（推薦）或 merge
$ git rebase upstream/main

# 4. 解決衝突（透過 IDE 或 git mergetool）
#    解決後執行
$ git add <files>
$ git rebase --continue

# 5. rebase 改寫歷史，需透過 force push 更新 PR
$ git push --force-with-lease
!
應使用 --force-with-lease 而非 --force 此規則於 Lesson 02 已說明——若同事於 rebase 期間 push 了內容，--force 會直接覆蓋；--force-with-lease 則會拒絕推送並提示。

依 reviewer 建議更新 PR

Reviewer 留下 comment 後，依其建議修改完畢的處理方式：

最簡便且最常使用的方式為直接 push 新 commit,新 commit 將自動納入該 PR。

bash 
$ git add <files>
$ git commit -m "refactor(button): rename loading prop to isLoading"
$ git push

合併政策由 maintainer 決定

review 通過後，最終由 maintainer 點擊 Merge 按鈕完成合併。不同專案採用不同的合併政策：

合併模式結果常見專案
Merge commit所有 commit 完整保留並新增一筆 merge commitLinux Kernel、Git 本身
Squash and merge整個 PR 壓縮為單一 commit多數新創公司、Vercel、Next.js
Rebase and merge所有 commit 線性接續至 main 後（不產生 merge commit）偏好線性歷史的團隊
i
PR 內的 commit 是否需整理？ 視專案政策而定。若採用 squash and merge，PR 內的 100 筆 commit 最終於 main 上僅顯示為 1 筆，commit message 影響有限。若採用 merge commit 或 rebase，每筆 commit 皆會出現於 main 歷史中，便值得投入時間整理。
