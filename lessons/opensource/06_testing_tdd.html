Lesson 06 · 自動化與測試篇

軟體測試

測試的目的並非追求 coverage 數字，而是提供使改動得以安全進行的保護網。本課將建立對測試類型、TDD 流程與覆蓋率的正確認知。

01 — 測試形狀總覽

金字塔、獎盃、蜂巢——三種測試結構

測試形狀（test shape）描述各類測試在整套測試套件中所佔的比例。在比較三種形狀之前，須先建立對各層測試類型的基本認識。

四種主要測試類型

以下以「使用者登入」功能為例，說明後續三種形狀中反覆出現的測試類型：

類型測試對象具體範例（登入功能）典型工具
Static
靜態檢查		 程式碼未執行，由編譯器、型別系統或 linter 於編寫期捕捉問題 TypeScript 編譯器發現呼叫 login(username) 時遺漏必填的 password 參數；ESLint 偵測未使用變數 TypeScript、ESLint
Unit
單元測試		 單一函式或元件本身的邏輯；資料庫、外部 API、其他模組等依賴以「假物件」（mock，預先寫好回傳值的替身）取代，使測試只驗證目標函式自身、執行快且結果穩定 驗證 hashPassword('abc123') 回傳長度為 60 的 bcrypt 字串；validateEmail('foo@bar') 回傳 false Jest、Vitest、pytest、JUnit
Integration
整合測試		 多個模組串接，搭配真實 DB、HTTP client、message queue 等依賴 對 POST /login 發出請求，驗證資料庫成功建立 session 紀錄、回應為 200 與合法 JWT Supertest、Testcontainers
E2E
端對端測試		 啟動真實瀏覽器或 client，從使用者視角執行完整流程 Playwright 開啟登入頁面、填入帳號密碼、點擊「登入」按鈕、驗證跳轉至 dashboard 並顯示使用者名稱 Playwright、Cypress、Selenium

除上述四種核心類型外，業界另有以下衍生分類，多由前四種類型組合而成：

類型說明典型工具
Contract
契約測試		 驗證服務間 API 的請求／回應契約（consumer / provider）是否相容，可於不啟動整個系統的前提下進行 Pact、OpenAPI schema
Smoke
冒煙測試		 部署後執行少量關鍵流程，確認系統可正常運作；通常為精選的 E2E 子集 少量精選 E2E
Regression
回歸測試		 持續累積的測試集合，確保已修補的 bug 不再復發；前四種類型皆可累積為回歸測試 —

範圍對照：同一個登入功能，三種測試怎麼測

讀完上表仍不易區分三者，因為差異不在「測試對象」的字面描述，而在「測試實際啟動了哪些真實元件」。下圖以登入流程為例，標示各類測試實際涵蓋的系統範圍：

圖 8.1 — 三種測試的系統覆蓋範圍對照
  • Unit——僅呼叫單一函式（如 hashPassword）。HTTP server 不啟動、資料庫不連線、瀏覽器不開啟；其他依賴皆以 mock 假物件取代。
  • Integration——啟動後端進程，發出真實 HTTP 請求至 POST /login，由真實程式碼讀寫真實資料庫。但不開啟瀏覽器、不執行前端 JS。
  • E2E——Playwright 開啟真實瀏覽器，載入前端網頁，模擬使用者點擊「登入」按鈕觸發前端發出 HTTP，後端再讀寫資料庫。整個系統皆為真實元件。
i
越底層的測試越快、越穩定；越上層越貼近真實使用情境，但成本也越高 三種測試形狀的差異，本質就在於「把測試比重放在哪一層」。

三種主流形狀

隨著測試工具與軟體架構的演進，業界逐步發展出三種主流形狀，各自對應不同的專案類型：

形狀提倡者核心觀點適用場景
金字塔 Mike Cohn（2009） 大量 unit、少量 E2E 後端 monolith、邏輯密集的服務
獎盃（Trophy） Kent C. Dodds（2018） 以整合測試為主，並以靜態檢查為基底 前端應用、TypeScript 專案
蜂巢（Honeycomb） Spotify Engineering 服務之間的整合測試最具價值 後端微服務架構
i
形狀的選擇取決於專案類型，而非個人偏好 接下來三節將分別說明各形狀的層級組成與設計動機。理解設計脈絡後再依專案性質做選擇，避免將單一形狀套用至所有情境。
02 — 金字塔

Mike Cohn 的測試金字塔（2009）

金字塔的核心理念：越底層的測試數量越多、執行越快、越穩定；越頂層的測試數量越少、執行越慢、越容易失效。傳統後端應用中，業務邏輯多集中於函式與類別層級，因此 unit test 是最具成本效益的測試類型。

圖 8.2 — Mike Cohn 測試金字塔
!
反金字塔（倒三角）為典型反模式 僅有 E2E、缺乏 unit test 的後果：test suite 動輒跑半小時、隨機失敗、debug 困難。此為新手團隊最常見的測試陷阱。
03 — 獎盃

Kent C. Dodds 的測試獎盃（2018）

Kent C. Dodds 於 React Testing Library 推出後提出此模型，反映前端測試生態的轉變。核心格言為：「Write tests. Not too many. Mostly integration.」

相對金字塔的三項改動

獎盃並非全新發明，而是針對前端應用對金字塔做出三項調整：

項目金字塔（2009）獎盃（2018）
新增基底層 無 新增 Static（TypeScript、ESLint），於編寫期即捕捉型別與語法問題
主體層的位置 底層的 Unit（約 70%）為主體 中段的 Integration 為主體（杯身最寬）
Unit 的角色 主力，涵蓋大多數邏輯 縮減至純函式、utilities、custom hooks；不再用於含 UI 的 component

之所以做這些改動，是因為前端 unit test 容易與 DOM 結構耦合（UI 微調即導致大量測試失敗），而 React Testing Library 提供了從使用者觀點測試元件的能力，使整合測試比 unit 更具價值。

圖 8.3 — Kent C. Dodds 測試獎盃
04 — 蜂巢

Spotify 的蜂巢模型

Spotify Engineering 於微服務架構中提出此模型。在分散式系統下，單一服務的內部邏輯通常單純，真正的複雜度來自服務之間的互動，因此將測試比重集中於「單一服務搭配真實依賴」的整合測試。

蜂巢使用的特殊命名

蜂巢並未沿用前述 Unit / Integration / E2E 三分法，而採用了 Spotify 內部的命名。三者與傳統類型的對應關係如下：

蜂巢用語對應傳統類型內容
Implementation Detail Tests
實作細節測試		 ≈ Unit 針對函式或類別的內部實作。蜂巢主張少寫，因為易與實作耦合，重構時容易破裂
Integration Tests
整合測試		 = Integration 單一服務內，搭配真實 DB、HTTP client、message queue 等依賴，驗證業務流程
Integrated Tests
跨服務測試		 ≈ E2E 橫跨多個服務的端對端流程，啟動成本高且易失效，僅針對核心鏈路
!
Integration 與 Integrated 一字之差，意義大不同 Integration（單一服務內）為主體，數量最多；Integrated（跨服務）為少量。讀英文原文時請特別留意尾綴 -ion / -ed 的差異。

05 — 撰寫好測試

TDD 流程、F.I.R.S.T 原則、AAA 結構

前四章說明的是「測試的整體配比」。本章轉向「測試本身」——四個彼此互補、可同時並用的概念：

概念關注層次回答的問題
TDD開發流程什麼時候寫測試？
Coverage覆蓋度量測試實際執行了多少程式碼？
F.I.R.S.T測試品質怎樣的測試算是好測試？
AAA單一測試結構一個測試的程式碼如何組織？

四者並無互斥：以 TDD 流程驅動撰寫測試，每個測試內部遵循 AAA 結構，整體須滿足 F.I.R.S.T 原則，並以 Coverage 作為輔助度量。

TDD 三步驟：Red → Green → Refactor

TDD（Test-Driven Development，測試驅動開發）由 Kent Beck 於《Test-Driven Development: By Example》（2002）中系統化的方法論。其為一個持續循環的流程：

圖 8.5 — TDD 三步驟循環

每次循環只處理一個測試

TDD 的核心紀律由 Robert C. Martin（Uncle Bob）整理為 Three Rules of TDD：

  1. 未有失敗的測試前，不可寫任何 production code
  2. 單一測試僅寫到「剛好失敗」即停（包含編譯錯誤）
  3. Production code 僅寫到「剛好通過該失敗測試」即停

因此每次 R-G-R 循環嚴格限制在單一測試的範圍內。多輪循環累積後，才會於 REFACTOR 步驟出現可抽出的重複結構——這正是 TDD 與「先列完所有測試再實作」（spec-first / test-first）的根本差異所在。

!
勿一次列完所有測試 若一次寫完整份規格的所有測試再開始實作，會失去三項 TDD 的核心價值：① REFACTOR 步驟失去抽取共通結構的素材、② 邊界情況無從於後續迴圈中發現、③ 開發者容易超出「剛好通過」的範圍而過度實作。此做法在業界稱為 spec-first / test-first，可視為一種有效的測試實務，但不屬於 Kent Beck 原始定義的 TDD。

實作範例：撰寫密碼驗證函式 validatePassword

規格：密碼須長度 ≥ 8 字元，且至少含一個數字。以下示範兩輪 RED → GREEN 累積出有意義的重構機會，再於第三步驟進行 refactor。

typescript 
// ===== Round 1（RED） =====
test('長度至少 8 字元才合法', () => {
  expect(validatePassword('abc')).toBe(false);
  expect(validatePassword('abcdefgh')).toBe(true);
});
// 執行 → 失敗（validatePassword 尚未實作）

// ===== Round 1（GREEN） =====
function validatePassword(p: string) {
  return p.length >= 8;
}
// 執行 → 通過

// ===== Round 2（RED） =====
test('必須包含至少一個數字', () => {
  expect(validatePassword('abcdefgh')).toBe(false);  // 無數字
  expect(validatePassword('abcdefg1')).toBe(true);
});

// ===== Round 2（GREEN） =====
function validatePassword(p: string) {
  return p.length >= 8 && /\d/.test(p);
}
// 執行 → 兩個測試皆通過

// ===== Round 3（REFACTOR） =====
// 兩條規則混在同一行 boolean 表達式中，可讀性下降。
// 測試保持不動，抽出具名規則：
const hasMinLength = (p: string) => p.length >= 8;
const hasDigit = (p: string) => /\d/.test(p);

function validatePassword(p: string) {
  return hasMinLength(p) && hasDigit(p);
}
// 執行 → 全部測試仍通過 ← REFACTOR 完成
i
TDD 真正的價值不在於「撰寫測試」 而在於「先釐清要實作的內容」。撰寫測試的當下，必須思考：「此函式的 input / output 為何？邊界情況有哪些？」此思考過程的價值遠高於測試本身。

Coverage——測試覆蓋率

Coverage 是衡量「測試實際執行到多少程式碼」的定量指標。工具（JS 的 Istanbul / c8、Python 的 coverage.py、Java 的 JaCoCo、Go 內建 go test -cover）會於測試執行時在程式碼中插入紀錄點，計算「執行到的部分 ÷ 總量」的百分比。依「總量」如何切分，常見四種指標：

指標計算依據
Line程式碼行數中被執行過的比例
Statement語句中被執行的比例（單一行可能含多個 statement）
Branch分支（if / else、switch）中被執行的比例——通常最具參考價值
Function函式中至少被呼叫過一次的比例
i
Coverage 數字並非測試品質的同義詞 Coverage 僅衡量「測試覆蓋了多少程式碼」，無法反映邏輯正確性——一個無 expect 卻呼叫過函式的測試亦能貢獻覆蓋率。業界以 80% 為健康度參考值（高品質開源專案多落於 80–90%），但追求 100% 邊際效益急遽下降，且易催生為了數字而撰寫的低價值測試。Coverage 應與 F.I.R.S.T、AAA 並用，作為輔助指標而非單一目標。

F.I.R.S.T——五項測試品質原則

由 Robert C. Martin（Uncle Bob）提出，每個字母代表一項原則：

字母原則實務意涵
Fast快速整套 unit test 應於數秒內完成，否則開發者將不願頻繁執行
Isolated獨立每個測試彼此獨立，不依賴執行順序、不互相影響
Repeatable可重複相同 input 永遠產生相同結果，不依賴時間、亂數或外部服務。違反此原則即為 flaky test（不穩定測試）——有時通過、有時失敗，會使團隊習慣性忽略 CI 紅燈
Self-validating自我驗證執行後直接顯示成功或失敗，無須人工檢視 log 判讀
Timely及時測試應緊隨 production code 撰寫，理想方式即為 TDD 的 R-G-R 循環

AAA——Arrange / Act / Assert

每個測試於結構上清楚劃分為三段，可顯著提升可讀性。三段各自的角色如下：

字母階段實務意涵
Arrange準備建立測試所需的初始狀態：建構物件、預設資料、設定 mock 回傳值。此段只做準備，不觸發任何被測行為
Act執行呼叫一次被測函式或觸發一次事件，即為本測試要驗證的「動作」。理想上僅一行
Assert驗證以 expect 檢查 Act 後的結果是否符合預期；若需多個 expect，應皆驗證同一件事

以下為三段清楚分隔的測試範例：

typescript 
test('logout: clears all user state', () => {
  // Arrange——設定測試情境
  const store = createStore({ user: { id: 42, name: 'Jason' } });

  // Act——觸發目標行為
  store.dispatch(logout());

  // Assert——驗證結果
  expect(store.getState().user).toBeNull();
  expect(store.getState().token).toBeNull();
});
✓
每個測試僅驗證單一事項 單一 test 內最好僅包含一個邏輯上的 assert（可有多個 expect，但皆驗證同一件事）。若發現需要第二組 act → assert，應將其拆分為新的測試。
