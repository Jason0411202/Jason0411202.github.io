01 — CI/CD 區別

Continuous Integration / Delivery / Deployment

三個都叫 CI/CD，但意義不同 — 是連續的三個階段：

名稱意義具體做了什麼
Continuous Integration 持續整合 每個 push / PR 自動跑測試 + lint，確保 main 永遠是綠的
Continuous Delivery 持續交付 CI 之後自動把通過的 build 送到 staging / artifact 倉庫，但 production 部署是手動按按鈕
Continuous Deployment 持續部署 連 production 部署也自動 — 合進 main 就直接上線
i
三者關係 CI 是 CD 的前提。CD 又分 Delivery (人按 button 才上線) 跟 Deployment (合 main 就上線)。Deployment 風險最高、需要最強 CI 守護。大多數團隊先做到 CI，再進到 Continuous Delivery，最後才 Continuous Deployment。
02 — GitHub Actions

GitHub Actions 結構

GitHub 內建的 CI/CD 平台，免費額度對開源專案幾乎無限。設定檔放在 .github/workflows/*.yml。

圖 9.1 — Workflow / Job / Step 三層結構

四個關鍵概念

  • Workflow — 一個 YAML 檔代表一個 workflow，定義什麼事件觸發 (on: push)
  • Job — 一個 workflow 內的執行單位，預設並行；可用 needs: 設依賴順序
  • Step — Job 內依序執行的步驟，可以是 shell 指令或 action
  • Action — 可重用的 step (例如 actions/checkout@v4)，可從 Marketplace 找
03 — 範例 workflow

典型 lint → test → build 完整範例

yaml 
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck

  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: ['18', '20', '22']
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm test --coverage
      - uses: codecov/codecov-action@v4

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - uses: actions/setup-node@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
i
幾個業界慣例 ① install --frozen-lockfile 確保 CI 用的依賴版本跟 lock 檔完全一致 ② 用 cache: 'pnpm' 加速 ③ matrix 跑多 Node 版本確保跨版本相容 ④ action 永遠 pin 主版號 (@v4)，不要 @latest 否則 breaking 會自動套上你 CI。
04 — Branch Protection

Branch Protection Rules

有 CI 但沒設保護規則 = 大家還是可以強推 main / 跳過 review。在 Settings → Branches 加保護規則：

規則效果
Require pull request before merging禁止直接 push main，一定要走 PR
Require approvals (1+)至少 N 人 approve 才能 merge
Dismiss stale approvals on new push有新 commit 後 approve 失效，要重 review
Require review from Code Owners動到 CODEOWNERS 指定的目錄要該 owner approve
Require status checks to passCI 必須綠燈才能 merge — 然後勾選哪些 check
Require branches to be up to datePR 必須 rebase 到最新 main 才能合 (避免 fast-forward 後爆炸)
Require linear history禁止 merge commit，強制 squash 或 rebase merge
Require signed commits強制 GPG / SSH 簽名 (高安全環境)
Do not allow bypassing連 admin 也不能跳過 — 真的卡住時才暫時關
!
沒設 branch protection 等於沒做 CI 如果開發者可以強推、可以跳過 CI 失敗、可以略過 review — 那 CI / review 流程就只是「禮貌建議」。Branch protection 把「該做的事」變成「必做的事」。
05 — Linter / Formatter

Linter 與 Formatter 的差別

  • Linter — 檢查程式碼邏輯 / 風格問題，例如未使用變數、不安全的型別、複雜度過高
  • Formatter — 只管排版，引號 / 縮排 / 換行 / 行寬

各語言對應工具

語言LinterFormatter
JavaScript / TypeScriptESLintPrettier (或 Biome 整合兩者)
PythonRuff (新) / Pylint / Flake8Ruff format / Black
Gogolangci-lintgofmt / goimports (內建)
RustClippyrustfmt
JavaCheckstyle / SpotBugsgoogle-java-format / Spotless
i
Ruff 是 Python 生態的新霸主 用 Rust 寫的，比 Pylint 快 10-100 倍。一個工具替代 Flake8 + isort + pyupgrade + Black。2024 後新 Python 專案幾乎都選它。
EditorConfig — 跨編輯器一致基礎排版 小工具

放一個 .editorconfig 在 repo 根目錄，所有支援的 IDE / 編輯器都會自動套用：

editorconfig 
root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true

[*.md]
trim_trailing_whitespace = false

不取代 Prettier / rustfmt，但讓「沒裝 formatter 的同事」至少縮排不會亂。

06 — Git Hooks

Git Hooks：在本地 commit 前先擋下垃圾

Git 內建 hooks (.git/hooks/) 在特定事件時執行 script。但 .git 不會被版本控制 = 同事 clone 時不會帶到 hook。所以實務上用第三方工具管理 hook：

三大方案比較

工具生態優點
Husky + lint-staged Node.js JS 生態事實標準；只 lint 改動的檔案 (lint-staged)
pre-commit Python (但跨語言) 支援所有語言；hook 設定一份 YAML 就能跨 repo 共用
Lefthook Go (但跨語言) 新世代，支援並行執行多個 hook，速度快

Husky + lint-staged 範例

bash 
# 1. 安裝
$ pnpm add -D husky lint-staged
$ pnpm exec husky init

# 2. 把 .husky/pre-commit 改成這樣
$ cat .husky/pre-commit
pnpm exec lint-staged

# 3. 在 package.json 加 lint-staged 設定
json 
// package.json
{
  "lint-staged": {
    "*.{ts,tsx,js,jsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    "*.{json,md,yml}": [
      "prettier --write"
    ]
  }
}
i
為什麼要 lint-staged 不直接 lint 全部 老 codebase 有 1000 個檔案，每次 commit 全 lint 一遍要 30 秒，沒人受得了。lint-staged 只 lint 你這次改動的檔案，1 秒內結束。同時逐步累積到 main 都乾淨。
為什麼不直接用 Git 原生 .git/hooks/？ 原因
  • 不會跟 repo 一起 clone — 別人拿到專案就沒 hook
  • 沒有跨平台處理 — Windows 跟 Unix shell 不同
  • 沒有 hook 管理 — 多個 hook 不能同時用一個事件
  • 沒有版本控制 — hook 改了沒人知道

所以 Husky / pre-commit / Lefthook 這些工具都是把 hook 寫在 repo 內、自動安裝到 .git/hooks/ — 解決上面所有問題。

07 — commitlint

commitlint：擋下不符規範的 commit message

Lesson 06 學的 Conventional Commits 規範要怎麼真正落實？讓 commitlint 在 commit-msg hook 自動驗證：

bash 
$ pnpm add -D @commitlint/cli @commitlint/config-conventional

# 建 commitlint.config.js
$ echo "export default { extends: ['@commitlint/config-conventional'] };" > commitlint.config.js

# 加 husky commit-msg hook
$ echo 'pnpm exec commitlint --edit "$1"' > .husky/commit-msg

之後若有人嘗試 commit fix bug：

output 
$ git commit -m "fix bug"

⧗   input: fix bug
✖   subject may not be empty [subject-empty]
✖   type may not be empty [type-empty]

✖   found 2 problems, 0 warnings
ⓘ   Get help: https://github.com/conventional-changelog/commitlint

husky - commit-msg script failed (code 1)
✓
本地 + CI 雙重把關 本地 hook 在 commit 當下擋；CI 上再用 wagoid/commitlint-github-action 在 PR 也檢查。本地擋下大多數，CI 處理「跳過 hook 推上來」的情況。
