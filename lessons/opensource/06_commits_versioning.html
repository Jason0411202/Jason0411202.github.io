02 — Conventional Commits 1.0

Conventional Commits 格式

規格網址：conventionalcommits.org/zh-hant/v1.0.0。基本格式：

commit 
<type>(<scope>)!: <subject>

<body 詳細說明，可省略>

<footer 例如 BREAKING CHANGE / Closes #123>

解構一個範例：

feat(auth)!: switch session storage to httpOnly cookies

之前用 localStorage 存 token 容易被 XSS 偷走，
改用 httpOnly cookie + CSRF token 機制。

BREAKING CHANGE: 前端不再能讀取 token，需透過新的
/api/me 端點取得當前使用者資訊。
Closes #421
  • type：feat = 新功能
  • scope：(auth) = 影響範圍 (可選)
  • !：標記 breaking change
  • subject：switch session storage to httpOnly cookies — 50 字內、祈使句、首字小寫、結尾不加句點
  • body：詳細說明 why (不是 what)
  • footer：BREAKING CHANGE 完整描述 + 連結 issue
03 — type 類型

常用的 type

Conventional Commits 規格只強制 feat 跟 fix，其他 type 是 Angular Commit Convention 的延伸，業界普遍採用：

type用途SemVer 對應
feat新功能MINOR (+0.1.0)
fix修 bugPATCH (+0.0.1)
docs純文件改動不升版
style格式 / 縮排 / 分號 (不影響邏輯)不升版
refactor重構 (不增功能也不修 bug)不升版
perf效能改善PATCH
test新增 / 修改測試不升版
build建置系統 / 依賴改動不升版
ciCI 設定改動不升版
chore其他雜事 (升 dep、調 config)不升版
revert還原某個 commit看還原什麼
任何 + !BREAKING CHANGEMAJOR (+1.0.0)
04 — Breaking change

標記 BREAKING CHANGE 的兩種方式

方式 A：subject 後加 ! (推薦，最顯眼)

commit 
feat(api)!: drop support for Node 16

Closes #999

方式 B：footer 寫 BREAKING CHANGE:

commit 
feat(api): require Node 18+

新版用了原生 fetch API。

BREAKING CHANGE: Node 16 不再支援，使用者需升級到 Node 18+。
!
建議兩種同時用 ! 在 git log 一行式很顯眼；BREAKING CHANGE: footer 提供完整說明 + 給工具讀取。兩者組合最完整。
05 — 好/壞範例

好/壞 commit message 對照

✗糟糕
fix bug
update
WIP
asdfasdf
finally working
✓好的
fix(profile): handle null avatar URL
refactor(api): extract pagination logic
perf(search): add result caching
docs(readme): update install steps
i
subject 寫法心法 ① 用祈使句 (add 不是 added) — 像在給 git 下命令
② 首字小寫、結尾不加句點
③ 50 字內，超過寫到 body
④ 想想看：在 changelog 裡讓使用者讀，看得懂嗎？
06 — SemVer

Semantic Versioning 2.0.0

規格：semver.org/lang/zh-TW。版號格式：

圖 6.2 — SemVer 版號三段式

三段升版規則

  • MAJOR (主版號)：有 Breaking Change 時 +1，MINOR 與 PATCH 重設為 0
  • MINOR (次版號)：新增向下相容的功能時 +1，PATCH 重設為 0
  • PATCH (修訂號)：修向下相容的 bug 時 +1

初始版本與 0.x

0.x.y 版本表示「還在快速迭代、API 可能任何時候改」。一旦發 1.0.0 就要對 API 穩定性負責，所以發 1.0 是個重大決定 —— 很多開源專案故意不發 1.0 (例如 webpack 4 之前)。

i
為什麼要嚴格 SemVer？ 因為使用者的 package.json 可能寫 "^1.4.0" 表示「自動更新到 1.x.y 都可以」。如果你違反 SemVer 在 1.4.5 偷塞 breaking change，全世界依賴你的人都會炸。
預發布版本 + Build metadata 深入
semver 
2.0.0-alpha.1     ← 早期測試
2.0.0-beta.3      ← 公測
2.0.0-rc.1        ← release candidate (即將正式)
2.0.0             ← 正式

1.4.5+exp.sha.5114f85   ← + 後是 build metadata，不影響版本順序

預發布版本排序：alpha < beta < rc < (正式)。

07 — Keep a Changelog

Keep a Changelog 1.1.0

規格：keepachangelog.com/zh-TW/1.1.0。它的主張是 changelog 是寫給人類看的，不是給機器解析的 git log。

固定六種分類

分類意義
Added新增的功能
Changed變更既有功能
Deprecated標記為即將移除
Removed已移除的功能
Fixed修正的 bug
Security安全性修補

標準格式範例

markdown 
# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

## [1.4.0] - 2026-04-15

### Added
- 新增使用者 profile 頁面 (#421)
- API 支援 GraphQL queries (#430)

### Fixed
- 修正登出後 token 殘留問題 (#418)

### Security
- 升級 axios 到 1.6.5 修補 CVE-2024-12345

## [1.3.2] - 2026-03-20

### Fixed
- 修正手機版 navbar overflow

[Unreleased]: https://github.com/owner/repo/compare/v1.4.0...HEAD
[1.4.0]: https://github.com/owner/repo/compare/v1.3.2...v1.4.0
[1.3.2]: https://github.com/owner/repo/compare/v1.3.1...v1.3.2
i
頂端永遠有 [Unreleased] 區塊 開發中的改動先放這個區塊，正式 release 時再把它改成版號 + 日期，並建立新的 [Unreleased]。
08 — 自動化工具

把整條鏈自動化的工具

有了 Conventional Commits 的格式，工具可以幫你做剩下的事：

工具做什麼定位
commitlint commit 時驗證 message 格式 守門員，配合 Husky 在 git commit 攔下不合格的
Commitizen 互動式幫你寫 commit (cz 指令) 新人友善，不用記格式
semantic-release 讀 commit 自動算下一版號 + 產 changelog + 發 release JS 生態系最主流，CI 上跑
changesets 類似 semantic-release，但 monorepo 友善 pnpm workspace / Turborepo 多套件用
Release Please Google 的自動 release PR 工具 多語言支援，產出 release PR 等人 approve
i
典型自動化鏈 feat: ... commit → 推上 main → CI 跑 semantic-release → 自動 bump 版號 → 產 CHANGELOG.md → 打 tag → 發 GitHub Release → publish 到 npm。整條不需手動介入。
Gitmoji、Angular Convention 與其他變體 歷史

Angular Commit Convention (2014) 是 Conventional Commits 的祖先 — Angular 團隊內部規範，後來被抽出來成為通用標準。

Gitmoji (gitmoji.dev) — 用 emoji 取代 type，例如 :sparkles: 新功能 / :bug: 修 bug。視覺友善但機器不易解析，所以多數大型專案還是用文字 type。

實務上，Conventional Commits 是 2026 的事實標準，幾乎所有自動化工具都讀這個格式。
