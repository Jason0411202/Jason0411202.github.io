Lesson 01 · Git 基礎篇

Git 日常實務速成

從四區心智模型出發，把每天會用到的 Git 指令一次學會。不是教你 Git 物件原理，是讓你今天就能在團隊裡正常工作。

01 — 總覽

四個區、一張圖

Git 的所有指令幾乎都在做同一件事：把檔案在四個區之間搬來搬去。理解這張圖，你就懂 80% 的 Git 指令在幹嘛。

圖 1.1 — Git 四區與主要流向
i
記住這四個動作就好 編輯 → add 到 Staging → commit 進 Local → push 上 Remote。所有複雜指令都是這條主線的變體。
02 — 基本操作

每天做事的標準節奏

無論在公司或開源專案，一天的 Git 操作會像下面這樣循環：

1
開始前先同步主分支
避免在過時的程式碼上開發,把遠端最新進度拉下來再開工。
2
看當前狀態
git status 是 Git 的儀表板，動任何指令前先看一眼。
3
挑要進這次 commit 的改動
git add <file> 一個一個加比 git add . 安全很多，能避免把 debug 用的 console.log 一起送進去。
4
提交
git commit -m "..."。message 寫法之後 Lesson 06 會專門講。
5
推上去
第一次推某個分支用 git push -u origin <branch> 設定上游，之後直接 git push 就好。
bash 
# 1. 同步主分支
$ git checkout main
$ git pull

# 2. 開新分支做事
$ git checkout -b feat/user-profile

# 3. 寫程式 ... 之後看狀態
$ git status

# 4. 挑檔案 stage
$ git add src/profile.ts src/profile.test.ts

# 5. commit
$ git commit -m "feat(profile): add user profile page"

# 6. 推上去 (第一次推這個分支要 -u)
$ git push -u origin feat/user-profile

add 的三種選法

指令行為什麼時候用
git add file.ts加單一檔案知道自己要的是哪幾個檔案 (推薦)
git add -p逐塊互動式選擇同一檔案內想拆成兩個 commit 時
git add .當前目錄全部確認過 status 沒有 debug 殘留時才用

執行 git add -p 後,Git 會把改動切成一塊塊「hunk」依序問你要不要 stage,長得像這樣:

output 
diff --git a/src/auth.ts b/src/auth.ts
@@ -10,3 +10,5 @@ export function login(user) {
   if (!user.password) {
     throw new Error("password required");
   }
+  if (!user.email) {
+    throw new Error("email required");
+  }
(1/2) Stage this hunk [y,n,q,a,d,s,e,?]?

看完這塊改動後,輸入單一字母按 Enter 即可。常用的有:

  • y (yes):stage 這塊。
  • n (no):跳過這塊,不 stage。
  • s (split):把這塊拆成更小塊再問一次,適合「同一塊裡有想要的也有不想要的」。
  • e (edit):用編輯器手動挑要 stage 的行 —— 留下要的、把不要的行刪掉再存檔。
  • a (all) / d (done):一次接受/略過剩下所有 hunk。
  • q (quit):立刻結束。? 隨時叫出說明。

實務最常用的就是 y、n、s 三個鍵。例如同一個檔案裡同時改了「正式功能」跟「順手修的 typo」,可以用 git add -p 把功能部分先 stage、commit;剩下 typo 再 commit 一筆,做出兩個各司其職的 commit。

03 — git blame

看 commit 歷史的兩個工具

Git Graph (VSCode 擴充套件) — 視覺化分支與 commit

原始的 git log 在 terminal 裡看起來又長又難讀。實務上,大多數開發者會直接在 VSCode 裝 Git Graph 擴充套件,用視覺化介面看歷史。

  • 裝完後在 VSCode 左下角狀態列、或 Source Control 面板上方點 View Git Graph,整個倉庫的分支圖一目了然。
  • 點任一個 commit:右側會顯示 commit message、作者、被改的檔案清單,點檔案就能直接看 diff。
  • 右鍵 commit:可以直接做 checkout、cherry-pick、revert、reset 等操作,不用記指令。
  • 分支合併、rebase、tag 的關係用線條畫出來,比 git log --graph --all 直觀十倍。

git blame — 看每一行是誰改的

不是用來找戰犯,是用來找「為什麼這行寫成這樣」的線索。

bash 
$ git blame src/auth.ts             # 整個檔案
$ git blame -L 42,55 src/auth.ts    # 只看 42 ~ 55 行

輸出每一行有四個欄位,從左到右是 commit SHA、作者、提交時間、(行號) 該行內容:

output 
^a1b2c3d (Alice  2024-08-12 14:32:01 +0800  42) export function login(user) {
e4f5a6b7 (Bob    2025-02-03 09:11:45 +0800  43)   if (!user.email) {
e4f5a6b7 (Bob    2025-02-03 09:11:45 +0800  44)     throw new Error("email required");
^a1b2c3d (Alice  2024-08-12 14:32:01 +0800  45)   }

找到可疑那行的 SHA 後,直接 git show e4f5a6b7 就能看到當時整筆 commit 的 message 跟改了什麼,線索就接起來了。SHA 開頭有 ^ 的(像上面的 ^a1b2c3d)代表這行從專案最早的 commit 就存在、沒被改過;沒有 ^ 的(像 e4f5a6b7)則是後來某個 commit 才新增或修改的那行。

04 — git diff

比較分支差異

單檔或工作區的差異,VSCode 的 Source Control 面板直接點檔案就能看,平常不太會用 git diff。真正在 terminal 還會用到的場景,是比較兩個分支的整體差異 —— 例如送 PR 前想確認自己這條分支跟 main 到底差了什麼。

指令比的是用在哪
git diff main..feat/x 分支 vs 分支 「我這條 PR 跟 main 差了什麼」

讀懂 diff 輸出

下面是典型的 diff 輸出:

output 
diff --git a/src/auth.ts b/src/auth.ts
index 1a2b3c4..5d6e7f8 100644
--- a/src/auth.ts
+++ b/src/auth.ts
@@ -10,6 +10,8 @@ export function login(user) {
   if (!user.password) {
     throw new Error("password required");
   }
-  return authenticate(user);
+  if (!user.email) {
+    throw new Error("email required");
+  }
+  return authenticate(user.email, user.password);
 }
  • diff --git a/... b/...:被比較的檔案路徑,a/ 是舊版、b/ 是新版。
  • index 1a2b3c4..5d6e7f8 100644:兩邊的內部物件 SHA 與檔案權限 (100644 是一般檔案),平常不用管。
  • --- a/... / +++ b/...:標示哪個是舊、哪個是新,接下來 - 對應這個舊檔、+ 對應那個新檔。
  • @@ -10,6 +10,8 @@:稱為 hunk header,意思是「舊版第 10 行起的 6 行,對應新版第 10 行起的 8 行」;後面跟著的是該函式或區塊的 context (例如 export function login(user) {)。
  • - 開頭:被刪掉的行;+ 開頭:新加的行;沒符號的行:沒動,只是上下文方便閱讀。
05 — git --amend 參數

剛 commit 完發現少改一個地方

不要再開一個 fix typo 的 commit，用 --amend 補進上一筆：

bash 
# 修了檔案後
$ git add src/forgot.ts

# 補進上一個 commit (message 不變)
$ git commit --amend --no-edit

# 想改 message 也可以
$ git commit --amend -m "feat(profile): add user profile page with avatar"
!
已經 push 出去的 commit 不要 amend --amend 會改變 commit 的 SHA，等於 rewrite history。如果這個 commit 已經 push、別人已經 pull 走，他們的 git 會抓狂。
06 — git stash

stash：手邊改一半，但要切去別處救火

主管丟一個線上 bug 過來,你手邊有 30% 沒 commit 的工作不想丟掉,這時候就用 stash。

最基本的用法

git stash 預設只會暫存已被 Git 追蹤的檔案(也就是 working tree 改過的、或已 stage 的)。新建的 untracked 檔案不會被收進去 —— 這是初學者最容易踩到的坑。如果你想把新檔案也一起暫存,要加上 -u (或 --include-untracked)。

bash 
# 最簡單的寫法,Git 會自動產生 message,例如
#   "WIP on feat/user-profile: a1b2c3d feat: add login"
$ git stash

# 加上 -m 自己取個有意義的名字 (推薦,在 stash list 裡好認)
$ git stash push -m "WIP: profile avatar upload"

# 連 untracked 新檔案也一起收進去 (預設不收!)
$ git stash push -u -m "WIP with new files"
!
-m 不是必須,但強烈建議加 沒加 -m 時 Git 會用「WIP on <branch>: <last commit>」自動產 message,所有 stash 看起來長一個樣。一旦 stash 累積到三、四筆就分不清哪個是哪個。git stash (不帶 push) 跟 git stash push 不加 -m 行為相同,都會用自動 message。

切去別處、回來拿回改動

bash 
# 切去主分支處理線上 bug
$ git checkout main
# ... 修 bug、push、merge ...

# 回來看看 stash 列表
$ git checkout feat/user-profile
$ git stash list
stash@{0}: On feat/user-profile: WIP: profile avatar upload
stash@{1}: On feat/user-profile: WIP with new files

# 拿回最新一筆 stash,並從列表移除
$ git stash pop

apply vs pop:差在會不會刪

兩個指令都會把 stash 的改動套用回 working tree,差別只有一個:pop 套用後會把那筆 stash 從列表刪掉,apply 則保留。

情境用哪個
確定這次套用會成功、之後再也不需要這筆暫存 git stash pop
不確定會不會衝突、想保留一份備份 git stash apply
想把同一筆 stash 套用到多個分支 git stash apply (可以重複用)
bash 
# 套用最新一筆 stash@{0},stash 仍保留在列表
$ git stash apply

# 套用指定編號的 stash
$ git stash apply stash@{2}

# 套用後確認沒問題,再手動刪掉
$ git stash drop stash@{0}

# 一次清掉所有 stash (謹慎使用)
$ git stash clear
i
新手建議養成 apply + drop 的習慣 pop 在套用過程如果發生衝突,stash 還是會被刪掉,你需要靠 git stash show -p stash@{0} 從 reflog 找回來,很麻煩。apply 確認沒問題後再 drop 是更穩的流程。
進階:stash 只暫存一部分檔案 深入
bash 
# 只 stash 已 stage 的部分,working tree 的改動留著繼續寫
$ git stash push --staged -m "only staged"

# 指定要 stash 哪幾個檔案
$ git stash push -m "only profile" src/profile.ts src/profile.test.ts

# 看某筆 stash 改了什麼
$ git stash show -p stash@{0}
07 — git restore

用 restore 從特定 commit 還原檔案

i
日常的取消改動,直接用 VSCode 的 Source Control 面板就好 像是「丟掉 working tree 的改動」、「把已 stage 的取消 stage」、「整個資料夾還原」這類操作,VSCode 的 Source Control 面板都有對應按鈕(改動旁邊的 ↶ Discard Changes、− Unstage 等),不用記指令。terminal 真正會用到的場景,是從某個歷史 commit 把單一檔案還原回來 —— 因為 GUI 對這個操作沒那麼順手。

從特定 commit 還原某個檔案

場景:三天前某筆 commit 的 src/auth.ts 是對的,後面幾筆把它改壞了。你不想 revert 整筆 commit (會連帶其他檔案),只想把這一個檔案的內容還原成那筆 commit 當時的樣子,其他檔案不動。

bash 
# 語法:從指定 commit 還原某個檔案到 working tree
$ git restore --source=<commit> <file>

# 例 1:把 src/auth.ts 還原成 commit a1b2c3d 當時的內容
$ git restore --source=a1b2c3d src/auth.ts

# 例 2:還原成「上一筆 commit 的上一筆」(HEAD~2) 的版本
$ git restore --source=HEAD~2 src/auth.ts

# 例 3:還原成 main 分支當前的版本
$ git restore --source=main src/config.ts

執行後檔案內容就被覆蓋成指定 commit 那個時間點的樣子,接著像平常一樣 git add + git commit 就能把這次「還原」變成新的一筆 commit。整段歷史不會被改寫,team 的人 pull 下來看到的是「你新加了一筆把 auth.ts 改回去的 commit」,完全乾淨。

i
怎麼找到 commit SHA? 用前面教的 Git Graph:在 VSCode 的 Git Graph 面板裡找到目標 commit,右鍵 → Copy Commit Hash to Clipboard,貼到指令裡就好。也可以用 git log --oneline src/auth.ts 只看影響過這個檔案的歷史,SHA 就在每行最前面。
!
會直接覆蓋 working tree,當前未 commit 的改動會不見 執行前先 git status 確認該檔案沒有未保存的修改,或先 git stash 收起來。
