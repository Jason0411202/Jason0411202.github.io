Lesson 01 · Git 基礎篇

Git 日常實務速成

從四區心智模型出發，把每天會用到的 Git 指令一次學會。不是教你 Git 物件原理，是讓你今天就能在團隊裡正常工作。

01 — 心智模型

四個區、一張圖

Git 的所有指令幾乎都在做同一件事：把檔案在四個區之間搬來搬去。理解這張圖，你就懂 80% 的 Git 指令在幹嘛。

圖 1.1 — Git 四區與主要流向
i
記住這四個動作就好 編輯 → add 到 Staging → commit 進 Local → push 上 Remote。所有複雜指令都是這條主線的變體。
02 — 每日工作流

每天做事的標準節奏

無論在公司或開源專案，一天的 Git 操作會像下面這樣循環：

1
開始前先同步主分支
避免在過時的程式碼上開發。--rebase 讓本地 commit 接在最新主分支後面，歷史更乾淨。
2
看當前狀態
git status 是 Git 的儀表板，動任何指令前先看一眼。
3
挑要進這次 commit 的改動
git add <file> 一個一個加比 git add . 安全很多，能避免把 debug 用的 console.log 一起送進去。
4
提交
git commit -m "..."。message 寫法之後 Lesson 06 會專門講。
5
推上去
第一次推某個分支用 git push -u origin <branch> 設定上游，之後直接 git push 就好。
bash 
# 1. 同步主分支
$ git checkout main
$ git pull --rebase

# 2. 開新分支做事
$ git checkout -b feat/user-profile

# 3. 寫程式 ... 之後看狀態
$ git status

# 4. 挑檔案 stage
$ git add src/profile.ts src/profile.test.ts

# 5. commit
$ git commit -m "feat(profile): add user profile page"

# 6. 推上去 (第一次推這個分支要 -u)
$ git push -u origin feat/user-profile

add 的三種選法

指令行為什麼時候用
git add file.ts加單一檔案知道自己要的是哪幾個檔案 (推薦)
git add -p逐塊互動式選擇同一檔案內想拆成兩個 commit 時
git add .當前目錄全部確認過 status 沒有 debug 殘留時才用
!
不要養成 git add . + commit -am 的習慣 這條黃金組合是業界 PR 被退件的常見原因 —— 你會把暫時 print、TODO 註解、本地實驗檔一起 commit 上去。git add -p 是真正的進階使用者習慣。
03 — 看歷史

看 commit 歷史的四個工具

git log — 看時間線

原始的 git log 太囉嗦。實務上幾乎都用這個別名：

bash 
# 一行一個 commit + 分支圖 + 全部分支
$ git log --oneline --graph --all --decorate

# 設成永久別名 (推薦)
$ git config --global alias.lg "log --oneline --graph --all --decorate"

# 之後就只要打
$ git lg

git show — 看單個 commit 的完整內容

bash 
$ git show a1b2c3d                # 看那個 commit 改了什麼
$ git show HEAD                   # 看最新的 commit
$ git show HEAD~3                 # 看 3 個 commit 之前那個

git blame — 看每一行是誰改的

不是用來找戰犯，是用來找「為什麼這行寫成這樣」的線索。

bash 
$ git blame src/auth.ts             # 整個檔案
$ git blame -L 42,55 src/auth.ts    # 只看 42 ~ 55 行

git log -p — 看歷史 + 每筆 diff

bash 
$ git log -p --follow src/auth.ts   # 這個檔案歷年來的所有改動 + 內容
i
--follow 為什麼重要 檔案被改名後，預設的 git log 只看得到改名後的歷史。--follow 會追溯改名前的內容，看到完整故事。
04 — 看差異

diff 的四種模式

Git 有四個區，所以 diff 自然有四種比法。能熟練切換是進階使用者的標誌。

指令比的是用在哪
git diff Working Tree ↔ Staging 「我改了什麼還沒 stage？」
git diff --staged Staging ↔ HEAD 「commit 前再看一次這次到底要送什麼」
git diff HEAD Working Tree + Staging ↔ HEAD 「我跟最後一次 commit 的全部差異」
git diff main..feat/x 分支 vs 分支 「我這條 PR 跟 main 差了什麼」
i
--staged vs --cached 兩者完全等價，--staged 是新版的別名。看到任何文章混用不必擔心。
05 — 修補最近的改動

剛 commit 完發現少改一個地方

不要再開一個 fix typo 的 commit，用 --amend 補進上一筆：

bash 
# 修了檔案後
$ git add src/forgot.ts

# 補進上一個 commit (message 不變)
$ git commit --amend --no-edit

# 想改 message 也可以
$ git commit --amend -m "feat(profile): add user profile page with avatar"
!
已經 push 出去的 commit 不要 amend --amend 會改變 commit 的 SHA，等於 rewrite history。如果這個 commit 已經 push、別人已經 pull 走，他們的 git 會抓狂。只有自己分支沒推出去前才能 amend。已 push 後仍要 amend 的話，請改用 git push --force-with-lease (Lesson 02 會說明)。
06 — 暫存

stash：手邊改一半，但要切去別處救火

主管丟一個線上 bug 過來，你手邊有 30% 沒 commit 的工作不想丟掉，這時候就用 stash。

bash 
# 把當前所有沒 commit 的改動暫存起來 (連未 stage 的都收)
$ git stash push -m "WIP: profile avatar upload"

# 切去主分支處理線上 bug
$ git checkout main
# ... 修 bug、push、merge ...

# 回來，把暫存的改動拿回來
$ git checkout feat/user-profile
$ git stash list
stash@{0}: On feat/user-profile: WIP: profile avatar upload

$ git stash pop                 # 拿回來並從 stash 列表移除
# 如果不確定要不要拿回來，用 git stash apply (套用但不刪)
進階：stash 只暫存一部分檔案 / 包含 untracked 檔 深入 
# 只 stash 已 stage 的部分
$ git stash push --staged -m "only staged"

# 連 untracked 新檔案也一起收 (預設不收)
$ git stash push -u -m "with untracked"

# 指定要 stash 哪些檔案
$ git stash push -m "only profile" src/profile.ts
07 — 取消改動

取消改動：用新的 restore，不要再用 checkout --

Git 2.23 (2019) 後，把以前 checkout 一個指令做太多事拆成兩個：

想做什麼新指令 (推薦)舊指令 (避免)
放棄 working tree 的改動 git restore <file> git checkout -- <file>
把已 stage 的取消 stage (改動還在) git restore --staged <file> git reset HEAD <file>
切分支 git switch <branch> git checkout <branch>
i
為什麼新指令值得學 checkout 既能切分支又能還原檔案，新手常常不小心打錯就把檔案救不回來。restore 和 switch 各司其職，誤操作風險小很多。新團隊現在都建議用新版。
✗舊寫法
git checkout -- src/auth.ts 一個指令切分支也是 checkout，還原檔案也是 checkout，新人一直搞混。
✓新寫法
git restore src/auth.ts 指令名稱直接告訴你它在做什麼。
08 — .gitignore

絕對不能 commit 進去的東西

每個專案都該在第一個 commit 就放好 .gitignore，避免下面這些東西不小心進入版本控制：

✗絕對不能進去
  • .env / .env.local — 含 API key 與密碼
  • *.pem / id_rsa — 私鑰
  • node_modules/ — 數萬個檔案，靠 lock 檔還原即可
  • __pycache__/ / *.pyc — Python 編譯快取
  • dist/ / build/ — 編譯產出
  • .DS_Store / Thumbs.db — 作業系統垃圾
  • .idea/ / .vscode/ (團隊不共用時)
✓應該進去
  • .env.example — 列出有哪些變數但不寫值
  • package-lock.json / pnpm-lock.yaml 等鎖檔
  • .gitignore 本身
  • .editorconfig / .prettierrc 等共用設定
  • 原始碼、測試、文件

從 GitHub 官方範本起手

不用自己一行一行寫，github/gitignore 倉庫收錄了主流語言/框架的官方 .gitignore 範本，直接複製過來再改。

已經不小心 commit 進去 .env 怎麼辦？ 救命

兩件事必須馬上做：

  1. 立刻把所有可能洩漏的 key、密碼換掉。Git 歷史上的東西就算 force push 移除，也可能已被人 clone / fork / scrape。
  2. 用 git-filter-repo (官方推薦工具，取代舊的 filter-branch) 從歷史中移除該檔案。

細節請見 GitHub Docs：Removing sensitive data from a repository。
