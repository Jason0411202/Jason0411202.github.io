Lesson 01 · Git 基礎篇

Git 日常實務速成

從四區心智模型出發，掌握每天會用到的 Git 指令。本課不深入 Git 內部物件原理，重點放在能立即應用於團隊協作的實務操作。

01 — 總覽

四個區、一張圖

Git 的多數指令都在做同一件事：讓檔案在四個區之間流動。掌握這張圖，後續絕大多數指令的行為都能對應到其中某一段流向。

圖 1.1 — Git 四區與主要流向
i
核心流程僅四個動作 編輯 → add 到 Staging → commit 進 Local → push 上 Remote。後續學到的進階指令，多半都是這條主線的延伸或變體。
02 — 基本操作

每天做事的標準節奏

無論在公司或開源專案，一天的 Git 操作會像下面這樣循環：

1
開始前先同步主分支
避免在過時的程式碼上開發，先把遠端最新進度同步下來再開始作業。
2
確認當前狀態
git status 是 Git 的儀表板，執行任何指令前都應先檢視當前狀態。
3
挑選要納入這次 commit 的改動
git add <file> 逐檔指定比 git add . 安全許多，能避免把 debug 用的 console.log 一併提交。
4
提交
git commit -m "..."。commit message 應遵循下一節介紹的 Conventional Commits 格式。
5
推送到遠端
首次推送新分支時使用 git push -u origin <branch> 設定上游分支，之後僅需執行 git push 即可。
bash 
# 1. 同步主分支
$ git checkout main
$ git pull

# 2. 開新分支做事
$ git checkout -b feat/user-profile

# 3. 寫程式 ... 之後看狀態
$ git status

# 4. 挑檔案 stage
$ git add src/profile.ts src/profile.test.ts

# 5. commit
$ git commit -m "feat(profile): add user profile page"

# 6. 推上去 (第一次推這個分支要 -u)
$ git push -u origin feat/user-profile

add 的三種選法

指令行為什麼時候用
git add file.ts加單一檔案知道自己要的是哪幾個檔案 (推薦)
git add -p逐塊互動式選擇同一檔案內想拆成兩個 commit 時
git add .當前目錄全部確認過 status 沒有 debug 殘留時才用

執行 git add -p 後，Git 會將改動切成多個「hunk」（程式區塊），逐塊詢問是否 stage，輸出形式如下：

output 
diff --git a/src/auth.ts b/src/auth.ts
@@ -10,3 +10,5 @@ export function login(user) {
   if (!user.password) {
     throw new Error("password required");
   }
+  if (!user.email) {
+    throw new Error("email required");
+  }
(1/2) Stage this hunk [y,n,q,a,d,s,e,?]?

檢視該區塊改動後，輸入單一字母並按 Enter 即可。常用選項如下：

  • y (yes)：stage 此區塊。
  • n (no)：跳過此區塊，不 stage。
  • s (split)：將此區塊拆成更小的單位再次詢問，適用於「同一區塊內既有想保留、也有想排除的內容」。
  • e (edit)：開啟編輯器手動挑選要 stage 的行——保留所需的行，刪除不需要的行後存檔。
  • a (all) / d (done)：一次接受或略過剩餘所有 hunk。
  • q (quit)：立即結束。? 可隨時呼叫說明。

實務上最常使用的是 y、n、s 三個鍵。例如同一檔案中同時包含「正式功能」與「順帶修正的 typo」，可透過 git add -p 先 stage 功能部分並 commit，剩下的 typo 再單獨 commit，得到兩個職責清楚的 commit。

03 — Commit Message 規範

Conventional Commits：業界事實標準

commit message 不僅是給自己看的紀錄，更是團隊協作、code review、產生 changelog 的重要依據。Conventional Commits 1.0 為當前的事實標準，幾乎所有自動化工具皆採用此格式。基本格式如下：

commit 
<type>(<scope>): <subject>

<body 詳細說明，可省略>

<footer 例如 Closes #123>

解構一個範例：

feat(auth): switch session storage to httpOnly cookies

之前用 localStorage 存 token 容易被 XSS 偷走，
改用 httpOnly cookie + CSRF token 機制。

Closes #421
  • type：feat 表示新功能
  • scope：(auth) 表示影響範圍（可選）
  • subject：switch session storage to httpOnly cookies——50 字以內、祈使句、首字小寫、結尾不加句點
  • body：詳細說明 why（而非 what）
  • footer：相關 issue 連結或其他 metadata

常用的 type

Conventional Commits 規格僅強制 feat 與 fix，其他 type 來自 Angular Commit Convention 的延伸，業界普遍採用：

type用途
feat新功能
fixbug 修補
docs純文件改動
style格式、縮排、分號等不影響邏輯的調整
refactor重構（不新增功能、不修補 bug）
perf效能優化
test新增或修改測試
build建置系統或依賴變動
ciCI 設定變動
chore其他維運事項（升級 dep、調整 config）
revert還原某筆 commit
i
subject 撰寫要點 ① 採用祈使句（add 而非 added）——如同對 git 下達指令
② 首字小寫，結尾不加句點
③ 控制於 50 字以內，較長內容置於 body
④ 自我檢視：當此訊息出現在 changelog 中供使用者閱讀時，是否易於理解？
04 — git blame

看 commit 歷史的兩個工具

Git Graph (VSCode 擴充套件) — 視覺化分支與 commit

原生的 git log 在 terminal 中輸出冗長且不易閱讀。實務上多數開發者會在 VSCode 安裝 Git Graph 擴充套件，透過視覺化介面檢視歷史。

  • 安裝後可從 VSCode 左下角狀態列或 Source Control 面板上方點選 View Git Graph，整個倉庫的分支結構一目了然。
  • 點選任一 commit：右側會顯示 commit message、作者及異動檔案清單，點選檔案即可直接檢視 diff。
  • 於 commit 上按右鍵：可直接執行 checkout、cherry-pick、revert、reset 等操作，無須記憶指令。
  • 分支合併、rebase、tag 的關係以線條繪製呈現，較 git log --graph --all 直觀許多。

git blame — 檢視每一行的修改紀錄

此指令的用途並非追究責任，而是查找「為何此行如此撰寫」的歷史線索。

bash 
$ git blame src/auth.ts             # 整個檔案
$ git blame -L 42,55 src/auth.ts    # 只看 42 ~ 55 行

輸出的每一行包含四個欄位，由左至右依序為 commit SHA、作者、提交時間、(行號) 該行內容：

output 
^a1b2c3d (Alice  2024-08-12 14:32:01 +0800  42) export function login(user) {
e4f5a6b7 (Bob    2025-02-03 09:11:45 +0800  43)   if (!user.email) {
e4f5a6b7 (Bob    2025-02-03 09:11:45 +0800  44)     throw new Error("email required");
^a1b2c3d (Alice  2024-08-12 14:32:01 +0800  45)   }

取得目標行的 SHA 後，執行 git show e4f5a6b7 即可檢視當時整筆 commit 的 message 與異動內容，將線索串接起來。SHA 開頭帶 ^ 者（如上例的 ^a1b2c3d）代表此行自專案最初的 commit 即存在且未曾修改；未帶 ^ 者（如 e4f5a6b7）則為後續某個 commit 新增或修改的內容。

05 — git diff

比較分支差異

單檔或工作區的差異，於 VSCode 的 Source Control 面板點選檔案即可檢視，日常較少使用 git diff。在 terminal 中真正會用到的情境，是比較兩個分支的整體差異——例如送出 PR 前確認自己的分支與 main 之間的具體變動。

指令比的是用在哪
git diff main..feat/x 分支 vs 分支 「我這條 PR 跟 main 差了什麼」

解讀 diff 輸出

典型的 diff 輸出格式如下：

output 
diff --git a/src/auth.ts b/src/auth.ts
index 1a2b3c4..5d6e7f8 100644
--- a/src/auth.ts
+++ b/src/auth.ts
@@ -10,6 +10,8 @@ export function login(user) {
   if (!user.password) {
     throw new Error("password required");
   }
-  return authenticate(user);
+  if (!user.email) {
+    throw new Error("email required");
+  }
+  return authenticate(user.email, user.password);
 }
  • diff --git a/... b/...：被比較的檔案路徑，a/ 為舊版、b/ 為新版。
  • index 1a2b3c4..5d6e7f8 100644：兩側的內部物件 SHA 與檔案權限（100644 表示一般檔案），日常閱讀無須關注。
  • --- a/... / +++ b/...：標示新舊檔案；後續以 - 開頭的行對應舊檔、+ 開頭的行對應新檔。
  • @@ -10,6 +10,8 @@：稱為 hunk header，意指「舊版第 10 行起的 6 行，對應新版第 10 行起的 8 行」；其後緊接該函式或區塊的 context（例如 export function login(user) {）。
  • - 開頭：被刪除的行；+ 開頭：新增的行；無符號的行：未變動，僅作為上下文方便閱讀。
06 — git --amend 參數

剛 commit 後發現遺漏的修改

不必另開一筆 fix typo 的 commit，使用 --amend 將遺漏的改動併入上一筆：

bash 
# 修了檔案後
$ git add src/forgot.ts

# 補進上一個 commit (message 不變)
$ git commit --amend --no-edit

# 想改 message 也可以
$ git commit --amend -m "feat(profile): add user profile page with avatar"
!
已 push 出去的 commit 不應 amend --amend 會改變 commit 的 SHA，等同於 rewrite history。若該 commit 已被推送、且其他成員已 pull 至本地，他們的本地歷史將與遠端對不上，後續同步會出現衝突。
i
確需推送 amend 後的 commit？請使用 --force-with-lease 個人使用的 PR 分支經 amend 後，push 會被 Git 擋下（因為本地與遠端的歷史對不上）。git push --force 雖能強制推送，但會無條件覆寫遠端——若同事同期亦對該分支推送過內容，將被覆蓋且無任何警告。改用 git push --force-with-lease 較為安全：它會比對「遠端當前位置」與「你上次 fetch 時的位置」，一致時才推送，否則直接拒絕並提示。此為 force push 的安全版本，建議養成預設使用的習慣——後續 Lesson 02 介紹 rebase 改寫歷史時亦會用到。
07 — git stash

stash：暫存未完成的工作，先處理其他事項

當前分支上有尚未 commit 的進度，但臨時被指派去處理另一條分支上的工作（例如線上 bug），這種情況就適合使用 stash。

基本用法

git stash 預設僅會暫存已被 Git 追蹤的檔案（即 working tree 已修改或已 stage 的檔案）。新建的 untracked 檔案不會被納入——這是初學者最容易忽略的細節。若需連同新檔案一併暫存，須加上 -u（或 --include-untracked）。

bash 
# 最簡單的寫法,Git 會自動產生 message,例如
#   "WIP on feat/user-profile: a1b2c3d feat: add login"
$ git stash

# 加上 -m 自己取個有意義的名字 (推薦,在 stash list 裡好認)
$ git stash push -m "WIP: profile avatar upload"

# 連 untracked 新檔案也一起收進去 (預設不收!)
$ git stash push -u -m "WIP with new files"
!
-m 並非必填，但強烈建議加上 未加 -m 時，Git 會以「WIP on <branch>: <last commit>」格式自動產生 message，所有 stash 的描述幾乎相同。當 stash 累積至三、四筆時，將難以辨識各自的內容。git stash（省略 push）與 git stash push 在不加 -m 時行為一致，皆會使用自動產生的 message。

切換到其他分支處理事項，再返回取回改動

bash 
# 切換到主分支處理線上 bug
$ git checkout main
# ... 修 bug、push、merge ...

# 返回原分支並檢視 stash 列表
$ git checkout feat/user-profile
$ git stash list
stash@{0}: On feat/user-profile: WIP: profile avatar upload
stash@{1}: On feat/user-profile: WIP with new files

# 取回最新一筆 stash 並從列表移除
$ git stash pop

apply 與 pop 的差異：是否從列表移除

兩個指令皆會將 stash 的改動套用回 working tree，唯一差異在於：pop 套用後會將該筆 stash 從列表移除，apply 則會保留。

情境建議指令
確定本次套用會成功，且不再需要該筆暫存 git stash pop
不確定是否會發生衝突，希望保留備份 git stash apply
需將同一筆 stash 套用至多個分支 git stash apply（可重複使用）
bash 
# 套用前先檢視某筆 stash 的內容（-p 顯示完整 diff）
$ git stash show -p stash@{0}

# 套用最新一筆 stash@{0}，該 stash 仍保留於列表
$ git stash apply

# 套用指定編號的 stash
$ git stash apply stash@{2}

# 套用並確認無誤後，再手動移除該筆 stash
$ git stash drop stash@{0}

# 一次清空所有 stash（請謹慎使用）
$ git stash clear
✓
新手建議養成 apply + drop 的習慣 pop 在套用過程若發生衝突，stash 仍會被移除，需透過 git stash show -p stash@{0} 從 reflog 救回，流程較為繁瑣。先以 apply 套用、確認無誤後再 drop，是更為穩妥的做法。
08 — git restore

使用 restore 從特定 commit 還原檔案

日常的取消改動，直接於 VSCode 的 Source Control 面板操作即可。諸如「丟棄 working tree 的改動」、「取消已 stage 的檔案」、「整個資料夾還原」等操作，Source Control 面板皆有對應按鈕（改動旁的 ↶ Discard Changes、− Unstage 等），無須記憶指令。terminal 真正會用到的情境，是從某個歷史 commit 將單一檔案還原回來——此操作 GUI 較不直接。

從特定 commit 還原某個檔案

典型情境：三天前某筆 commit 的 src/auth.ts 是正確版本，後續幾筆 commit 將其改壞。此時不希望 revert 整筆 commit（會牽動其他檔案），僅希望將該檔案的內容還原為當時那筆 commit 的版本，其他檔案不受影響。

bash 
# 語法：從指定 commit 將某個檔案還原至 working tree
$ git restore --source=<commit> <file>

# 例 1：將 src/auth.ts 還原為 commit a1b2c3d 當時的內容
$ git restore --source=a1b2c3d src/auth.ts

# 例 2：還原為「上上一筆 commit」（HEAD~2）的版本
$ git restore --source=HEAD~2 src/auth.ts

# 例 3：還原為 main 分支當前的版本
$ git restore --source=main src/config.ts

執行後，檔案內容會被覆蓋為指定 commit 當時的版本。接著如同平常一樣執行 git add 與 git commit，即可將此次「還原」記錄為一筆新的 commit。整段歷史不會被改寫，團隊成員 pull 下來看到的是「新增一筆將 auth.ts 還原的 commit」，操作乾淨且可追溯。

i
如何取得 commit SHA？ 使用前述的 Git Graph：於 VSCode 的 Git Graph 面板找到目標 commit，右鍵 → Copy Commit Hash to Clipboard，貼入指令即可。亦可使用 git log --oneline src/auth.ts 僅檢視影響過該檔案的歷史，SHA 顯示於每行開頭。
!
此操作會直接覆蓋 working tree，當前未 commit 的改動將會遺失 執行前先以 git status 確認該檔案沒有未儲存的修改，或先用 git stash 暫存。
