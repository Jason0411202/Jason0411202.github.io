01 — 為什麼需要文件

程式碼回答「怎麼做」，文件回答「為什麼」

大型專案維護幾年後最大的痛點：沒人記得當初為什麼這樣設計。看 git log 只能看到 what，看不到 why。

  • 「這個 mutex 為什麼放這？拿掉好像也沒事？」 → 拿掉一週後線上爆炸
  • 「為什麼選 Postgres 不選 MongoDB？」 → 沒人記得當初討論
  • 「這個 API 為什麼設計成 RESTful 不是 GraphQL？」 → 新人不知道，重做了一遍

解法：RFC 跟 ADR 兩種文件。一個用在「動工前廣泛徵詢意見」，一個用在「決定後永久存檔說明」。

02 — RFC

RFC：Request for Comments

「請求意見」— 在做大決策前，把想法寫成文件邀請眾人 review。傳統來自：

  • IETF (1969 起) — 網路標準 (HTTP、TCP/IP 都是 RFC)
  • Python PEP (Python Enhancement Proposal) — 2000 起
  • Rust RFCs — 2014 起，現代開源專案的 RFC 標竿
  • TC39 proposals — JavaScript 標準變更流程
  • React / Vue — 也都建立了 RFC 流程

典型 RFC 流程

1
寫 RFC 草稿
複製專案的 RFC 模板填好，開 PR 進 rfcs/ 目錄。
2
公開討論期 (通常 2-4 週)
社群成員在 PR 留言討論，作者持續修改。
3
FCP — Final Comment Period
核心團隊提議「進入最終評論期」，再給 7-10 天提異議。
4
Accepted / Rejected
合併 PR (accepted) 或關閉 (rejected)。Accepted 的 RFC 拿到永久編號。
5
追蹤實作 issue
RFC 被接受不代表立刻實作，會開 tracking issue 追進度。
03 — RFC 範本

實務 RFC 範本 (Rust 風格)

# RFC: 改用 httpOnly cookie 取代 localStorage 存 session

- Start Date: 2026-04-15
- Authors: @jason
- RFC PR: #421
- Tracking Issue: (待填)

## Summary
把 session token 從 localStorage 改放到 httpOnly cookie，
配合 CSRF token 機制防止 XSS 偷取 session。

## Motivation
目前用 localStorage 存 token 有兩個問題：
1. 任何 XSS 都能讀取 token (高安全風險)
2. SSR 時無法在伺服器端讀取 (需另外打 API 確認登入狀態)

## Detailed Design
1. 後端在登入成功時 Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax
2. 前端不再讀取 token，改用 /api/me 取得當前使用者資訊
3. 表單提交需附 CSRF token (從 cookie 讀，header 送)
4. 提供遷移路徑：兩種方式並存 30 天，30 天後關閉舊路徑

## Drawbacks
- BREAKING CHANGE，所有依賴前端讀 token 的程式都要改
- 跨子網域共享 session 需重新規劃 cookie domain

## Alternatives Considered
- A. 維持現狀但加 CSP (擋大部分但不是全部 XSS) → 不夠
- B. 使用 SameSite=Strict (但會讓外部連結登入失敗) → 不夠

## Unresolved Questions
- 行動 App 怎麼處理？(目前共用 web 後端 API)
- Service Worker 是否需要存取 token？
i
RFC 寫得好的關鍵 「Alternatives Considered」區塊最重要 — 證明你有想過其他方案才提這個。沒有這段的 RFC 通常會被退回。「Drawbacks」誠實列出缺點也加分，比假裝沒有缺點可信度高。
04 — ADR

ADR：Architecture Decision Record

由 Michael Nygard 在 2011 年提出 (原文)。ADR 是 RFC 的「成果記錄」版本 — RFC 用在動工前討論，ADR 用在「決定後永久存檔說明這次選擇了什麼、為什麼、結果是什麼」。

RFC vs ADR

面向RFCADR
時機決策前 (邀請討論)決策後 (記錄歷史)
長度長 (5-30 頁都有)短 (1-2 頁)
受眾當下參與討論的人未來幾年讀 codebase 的人
狀態變化Draft → Accepted / RejectedProposed → Accepted → (Superseded)
存放位置rfcs/docs/adr/ 或 docs/decisions/
i
實務上很多團隊只用 ADR 不用 RFC 開源 / 大型公開專案：兩者都要 (社群參與)。中小團隊內部專案：只用 ADR 就夠 (討論在 Slack / 會議，記錄在 ADR)。
05 — ADR 範例

Michael Nygard 格式：完整 ADR 範例

展開查看完整 ADR 範例 範本
markdown 
# ADR-007: 使用 PostgreSQL 作為主要資料庫

Date: 2026-04-15

## Status
Accepted

## Context
我們需要為新系統選擇資料庫。需求：
- 預期 100k 筆資料起步，未來 5 年成長到 10M 筆
- 有複雜的 JOIN 查詢 (報表用)
- 需要事務 (transaction) 處理金流相關操作
- 團隊主力工程師有 SQL 經驗
- 部署目標是 AWS / GCP，需要 managed service 選項

候選：
- PostgreSQL
- MySQL
- MongoDB

## Decision
選擇 PostgreSQL 14。

理由：
1. 對複雜 JOIN 與事務的支援最完整
2. 支援 JSON 欄位，未來如有半結構化需求也能處理
3. AWS RDS / Cloud SQL 都有 managed 服務
4. 工程師熟悉度高，學習成本最低
5. 開源且授權友善 (PostgreSQL License)

## Consequences

### 正面
- 不需擔心 ACID 限制
- 大量現成生態 (PostGIS, TimescaleDB) 之後可擴充
- 招募市場容易找懂的人

### 負面
- 比 MongoDB 在純 schema-less 場景靈活度低
- 需要團隊投入 schema migration 流程設計
- 寫操作擴展時需要 read replica + sharding (不是初期問題)

### 後續行動
- 用 Prisma 作為 ORM (見 ADR-008)
- migration 流程用 Prisma Migrate
- 上線前必須建立 backup + PITR (point-in-time recovery)

ADR 的 Status 流轉

Status意義
Proposed提出但還在討論
Accepted確定要做
Deprecated不再推薦但仍在用
Superseded by ADR-X被新的 ADR-X 取代
i
關鍵原則：ADR 永不刪除 就算決策被推翻，也是把舊 ADR 改成 Superseded by ADR-N 而不是刪掉。讓「歷史脈絡可追溯」是 ADR 的核心價值。

工具：adr-tools 提供 CLI 自動建檔；adr.github.io 整理多種變體格式。

07 — Hotfix 流程

線上爆炸：Hotfix 完整流程

1
確認問題影響範圍
先看 monitoring / 錯誤追蹤 (Sentry / Datadog)。是部分使用者還是全炸？影響哪個功能？
2
考慮 rollback 而不是 forward fix
如果剛 deploy 沒多久，回退到上一版往往比現場修快。「forward fix」(往前推新 commit 修) 適合 rollback 不可行時。
3
從 production 對應的 tag / branch 開 hotfix branch
不是從 main 最新開 — main 可能已有未上線的東西。從 v1.4.2 tag 開 hotfix/v1.4.3。
4
最小修改 + 補一個 regression test
只改要修的，不要順便 refactor。補測試確保這個 bug 不會再出現。
5
PR + review + CI 過 + merge + 打 patch tag
即使緊急，review 不能跳過 — 線上爆炸時的 patch 最容易引入新 bug。打 v1.4.3 tag。
6
deploy 到 production
確認 monitoring 顯示問題消失。
7
把 hotfix 合回 main / develop
否則下次 deploy main 時這個 bug 會再回來 (regression)。
8
寫 incident postmortem
Blameless postmortem — 不究責、找根因、列 action items。Google SRE 流程。寫成 Markdown 進 ADR 或 incidents 目錄。
08 — Release Notes

Release Notes：自動產 vs 手寫

自動產 (Auto-generated)

有 Conventional Commits 後，工具能直接從 git log 產出 release notes：

  • semantic-release — 自動 bump 版號 + 產 CHANGELOG.md + 發 GitHub Release
  • changesets — 適合 monorepo
  • release-drafter — 自動 draft GitHub Release，等人 review
  • git-cliff — Rust 寫的高速生成器

手寫 (Curated)

大型 release (主版本 / 旗艦功能) 通常會手寫一份「給人讀的」release notes，跟自動產的 changelog 並存。例如 Next.js 15、Rust 1.75 的 release blog 都是手寫的，會：

  • 強調最重要的 3-5 個改動
  • 提供使用範例
  • 列出 breaking change 與遷移路徑
  • 感謝貢獻者
i
業界做法 通常兩個都要：自動產 CHANGELOG.md 給機器讀 / 想看完整列表的進階使用者；手寫 release blog 給一般使用者。
Multi-repo vs Monorepo / Release Train 模型 深入

Multi-repo：每個專案一個 repo，獨立版本 / 發版。多數小團隊用這個。

Monorepo：多個 package 在同一個 repo。Google 整個公司用一個 monorepo；Vercel 的 turborepo / Nx 是 JS 生態工具。優點：跨 package 重構容易、共享 CI；缺點：版本管理複雜，需要 changesets / nx release 這種工具。

Release Train (Android / Chrome 用)：固定週期 (例如 6 週) 發新版，這個週期內合進去的就有，沒合到的等下班車。優點：節奏穩定、deadline 明確；缺點：feature 沒趕上要等。
