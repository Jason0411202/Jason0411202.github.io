RFC、ADR、Release Management

01 — 為什麼需要文件

程式碼回答「如何做」，文件回答「為何如此」

大型專案維護數年後最大的痛點：無人記得當初的設計理由。git log 僅能呈現 what，無法呈現 why。

  • 「此 mutex 為何放置於此？移除後似乎亦無異常？」→ 移除後一週發生線上事故
  • 「為何選用 Postgres 而非 MongoDB？」→ 無人記得當初的討論內容
  • 「此 API 為何設計為 RESTful 而非 GraphQL？」→ 新成員不知，又重新討論一輪

解法：RFC 與 ADR 兩種文件。前者用於「動工前廣泛徵詢意見」，後者用於「決策後永久存檔說明」。

02 — RFC

RFC：Request for Comments

「請求意見」——於進行重大決策前，將想法整理成文件邀請各方 review。其源頭包含：

  • IETF（1969 年起）——網路標準（HTTP、TCP/IP 等皆為 RFC）
  • Python PEP（Python Enhancement Proposal）——2000 年起
  • Rust RFCs——2014 年起，為現代開源專案 RFC 流程的標竿
  • TC39 proposals——JavaScript 標準的變更流程
  • React、Vue——亦皆已建立 RFC 流程

典型的 RFC 流程

1
撰寫 RFC 草稿
複製專案的 RFC 範本填寫完成，並以 PR 提交至 rfcs/ 目錄。
2
公開討論期（通常 2 至 4 週）
社群成員於 PR 留言討論，作者持續修訂內容。
3
FCP——Final Comment Period
核心團隊提議「進入最終評論期」，再給予 7 至 10 天的異議窗口。
4
Accepted 或 Rejected
合併 PR（accepted）或關閉 PR（rejected）。Accepted 的 RFC 獲得永久編號。
5
追蹤實作 issue
RFC 被接受並不代表立即實作，將另開 tracking issue 追蹤進度。
03 — RFC 範本

實務 RFC 範本（Rust 風格）

# RFC: 以 httpOnly cookie 取代 localStorage 存放 session

- Start Date: 2026-04-15
- Authors: @jason
- RFC PR: #421
- Tracking Issue: （待填）

## Summary
將 session token 由 localStorage 移至 httpOnly cookie，
並配合 CSRF token 機制，避免 XSS 竊取 session。

## Motivation
目前以 localStorage 存放 token 存在兩個問題：
1. 任何 XSS 皆可讀取 token（高安全風險）
2. SSR 時無法於伺服器端讀取（需額外呼叫 API 確認登入狀態）

## Detailed Design
1. 後端於登入成功時設定 Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax
2. 前端不再讀取 token，改以 /api/me 取得當前使用者資訊
3. 表單提交須附帶 CSRF token（自 cookie 讀取，於 header 送出）
4. 提供遷移路徑：兩種方式並存 30 天，30 天後關閉舊路徑

## Drawbacks
- BREAKING CHANGE，所有依賴前端讀 token 的程式皆需調整
- 跨子網域共享 session 須重新規劃 cookie domain

## Alternatives Considered
- A. 維持現狀並加上 CSP（可擋下多數但非全部的 XSS）→ 不足
- B. 採用 SameSite=Strict（但外部連結登入將失敗）→ 不足

## Unresolved Questions
- 行動 App 如何處理？（目前共用 web 後端 API）
- Service Worker 是否需要存取 token？
i
撰寫優質 RFC 的關鍵 「Alternatives Considered」區塊最為重要——證明已評估過其他方案後才提出此提案。缺少此區塊的 RFC 通常會被退回。「Drawbacks」誠實列出缺點亦是加分項，較刻意忽略缺點更具說服力。
04 — ADR

ADR：Architecture Decision Record

由 Michael Nygard 於 2011 年提出（原文）。ADR 是 RFC 的「成果記錄」版本——RFC 用於動工前的討論，ADR 用於「決策後永久存檔，說明此次選擇了什麼、為何如此、後續結果為何」。

RFC 與 ADR 的比較

面向RFCADR
使用時機決策前（邀請討論）決策後（記錄歷史）
篇幅較長（5 至 30 頁）較短（1 至 2 頁）
受眾當下參與討論的成員未來數年閱讀 codebase 的成員
狀態變化Draft → Accepted 或 RejectedProposed → Accepted →（可能 Superseded）
存放位置rfcs/docs/adr/ 或 docs/decisions/
i
實務上許多團隊僅採用 ADR，不使用 RFC 開源或大型公開專案：兩者皆需（涉及社群參與）。中小型團隊的內部專案：僅採用 ADR 即可（討論於 Slack 或會議進行，結論記錄於 ADR）。
05 — ADR 範例

Michael Nygard 格式：完整 ADR 範例

展開查看完整 ADR 範例 範本
markdown 
# ADR-007: 採用 PostgreSQL 作為主要資料庫

Date: 2026-04-15

## Status
Accepted

## Context
新系統需選擇資料庫。需求如下：
- 預期 100k 筆資料起步，未來 5 年成長至 10M 筆
- 包含複雜的 JOIN 查詢（用於報表）
- 需要事務（transaction）處理金流相關操作
- 團隊主力工程師具備 SQL 經驗
- 部署目標為 AWS 或 GCP，需具備 managed service 選項

候選方案：
- PostgreSQL
- MySQL
- MongoDB

## Decision
選擇 PostgreSQL 14。

理由：
1. 對複雜 JOIN 與事務的支援最為完整
2. 支援 JSON 欄位，未來如有半結構化需求亦可處理
3. AWS RDS 與 Cloud SQL 皆提供 managed 服務
4. 工程師熟悉度高，學習成本最低
5. 開源且授權友善（PostgreSQL License）

## Consequences

### 正面影響
- 無須擔心 ACID 限制
- 豐富的現有生態系（PostGIS、TimescaleDB），未來可擴充
- 招募市場上具備此技能的人才較多

### 負面影響
- 於純 schema-less 場景下，靈活度低於 MongoDB
- 團隊需投入心力設計 schema migration 流程
- 寫入擴展時需要 read replica 與 sharding（非初期重點）

### 後續行動
- 採用 Prisma 作為 ORM（見 ADR-008）
- migration 流程採用 Prisma Migrate
- 上線前須建立 backup 與 PITR（point-in-time recovery）

ADR 的 Status 流轉

Status意義
Proposed已提出但仍在討論中
Accepted已確認執行
Deprecated不再推薦，但仍在使用中
Superseded by ADR-X被新的 ADR-X 取代
i
關鍵原則：ADR 永不刪除 即使決策被推翻，亦應將舊 ADR 改為 Superseded by ADR-N，而非直接刪除。確保「歷史脈絡可追溯」為 ADR 的核心價值。

工具：adr-tools 提供 CLI 自動建檔；adr.github.io 彙整多種變體格式。

07 — Hotfix 流程

線上事故：Hotfix 完整流程

1
確認問題影響範圍
先檢視 monitoring 與錯誤追蹤系統（Sentry、Datadog）。受影響的是部分使用者還是全部？影響哪些功能？
2
優先考慮 rollback 而非 forward fix
若 deploy 後不久即發現問題，回退至上一版往往較現場修補更為迅速。「forward fix」（推送新 commit 修補）適用於 rollback 不可行的情況。
3
自 production 對應的 tag 或 branch 切出 hotfix branch
不應自 main 最新狀態切出——main 可能已包含尚未上線的內容。建議自 v1.4.2 tag 切出 hotfix/v1.4.3。
4
最小修改並補上 regression test
僅修改必要部分，避免一併進行 refactor。補上測試以確保此 bug 不再復發。
5
PR、review、CI 通過、merge、打 patch tag
即使情況緊急，review 不應略過——緊急修補階段的 patch 最容易引入新的 bug。打上 v1.4.3 tag。
6
部署至 production
確認 monitoring 顯示問題已消除。
7
將 hotfix 合併回 main 或 develop
未合併回主分支，下次 deploy main 時該 bug 將再次出現（regression）。
8
撰寫 incident postmortem
Blameless postmortem——不究責、找出根因、列出 action items。源於 Google SRE 流程。可撰寫為 Markdown 並置入 ADR 或 incidents 目錄。
08 — Release Notes

Release Notes：自動產生與人工撰寫

自動產生（Auto-generated）

採用 Conventional Commits 後，工具可直接由 git log 生成 release notes：

  • semantic-release——自動 bump 版號、產生 CHANGELOG.md、發布 GitHub Release
  • changesets——適用於 monorepo
  • release-drafter——自動 draft GitHub Release，等待人工 review
  • git-cliff——以 Rust 撰寫的高速生成器

人工撰寫（Curated）

大型 release（主版本或旗艦功能）通常會撰寫一份「面向人類讀者」的 release notes，與自動產生的 changelog 並存。例如 Next.js 15、Rust 1.75 的 release blog 皆為人工撰寫，內容包含：

  • 強調最重要的 3 至 5 項改動
  • 提供使用範例
  • 列出 breaking change 與遷移路徑
  • 感謝貢獻者
i
業界做法 通常兩者並用：自動產生 CHANGELOG.md 供機器讀取與進階使用者檢視完整列表；人工撰寫 release blog 供一般使用者閱讀。
Multi-repo、Monorepo 與 Release Train 模型 深入

Multi-repo：每個專案獨立一個 repo，版本與發版皆獨立進行。多數小型團隊採用此模式。

Monorepo：多個 package 共用同一個 repo。Google 整家公司使用單一 monorepo；Vercel 的 turborepo 與 Nx 為 JS 生態的相關工具。優點：跨 package 重構容易、共享 CI；缺點：版本管理較為複雜，需仰賴 changesets 或 nx release 等工具。

Release Train（Android、Chrome 採用）：以固定週期（例如 6 週）發布新版，週期內合併進來的功能將納入此版，未及完成者則等待下一班次。優點：節奏穩定、deadline 明確；缺點：未趕上週期的 feature 須等待下一週期。
