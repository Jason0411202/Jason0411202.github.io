Code Review 文化與技巧

Code review 不是「揪錯」，是知識傳遞 + 風險控管。這堂教你怎麼當好 reviewer 與好 author，以及業界共通的留言規範。

01 — 為什麼要 review

Code review 的真正目的

很多人以為 review 是來「找 bug」的。其實找 bug 只是副產物。Google Engineering Practices 對 review 目的的官方說法：

  1. 維護程式碼品質 — 不只正確，還要可讀、可維護
  2. 知識共享 — 至少有 2 個人懂這段程式碼
  3. 建立團隊規範 — 透過討論累積一致風格
  4. 傳承 — 新人看 review 學資深者的思考
i
Reviewer 不是把關者，是合作者 Reviewer 跟 author 是同一隊的，目標都是讓程式碼變好。如果用「審判 vs 被審判」心態做 review，整個團隊文化就崩了。
02 — Reviewer 心法

當 Reviewer 該注意什麼

Reviewer 的優先順序 (Google Style Guide)

  1. 設計 — 整體架構與這次改動的取捨對嗎？
  2. 功能性 — 行為符合需求嗎？邊界情況處理了嗎？
  3. 複雜度 — 有沒有 over-engineer？讀的人能理解嗎？
  4. 測試 — 有測試嗎？測試本身寫得好嗎？
  5. 命名 — 變數 / 函式 / 檔案命名清楚嗎？
  6. 註解 — 註解解釋「為什麼」(WHY) 而非「做什麼」(WHAT)？
  7. 風格 — 符合 style guide？(這個應該是 linter 自動處理)

Reviewer 速度

Google 的內部標準：收到 review 請求後 1 個工作天內要給回應。即使做不完整 review，也要先 ack「收到，X 點前回覆」。

!
不要拖延 review PR 卡在 review 越久，author 越痛苦：① 衝突越來越多 ② 上下文記憶流失 ③ 後面工作被擋住。review 不是「我有空再做」的事，是團隊輸出速度的關鍵環節。

找不到問題該怎麼做？

Reviewer 不需要「至少挑一個毛病」。如果整份 PR 看起來都好，直接 approve 並寫一句肯定的話就好。Google 內部規定：「如果一個 reviewer 覺得整份 CL (change list) 至少跟現有 codebase 一樣好，就應該 LGTM (Looks Good To Me)」。

03 — Author 禮儀

當 Author 怎麼讓自己 PR 順利通過

1
PR 越小越好
理想 PR：< 400 行 diff。超過 1000 行的 PR review 品質會急速下降，建議拆成多個。
2
寫好 PR 描述
背景 / 改動 / 怎麼測 / 是否 breaking — Lesson 04 的範本。
3
先自己 review 一遍
送出前自己看一次 GitHub diff 介面，常會發現 debug print、TODO、命名問題。
4
回覆每一個 comment
不要靜默改完就 push。每條 comment 至少回 Done / Won't fix because... / Discussed with X offline。
5
不同意時 push back，但禮貌
Reviewer 也會錯。有理由就提出來討論，不是無條件接受所有建議。
6
避免被 review 後再大改
review 通過後再大改範圍，整個流程要重來，等於浪費 reviewer 的時間。
i
不要把 review 當成個人攻擊 Reviewer 是針對「程式碼」不是「你」。即使留言用詞犀利，把焦點拉回 — 這段程式碼有什麼可以更好的？大多數時候你會發現意見很合理。
04 — 留言規範

Conventional Comments — 用 prefix 表達意圖

同樣是 review 留言，下面這兩個的訊號完全不同：

✗含糊
這裡名字不太好

Author 不知道這是 must-fix 還是建議？要改成什麼？必要嗎？

✓明確
nit: handle → handleSubmit 會更明確，可選

nit: = 雞蛋裡挑骨頭、可選；author 知道不必修也能合併。

業界共通的 prefix

Prefix意思範例
blocking: 必須改，否則不能合併 blocking: 這會讓 production 上的 user 看到 stack trace
suggestion: 建議改，author 可以選擇接受 suggestion: 用 Map 取代 if-else 鏈會更好維護
nit: (nitpick) 雞蛋裡挑骨頭、純風格 nit: 這個變數可以早一行宣告
question: 提問，不是要你改 question: 這個 timeout 設 5 秒的理由是？
thought: 想到的點，純分享 thought: 之後可能可以抽成 hook
praise: 稱讚 (鼓勵 reviewer 也寫這種!) praise: 這段抽象做得很乾淨 👍
i
praise 留言其實很重要 Code review 文化不是只批評。看到好的設計記得說出來 — author 知道自己做對了什麼，整個團隊 morale 也會好。研究顯示「正面 review 留言」跟「員工留任率」高度相關。
05 — 三種 review 結論

Approve / Request changes / Comment 怎麼選

GitHub PR review 有三個按鈕，各自意義：

選項效果什麼時候用
Approve +1 通過票，CI / 設定到位後可合併 看完了沒問題；或只剩 nit 不影響合併
Request changes 擋住 merge 直到 author 解決並再次 review 有 blocking 問題，沒改不能合
Comment 留下意見但不投票 路過想說兩句、無法完整 review、提問
!
Request changes 要慎用 有些團隊文化把 request changes 當成「強烈反對」。如果只是 nit 級別的問題，用 Approve + 內含 nit comment 比較好 — 表示「這些是建議，可以採納可以不採納，但不擋你合」。Request changes 留給真正會破壞東西的問題。
06 — Suggested changes

GitHub Suggested Changes — 一鍵接受改動

GitHub 留言時用一個特殊 code block 寫建議的修改，author 就能在網頁上直接 commit：

markdown 
```suggestion
const handleSubmit = async (event) => {
  event.preventDefault();
  await onSubmit(formData);
};
```

Author 看到後會出現一個按鈕「Add suggestion to batch」或「Commit suggestion」，按下去就直接 push 一個 commit，不用本地操作。

i
什麼時候用 suggestion 最好 Typo、命名建議、簡單重構 — 反正比文字描述更精準。複雜邏輯不適合用 suggestion，因為 author 應該理解後自己寫，不是無腦接受。
07 — CODEOWNERS

CODEOWNERS：自動指派 reviewer

大型專案會在 repo 根目錄或 .github/ 放一個 CODEOWNERS 檔，定義「哪個目錄改動時自動找誰 review」：

codeowners 
# 預設所有改動都要這個團隊 review
*       @your-org/core-team

# 前端目錄找前端團隊
/web/   @your-org/frontend
/mobile/ @your-org/mobile

# 安全相關必須找安全團隊
/auth/  @your-org/security
*.pem   @your-org/security

# 文件可以任何 docs 團隊成員
/docs/  @your-org/docs

當 PR 改到對應目錄，GitHub 會自動加那些人 / 團隊到 reviewer。配合 branch protection 的 Require review from Code Owners，可以強制必須由相關專家 review 才能合併。

進階：CODEOWNERS 語法陷阱 深入
  • 後寫的優先 — 跟 .gitignore 一樣，下面的規則覆蓋上面的
  • 路徑開頭的 / 表示 repo 根目錄 — /auth/ 只匹配根的 auth/，不匹配 src/auth/
  • 沒有開頭 / 表示任何位置 — auth/ 會匹配 src/auth/ 也會匹配 tests/auth/
  • 個人 owner 必須有寫權限 — 沒寫權限的人不能成為 code owner
Async vs Sync review、Pair review、Review fatigue 深入

Async review (大多數)：reviewer 自己有空時看，留 comment，author 改完再請 reviewer 看。適合分散團隊。

Sync review：兩人坐一起 (或視訊) 一行一行討論。適合複雜 / 高風險改動，一次解掉所有疑問。

Pair review：兩人一起寫程式 + review，適合新人 onboarding。

Review fatigue 對策：① 一次 review 上限 60 分鐘 / 400 行 ② 大 PR 拆小 ③ reviewer rotation 不要永遠同幾個人 ④ 自動化 (linter / CI) 處理機械性檢查，讓人類 review 集中在真正重要的東西。
