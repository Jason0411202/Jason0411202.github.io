Code Review 文化與技巧

Code review 並非「挑錯」，而是知識傳遞與風險控管。本課將介紹擔任 reviewer 與 author 的角色職責，以及業界共通的留言規範。

01 — 為什麼要 review

Code review 的核心目的

許多人誤以為 review 的主要任務是「找 bug」。實際上找 bug 只是副產物。Google Engineering Practices 對 review 目的的官方定義如下：

  1. 維護程式碼品質——不僅追求正確性，更須具備可讀性與可維護性
  2. 知識共享——確保至少有 2 位成員理解該段程式碼
  3. 建立團隊規範——透過討論累積一致的撰寫風格
  4. 傳承——新人透過 review 過程學習資深成員的思考方式
i
Reviewer 是合作者而非把關者 Reviewer 與 author 站在同一陣線，目標皆為提升程式碼品質。若以「審判」心態進行 review，將破壞團隊協作文化。
02 — Reviewer 心法

Reviewer 的工作要點

Reviewer 的審查優先順序（Google Style Guide）

  1. 設計——整體架構與此次改動的取捨是否合理？
  2. 功能性——行為是否符合需求？邊界情況是否已處理？
  3. 複雜度——是否存在 over-engineering？是否便於後續閱讀？
  4. 測試——是否包含測試？測試本身的撰寫品質如何？
  5. 命名——變數、函式、檔案命名是否清晰？
  6. 註解——註解是否解釋「為什麼」（WHY）而非「做什麼」（WHAT）？
  7. 風格——是否符合 style guide？（此項應交由 linter 自動處理）

Reviewer 的回應時效

Google 的內部標準：收到 review 請求後須於 1 個工作天內給予回應。即使無法完成完整 review，也應先回覆「已收到，將於 X 時前回覆」。

!
避免延遲 review PR 卡在 review 階段時間越長，對 author 的影響越大：① 衝突日益累積；② 上下文記憶流失；③ 後續工作受阻。review 並非「有空再處理」的工作，而是影響團隊產出速度的關鍵環節。

未發現問題時的處理方式

Reviewer 並無「至少挑出一個問題」的義務。若整份 PR 內容良好，直接 approve 並附上一句肯定的留言即可。Google 內部規範：「若 reviewer 認為整份 CL（change list）至少與現有 codebase 同等優秀，即應給予 LGTM（Looks Good To Me）」。

03 — Author 禮儀

Author 提升 PR 通過率的做法

1
PR 規模越小越佳
理想 PR 規模：低於 400 行 diff。超過 1000 行的 PR，review 品質將顯著下降，建議拆分。
2
完整撰寫 PR 描述
包含背景、改動內容、測試方式、是否為 breaking change——可參考 Lesson 04 的範本。
3
送出前先自我 review
送出前於 GitHub diff 介面再次檢視，往往能發現 debug print、TODO 與命名問題。
4
回覆每一條 comment
勿在無回覆下逕自修改後 push。每條 comment 至少回覆 Done、Won't fix because... 或 Discussed with X offline。
5
不同意時提出討論，並保持禮貌
Reviewer 亦可能有誤。具備合理依據時應提出討論，而非無條件接受所有建議。
6
避免 review 通過後又大幅改動
review 通過後若再大幅改動範圍，整個流程須重新進行，將造成 reviewer 的時間浪費。
i
勿將 review 視為個人攻擊 Reviewer 針對的是「程式碼」而非「個人」。即使留言用詞較為直接，仍應將焦點拉回——此段程式碼是否仍有可改進之處？多數情況下，建議皆具合理性。
04 — 留言規範

Conventional Comments——以 prefix 表達意圖

同樣是 review 留言，以下兩種寫法傳達的訊息截然不同：

✗模糊
這個命名不太好

Author 無從判斷這是必須修正還是僅為建議？應改為什麼？是否為合併的必要條件？

✓明確
nit: handle → handleSubmit 會更明確，非必要

nit: 表示細節建議、可選；author 即可判斷不修改亦能合併。

業界共通的 prefix

Prefix含義範例
blocking: 必須修正，否則不應合併 blocking: 此寫法會使 production 上的使用者看到 stack trace
suggestion: 建議修改，author 可自行決定是否採納 suggestion: 以 Map 取代 if-else 鏈將更易於維護
nit: （nitpick）細節建議，純風格層面 nit: 此變數可提前一行宣告
question: 提問，並非要求修改 question: 此處將 timeout 設為 5 秒的理由為何？
thought: 想法分享，無修改要求 thought: 後續或可抽成 hook
praise: 稱讚（鼓勵 reviewer 多加使用） praise: 此處的抽象設計相當乾淨
✓
praise 留言不可或缺 Code review 文化不應僅止於批評。看到良好的設計時應主動表達——author 得以了解自身做得正確的部分，亦能提升整體團隊士氣。研究顯示，正面的 review 留言與員工留任率呈現高度相關。
05 — 三種 review 結論

Approve / Request changes / Comment 的選用

GitHub PR review 提供三個按鈕，各自的意義如下：

選項效果適用情境
Approve +1 通過票，CI 與設定到位後即可合併 檢視後並無問題；或僅剩 nit 級別建議，不影響合併
Request changes 阻擋 merge，直至 author 處理完畢並再次 review 存在 blocking 問題，未修正前不應合併
Comment 留下意見但不投票 偶然瀏覽提出建議、無法完整 review、純粹提問
!
Request changes 應審慎使用 部分團隊文化將 request changes 視為「強烈反對」。若僅為 nit 級別的問題，建議使用 Approve 並附上 nit comment——傳達「以下為建議，可採納或不採納，但不阻擋合併」之意。Request changes 應保留予確實會造成破壞性影響的問題。
06 — Suggested changes

GitHub Suggested Changes——一鍵接受改動

於 GitHub 留言時使用特殊的 code block 撰寫建議修改內容，author 即可於網頁上直接 commit：

markdown 
```suggestion
const handleSubmit = async (event) => {
  event.preventDefault();
  await onSubmit(formData);
};
```

Author 將看到「Add suggestion to batch」或「Commit suggestion」按鈕，點擊後即可直接 push 一筆 commit，無須本地操作。

i
suggestion 的適用時機 適合用於 typo、命名建議、簡單重構等場景，較文字描述精確。複雜邏輯不建議使用 suggestion，因為 author 應在理解後自行撰寫，而非無條件採納。
07 — CODEOWNERS

CODEOWNERS：自動指派 reviewer

大型專案會在 repo 根目錄或 .github/ 下放置一個 CODEOWNERS 檔，用於定義「特定目錄變動時自動指派由誰 review」：

codeowners 
# 預設所有改動皆由此團隊 review
*       @your-org/core-team

# 前端目錄指派前端團隊
/web/   @your-org/frontend
/mobile/ @your-org/mobile

# 安全相關必須由安全團隊 review
/auth/  @your-org/security
*.pem   @your-org/security

# 文件由 docs 團隊任一成員 review 即可
/docs/  @your-org/docs

當 PR 改動對應目錄時，GitHub 將自動將相關人員或團隊加為 reviewer。搭配 branch protection 的 Require review from Code Owners，可強制要求由相關領域專家 review 後方能合併。

進階：CODEOWNERS 的語法注意事項 深入
  • 後寫者優先——與 .gitignore 相同，下方規則會覆蓋上方規則
  • 路徑開頭的 / 代表 repo 根目錄——/auth/ 僅匹配根目錄下的 auth/，不會匹配 src/auth/
  • 未加開頭 / 代表任意位置——auth/ 將同時匹配 src/auth/ 與 tests/auth/
  • 個人 owner 須具備寫入權限——無寫入權限者不得成為 code owner
Async 與 Sync review、Pair review 與 Review fatigue 深入

Async review（最常採用）：reviewer 於可用時段自行檢視並留下 comment，author 修改後再請 reviewer 複查。適合分散式團隊。

Sync review：雙方共處（或視訊）逐行討論。適合複雜或高風險的改動，可一次釐清所有疑問。

Pair review：兩人共同撰寫並進行 review，適用於新人 onboarding。

Review fatigue 的緩解措施：① 單次 review 控制在 60 分鐘或 400 行以內；② 將大 PR 拆解為較小的 PR；③ 進行 reviewer rotation，避免長期集中於少數成員；④ 透過自動化工具（linter、CI）處理機械性檢查，讓人工 review 聚焦於真正重要的部分。
