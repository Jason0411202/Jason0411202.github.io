Lesson 08 · 自動化與測試篇

CI/CD、Hooks、Lint

人為操作難免遺漏 lint、測試或格式檢查。將這類例行任務交由機器執行——從本地 git hook 到雲端 CI/CD 的完整自動化鏈。

01 — CI/CD 區別

Continuous Integration / Delivery / Deployment

三者皆稱為 CI/CD，但意涵不同——屬於連續性的三個階段：

名稱意義具體執行內容
Continuous Integration 持續整合 每次 push 或 PR 自動執行測試與 lint，確保 main 永遠處於綠燈狀態
Continuous Delivery 持續交付 CI 通過後自動將 build 產物推送至 staging 或 artifact 倉庫，但 production 部署仍須人工觸發
Continuous Deployment 持續部署 連同 production 部署亦自動化——合併進 main 後即直接上線
i
三者的關係 CI 為 CD 的前提。CD 又分為 Delivery（須人工觸發部署）與 Deployment（合併 main 即自動上線）。Deployment 風險最高，需要最強的 CI 防護。多數團隊會先建立 CI，再推進至 Continuous Delivery，最後才導入 Continuous Deployment。
02 — GitHub Actions

GitHub Actions 結構

GitHub 內建的 CI/CD 平台，對開源專案提供近乎無限的免費額度。設定檔置於 .github/workflows/*.yml。

圖 9.1 — Workflow / Job / Step 三層結構

四個關鍵概念

  • Workflow——一份 YAML 檔代表一個 workflow，定義由什麼事件觸發（on: push）
  • Job——workflow 內的執行單位，預設並行執行；可透過 needs: 設定依賴順序
  • Step——Job 內依序執行的步驟，可為 shell 指令或 action
  • Action——可重複使用的 step（例如 actions/checkout@v4），可於 Marketplace 取得
03 — 範例 workflow

典型 lint → test → build 完整範例

yaml 
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck

  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: ['18', '20', '22']
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm test --coverage
      - uses: codecov/codecov-action@v4

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - uses: actions/setup-node@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
✓
數項業界慣例 ① install --frozen-lockfile 確保 CI 使用的依賴版本與 lock 檔完全一致；② 透過 cache: 'pnpm' 加速依賴安裝；③ 以 matrix 執行多個 Node 版本，確保跨版本相容；④ action 永遠固定主版號（@v4），避免使用 @latest，否則上游的 breaking change 將自動套用至 CI。
04 — Branch Protection

Branch Protection Rules

具備 CI 但未設定保護規則時，成員仍可強制推送 main 或略過 review。應於 Settings → Branches 設定保護規則：

規則效果
Require pull request before merging禁止直接 push main，所有變更須透過 PR
Require approvals（1+）須至少 N 人 approve 後才能 merge
Dismiss stale approvals on new push有新 commit 進入後，先前的 approve 失效，須重新 review
Require review from Code Owners動到 CODEOWNERS 指定的目錄時須由該 owner approve
Require status checks to passCI 必須通過才能 merge——可指定須通過的 check 項目
Require branches to be up to datePR 必須 rebase 至最新 main 後才能合併（避免 fast-forward 後產生問題）
Require linear history禁止 merge commit，強制使用 squash 或 rebase merge
Require signed commits強制 GPG 或 SSH 簽名（高安全等級環境）
Do not allow bypassing連 admin 亦不得跳過——僅於確實受阻時暫時關閉
!
未設定 branch protection 等同於未建立 CI 若開發者可強制推送、可略過 CI 失敗、可繞過 review，CI 與 review 流程僅淪為「禮貌建議」。Branch protection 將「應做的事」轉化為「必做的事」。
05 — Linter / Formatter

Linter 與 Formatter 的差異

  • Linter——檢查程式碼的邏輯與風格問題，例如未使用的變數、不安全的型別、複雜度過高等
  • Formatter——僅處理排版，例如引號、縮排、換行、行寬等

各語言對應工具

語言LinterFormatter
JavaScript / TypeScriptESLintPrettier（或以 Biome 整合兩者）
PythonRuff（新）/ Pylint / Flake8Ruff format / Black
Gogolangci-lintgofmt / goimports（內建）
RustClippyrustfmt
JavaCheckstyle / SpotBugsgoogle-java-format / Spotless
i
Ruff 為 Python 生態的新主流 以 Rust 撰寫，速度較 Pylint 快 10 至 100 倍。單一工具即可取代 Flake8、isort、pyupgrade 與 Black 的功能。2024 年後新建立的 Python 專案多採用此工具。
EditorConfig——跨編輯器的基礎排版一致性 輔助工具

於 repo 根目錄置入 .editorconfig，所有支援的 IDE 與編輯器將自動套用：

editorconfig 
root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true

[*.md]
trim_trailing_whitespace = false

不取代 Prettier 或 rustfmt，但可確保「未安裝 formatter 的成員」至少能維持基本縮排一致。

06 — Git Hooks

Git Hooks：在本地 commit 前先攔截不合格內容

Git 內建 hooks（.git/hooks/）可於特定事件時執行 script。然而 .git 不納入版本控制——成員 clone 時不會帶入 hook。實務上採用第三方工具管理 hook：

三大方案比較

工具生態優點
Husky + lint-staged Node.js JS 生態的事實標準；可僅針對改動的檔案執行 lint（lint-staged）
pre-commit Python（跨語言適用） 支援所有語言；hook 以單份 YAML 定義即可跨 repo 共用
Lefthook Go（跨語言適用） 新世代工具，支援並行執行多個 hook，速度更快

Husky + lint-staged 範例

bash 
# 1. 安裝
$ pnpm add -D husky lint-staged
$ pnpm exec husky init

# 2. 將 .husky/pre-commit 內容調整如下
$ cat .husky/pre-commit
pnpm exec lint-staged

# 3. 於 package.json 新增 lint-staged 設定
json 
// package.json
{
  "lint-staged": {
    "*.{ts,tsx,js,jsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    "*.{json,md,yml}": [
      "prettier --write"
    ]
  }
}
i
為何採用 lint-staged 而非全量 lint 舊有 codebase 動輒上千個檔案，每次 commit 全量 lint 需耗時數十秒，影響開發流暢度。lint-staged 僅針對本次改動的檔案執行 lint，可於 1 秒內完成，並能逐步使 main 維持整潔。
為何不直接使用 Git 原生 .git/hooks/？ 原因
  • 不會隨 repo 一同 clone——其他成員 clone 後即缺少 hook
  • 缺乏跨平台處理——Windows 與 Unix shell 行為不同
  • 缺乏 hook 管理機制——同一事件無法同時掛載多個 hook
  • 缺乏版本控制——hook 變更後其他成員無從得知

因此 Husky、pre-commit、Lefthook 等工具皆採用「將 hook 寫於 repo 內，並自動安裝至 .git/hooks/」的方式，解決上述所有問題。

07 — commitlint

commitlint：攔截不符規範的 commit message

Lesson 01 介紹的 Conventional Commits 規範如何於團隊中切實落實？可透過 commitlint 於 commit-msg hook 自動驗證：

bash 
$ pnpm add -D @commitlint/cli @commitlint/config-conventional

# 建立 commitlint.config.js
$ echo "export default { extends: ['@commitlint/config-conventional'] };" > commitlint.config.js

# 新增 husky commit-msg hook
$ echo 'pnpm exec commitlint --edit "$1"' > .husky/commit-msg

後續若有成員嘗試以 fix bug commit：

output 
$ git commit -m "fix bug"

⧗   input: fix bug
✖   subject may not be empty [subject-empty]
✖   type may not be empty [type-empty]

✖   found 2 problems, 0 warnings
ⓘ   Get help: https://github.com/conventional-changelog/commitlint

husky - commit-msg script failed (code 1)
✓
本地與 CI 雙重把關 本地 hook 於 commit 當下攔截；CI 端再以 wagoid/commitlint-github-action 於 PR 階段檢查。本地阻擋多數情況，CI 則處理「跳過 hook 強行推送」的例外。
