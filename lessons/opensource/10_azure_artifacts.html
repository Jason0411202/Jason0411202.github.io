Azure Artifacts 套件治理

Azure Artifacts 為 Azure DevOps 的套件管理 hub。本課介紹 Feed 設計、六種支援的套件格式（npm / NuGet / Maven / Python / Cargo / Universal）、Upstream Sources 自動快取機制、Views 三段式 promotion 流程，以及 Retention Policy。

01 — 為什麼需要

為什麼企業需要內部套件 feed

直接從 npmjs.com、PyPI、Maven Central 拉套件雖然方便，卻有以下風險：

  • 供應鏈風險——惡意套件、typosquatting、被入侵的維護者帳號（如 xz utils 事件）
  • 授權合規——某些 license（GPL 等 copyleft）禁止用於商業產品；需在引入前審核
  • build reproducibility——公共 registry 上的套件可能被 unpublish（npm 著名的 left-pad 事件）
  • 網路延遲與成本——大型 monorepo 每天數百次 build，全走外網拉取效率不佳
  • 地端 / VNet 內 build——若 build agent 位於封閉網路，無法直接連到公網

內部 feed 解決上述問題——build agent 一律從內部 feed 取套件，feed 統一管理「哪些版本可用、哪些被禁用」。

02 — Feed 概念

Feed:組織內的套件儲存庫

Feed 是 Azure Artifacts 的最基本單位——可視為一個獨立的 package registry，內部可同時存放多種套件格式。Feed 的 scope 有兩種：

Scope說明適用
Project-scopedFeed 隸屬於某個 Project；URL 含 project 名稱單一 Project 自用
Organization-scopedFeed 跨整個 Organization 共用共用 SDK、共用工具型套件，多 Project 引用

Feed 內的權限

  • Reader——可下載套件
  • Collaborator——可從 upstream 拉取套件至 feed
  • Contributor——可直接 publish 套件
  • Owner——管理權限與設定
i
常見的 Feed 切分策略 ① 每個 Project 一個 feed（隔離）；② 共用「core libraries」org-scoped feed + 各 Project 各自 feed（混合）；③ 全公司一個 org-scoped feed（集中管理但權限細粒度需細心）。多數企業採策略 ②。
03 — 套件格式

支援的六種套件格式

單一 feed 即可同時容納以下六種：

格式用戶端工具典型用途
npmnpm / pnpm / yarnJavaScript / TypeScript 套件
NuGetnuget / dotnet.NET 套件
Mavenmvn / gradleJava / Kotlin / Scala 套件
Pythonpip / twinePython wheel / sdist
CargocargoRust crate
Universal Packagesaz artifacts universal任意大型二進位檔（ML model、container layer cache、build artifact）
i
Universal Packages 的定位 並非主流 ecosystem 已有的標準格式（如 OCI image）；而是「無對應 package manager 的二進位想要進入 feed」的萬用容器，例如機器學習模型 weights、編譯後的 binary tarball。容量上限大、可分塊傳輸，但用戶端僅有 az CLI。

引用內部 feed:以 npm 為例

於專案根目錄置入 .npmrc：

ini 
; @my-org scope 走內部 feed,其餘走公共 registry
@my-org:registry=https://pkgs.dev.azure.com/<org>/<project>/_packaging/my-feed/npm/registry/
always-auth=true

本機開發者執行 npm install 時，Azure Artifacts CLI 會引導完成 token 寫入；CI 環境則由 npmAuthenticate@0 task 自動處理。

04 — Upstream Sources

Upstream Sources:自動快取公共套件

Feed 可掛載多個 upstream source——當 client 索取的套件不存在於 feed 中時，Azure Artifacts 會：

  1. 依 feed 設定的順序，逐一向 upstream（npmjs.com、PyPI、Maven Central 或其他 feed）查詢
  2. 找到後自動下載至 feed 並回傳給 client
  3. 後續對相同版本的請求一律從 feed 取出，無須再外連
圖 12.1 — Upstream sources 自動快取流程

查詢順序

  1. 直接 publish 至 feed 的套件
  2. 過去已從 upstream 快取的套件
  3. 依 upstream 設定順序，逐一查詢
✓
掛上 upstream 後 build 行為對開發者透明 開發者照常 npm install lodash，看不見「來自 npmjs 或來自 feed cache」的差異。但對組織而言：① 套件被永久保留，npmjs 即使該版本被 unpublish 也仍可下載；② 可基於 audit log 追溯哪些公共套件被使用；③ 後續整網路斷網仍可繼續 build。
!
Maven snapshot 不支援 upstream Maven snapshot 版本（如 1.0.0-SNAPSHOT）的語意是「同一版本號可被重複覆蓋」，與 Azure Artifacts 套件 immutable 的設計衝突。upstream 僅支援已發佈的正式版本，snapshot 須直接連 Maven Central 或自行 publish 至 feed。
05 — Views 與 Promotion

Views:套件三段式 promotion 流程

每個 feed 預設提供三個 view，作為套件「成熟度」的階段：

View用途進入方式
@Local所有 publish 與 upstream 快取的套件，預設皆進此 viewpublish 或 upstream 拉取後自動進入
@Prerelease標記為「測試中」的版本由 maintainer 從 @Local promote 至此
@Release已驗證、可生產使用的穩定版本由 maintainer 從 @Prerelease 或 @Local promote 至此

引用 view 的 URL 後綴差異：

@Local      → .../_packaging/my-feed/npm/registry/
@Prerelease → .../_packaging/my-feed@Prerelease/npm/registry/
@Release    → .../_packaging/my-feed@Release/npm/registry/

典型應用：開發環境引 @Prerelease 嘗試新版本；生產環境鎖死 @Release，只看見經 QA 簽核過的版本。promote 動作不複製檔案，僅在 metadata 上加標籤，零儲存成本。

i
三 view 設計取代了傳統「dev / staging / prod feed」三套** 舊作法須維護三個獨立 feed 並手動同步；單一 feed + 三 view 既節省儲存又避免同步漂移，且權限可細分到 view 級別（例：開發者可 promote 至 @Prerelease，但 promote 至 @Release 須 release manager 同意）。
06 — Pipeline 發佈

於 Pipeline 中發佈套件

內部套件多由 CI pipeline 自動發佈。以 npm 為例：

yaml 
steps:
  - task: UseNode@1
    inputs: { version: '20.x' }

  # 1. 寫入認證至 ~/.npmrc
  - task: npmAuthenticate@0
    inputs:
      workingFile: .npmrc           # repo 中已存在的 .npmrc

  # 2. 安裝、build、test
  - script: npm ci
  - script: npm run build
  - script: npm test

  # 3. 設定版本(以 build number 為 suffix)
  - script: npm version $(Build.BuildNumber) --no-git-tag-version

  # 4. publish 至 @Local view
  - script: npm publish

publish 成功後，套件即出現於 Artifacts → 該 feed → @Local。若同時欲 promote 至 @Prerelease，可於下一個 step 呼叫 az CLI：

bash 
$ az artifacts universal promote \
    --organization https://dev.azure.com/myorg \
    --feed my-feed \
    --name @my-org/utils \
    --version 1.2.3 \
    --view Prerelease
07 — Retention

Retention Policy:控管儲存配額

Azure Artifacts 依儲存量計費；長期累積的舊版本會持續消耗配額。每個 feed 可設定 retention：

設定說明
Maximum versions per package每個套件最多保留 N 版；超出時最舊的被刪除
Days to keep最後一次被下載 N 天後即可被回收
Promoted package protection已 promote 至 @Prerelease 或 @Release 的版本不受 retention 影響，永久保留
✓
典型設定:每套件保留 50 版 + 30 天未被下載即可回收 搭配「promoted package protection」，正式釋出的版本永遠保留，僅大量未被採用的 CI 中間版本會被清掉。如此既可滿足 audit 需求，也避免儲存配額爆炸——後者於大型 monorepo 是常見痛點。
