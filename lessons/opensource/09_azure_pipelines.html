Azure Pipelines YAML 與部署

Azure Pipelines 是 Azure DevOps 的 CI/CD 引擎。本課從 YAML 結構出發，逐步介紹 Trigger、Stage / Job / Step、Service Connection、Variable Group + Key Vault、Environments + Approvals、Templates 與 Deployment Strategies——一份 pipeline.yml 看完即可寫得出來。

01 — YAML vs Classic

為什麼 2026 後一律寫 YAML

Azure Pipelines 歷史上有兩種設定模型：

  • Classic UI Editor——拖拉式介面，設定儲存於 Azure DevOps 內部資料庫
  • YAML Pipelines——設定檔以 azure-pipelines.yml 存放於 repo，隨程式碼版本控制

Microsoft 已宣告所有新功能僅實作於 YAML pipeline。具體列舉如下，這些能力於 Classic 不可用：

  • Pipeline templates 與參數化
  • Environment approvals 與其他 checks
  • Container jobs
  • Pipeline resources（跨 pipeline 觸發）
  • Stages 之間的 dependency expression
!
Classic pipeline 仍可運行，但新建議一律寫 YAML 既有 Classic pipeline 可繼續使用，但若有任何修改機會即建議遷移至 YAML——除了享受新功能，pipeline 設定亦會進入 PR review 流程，與程式碼一同被審視。
02 — YAML 結構

Pipeline / Stage / Job / Step 四層

Azure Pipelines 的 YAML 採四層結構：

圖 11.1 — Pipeline → Stages → Jobs → Steps

多 stage YAML 骨架

yaml 
# azure-pipelines.yml
trigger:
  branches:
    include: [main, release/*]

pool:
  vmImage: ubuntu-latest

variables:
  buildConfiguration: 'Release'

stages:
  - stage: Build
    jobs:
      - job: Compile
        steps:
          - task: UseDotNet@2
            inputs: { version: '8.x' }
          - script: dotnet build --configuration $(buildConfiguration)
          - script: dotnet test --logger trx
          - task: PublishTestResults@2
            inputs: { testResultsFormat: VSTest, testResultsFiles: '**/*.trx' }
          - task: PublishPipelineArtifact@1
            inputs: { targetPath: '$(Build.ArtifactStagingDirectory)', artifact: drop }

  - stage: Deploy_Dev
    dependsOn: Build
    jobs:
      - deployment: DeployToDev
        environment: 'dev'
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureWebApp@1
                  inputs:
                    azureSubscription: 'svc-conn-dev'
                    appName: myapp-dev
                    package: $(Pipeline.Workspace)/drop/**/*.zip

關鍵詞速查

  • job——一般執行單位；於單一 agent 上跑所有 step
  • deployment——特殊 job 類型，指向一個 environment、支援部署策略（runOnce / rolling / canary）
  • task——預封裝步驟，例如 AzureWebApp@1、UseDotNet@2；數字為主版號
  • script / bash / pwsh——直接執行 shell 指令
03 — Triggers

CI Trigger 與 PR Trigger

Pipeline 由「事件」觸發。最常見三種：

觸發類型YAML 區塊典型情境
CI triggertrigger:push 至特定分支即啟動，常用於合併後自動部署 dev
PR triggerpr:建立 / 更新 PR 時啟動，用於 build validation 與 review feedback
Scheduled triggerschedules:cron 規則，例如每晚跑完整 E2E test suite
Pipeline resource triggerresources.pipelines:另一條 pipeline 完成時啟動，例如 build pipeline 成功後跑 deployment pipeline
yaml 
trigger:
  branches:
    include: [main, release/*]
    exclude: [feature/experimental]
  paths:
    exclude: [docs/*, README.md]   # 純文件修改不啟動

pr:
  branches:
    include: [main]

schedules:
  - cron: '0 22 * * *'           # UTC 每晚 22:00
    displayName: Nightly E2E
    branches:
      include: [main]
04 — Agents

Microsoft-hosted vs Self-hosted Agents

Pipeline 須在 agent 上執行。兩種選擇：

類型優點缺點
Microsoft-hosted 無需維護；每次執行皆為乾淨 VM；常見映像（ubuntu-latest、windows-latest、macOS-latest）皆已預裝主流工具鏈 免費方案有月度分鐘上限；無法存取內網資源；環境受限於 Microsoft 提供的映像
Self-hosted 可置於 VNet 內存取內部服務；可預裝特殊工具（GPU driver、商用 SDK）；無分鐘限制 需自行維護 OS patch、agent 版本、磁碟空間；可能殘留前次 build 狀態
i
Managed DevOps Pools 為新中間選項 2025 年推出的 Managed DevOps Pools 提供「self-hosted 的彈性 + Microsoft-hosted 的免維護」——Microsoft 代管 VM 池，可自訂映像並接入 VNet。對中大型企業是常見折衷方案。
05 — Variables 與 Key Vault

Variables、Variable Groups 與 Key Vault 整合

變數有三種來源層級，由高至低優先：

來源定義位置用途
Runtime parameterYAML parameters:Queue pipeline 時人為輸入
YAML inline variableYAML variables:環境無關常數，例如 buildConfiguration: Release
Variable groupPipelines → Library → Variable groups跨 pipeline 共用；可標記為 secret；可連結 Key Vault

連結 Key Vault

機敏資料絕不應寫在 YAML 或 variable group 的文字欄位中。正確做法：將 secret 存於 Azure Key Vault，於 variable group 切到「Link secrets from an Azure key vault as variables」，授權 service connection 後選擇要曝露的 secret 名稱：

設定流程:

1. Azure Portal:建立 Key Vault,放入 secret
   例:secret 名稱 db-connection-string,值 = "Server=...;Password=..."

2. Azure DevOps:Project Settings → Service connections
   新建 Azure Resource Manager 連線,使用 Service Principal 或 Managed Identity

3. Azure DevOps:Pipelines → Library → 新建 Variable group
   開啟「Link secrets from an Azure key vault」
   選 service connection、選 Key Vault、勾選要 expose 的 secret

4. YAML 引用:
   variables:
     - group: db-secrets
   steps:
     - script: echo "$(db-connection-string)"
       env:
         DB_CONN: $(db-connection-string)
!
Key Vault secret 名稱的 dash 會自動轉 underscore Key Vault 中名為 db-connection-string 的 secret 暴露至環境變數時會成為 db_connection_string。引用時兩種寫法都行——YAML $() 語法仍用原始名稱，但若要當作 env var 在腳本內以 $VAR 形式讀，須以 underscore 寫。

為什麼 Key Vault 優於 Variable Group 純 secret

  • 單一來源——secret 在 Key Vault 輪替後，所有依賴的 pipeline 自動取得新值，無須更新
  • Audit trail——Key Vault 內建存取記錄，誰於何時取走哪個 secret 可追溯
  • 細粒度權限——可由 Key Vault access policy 控制不同 pipeline 看到不同 secret
  • 強化保護——可設定 soft-delete、purge protection，避免誤刪
06 — Environments 與 Approvals

Environments、Approvals 與 Checks

Environment 是「部署目標」的邏輯抽象——例如 dev、staging、prod。`deployment` job 透過 environment: 欄位指向之，所有部署紀錄、批准與檢查皆掛於 environment 上。

Pipelines → Environments → 新建「prod」→ Approvals and checks → 新增:

✓ Approvals       :指派 1 至 N 位人員,部署前須點按鈕
✓ Branch control  :只允許 main / release/* 部署本 environment
✓ Business hours  :僅在指定時段內部署(例:平日 09:00–17:00)
✓ Exclusive lock  :同時最多一條 pipeline 部署本 environment
✓ Required template :pipeline 必須 extends 自指定模板(集中安全策略)
✓ Invoke REST API :對外部系統送 webhook,由其回應 pass/fail 決定能否部署
✓ Query work items:檢查相關 work item 是否處於合法狀態
✓
Approvals 與 Checks 不寫於 YAML，而是綁在 Environment 上 這個設計使「誰能部署什麼」由平台團隊集中控管，application team 即使能修改 YAML 亦無法繞過。這是 Azure Pipelines 與 GitHub Actions 重要差異——GitHub 中相當功能須透過 environment + repo settings 拼湊。
07 — Templates

Templates:Pipeline 模組化

大型組織常有「所有應用都該跑安全掃描、製品上傳」之類的標準化需求。Templates 允許將共用片段抽出為獨立檔案：

三種模板用法

用法適用
Includes（steps: - template: foo.yml）把 steps、jobs 或 stages 段落抽出共用
Extends（extends: template: base.yml）父模板提供完整骨架；子 pipeline 僅填入差異參數。常配合 Required Template check 強制使用
Parameters（parameters:）讓 template 接受參數（string / number / object / step list 等）

Extends 模板範例

yaml 
# templates/secure-pipeline.yml(平台團隊維護)
parameters:
  - name: buildSteps
    type: stepList
    default: []

stages:
  - stage: SecurityScan
    jobs:
      - job: scan
        steps:
          - task: SonarCloudPrepare@1
          - task: CredScan@3
          - task: SonarCloudAnalyze@1
  - stage: Build
    jobs:
      - job: build
        steps: ${{ parameters.buildSteps }}

# azure-pipelines.yml(application team 撰寫)
extends:
  template: templates/secure-pipeline.yml
  parameters:
    buildSteps:
      - script: npm ci
      - script: npm run build

配合 Environment 的「Required template」check，可強制所有部署 prod 的 pipeline 必須 extends 自此模板——應用團隊無法跳過安全掃描階段。

08 — Deployment Strategies

三種內建部署策略

`deployment` job 的 strategy: 欄位支援三種內建策略：

策略行為適用
runOnce對所有目標一次部署完畢dev / staging；節點少；可接受短暫 downtime
rolling分批部署，可設 maxParallel: 25%，每批完成後再進下一批VM Scale Set、Kubernetes 多 replica
canary先部署小比例（例 5% / 20% / 50% / 100%），每階段間可暫停觀察 metrics高流量、需 progressive rollout 的 prod 服務

三種策略皆提供四個生命週期 hook：preDeploy、deploy、routeTraffic、postRouteTraffic，外加 on: success 與 on: failure——可在任一階段插入 smoke test 或 rollback 邏輯。

yaml 
- stage: Deploy_Prod
  jobs:
    - deployment: DeployToProd
      environment: 'prod'
      strategy:
        canary:
          increments: [10, 25, 50, 100]
          preDeploy:
            steps:
              - script: ./scripts/pre-deploy-check.sh
          deploy:
            steps:
              - task: KubernetesManifest@1
                inputs:
                  action: deploy
                  strategy: canary
                  percentage: $(strategy.increment)
          postRouteTraffic:
            steps:
              - script: ./scripts/smoke-test.sh
          on:
            failure:
              steps:
                - task: KubernetesManifest@1
                  inputs: { action: reject, strategy: canary }
