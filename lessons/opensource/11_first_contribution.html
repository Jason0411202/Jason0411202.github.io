01 — 找專案

適合貢獻專案的取得管道

來源適用對象
firstcontributions.github.io 初次練習——並非真實貢獻，但可完整體驗 fork → PR → merge 流程
goodfirstissue.dev 依語言彙整各專案的 good first issue 標籤
github.com/topics/good-first-issue GitHub 內建 topic 搜尋
Hacktoberfest 每年 10 月舉辦的活動，提交 4 個合格 PR 即可獲得紀念品
日常使用的工具 最佳起點——日常使用時即累積使用經驗與痛點，可選擇可勝任的 issue 著手
✓
建議從日常使用的工具著手 已具備該工具的領域知識、實際 use case 與優缺點認知，相較於選擇陌生的熱門專案，可顯著縮短入手時間。
02 — 判斷專案

5 分鐘評估一個專案是否值得貢獻

1
檢視 CONTRIBUTING.md 是否清晰
缺少此文件代表流程不明，貢獻過程容易受阻。內容越詳盡越友善。
2
檢視最近 30 天的活動量
觀察 commit、PR 與 issue 的留言頻率。長期無動靜的專案，PR 提交後恐無人處理。
3
檢視 PR 的平均合併時間
點選 Pull requests，檢視近期合併 PR 從開啟至合併的時間。一週內較佳；一個月以上代表等候時間可能較長。
4
檢視 Issue 的回應狀況
最近 10 個 issue 是否有 maintainer 留言？若皆無回應，代表貢獻友善度較低。
5
檢視 CI 通過率
點選 Actions tab。若多數 build 皆失敗，代表專案維護不穩定，自身的 PR 也難以通過。
6
檢視社群氛圍
隨機檢視 5 個 issue 或 PR 的留言。Maintainer 是否禮貌？是否頻繁拒絕新進貢獻者？應排除氛圍明顯不友善的專案。
03 — 授權

授權快速辨識（5 種常見族群）

授權核心限制適用場景
MIT幾乎無限制（僅需保留 copyright）個人或商業皆可自由使用，限制最寬鬆
Apache 2.0類似 MIT，並加上專利條款大型專案（k8s、Apache 系列）
BSD-2 / BSD-3類似 MITBSD 系統與學術界傳統
GPL v2 / v3copyleft——衍生作品亦須以 GPL 開源Linux Kernel、Bash 等
AGPL v3GPL 強化版——連網路服務亦須開源MongoDB（舊版）、Grafana（舊版）
!
商業使用須留意 GPL 與 AGPL 公司專案使用 GPL 函式庫，整個專案可能須一併開源。AGPL 的影響更為廣泛——連內部 SaaS 服務的原始碼也須公開。商業專案多選擇 MIT 或 Apache 2.0 授權的依賴。但此並不影響貢獻 GPL 專案，僅須清楚理解授權的內涵。

若不確定如何為自身開源專案選擇授權：choosealicense.com 是 GitHub 官方提供的決策樹工具。

04 — 必讀檔案

送出 PR 前必讀的 5 個檔案

檔案檢視重點
README.md 專案的目的、開發環境設定方式、使用者群
CONTRIBUTING.md 貢獻流程、PR 命名規則、commit 規範、是否需要 DCO 簽署
CODE_OF_CONDUCT.md 社群行為準則。多數專案採用 Contributor Covenant
SECURITY.md 發現漏洞時的通報方式（Lesson 10）
LICENSE 授權條款（前述已介紹）
i
DCO（Developer Certificate of Origin） 部分專案（Linux Kernel、Docker、CNCF 系列）要求使用 git commit -s 加上 Signed-off-by: footer，以聲明「貢獻者具備提交此段程式碼的權利」。應確認 CONTRIBUTING 中是否有相關規範。
05 — 認領 issue

避免認領 issue 時撞單的方式

看到一個 good first issue 時，如何避免「實作到一半發現他人亦在進行」？

  1. 檢視 issue 留言區——確認尚無人聲明 "I'm working on this"
  2. 檢視 Linked PRs——確認是否已有 draft PR 連結至該 issue
  3. 留言聲明「將著手處理此 issue，預計 N 天內提交 PR」——等 maintainer 確認後再動工
  4. 若一兩天無回應可直接動工——大型專案 maintainer 可能事務繁忙；Open Source 重視行動，付諸實作通常較持續等待更有效益
!
避免過度承諾 若僅為「有興趣或許會嘗試」，不應輕易留下 "I'll work on this" 後即無下文。此舉會使 issue 凍結，他人無法接手。實際動工後才宜認領。
06 — 完整 Checklist

從零至 PR 被合併的完整 Checklist

將前 11 堂的內容彙整為一份可執行的清單，每項皆對應特定 Lesson 編號。

① 起手準備（Lesson 04） 10 分鐘
  • □ 於 GitHub 點選 Fork 按鈕，fork 至個人帳號
  • □ git clone 個人 fork 後的版本
  • □ git remote add upstream <原始 repo>
  • □ git remote -v 確認 origin 與 upstream 皆已設定
② 同步上游（Lesson 04） 2 分鐘
  • □ git checkout main
  • □ git fetch upstream
  • □ git rebase upstream/main
  • □ git push origin main（使 GitHub 上的個人 fork 同步至最新）
③ 切分支並設定本地環境（Lesson 01、09） 約 30 分鐘
  • □ 自最新 main 切出 feature 分支 git checkout -b fix/issue-234
  • □ 依 README 安裝依賴（npm ci、poetry install 等）
  • □ 執行現有測試以確認環境正常（npm test）
  • □ 確認是否需手動安裝 Husky 或 pre-commit——若無自動安裝，須執行 pnpm prepare 或 pre-commit install
④ 動工——TDD 流程（Lesson 07） 主要時間
  • □ 先撰寫一個失敗的測試（RED）
  • □ 以最小幅度的實作通過測試（GREEN）
  • □ 整理程式碼（REFACTOR）
  • □ 重複前述循環，直至功能完成
  • □ 執行全套測試，確認未影響其他部分
⑤ 撰寫 commit（Lesson 01） 5 分鐘
  • □ git status 確認異動的檔案
  • □ git add <file> 逐檔加入（避免 debug 殘留）
  • □ git diff --staged 進行最終檢查
  • □ 採用 Conventional Commits 格式：fix(scope): subject
  • □ Body 清楚說明 why（而非 what）
  • □ Footer 加上 Closes #234
⑥ Push 並開啟 PR（Lesson 04） 10 分鐘
  • □ git push -u origin fix/issue-234
  • □ 於 GitHub 點選 "Compare & pull request"
  • □ 採用標準 PR 描述格式：背景、改動、測試方式、截圖、是否為 breaking
  • □ 連結 issue：Closes #234
  • □ 尚未準備好被 review 時，建立 Draft PR
  • □ 確認 CI 全部通過
⑦ 處理 Code Review（Lesson 05） 數天
  • □ 收到留言後不急於回覆，先完整閱讀所有 comment
  • □ 每條 comment 至少回覆 Done、Won't fix because... 或 Discussed offline
  • □ 不同意時若具備合理依據應提出討論，而非無條件接受
  • □ 修改完成後 push 新 commit（暫不改寫歷史，待 review 通過再整理）
  • □ 所有 comment 處理完畢後留下 "Ready for re-review"
⑧ 處理衝突（Lesson 02、04） 視情況而定
  • □ git fetch upstream
  • □ git rebase upstream/main
  • □ 於 IDE 解決衝突
  • □ git add <files> + git rebase --continue
  • □ git push --force-with-lease（不應使用 --force）
⑨ 合併與完成 5 分鐘
  • □ 等待 maintainer 合併（依 CONTRIBUTING 的政策決定採用 squash、merge 或 rebase）
  • □ Issue 自動關閉（因 PR 描述包含 Closes #234）
  • □ 將本地 fork 的 main 重新同步至 upstream/main
  • □ 刪除個人的 feature branch
  • □ 完成後可將該 PR 納入個人作品集或 LinkedIn 紀錄
07 — 常見陷阱

新手最常見的 8 個錯誤

  1. 對個人 main 執行 force push——嚴重風險，整個分支的歷史將遭破壞。應始終於 feature branch 操作，不應動到 main。
  2. PR 規模過大——1000 行以上的 PR 將難以 review。每個 PR 應對應單一目的，盡量控制於 400 行 diff 以內。
  3. commit 歷史紊亂——fix、asdf、finally working 等粗略的 commit 散布其中。送出 PR 前應以 rebase -i 進行整理。
  4. 未回應 review comment——修改後逕自 push 而未留言，reviewer 無從得知採納哪些建議。每條至少回覆 Done。
  5. 遺忘同步 upstream——數週未同步將導致 PR 衝突堆積。每次切出新分支前皆應先執行 fetch 與 rebase upstream。
  6. 提交未經測試的程式碼——push 前應於本地執行 npm test 與 npm run lint。CI 紅燈將使 reviewer 不願審閱該 PR。
  7. 忽略 CONTRIBUTING 的 commit 規範——部分專案要求 conventional commits，部分要求 DCO sign-off。違反規範將被退件重做。
  8. PR 描述僅寫 "fix"——reviewer 無從得知 PR 的內容。Lesson 04 的標準範本應確實填寫。
08 — 好/壞範例

第一次貢獻雙重對照

Issue 範例

✗不佳的第一個 issue
不能用

缺少重現步驟、環境資訊與「預期 vs 實際」描述。Maintainer 僅能回覆「請補充資訊」。

✓良好的第一個 issue
[Bug] DatePicker placeholder 於 Safari 17.4 顯示為亂碼

包含完整重現步驟、環境資訊、截圖與可能成因的推測。

PR 範例

✗不佳的第一個 PR
title: update
desc:（空白）
commits: fix · WIP · finally · oops

Reviewer 無從得知 PR 內容，將被直接退件。

✓良好的第一個 PR
title: fix(date-picker): handle Safari 17.4 placeholder render
desc: 完整 5 段描述
commits: 1 個結構清晰的 conventional commit

Reviewer 可於數秒內掌握脈絡，直接進入程式碼審視。

09 — 總回顧

11 堂課重點反向問答

Lesson 01：Working Tree、Staging、Local Repo 與 Remote 各為何？
編輯中的檔案 → add 進入 Staging → commit 進入 Local → push 至 Remote。
Lesson 02：rebase 的黃金守則為何？
不應對「已 push 至他人會使用的公開分支」執行 rebase。
Lesson 03：8 人 SaaS 團隊應採用何種分支策略？
GitHub Flow——main 加上短命 feature branch 與 PR。
Lesson 04：origin 與 upstream 的差異為何？
origin 為個人 fork 的版本；upstream 為原始專案。
Lesson 05：Reviewer 僅剩 nit 級問題時應 Approve 或 Request changes？
Approve 並留下 nit comment。Request changes 應保留予實際 blocking 的問題。
Lesson 06：good first issue 標籤的意涵為何？
經 maintainer 評估，範圍適合新人、具備清楚說明、無須熟悉整體 codebase 的 issue。
Lesson 07：TDD 三步驟的順序為何？
Red（撰寫失敗的測試）→ Green（以最小幅度的實作通過測試）→ Refactor（整理）。
Lesson 08：Continuous Delivery 與 Continuous Deployment 的差異為何？
Delivery 為自動部署至 staging，但 production 須人工觸發；Deployment 則為合併 main 後直接部署至 production。
Lesson 09：ADR 的決策被推翻後應如何處理？
將 Status 改為 Superseded by ADR-N，保留所有內容，永不刪除。
Lesson 10：xz utils 後門的最大教訓為何？
開源專案的單一維護者為攻擊者鎖定的目標，需採用多人維護、SBOM 與 artifact signing。
Lesson 11：開始動工前應先閱讀哪 5 個檔案？
README.md、CONTRIBUTING.md、CODE_OF_CONDUCT.md、SECURITY.md 與 LICENSE。
10 — 畢業任務

三個畢業任務（難度遞增）

Level 1——使用 first-contributions 進行練習

前往 firstcontributions/first-contributions，完整走過一次 fork → clone → 修改 → push → PR 流程。雖然僅是於 list 加上個人名稱，但可完整體驗整套流程。約 30 分鐘可完成。

Level 2——修正常用工具 docs 中的 typo

挑選一個日常使用的工具（個人編輯器外掛、公司使用的 npm 套件、Vue / React / Vite 等）。於 docs/ 中查找錯字、過時範例或缺漏的中文翻譯，提交一個小型 PR。重點在於實際擁有「一個被合併的開源 PR」。約 1 至 2 小時可完成。

Level 3——修補一個 good first issue 中的小 bug

於 goodfirstissue.dev 挑選一個語言熟悉且能理解的 small bug。完整走過 TDD、Conventional Commits 與 PR 流程，並回應 review。此為「實質貢獻」的起點。所需時間視 bug 規模而定，約 1 天至 1 週。

✓
三項任務皆完成即代表課程結業 至此已具備加入大型開源專案或於公司投入大型專案的協作能力。後續即為持續累積實戰經驗。持續送出 PR、持續 review 他人的 PR——此為進步最快的途徑。
