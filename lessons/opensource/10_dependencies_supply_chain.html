Lesson 10 · 大型專案實務篇

依賴管理 與供應鏈安全

現代專案中，多數程式碼來自外部依賴。依賴即風險——本課將透過 left-pad 至 xz utils 後門等案例，介紹相應的防禦機制。

01 — 為什麼依賴是風險

自行撰寫的程式碼僅佔 10%，其餘 90% 來自外部

現代 Node.js 專案執行 npm install 後，node_modules 動輒數百 MB、包含上千個套件。Python、Rust、Go 等生態亦類似。每一個依賴都帶來：

  • 未知的 bug——影響 production 環境
  • 未知的安全漏洞——成為攻擊者的入口
  • 未知的維護狀況——作者可能在某日停止維護
  • 傳遞性依賴——直接安裝 10 個，間接帶入 800 個
!
新增依賴前應自問三個問題 ① 此功能自行實作需多久？（耗時極短者不應引入依賴）② 此套件近一年是否活躍？（檢視 GitHub 的 commit 頻率與 issue 回覆狀況）③ 其傳遞性依賴為何？（可用 npm view <pkg> dependencies 檢視）
02 — 真實案例

三起讓全球意識到「依賴即風險」的事件

2016——left-pad 事件

Azer Koçulu 因 npm 與另一家公司之間的商標糾紛，將自身所有套件下架，包括 left-pad（一個僅 17 行的字串填補函式）。結果使 JavaScript 生態系大規模建置失敗——Babel、React、Node 等專案皆受影響。

教訓：避免採用「規模僅有十幾行」的依賴；npm 亦因此引入「unpublish 限制」（72 小時內、無人下載才能撤回）。

2018——event-stream 事件

受歡迎的 npm 套件 event-stream，原作者因無暇維護，將權限交予一位熱心的「貢獻者」。新維護者接手後新增依賴 flatmap-stream，其中暗藏竊取 Bitcoin 錢包私鑰的後門，影響數百萬下載量。

教訓：套件「轉移維護權」屬供應鏈攻擊的高風險點。應於 package.json 中鎖定依賴版本，並仔細檢視自動更新 PR 的 diff 內容。

2024——xz utils 後門（CVE-2024-3094）

近年最嚴重的供應鏈攻擊。攻擊者耗費 2 年偽裝成善意貢獻者，逐步取得 xz utils（Linux 廣泛依賴的壓縮函式庫）的維護權，並於 5.6.0 與 5.6.1 中植入後門，可透過 SSH 遠端執行任意指令。

所幸微軟工程師 Andres Freund 觀察到 SSH 反應延遲 0.5 秒，因而提早發現。若再晚一至兩週、xz 進入 Debian 或 Ubuntu 穩定版，全球 Linux 伺服器皆可能遭入侵。

教訓：開源專案的單一維護者為攻擊者鎖定的目標。社群與政府開始重視「關鍵開源專案應由多人維護」，並推動 SBOM、SLSA 與 artifact signing 等防護機制。

03 — 鎖檔

Lock 檔的意義

缺少鎖檔時，install 取得的依賴版本可能每次皆不同——本機運作正常，CI 上卻發生問題。鎖檔記錄每個套件的精確版本與完整依賴樹。

生態鎖檔備註
npmpackage-lock.jsonnpm 5+ 預設啟用
pnpmpnpm-lock.yaml記錄完整依賴樹與完整性 hash
yarnyarn.lockYarn 1 與 Yarn Berry 格式不同
Python（pip + Pipenv）Pipfile.lock但 pip 本身並未強制鎖定
Python（Poetry）poetry.lock建議以 Poetry 或 uv 取代純 pip
RustCargo.lockLibrary 不應 commit；binary 則應 commit
Gogo.sum記錄 hash 以確保完整性
✓
鎖檔必須納入版本控制 此為 Lesson 01 提及的鐵律。缺少鎖檔等同於團隊成員與每次 CI 執行皆使用不同版本的依賴，bug 將難以重現。

CI 上應使用「frozen」模式

所有套件管理器皆提供「不解析、嚴格依 lock 檔安裝」的旗標：

bash 
$ npm ci                                # 而非 npm install
$ pnpm install --frozen-lockfile
$ yarn install --immutable
$ poetry install --no-update
$ cargo build --locked
04 — 漏洞掃描

主動掃描已知漏洞

各語言生態皆提供官方的漏洞資料庫與 CLI 工具：

生態指令資料來源
npmnpm auditGitHub Advisory Database
pnpmpnpm audit同上
Pythonpip-auditPython Packaging Authority 與 OSV
Rustcargo auditRustSec Advisory Database
GogovulncheckGo Vulnerability Database
跨語言OWASP Dependency-CheckNVD（美國國家漏洞資料庫）
output 
$ npm audit

axios  <1.6.0
Severity: high
SSRF in axios - https://github.com/advisories/GHSA-xxxx-xxxx
fix available via `npm audit fix`
node_modules/axios

3 vulnerabilities (1 moderate, 2 high)
✓
將 audit 納入 CI 於 CI 中執行 npm audit --audit-level=high，high 與 critical 等級的漏洞將使 build 失敗。較低風險（low、moderate）通常不阻擋 build，但保留紀錄。
05 — 自動更新

Dependabot 與 Renovate 的比較

手動追蹤依賴更新過於耗時，業界普遍採用機器人自動開 PR 進行升版：

面向DependabotRenovate
歸屬GitHub 內建（2019 年收購）Mend（前 WhiteSource）開源加 SaaS
設定簡單（單一 yml 檔案）強大但較複雜
PR 策略每個依賴單獨一個 PR可配置批次或群組更新
排程固定間隔（daily、weekly）支援 cron、時區與 quiet hours
支援生態涵蓋主流語言覆蓋更廣（Docker base image、GitHub Actions、Terraform）
適用多數專案使用此選項即可適用於大型 monorepo 或需精細控制的場景

Dependabot 設定範例

yaml 
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    groups:
      eslint:
        patterns: ["eslint*"]
      testing:
        patterns: ["vitest*", "@testing-library/*"]
    open-pull-requests-limit: 5

  # GitHub Actions 亦需更新
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
!
不應未經審視即合併 bot 開的 PR 即使 CI 通過，major version 升級必須檢視 release notes——breaking change 可能無法被 CI 偵測（例如新版改變某 hook 的副作用）。Patch 與 minor 雖較為安全，仍應快速檢視 changelog。
06 — CVE / GHSA

解讀 CVE 與 GHSA 識別碼

系統格式發放單位
CVE CVE-2024-3094 MITRE 與美國 CISA——跨平台的通用識別碼
GHSA GHSA-xxxx-xxxx-xxxx GitHub Advisory——開源套件專用
NPM advisory 數字 ID npm 早期格式（現多已遷移至 GHSA）

CVSS 評分

每個 CVE 皆會給予 0 至 10 的 CVSS（Common Vulnerability Scoring System）分數：

  • 9.0-10.0 Critical——立即處理
  • 7.0-8.9 High——本週內處理
  • 4.0-6.9 Medium——排入近期 sprint
  • 0.1-3.9 Low——視情況評估
i
分數高並不必然影響本專案 例如某漏洞需要「使用者已被釣魚並啟用特定設定」才能利用，CVSS 仍可能評為 7.0，但實際影響極為有限。應檢視完整描述，評估 attack vector 於本專案的使用情境下是否可達，再決定處理優先級。
07 — SECURITY.md

SECURITY.md：負責任的揭露流程

專案上線後，必須讓外界明確知道發現漏洞時的通報對象與方式。否則安全研究員可能直接公開 issue（zero-day），反而使使用者陷入風險。

業界慣例為於 repo 根目錄置入 SECURITY.md：

markdown 
# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 2.x     | :white_check_mark: |
| 1.x     | :white_check_mark: (until 2026-12) |
| < 1.0   | :x:                |

## Reporting a Vulnerability

請不要透過公開 issue 通報安全漏洞。

請改用以下任一方式：

1. GitHub Private Vulnerability Reporting (推薦)
   到 Security 分頁按「Report a vulnerability」
2. Email: security@example.com (PGP key: 0xABCD1234)

我們會在 48 小時內回覆，90 天內發補丁。
依照 [CVD policy](https://...) 處理 — 在補丁發布前不會公開細節。

## Hall of Fame
感謝以下發現過漏洞的研究員：
- @researcher1 (CVE-2024-1234)
- @researcher2 (CVE-2024-5678)
i
採用 GitHub 的 Private Vulnerability Reporting GitHub 於 2022 年推出的功能——前往 repo Settings → Security → Private vulnerability reporting → Enable。研究員可於 web UI 私下提交漏洞，maintainer 於 private fork 修補後再行公開。較 email 流程更為流暢。
08 — SBOM / SLSA

SBOM 與 SLSA——供應鏈安全的下一步

SBOM（Software Bill of Materials）

「軟體物料清單」——列出一份軟體使用了哪些依賴、版本與授權。美國 2021 年行政命令 14028 後，成為政府採購軟體的必要文件。

主流 SBOM 格式：

  • SPDX——Linux Foundation 發起，為 ISO 標準
  • CycloneDX——OWASP 維護，較聚焦於安全議題

產生 SBOM 的方式：syft <repo>（Anchore 開源工具）或各 build tool 的內建支援。

SLSA（Supply-chain Levels for Software Artifacts）

Google 主導的供應鏈安全等級框架，分為四級：

  • SLSA 1——具備自動化建置與可追溯來源
  • SLSA 2——使用 hosted CI（GitHub Actions、GCB）並進行簽名
  • SLSA 3——不可竄改的 build（含 ephemeral runner）
  • SLSA 4——最高等級，包含 two-party review 與可重現建置

Sigstore——開源簽章方案

Sigstore 為 Linux Foundation 的開源 artifact 簽章方案（含 Cosign、Fulcio、Rekor）。npm、PyPI、GitHub Actions 皆已支援。npm 自 2023 年起對熱門套件強制要求 provenance——證明該套件由哪個 CI 與哪個 commit 建置而成。

private registry 與 internal mirror 深入

大型企業會於內部建置 mirror（Verdaccio、Artifactory、Nexus），原因如下：

  • 下載速度較快（內部網路）
  • 僅允許使用核准過的套件（黑名單或白名單）
  • 保留審計紀錄
  • 離線環境亦可使用
  • 避免上游下架的影響（left-pad 事件的後續影響）
版本鎖定策略：caret、tilde 與 exact 細節
"axios": "1.4.5"完全鎖定
"axios": "~1.4.5"允許 1.4.x（僅 patch）
"axios": "^1.4.5"允許 1.x.y（minor 與 patch，預設值）
"axios": "*"任何版本，不應採用

應用程式：採用 ^ 或 ~ 並搭配 lock file 鎖定。Library：通常採用 ^，給予使用者較大彈性。
