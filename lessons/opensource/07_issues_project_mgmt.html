Issue 撰寫 與專案管理

Issue 不是抱怨區，是協作的最小單位。這堂教你寫出工程師願意處理的 issue，並看懂大型專案的標籤、Milestone、Triage 系統。

01 — Issue 三大類

Issue 不只是「報 bug」

大型專案的 issue 通常分三大類，各有不同目的與寫法：

類型用途標籤
Bug Report回報軟體錯誤行為type: bug
Feature Request提議新功能 / 改善type: feature / enhancement
Question / Discussion使用問題、設計討論type: question (許多專案會引導到 GitHub Discussions)
i
Question 該去 issue 還是 Discussions？ GitHub Discussions 是 2020 推出的論壇式功能，適合「使用方法」的提問。Issue 留給「需要 maintainer 採取行動」的事 — 改 code、改 docs、評估 feature。Stack Overflow 也是常見導向。看專案 CONTRIBUTING 怎麼說。
02 — Bug Report

業界通用的 Bug Report 結構

一個能被處理的 bug report，必須包含這幾個區塊：

# 標題：[Bug] 一句話描述問題

## Describe the bug
登出後 token 沒清乾淨，下次登入別的帳號會看到上一個使用者的資料。

## Steps to Reproduce
1. 用 user A 登入
2. 點右上角的「登出」
3. 用 user B 登入
4. 看到首頁顯示 user A 的 profile

## Expected behavior
登出後應該完全清掉所有 user 資料，登入新帳號應顯示新使用者資料。

## Actual behavior
頁面短暫閃過 user A 的 avatar / name，再 refresh 才正常。

## Environment
- OS: macOS 14.5
- Browser: Chrome 124.0.6367
- App version: 1.4.2
- Node: 18.19.0

## Screenshots / Video
(附錄影或截圖)

## Additional context
我猜可能是 React Query cache 沒清。第一次發現是 2026-04-25 上線後。
!
沒有重現步驟 = 無解 Maintainer 不能重現的 bug 就無法 debug。「有時候會壞」是最差的 bug report。提供確切步驟、影片、log、版本資訊，才能進入 triage 流程。
03 — Feature Request

Feature Request 結構

新功能要求要說痛點而不是解法。直接寫「請加 X 功能」很容易被拒，因為 maintainer 看不到使用情境。

# 標題：[Feature] Allow custom date format in <DatePicker>

## Is your feature request related to a problem?
我們的應用要顯示中文日期格式 (民國年)，目前 DatePicker 只支援
ISO 格式，需要在 component 外面額外做轉換。

## Describe the solution you'd like
新增一個 `format` prop 讓使用者傳入 day.js / date-fns 的格式字串：
```tsx
<DatePicker format="民國 YYY 年 M 月 D 日" />
```

## Describe alternatives you've considered
- 用 wrapper component 自己做轉換 (現在的做法，但 onChange 處理麻煩)
- 用 i18n library 整合 (太重)

## Additional context
看到隔壁的 react-aria/Calendar 有類似功能，可以參考。
i
「我願意動手做」會大幅提升被接受機率 在 issue 結尾加一句「我願意送 PR」，maintainer 心中的成本馬上降一半。
04 — 對照

好/壞 Issue 對照

✗糟糕
不能用

無標題、無重現、無環境。Maintainer 只能回「請補充資訊」。

✓好的
[Bug] Logout doesn't clear React Query cache (v1.4.2)

標題告訴你「什麼壞了 + 哪個版本」，maintainer 不用點進去就能初步分類。

05 — Issue Template

Issue / PR 模板：把好習慣自動化

專案根目錄放 .github/ISSUE_TEMPLATE/ 後，使用者按「New Issue」會跳出選單，引導他們填好結構：

yaml 
# .github/ISSUE_TEMPLATE/bug_report.yml
name: 🐞 Bug Report
description: 回報軟體錯誤行為
labels: ["type: bug", "status: triage"]
body:
  - type: textarea
    id: describe
    attributes:
      label: Describe the bug
      description: 簡短描述問題現象
    validations: { required: true }
  - type: textarea
    id: steps
    attributes:
      label: Steps to Reproduce
      value: |
        1.
        2.
        3.
    validations: { required: true }
  - type: input
    id: version
    attributes:
      label: App Version
    validations: { required: true }

PR 模板也類似 — 在 .github/PULL_REQUEST_TEMPLATE.md 寫好結構，使用者開 PR 時會自動帶入。

i
模板省下 80% 的「請補資訊」往返 有了模板，必填欄位空著就不能送出。Maintainer 收到的 issue 質量會大幅提升。
06 — 標籤系統

大型專案的標籤 (Label) 分類法

看一下 Kubernetes 或 VS Code 的 issue 列表，會發現標籤被分成幾個維度，透過 prefix 區隔：

維度常見 prefix範例
類型type:type: bug type: feature type: docs
優先級priority:priority: critical priority: high
狀態status:status: triage status: in-progress status: needs-review
區域area: 或 scope:area: frontend area: api
邀請貢獻(無 prefix)good first issue help wanted
難度difficulty:difficulty: easy
i
good first issue 與 help wanted 是 GitHub 官方推薦標籤 GitHub 會主動把帶有這兩個標籤的 issue 推到「explore」或「first contributions」入口。新人想找專案貢獻時，這是最好的入口。
07 — Milestone

Milestone：把 issue 綁定到 release

Milestone 是 GitHub 內建的「一組 issue/PR 屬於同一個 release」機制。例如建一個 v1.5.0 milestone，把要進這版的 issue 全部 assign 過去，可以一目瞭然進度。

08 — GitHub Projects

GitHub Projects (Projects v2)

2022 年推出的新版 Projects 已成為 GitHub 自家 issue 跨 repo 管理的主力工具。可以用三種視圖看同一份資料：

視圖適用場景
Table看全貌、批次編輯欄位 (像 Notion / Airtable)
Board (Kanban)看流程：Backlog → Todo → In Progress → Done
Roadmap看時間軸：哪些工作壓在哪幾週

Projects 可以加自訂欄位 (Status / Priority / Sprint / Estimate / Assignee)，並且跨 repo聚合 issue。一個 Project 可以同時管 frontend repo + backend repo + docs repo 的 issue。

Linear / Jira 跟 GitHub Projects 的取捨 產品比較
  • GitHub Projects：完全整合 GitHub issue / PR；簡單、免費、不需切換工具；但 sprint planning / 報表能力較弱。
  • Linear：UI / 速度業界標竿，cycle (sprint) 概念內建好，適合產品團隊。需付費。
  • Jira：企業老牌，自訂能力極強但設定複雜，常被工程師討厭。多在大企業 / 受監管產業。

小到中型團隊用 GitHub Projects 通常就夠了。

09 — Triage

Issue Triage 流程

大型專案每天可能收到幾十個 issue，靠 maintainer 一個個處理會炸。Triage (分流) 是把進來的 issue 快速分類處理的流程：

1
標 status: triage
所有新 issue 預設這個狀態，等人來看。
2
驗證重現性
能重現 → 標 status: confirmed；不能 → 留言請 reporter 補資訊，標 needs-info。
3
指派優先級 / 區域 / Milestone
priority: high / area: frontend / 綁到 v1.5.0 milestone
4
如果適合外部貢獻，標 good first issue
明確、範圍小、有重現步驟 → 邀請社群來修。
5
不打算做的 issue 直接 close + 解釋
超過範圍 / 重複 / 設計上不收 → 給明確理由然後 close。讓 backlog 不要爆炸。

Stale bot：自動關閉沒人理的 issue

很多大型專案 (如 React、TypeScript) 用 actions/stale：超過 N 天沒回應的 issue 自動標 stale，再過 M 天自動 close。讓「永遠在那但沒人在乎」的 issue 不會堆積。

!
Stale bot 也有副作用 有些 issue 是真的重要但 maintainer 沒空看，被自動 close 後社群會生氣。所以 stale bot 通常配合白名單 — 標了 pinned / priority: high 的 issue 不會被自動關。
