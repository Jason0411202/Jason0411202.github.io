測試金字塔 與 TDD

測試不是「為了 coverage 數字」，是讓改動安全的安全網。這堂建立你對測試類型、TDD 流程、覆蓋率的正確直覺。

01 — 測試金字塔

Mike Cohn 的測試金字塔 (2009)

金字塔的中心思想：越底層的測試越多、越快、越穩定；越頂層的測試越少、越慢、越容易壞。

圖 8.1 — Mike Cohn 測試金字塔
!
反金字塔 (倒三角) 是反模式 只有 E2E、沒有 unit test = test suite 跑半小時、隨機壞、debug 困難。是新手團隊最常踩的測試陷阱。
02 — 其他形狀

獎盃、蜂巢 — 金字塔不是唯一答案

形狀提倡者核心觀點
金字塔 Mike Cohn (2009) 多 unit、少 E2E (傳統)
獎盃 (Trophy) Kent C. Dodds (2018) 整合測試最重要，加上靜態檢查 (TypeScript) 為基底
蜂巢 (Honeycomb) Spotify Engineering 後端微服務環境下，整合測試 (服務間) 才有最高價值
i
選哪個形狀？看你做什麼 前端：獎盃 (UI 變化快，unit 容易過度耦合 DOM；用 React Testing Library + 整合測試 + Playwright)。
後端 monolith：金字塔仍然是好預設。
後端微服務：蜂巢 — 服務內邏輯測整合，服務間測 contract。
03 — 測試類型

實務常見的測試類型

類型測什麼速度典型工具
Unit 單一函式 / 元件，外部依賴 mock 掉 毫秒 Jest / Vitest / pytest / JUnit
Integration 多個模組串接，例如 API + DB 幾十毫秒 ~ 秒 Supertest / Testcontainers
Contract 服務 API 的契約 (consumer/provider) 秒 Pact / OpenAPI schema
E2E 真的開瀏覽器，從使用者角度測整套流程 秒 ~ 分鐘 Playwright / Cypress / Selenium
Smoke 「冒煙測試」— 部署後跑幾個關鍵 path 確認沒整個炸 秒 少量精選 E2E
Regression 確保以前修好的 bug 沒回來 — 所有上述類型 + 持續累積
04 — TDD

TDD 三步驟：Red → Green → Refactor

由 Kent Beck 在《Test-Driven Development: By Example》(2002) 系統化的方法。一個無限循環：

圖 8.2 — TDD 三步驟循環

實戰範例：實作一個 add 函式

typescript 
// 步驟 1 (RED): 先寫測試
test('add: 兩個數相加', () => {
  expect(add(1, 2)).toBe(3);
});

// 跑測試 → 失敗 (add 還沒實作) ← RED

// 步驟 2 (GREEN): 寫最少實作
function add(a: number, b: number) {
  return a + b;
}

// 跑測試 → 通過 ← GREEN

// 步驟 3 (REFACTOR): 整理程式碼
// 這次太簡單沒得 refactor，但複雜情境會在這步抽函式 / 改命名

// 然後回到步驟 1 寫下一個測試 (例如負數相加)
i
TDD 真正的價值不是「寫測試」 是「先想清楚要做什麼」。寫測試的當下，你被迫思考：「這個函式的 input/output 是什麼？邊界情況是什麼？」這個思考過程比測試本身更值錢。
05 — F.I.R.S.T

好測試的 F.I.R.S.T 原則

Robert C. Martin (Uncle Bob) 提出，每個字母代表一個原則：

字母原則實務含義
Fast快整套 unit test 應該秒殺，否則開發者就不會常跑
Isolated孤立每個測試獨立、測試間不依賴順序、不互相影響
Repeatable可重複同樣 input 永遠同樣結果，不依賴時間、亂數、外部服務
Self-validating自我驗證跑完直接告訴你成功/失敗，不需要人類去看 log 判斷
Timely及時測試應該緊鄰 production code 寫，最好就是 TDD 的 R-G-R
!
flaky test (不穩定測試) 是違反 Repeatable 的元兇 有時候過、有時候掛 — 通常是依賴時間、依賴順序、依賴網路。flaky test 比沒測試更糟 — 它會訓練團隊忽略 CI 紅燈，最後真的 bug 也被當 flaky 重跑。發現 flaky 立刻修，修不好就先 skip + 開 issue 排修。
06 — AAA 模式

AAA — Arrange / Act / Assert

每個測試在結構上明顯分三段，閱讀性大增：

typescript 
test('logout: clears all user state', () => {
  // Arrange — 設定情境
  const store = createStore({ user: { id: 42, name: 'Jason' } });

  // Act — 觸發行為
  store.dispatch(logout());

  // Assert — 驗證結果
  expect(store.getState().user).toBeNull();
  expect(store.getState().token).toBeNull();
});
i
一個測試只測一件事 一個 test 內最好只有一個邏輯上的 assert (可以多個 expect 但都驗證同一件事)。如果你發現要寫第二個 act → assert，那是新測試了。
07 — Coverage

Coverage 80% 的真相

四種 coverage 指標

指標意義
Line程式碼有多少 % 的行被執行過
Statement多少 % 的語句被執行 (一行可能有多個 statement)
Branch多少 % 的分支 (if-else / switch) 走過 — 通常是最重要的指標
Function多少 % 的函式至少被呼叫過一次

為什麼 80% 是常見目標

  • 低於 60% — 太多區域沒被測試保護，重構很危險
  • 60-80% — 健康，重要邏輯都被覆蓋
  • 80-90% — 高品質開源專案的水準
  • 追求 100% — 邊際效益急速下降，且容易出現「為了 coverage 而寫的低價值測試」
!
Coverage 高不代表測試好 下面這個測試 coverage 100%，但毫無價值： 
test('add', () => {
  add(1, 2);  // 沒有 assert，純執行
});
Coverage 是「測試覆蓋多少行」，不是「邏輯正確性多少」。覆蓋率高的同時要看：是否有清楚 assert、是否測了邊界情況、是否有 mutation testing 驗證測試強度。

不該追求高 coverage 的場景

  • Prototype / spike code — 探索性程式碼丟掉就好
  • 純呼叫第三方 API 的 wrapper — 測試只會 mock 自己寫的東西
  • 複雜的 UI 微互動 — 視覺回歸 (visual regression) 比 unit test 有效
Mutation Testing — 測試你的測試 進階

Mutation testing 工具會故意改你的 production code (例如 > 改成 >=、刪掉 !)，然後跑你的測試。如果測試還通過 = 你的測試沒抓到這個 mutation = 測試強度不夠。

JS 工具：Stryker。Java：PIT。Python：mutmut。比 coverage 更精確衡量測試強度，但跑得慢，通常不會每次 CI 都跑。

Property-Based Testing — 用隨機輸入找邊界 進階

傳統測試：你寫 add(1, 2) === 3。Property-based 測試：你定義屬性 ∀ a, b: add(a, b) === add(b, a) (交換律)，框架自動生成數百組隨機輸入測試。

JS：fast-check。Haskell 的 QuickCheck 是這個概念的源頭。發現邊界 bug 的能力遠超人工寫測試。
