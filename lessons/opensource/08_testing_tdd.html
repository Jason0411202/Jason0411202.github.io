Lesson 08 · 自動化與測試篇

測試金字塔 與 TDD

測試的目的並非追求 coverage 數字，而是提供使改動得以安全進行的保護網。本課將建立對測試類型、TDD 流程與覆蓋率的正確認知。

01 — 測試金字塔

Mike Cohn 的測試金字塔（2009）

金字塔的核心理念：越底層的測試數量越多、執行越快、越穩定；越頂層的測試數量越少、執行越慢、越容易失效。

圖 8.1 — Mike Cohn 測試金字塔
!
反金字塔（倒三角）為反模式 僅有 E2E、缺乏 unit test 的後果：test suite 動輒跑半小時、隨機失敗、debug 困難。此為新手團隊最常見的測試陷阱。
02 — 其他形狀

獎盃與蜂巢——金字塔並非唯一解

形狀提倡者核心觀點
金字塔 Mike Cohn（2009） 大量 unit、少量 E2E（傳統做法）
獎盃（Trophy） Kent C. Dodds（2018） 以整合測試為主，並以靜態檢查（TypeScript）為基底
蜂巢（Honeycomb） Spotify Engineering 於後端微服務環境下，服務間的整合測試具有最高價值
i
形狀的選擇取決於專案類型 前端：採用獎盃（UI 變動頻繁，unit test 容易過度耦合 DOM；建議使用 React Testing Library 搭配整合測試與 Playwright）。
後端 monolith：金字塔仍為良好的預設選擇。
後端微服務：採用蜂巢——服務內邏輯以整合測試覆蓋，服務間以 contract 測試。
03 — 測試類型

實務上常見的測試類型

類型測試對象速度典型工具
Unit 單一函式或元件，外部依賴皆 mock 毫秒 Jest / Vitest / pytest / JUnit
Integration 多個模組的串接，例如 API + DB 數十毫秒至秒 Supertest / Testcontainers
Contract 服務 API 的契約（consumer / provider） 秒 Pact / OpenAPI schema
E2E 實際啟動瀏覽器，從使用者角度測試完整流程 秒至分鐘 Playwright / Cypress / Selenium
Smoke 「冒煙測試」——部署後執行少量關鍵 path 確認系統可正常運作 秒 少量精選的 E2E
Regression 確保過往已修補的 bug 不再復發 — 上述所有類型，並持續累積
04 — TDD

TDD 三步驟：Red → Green → Refactor

由 Kent Beck 於《Test-Driven Development: By Example》（2002）中系統化的方法論。其為一個持續循環的流程：

圖 8.2 — TDD 三步驟循環

實作範例：撰寫一個 add 函式

typescript 
// 步驟 1（RED）：先撰寫測試
test('add: 兩個數相加', () => {
  expect(add(1, 2)).toBe(3);
});

// 執行測試 → 失敗（add 尚未實作）← RED

// 步驟 2（GREEN）：以最小幅度的實作通過測試
function add(a: number, b: number) {
  return a + b;
}

// 執行測試 → 通過 ← GREEN

// 步驟 3（REFACTOR）：重整程式碼
// 此例過於簡單而無重構空間；於複雜情境下，此步驟會進行函式抽離與命名調整

// 接著回到步驟 1 撰寫下一個測試（例如負數相加）
i
TDD 真正的價值不在於「撰寫測試」 而在於「先釐清要實作的內容」。撰寫測試的當下，必須思考：「此函式的 input / output 為何？邊界情況有哪些？」此思考過程的價值遠高於測試本身。
05 — F.I.R.S.T

良好測試的 F.I.R.S.T 原則

由 Robert C. Martin（Uncle Bob）提出，每個字母代表一項原則：

字母原則實務意涵
Fast快速整套 unit test 應於數秒內完成，否則開發者將不願頻繁執行
Isolated獨立每個測試彼此獨立，不依賴執行順序、不互相影響
Repeatable可重複相同 input 永遠產生相同結果，不依賴時間、亂數或外部服務
Self-validating自我驗證執行後直接顯示成功或失敗，無須人工檢視 log 判讀
Timely及時測試應緊隨 production code 撰寫，理想方式即為 TDD 的 R-G-R 循環
!
flaky test（不穩定測試）為違反 Repeatable 的主要原因 有時通過、有時失敗——通常源於依賴時間、依賴順序或依賴網路。flaky test 比缺少測試更糟——它會使團隊習慣性忽略 CI 紅燈，最終連真實的 bug 也被當作 flaky 而重跑。發現 flaky 應立即修補；無法立即修補時應先 skip 並建立 issue 安排處理。
06 — AAA 模式

AAA——Arrange / Act / Assert

每個測試於結構上清楚劃分為三段，可顯著提升可讀性：

typescript 
test('logout: clears all user state', () => {
  // Arrange——設定測試情境
  const store = createStore({ user: { id: 42, name: 'Jason' } });

  // Act——觸發目標行為
  store.dispatch(logout());

  // Assert——驗證結果
  expect(store.getState().user).toBeNull();
  expect(store.getState().token).toBeNull();
});
✓
每個測試僅驗證單一事項 單一 test 內最好僅包含一個邏輯上的 assert（可有多個 expect，但皆驗證同一件事）。若發現需要第二組 act → assert，應將其拆分為新的測試。
07 — Coverage

Coverage 80% 目標的解析

四種 coverage 指標

指標意義
Line程式碼有多少百分比的行被執行
Statement多少百分比的語句被執行（單一行可能含多個 statement）
Branch多少百分比的分支（if-else、switch）被執行——通常為最重要的指標
Function多少百分比的函式至少被呼叫過一次

為何 80% 是常見的目標

  • 低於 60%——過多區域缺乏測試保護，重構風險高
  • 60-80%——體質健康，重要邏輯皆已覆蓋
  • 80-90%——高品質開源專案的水準
  • 追求 100%——邊際效益急遽下降，且容易出現「為了 coverage 而撰寫的低價值測試」
!
Coverage 高不等同於測試良好 以下測試的 coverage 為 100%，但實際毫無價值： 
test('add', () => {
  add(1, 2);  // 無 assert，純執行
Coverage 衡量的是「測試覆蓋了多少行程式碼」，而非「邏輯正確性」。在覆蓋率高的同時，仍須確認：是否包含明確的 assert？是否測試了邊界情況？是否透過 mutation testing 驗證測試強度？

不應追求高 coverage 的場景

  • Prototype 或 spike code——探索性程式碼，可隨時丟棄
  • 純呼叫第三方 API 的 wrapper——測試僅能 mock 自行撰寫的部分
  • 複雜的 UI 微互動——視覺回歸（visual regression）較 unit test 更為有效
Mutation Testing——測試的測試 進階

Mutation testing 工具會刻意修改 production code（例如將 > 改為 >=、移除 !），然後執行測試。若測試仍通過，代表測試未捕捉到該 mutation，亦即測試強度不足。

JS 工具：Stryker。Java：PIT。Python：mutmut。較 coverage 更精確衡量測試強度，但執行較慢，通常不會於每次 CI 執行。

Property-Based Testing——以隨機輸入發掘邊界 進階

傳統測試：撰寫 add(1, 2) === 3。Property-based 測試：定義屬性 ∀ a, b: add(a, b) === add(b, a)（交換律），框架將自動產生數百組隨機輸入進行測試。

JS：fast-check。Haskell 的 QuickCheck 為此概念的源頭。發掘邊界 bug 的能力遠勝於人工撰寫的測試。
