Issue 撰寫 與專案管理

Issue 並非抱怨區，而是協作的最小單位。本課將介紹如何撰寫工程師願意處理的 issue，並解析大型專案的標籤、Milestone 與 Triage 系統。

01 — Issue 三大類

Issue 不僅止於「回報 bug」

大型專案的 issue 通常分為三大類，各自具備不同目的與撰寫方式：

類型用途標籤
Bug Report回報軟體的錯誤行為type: bug
Feature Request提議新功能或改善建議type: feature 或 enhancement
Question / Discussion使用方面的提問、設計層面的討論type: question（許多專案會將此類引導至 GitHub Discussions）
i
Question 應提交於 issue 或 Discussions？ GitHub Discussions 為 2020 年推出的論壇式功能，適合「使用方法」類型的提問。Issue 應保留予「需要 maintainer 採取行動」的事項——程式碼修改、文件修改、feature 評估等。Stack Overflow 亦為常見的引導去處。實際做法以專案的 CONTRIBUTING 為準。
02 — Bug Report

業界通用的 Bug Report 結構

一份能被有效處理的 bug report，須包含以下幾個區塊：

# 標題：[Bug] 一句話描述問題

## Describe the bug
登出後 token 未完全清除，下次登入其他帳號時會短暫看到上一位使用者的資料。

## Steps to Reproduce
1. 以 user A 登入
2. 點選右上角的「登出」
3. 以 user B 登入
4. 首頁顯示 user A 的 profile

## Expected behavior
登出後應完全清除所有 user 相關資料，登入新帳號時應顯示新使用者的資料。

## Actual behavior
頁面短暫閃現 user A 的 avatar 與 name，刷新後才恢復正常。

## Environment
- OS: macOS 14.5
- Browser: Chrome 124.0.6367
- App version: 1.4.2
- Node: 18.19.0

## Screenshots / Video
（附上錄影或截圖）

## Additional context
推測可能是 React Query cache 未清除。首次發現於 2026-04-25 部署後。
!
缺少重現步驟即無從處理 無法重現的 bug 即無法 debug。「有時候會壞」屬於品質最低的 bug report。提供明確的重現步驟、影片、log 與版本資訊，才能進入 triage 流程。
03 — Feature Request

Feature Request 結構

新功能請求應描述痛點而非解法。直接寫「請新增 X 功能」常會被拒絕，因 maintainer 無從判斷實際的使用情境。

# 標題：[Feature] Allow custom date format in <DatePicker>

## Is your feature request related to a problem?
應用需顯示中文日期格式（民國年），目前 DatePicker 僅支援
ISO 格式，必須於 component 外額外進行轉換。

## Describe the solution you'd like
新增 `format` prop，使用者可傳入 day.js 或 date-fns 的格式字串：
```tsx
<DatePicker format="民國 YYY 年 M 月 D 日" />
```

## Describe alternatives you've considered
- 以 wrapper component 自行轉換（目前做法，但 onChange 處理較為繁瑣）
- 整合 i18n library（過於厚重）

## Additional context
react-aria/Calendar 具備類似功能，可作為參考。
i
「願意親自實作」可顯著提升被接受的機率 在 issue 末尾附上一句「願意送出 PR」，可大幅降低 maintainer 評估時的心理成本。
04 — 對照

好/壞 Issue 對照

✗不佳
不能用

無描述性標題、無重現步驟、無環境資訊。Maintainer 只能回覆「請補充資訊」。

✓良好
[Bug] Logout doesn't clear React Query cache (v1.4.2)

標題即傳達「問題為何、發生於哪個版本」，maintainer 無須點開即可進行初步分類。

05 — Issue Template

Issue 與 PR 模板：將良好習慣自動化

於專案根目錄置入 .github/ISSUE_TEMPLATE/ 後，使用者按下「New Issue」時會出現選單，引導其依結構填寫內容：

yaml 
# .github/ISSUE_TEMPLATE/bug_report.yml
name: 🐞 Bug Report
description: 回報軟體錯誤行為
labels: ["type: bug", "status: triage"]
body:
  - type: textarea
    id: describe
    attributes:
      label: Describe the bug
      description: 簡短描述問題現象
    validations: { required: true }
  - type: textarea
    id: steps
    attributes:
      label: Steps to Reproduce
      value: |
        1.
        2.
        3.
    validations: { required: true }
  - type: input
    id: version
    attributes:
      label: App Version
    validations: { required: true }

PR 模板的做法類似——於 .github/PULL_REQUEST_TEMPLATE.md 中寫好結構，使用者開 PR 時即會自動帶入。

✓
模板可省下大量「請補充資訊」的往返溝通 採用模板後，必填欄位若未填寫即無法提交。Maintainer 收到的 issue 品質將顯著提升。
06 — 標籤系統

大型專案的標籤（Label）分類法

觀察 Kubernetes 或 VS Code 的 issue 列表，可發現標籤被劃分為幾個維度，並以 prefix 加以區隔：

維度常見 prefix範例
類型type:type: bug type: feature type: docs
優先級priority:priority: critical priority: high
狀態status:status: triage status: in-progress status: needs-review
區域area: 或 scope:area: frontend area: api
邀請貢獻(無 prefix)good first issue help wanted
難度difficulty:difficulty: easy
i
good first issue 與 help wanted 為 GitHub 官方推薦標籤 GitHub 會主動將帶有這兩個標籤的 issue 推送至「explore」或「first contributions」入口。對於初次想參與貢獻的開發者而言，這是最佳的進入點。
07 — Milestone

Milestone：將 issue 綁定至特定 release

Milestone 為 GitHub 內建的「一組 issue / PR 屬於同一個 release」機制。例如建立 v1.5.0 milestone，將該版預計納入的 issue 全數指派至此，即可清晰掌握整體進度。

08 — GitHub Projects

GitHub Projects（Projects v2）

2022 年推出的新版 Projects 已成為 GitHub 跨 repo issue 管理的主力工具，可透過三種視圖檢視同一份資料：

視圖適用場景
Table檢視整體概況、批次編輯欄位（類似 Notion 或 Airtable）
Board（Kanban）追蹤流程：Backlog → Todo → In Progress → Done
Roadmap掌握時間軸：各項工作分配於哪些週次

Projects 可新增自訂欄位（Status、Priority、Sprint、Estimate、Assignee），並支援跨 repo聚合 issue。單一 Project 可同時管理 frontend、backend 與 docs 等多個 repo 的 issue。

Linear、Jira 與 GitHub Projects 的取捨 產品比較
  • GitHub Projects：完整整合 GitHub issue 與 PR；介面簡潔、免費、無須切換工具；但 sprint planning 與報表能力較為有限。
  • Linear：UI 與速度為業界標竿，cycle（sprint）概念內建完整，適合產品團隊。須付費。
  • Jira：老牌企業級工具，自訂能力極強但設定複雜，工程師接受度普遍較低。多用於大型企業或受監管產業。

中小型團隊使用 GitHub Projects 通常已能滿足需求。

09 — Triage

Issue Triage 流程

大型專案每日可能湧入數十個 issue，maintainer 逐一處理將難以負荷。Triage（分流） 是將進來的 issue 快速分類處理的流程：

1
標記 status: triage
所有新 issue 預設為此狀態，等待人員處理。
2
驗證重現性
可重現者 → 標記 status: confirmed；無法重現者 → 留言請 reporter 補充資訊，並標記 needs-info。
3
指派優先級、區域與 Milestone
例如 priority: high、area: frontend、綁定至 v1.5.0 milestone。
4
適合外部貢獻者，標記 good first issue
範圍明確、規模適中、具備重現步驟者，可邀請社群協助修補。
5
不予處理的 issue 應直接關閉並說明理由
超出範圍、重複、設計上不採納者 → 給予明確理由後關閉，避免 backlog 過度膨脹。

Stale bot：自動關閉長期未處理的 issue

許多大型專案（如 React、TypeScript）採用 actions/stale：超過 N 天無回應的 issue 將自動標記為 stale，再經過 M 天即自動關閉，避免「長期存在但無人關注」的 issue 堆積。

!
Stale bot 亦有副作用 部分 issue 確實重要，僅因 maintainer 暫時無暇處理，被自動關閉後可能引發社群反彈。因此 stale bot 通常配合白名單機制——已標記 pinned 或 priority: high 的 issue 不會被自動關閉。
