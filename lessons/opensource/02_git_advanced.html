01 — rebase

rebase：把分支「移植」到別人後面

rebase 字面意思是「換 base」 —— 把你的分支從原本接的點，搬到另一個點上。常見場景：你的 feature branch 是從兩週前的 main 開的，現在 main 已經往前跑了 30 個 commit，你想把自己的工作接到最新 main 後面。

圖 2.1 — rebase 把 X、Y 重新接到最新 main 後面
bash 
# 在 feature branch 上
$ git fetch origin
$ git rebase origin/main

# 如果有衝突，解決後
$ git add <檔案>
$ git rebase --continue

# 不想繼續了 (回到 rebase 前狀態)
$ git rebase --abort

互動式 rebase：整理自己的 commit 歷史

這是 rebase 的另一個用法，跟前面「移植分支」是完全不同的場景。前面的 git rebase main 是把分支接到別的 base；這裡的 git rebase -i 是針對同一條分支上自己的 commit做修改 —— 合併、改 message、刪掉、重排順序。共通點只是底層都會「重新建立 commit」這個動作。

送 PR 前最常用 — 把混亂的 commit 整理成乾淨的故事。（如果只想改最新一個 commit，用 git commit --amend 更快；rebase -i 是處理更早或一次多個 commit 的版本。）

bash 
# 整理最近 5 個 commit
$ git rebase -i HEAD~5

# 整理從 main 分出去後的所有 commit
$ git rebase -i origin/main

跳出來的編輯器會列出這些 commit，每行最前面寫一個動作：

i
怎麼操作這個編輯器 預設會開 Vim：i 進入編輯模式 → 把行首的 pick 改成 squash / fixup / reword 等動作（也可以整行上下拖移來改順序）→ 按 Esc 離開編輯模式 → 輸入 :wq 存檔離開。不熟 Vim 可以先 git config --global core.editor "code --wait" 換成 VS Code，存檔關掉分頁就會繼續。
動作意思用在哪
pick保留什麼都不改
reword / r保留改動但重寫 messagecommit message 寫得不夠好
squash / s跟上一個合併，合併兩邊的 message把多個有意義的小步驟收成一個大功能 commit
fixup / f跟上一個合併，丟掉這個的 message「修 typo / 修 lint / 修上一個 commit 的小錯」這種補丁
edit / e停在這個 commit 讓你重新改檔案想拆開或大改某個 commit
drop / d整個 commit 丟掉不該存在的實驗 commit

實際在編輯器裡長這樣 — 假設 git rebase -i HEAD~4，要把後面 3 個小修補通通合進第一個有意義的 commit：

git-rebase-todo 
# 編輯器打開後預設長這樣 (越上面越舊)
pick a1b2c3d feat(profile): add avatar upload
pick e4f5g6h fix typo
pick i7j8k9l fix lint
pick m0n1o2p tweak avatar size

# 把後面三個改成 fixup (沒新訊息) 或 squash (要合併訊息)
pick   a1b2c3d feat(profile): add avatar upload
fixup  e4f5g6h fix typo
fixup  i7j8k9l fix lint
squash m0n1o2p tweak avatar size

# 存檔離開後，4 個 commit 會合成 1 個，
# 並再跳一次編輯器讓你確認最終的 commit message

把 rebase 過的 commit 推上去:--force-with-lease

rebase 完成後,本地的 commit SHA 已經跟遠端對不起來,普通 git push 會被 Git 擋下來(它認為你會弄丟遠端的歷史)。要把改寫過的歷史推上去就得 force push,但純 --force 有風險 — 如果同事在你 rebase 期間剛好 push 了東西到同一支分支,你會直接覆蓋掉他的工作。

✗危險
git push --force
無條件覆蓋,同事剛 push 的 commit 會被你蓋掉,且不會警告。
✓推薦
git push --force-with-lease
「除非遠端跟我上次 fetch 看到的一樣,否則拒絕推」 — 同事有新 push 時會擋下來提醒你。
i
建議設成預設 alias git config --global alias.fpush "push --force-with-lease" — 之後習慣性打 git fpush,徹底避開 --force。同樣的安全考量也適用於 commit --amend、cherry-pick、reset 等任何會改寫 commit SHA 的操作 — 只要你接下來要 push,就用 --force-with-lease。
02 — cherry-pick

cherry-pick：只挑一個 commit 搬到別處

典型場景：main 上有個 bugfix，你想把它 backport 到 v2 維護分支，但 v2 跟 main 已經差很多，沒辦法整支 merge。

bash 
# 切到目標分支
$ git checkout release/v2.x

# 把 main 上某個 commit 拿過來
$ git cherry-pick a1b2c3d

# 一次 cherry-pick 多個
$ git cherry-pick a1b2c3d e4f5g6h

# 一個範圍 (含 e4f5g6h，不含 a1b2c3d)
$ git cherry-pick a1b2c3d..e4f5g6h

# 衝突解掉之後
$ git add <files>
$ git cherry-pick --continue
i
cherry-pick 到底搬了什麼？ 只搬「那個 commit 自己的 diff」（它跟父 commit 的差異），不會把歷史、父 commit 或其他不同步的內容一起搬過來。套用到目前分支後會產生一個全新的 commit（hash 不同），如果這段 diff 在目標分支的對應位置已經長得不一樣，就會出現衝突要手動解。
03 — reflog

reflog：救回任何「不見」的 commit

reflog 記錄你這台機器上所有 HEAD 與分支指針的移動 — commit、reset、rebase、checkout、fetch / pull 都會留下一筆（純本地紀錄，別人 push 到遠端的事不會出現）。就算 commit 從分支上消失、被 rebase 改寫、被 reset --hard 蓋掉，只要還沒 GC (預設 90 天)，它都還在 reflog 裡。

bash 
# 看 HEAD 最近 30 個移動
$ git reflog

a1b2c3d HEAD@{0}: reset: moving to HEAD~3
e4f5g6h HEAD@{1}: commit: feat(profile): add avatar upload
i7j8k9l HEAD@{2}: commit: fix(profile): handle empty name
...

# 我想救回 e4f5g6h 那個 commit
$ git checkout -b rescue HEAD@{1}
# 或直接
$ git reset --hard HEAD@{1}

「救回」具體在救什麼？

要理解 reflog 怎麼救人，先把 git 拆成三個東西看：

角色放什麼reset --hard 後
.git/objects/ commit 物件本體 — 檔案快照、message、作者、parent SHA 都在這 完全沒動，commit 還活著
分支指針 (main、feature...) 「這個分支現在指向哪個 SHA」 已搬走，所以 git log 看不到那些 commit
reflog 只記「HEAD 曾經指過哪些 SHA」的時間軸 — 不存任何 commit 內容 還在，會看到「reset: moving to ...」這一筆

救回流程：從 reflog 拿到 SHA（reflog 只給你「鑰匙」） → 用這把鑰匙找到 .git/objects/ 裡那個還活著的 commit → 用 reset 或 checkout 把分支指針重新接回去。

兩種救法影響範圍不同：

指令會發生什麼執行完的下一步
git checkout -b rescue <SHA>
推薦先用這個		 開一條新分支 rescue 指向那個 commit；原分支完全不動，git 歷史沒被改寫 在 rescue 確認內容沒問題，再 cherry-pick 或 merge 回原本的分支
git reset --hard <SHA>
已 push 過要小心		 當前分支指針 + HEAD + index + 工作目錄全部覆蓋成那個 commit 的狀態，本地分支歷史會被改寫 本地已是救回後的狀態；要同步到遠端需要 git push --force-with-lease (會影響其他協作者)

注意：untracked 或只 modify 沒 commit 的檔案不在 commit 物件裡，reflog 救不回。

04 — worktree

worktree：同一個 repo 開多個工作目錄

你正在 feature 分支寫到一半，主管要你立刻 review 一個 PR。傳統做法是 git stash + git switch pr/123，但 stash 容易忘記、切回來還要解 IDE 重新索引。worktree 直接幫你開出第二個目錄，兩個分支同時存在不互相干擾。

bash 
# 在隔壁目錄開一個新工作區，checkout 別的分支
$ git worktree add ../myrepo-review pr/123

# 開 IDE 進去 ../myrepo-review 看 PR，原本目錄完全不動

# 看完了，刪掉這個 worktree
$ git worktree remove ../myrepo-review

# 列出全部 worktree
$ git worktree list
i
用 worktree 取代多 clone 以前需要看別的分支時，常常會 clone 兩份同樣的 repo。worktree 讓你共用 .git，磁碟空間省一半，fetch / pull 也只要做一次。
05 — reset

reset：把分支指針往回搬

reset 的本質是「移動指針」 — 把 HEAD 跟當前分支搬到指定的 commit，被略過的 commit 就從這條分支上消失了 (還在 reflog，90 天內可救)。典型場景：剛 commit 完發現 message 寫錯、想把幾個小 commit 重新揉一揉、或本地實驗失敗想整個工作目錄還原到乾淨狀態。它會改寫歷史，所以原則只有一條：對還沒 push 的本地 commit 用，已 push 的公開分支改用下一節的 revert。

三種模式差別在動到哪幾層：

模式HEAD 動?Staging 動?Working Tree 動?用在哪
--soft ✓ 動 ✗ 不動 ✗ 不動 把幾個 commit 退回成「即將要 commit」的狀態 (改動已 staged)
--mixed (預設) ✓ 動 ✓ 動 (清空) ✗ 不動 退回 commit、把 staging 清空，但檔案改動還留在工作目錄
--hard ✓ 動 ✓ 動 ✓ 動 (覆寫) 連檔案都還原 — 會吃掉沒 commit 的改動，務必確認
bash 
# 把最近 3 個 commit 撤掉但保留改動 (預設 mixed)
$ git reset HEAD~3

# 退回成「已 staged」的狀態 (常配 commit --amend 重做最後一個 commit)
$ git reset --soft HEAD~1

# 連檔案內容也還原 (危險：沒 commit 的改動會消失)
$ git reset --hard HEAD~3

# 把整個 staging 清空 (不加參數等同 reset --mixed HEAD)
$ git reset

# 不小心 reset --hard 錯了？用 reflog 救回來
$ git reflog
$ git reset --hard HEAD@{1}
06 — revert

revert：用一個反向 commit 取消歷史

revert 不會把舊 commit 拔掉，而是產生一個新 commit，內容是「把那個 commit 的 diff 反過來做」。原本的歷史完全保留 — 別人 pull 下來不會發現有 commit 不見，只會看到分支多了一個「revert xxx」的新 commit。典型場景：已經 push 到 main 或團隊共用分支的 commit 要撤掉、production 上線後發現 bug 要立刻回退但保留紀錄、或想撤掉一次已經 merge 進主線的整支 feature。它是公開分支唯一安全的回退方式；本地還沒 push 的 commit 用上一節的 reset 比較乾淨，不會在歷史中多一個 revert commit。

bash 
# 撤掉某個 commit (產生一個反向 commit)
$ git revert a1b2c3d

# 連續撤掉多個 commit (會逐一產生反向 commit)
$ git revert a1b2c3d e4f5g6h

# 撤掉但先不立刻 commit，最後合併成一個反向 commit
$ git revert --no-commit a1b2c3d e4f5g6h
$ git commit -m "revert: rollback feature X"

# 撤掉一個 merge commit (要用 -m 1 指定保留主分支那一邊)
$ git revert -m 1 <merge-commit>

# revert 進行中發生衝突
$ git add <files>
$ git revert --continue
i
reset 還是 revert？一句話判斷 還沒 push → reset 隨你改。已 push 到別人會看到的分支 → 一律 revert，不要在公開歷史上動 reset。
