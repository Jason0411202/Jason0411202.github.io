Git 進階操作

PR 流程中經常會用到的進階指令：rebase、cherry-pick、reflog、worktree、reset、revert，本課將逐一說明其用途與適用場景。

01 — rebase

rebase：將分支「移植」至另一個基準點

rebase 字面意思是「換 base」——把分支從原本接續的 commit，重新接到另一個 commit 之後。常見場景：feature 分支從兩週前的 main 切出，期間 main 已新增 30 個 commit，希望將自己的工作接續到最新 main 之後。

圖 2.1 — rebase 把 X、Y 重新接到最新 main 後面
bash 
# 於 feature branch 上執行
$ git fetch origin
$ git rebase origin/main

# 若發生衝突，解決後執行
$ git add <檔案>
$ git rebase --continue

# 中止 rebase 並回到原始狀態
$ git rebase --abort

互動式 rebase：整理自己的 commit 歷史

此為 rebase 的另一種用法，與前述「移植分支」屬於不同場景。前述的 git rebase main 是將分支接續到不同的 base；此處的 git rebase -i 則是針對同一條分支上的 commit進行調整——合併、修改 message、刪除、重新排序。兩者的共通點僅在於底層都會「重新建立 commit」。

送出 PR 前最常使用此功能，將零散的 commit 整理為結構清晰的歷史。（若僅需修改最新一筆 commit，使用 git commit --amend 更為直接；rebase -i 適用於處理更早或多筆 commit 的情況。）

bash 
# 整理最近 5 個 commit
$ git rebase -i HEAD~5

# 整理從 main 分出去後的所有 commit
$ git rebase -i origin/main

執行後開啟的編輯器會列出這些 commit，每行最前方標示一個動作：

動作說明適用情境
pick保留不需修改
reword / r保留改動，重寫 messagecommit message 描述不夠精確
squash / s與上一筆合併，同時合併兩者的 message將多個有意義的小步驟整併為一筆功能性 commit
fixup / f與上一筆合併，捨棄該筆的 message修 typo、修 lint、修補上一筆 commit 的小錯誤
edit / e暫停於該 commit，可重新修改檔案需要拆解或大幅修改某筆 commit
drop / d整筆 commit 移除不應保留的實驗性 commit

編輯器中的實際內容如下——假設執行 git rebase -i HEAD~4，將後續三筆小修補全數合併至第一筆功能性 commit：

git-rebase-todo 
# 編輯器開啟後的預設內容（由上而下，由舊到新）
pick a1b2c3d feat(profile): add avatar upload
pick e4f5g6h fix typo
pick i7j8k9l fix lint
pick m0n1o2p tweak avatar size

# 將後續三筆改為 fixup（不保留 message）或 squash（合併 message）
pick   a1b2c3d feat(profile): add avatar upload
fixup  e4f5g6h fix typo
fixup  i7j8k9l fix lint
squash m0n1o2p tweak avatar size

# 存檔離開後，四筆 commit 將合併為一筆，
# 並再次開啟編輯器供確認最終的 commit message
i
編輯器操作方式 預設使用 Vim：按 i 進入編輯模式 → 將行首的 pick 改為 squash / fixup / reword 等動作（亦可整行上下移動以調整順序）→ 按 Esc 離開編輯模式 → 輸入 :wq 存檔離開。若不熟悉 Vim，可先以 git config --global core.editor "code --wait" 將編輯器切換為 VS Code，存檔關閉分頁後即會繼續執行。

推送 rebase 後的 commit：--force-with-lease

rebase 完成後，本地的 commit SHA 已與遠端對不上，一般的 git push 會被 Git 拒絕（避免覆蓋遠端的歷史）。要將改寫後的歷史推送出去必須 force push，但單純使用 --force 具有風險——若同事於 rebase 期間對同一分支 push 了內容，將會被直接覆寫。

✗危險
git push --force
無條件覆蓋；同事剛 push 的 commit 將被覆寫，且無任何警告。
✓推薦
git push --force-with-lease
僅當遠端的位置與上次 fetch 時一致才允許推送——若遠端有他人新 push，會直接拒絕並提示。
✓
建議設定為預設 alias git config --global alias.fpush "push --force-with-lease"——後續習慣性使用 git fpush，可徹底避開 --force。同樣的安全考量也適用於 commit --amend、cherry-pick、reset 等任何會改寫 commit SHA 的操作——只要後續需要 push，皆建議使用 --force-with-lease。
02 — cherry-pick

cherry-pick：將指定 commit 套用至其他分支

典型場景：main 上有一筆 bugfix，需 backport 至 v2 維護分支，但 v2 與 main 之間已存在大量差異，無法整支分支 merge。

bash 
# 切換至目標分支
$ git checkout release/v2.x

# 將 main 上某筆 commit 套用過來
$ git cherry-pick a1b2c3d

# 一次套用多筆
$ git cherry-pick a1b2c3d e4f5g6h

# 套用一個範圍（包含 e4f5g6h，不包含 a1b2c3d）
$ git cherry-pick a1b2c3d..e4f5g6h

# 解決衝突後繼續
$ git add <files>
$ git cherry-pick --continue
i
cherry-pick 實際套用了哪些內容？ 僅套用「該 commit 本身的 diff」（即與父 commit 的差異），不會連同歷史、父 commit 或其他相關內容一併帶入。套用至當前分支後會產生一筆全新的 commit（hash 不同）；若該 diff 在目標分支的對應位置已有不同內容，將會出現衝突需手動解決。
03 — reflog

reflog：救回看似「消失」的 commit

reflog 記錄本機上所有 HEAD 與分支指針的移動——commit、reset、rebase、checkout、fetch / pull 皆會留下紀錄（純本地紀錄，他人 push 至遠端的動作不會出現）。即使 commit 從分支上消失、被 rebase 改寫、或被 reset --hard 覆蓋，只要尚未被 GC（預設 90 天）回收，皆可從 reflog 中找回。

bash 
# 檢視 HEAD 最近 30 次移動紀錄
$ git reflog

a1b2c3d HEAD@{0}: reset: moving to HEAD~3
e4f5g6h HEAD@{1}: commit: feat(profile): add avatar upload
i7j8k9l HEAD@{2}: commit: fix(profile): handle empty name
...

# 救回 e4f5g6h 該筆 commit
$ git checkout -b rescue HEAD@{1}
# 或直接重設當前分支
$ git reset --hard HEAD@{1}

「救回」的對象究竟為何？

要理解 reflog 的救援機制，需先將 git 拆解為三個層次：

角色內容reset --hard 後的狀態
.git/objects/ commit 物件本體——檔案快照、message、作者、parent SHA 皆儲存於此 完全未動，commit 仍存在
分支指針（main、feature 等） 記錄「該分支當前指向哪個 SHA」 已移動，因此 git log 無法顯示那些 commit
reflog 僅記錄「HEAD 曾指向過哪些 SHA」的時間軸——不儲存任何 commit 內容 仍存在，會顯示「reset: moving to ...」一筆

救援流程：從 reflog 取得 SHA（reflog 僅提供「索引」）→ 透過該 SHA 於 .git/objects/ 中找到仍存在的 commit → 以 reset 或 checkout 將分支指針重新指向該 commit。

兩種救援方式的影響範圍如下：

指令影響範圍後續動作
git checkout -b rescue <SHA>
建議優先採用		 建立新分支 rescue 指向該 commit；原分支完全不受影響，git 歷史未被改寫 於 rescue 確認內容無誤後，再透過 cherry-pick 或 merge 整合回原分支
git reset --hard <SHA>
已 push 者需審慎評估		 當前分支指針、HEAD、index 與工作目錄一併覆蓋為該 commit 的狀態，本地分支歷史會被改寫 本地已恢復為救援後的狀態；同步至遠端需執行 git push --force-with-lease（將影響其他協作者）

注意：untracked 或僅修改未 commit 的檔案並未存於 commit 物件中，reflog 無法救回。

04 — worktree

worktree：同一倉庫開設多個工作目錄

當前在 feature 分支進行中，臨時需要切換至另一條分支進行 review。傳統做法是先 git stash 再 git checkout pr/123，但 stash 容易遺忘、切回後 IDE 還需重新索引。worktree 可直接開出第二個目錄，使兩個分支同時並存且互不干擾。

bash 
# 於相鄰目錄開設新工作區，並 checkout 指定分支
$ git worktree add ../myrepo-review pr/123

# 以 IDE 開啟 ../myrepo-review 進行 review，原目錄不受影響

# review 完成後，移除該 worktree
$ git worktree remove ../myrepo-review

# 列出所有 worktree
$ git worktree list
i
以 worktree 取代多重 clone 過去需同時檢視不同分支時，常會 clone 兩份相同的 repo。worktree 共用同一個 .git 目錄，可顯著節省磁碟空間，fetch / pull 也僅需執行一次。
05 — reset

reset：將分支指針回退至指定位置

reset 的本質是「移動指針」——將 HEAD 與當前分支移至指定的 commit，被跳過的 commit 隨之從該分支上消失（仍存於 reflog，90 天內可救回）。典型場景：commit 後發現 message 撰寫錯誤、需重新整併數筆小 commit，或本地實驗失敗需將工作目錄還原為乾淨狀態。此操作會改寫歷史，原則僅有一條：僅用於尚未 push 的本地 commit；已 push 至公開分支者請改用下一節介紹的 revert。

三種模式的差異在於影響的層級：

模式HEADStagingWorking Tree適用情境
--soft ✓ 移動 ✗ 保留 ✗ 保留 將數筆 commit 退回為「待 commit」狀態（改動仍處於 staged）
--mixed（預設） ✓ 移動 ✓ 清空 ✗ 保留 退回 commit 並清空 staging，但檔案改動仍保留於工作目錄
--hard ✓ 移動 ✓ 清空 ✓ 覆寫 連同檔案內容一併還原——將清除尚未 commit 的改動，務必確認
bash 
# 撤回最近 3 筆 commit 但保留改動（預設 mixed）
$ git reset HEAD~3

# 退回為「已 staged」狀態（常搭配 commit --amend 重做最後一筆 commit）
$ git reset --soft HEAD~1

# 連同檔案內容一併還原（高風險：未 commit 的改動將遺失）
$ git reset --hard HEAD~3

# 清空整個 staging（不加參數等同於 reset --mixed HEAD）
$ git reset

# 誤用 reset --hard 後的救援：透過 reflog 找回
$ git reflog
$ git reset --hard HEAD@{1}
06 — revert

revert：以反向 commit 取消既有歷史

revert 不會移除舊有 commit，而是產生一筆新 commit，內容為「將該 commit 的 diff 反向套用」。原始歷史完整保留——他人 pull 下來不會發現有 commit 消失，僅會看到分支上多了一筆「revert xxx」的新 commit。典型場景：已 push 至 main 或團隊共用分支的 commit 需撤回、production 上線後發現 bug 須立即回退但保留紀錄、或需撤回一條已 merge 進主線的 feature 分支。此為公開分支上唯一安全的回退方式；本地尚未 push 的 commit 改用上一節的 reset 較為乾淨，不會在歷史中留下 revert commit。

bash 
# 撤回某筆 commit（產生一筆反向 commit）
$ git revert a1b2c3d

# 連續撤回多筆 commit（將逐一產生反向 commit）
$ git revert a1b2c3d e4f5g6h

# 撤回但暫不 commit，最後合併為單一反向 commit
$ git revert --no-commit a1b2c3d e4f5g6h
$ git commit -m "revert: rollback feature X"

# 撤回一筆 merge commit（須以 -m 1 指定保留主分支端）
$ git revert -m 1 <merge-commit>

# revert 進行中發生衝突
$ git add <files>
$ git revert --continue
i
reset 或 revert 的判斷準則 尚未 push → 使用 reset 自由調整。已 push 至他人可見的分支 → 一律使用 revert，不應於公開歷史上執行 reset。
