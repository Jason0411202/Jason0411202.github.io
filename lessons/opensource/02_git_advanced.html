01 — rebase

rebase：把分支「移植」到別人後面

rebase 字面意思是「換 base」 —— 把你的分支從原本接的點，搬到另一個點上。常見場景：你的 feature branch 是從兩週前的 main 開的，現在 main 已經往前跑了 30 個 commit，你想把自己的工作接到最新 main 後面。

圖 2.1 — rebase 把 X、Y 重新接到最新 main 後面
bash 
# 在 feature branch 上
$ git fetch origin
$ git rebase origin/main

# 如果有衝突，解決後
$ git add <檔案>
$ git rebase --continue

# 不想繼續了 (回到 rebase 前狀態)
$ git rebase --abort
02 — interactive rebase

互動式 rebase：整理自己的歷史

送 PR 前最常用 — 把混亂的 commit 整理成乾淨的故事。

bash 
# 整理最近 5 個 commit
$ git rebase -i HEAD~5

# 整理從 main 分出去後的所有 commit
$ git rebase -i origin/main

跳出來的編輯器會列出這些 commit，每行最前面寫一個動作：

動作意思用在哪
pick保留什麼都不改
reword / r保留改動但重寫 messagecommit message 寫得不夠好
squash / s跟上一個合併，合併兩邊的 message把 3 個 commit 合成 1 個有意義的 commit
fixup / f跟上一個合併，丟掉這個的 message「修上一個 commit 的小錯」這種補丁
edit / e停在這個 commit 讓你重新改檔案想拆開或大改某個 commit
drop / d整個 commit 丟掉不該存在的實驗 commit
i
squash 與 fixup 的差別很重要 兩者都合併 commit，但 squash 會打開編輯器讓你寫合併後的新 message，fixup 直接用上一個 commit 的 message。實務上 「修 typo / 修 lint」這種 commit 用 fixup，把多個有意義小步驟收成一個大功能用 squash。
--autosquash：自動排好 fixup commit 技巧 
# commit 的時候用 --fixup 指定要修哪個
$ git commit --fixup=a1b2c3d

# 之後 rebase 時加 --autosquash，git 自動排好順序
$ git rebase -i --autosquash origin/main

這個組合在邊寫程式邊修小錯時特別省力，不用手動拖動 fixup 行到目標 commit 旁邊。

03 — Golden Rule

rebase vs merge：黃金守則

!
The Golden Rule of Rebasing 不要對「已經 push 到別人會用的公開分支」做 rebase。因為 rebase 會改寫 commit SHA，等於從別人的角度看歷史突然變了，他們的本地 git 會迷路。
情境該用原因
只有自己用的 feature branch，要跟上 main rebase 歷史線性、乾淨
把 feature 合進 main merge (PR merge) 保留 PR 完整脈絡
合別人正在用的共用分支 merge 不能改寫別人的歷史
送 PR 前整理自己的 commit rebase -i 清理混亂的 WIP commits
04 — cherry-pick

cherry-pick：只挑一個 commit 搬到別處

典型場景：main 上有個 bugfix，你想把它 backport 到 v2 維護分支，但 v2 跟 main 已經差很多，沒辦法整支 merge。

bash 
# 切到目標分支
$ git checkout release/v2.x

# 把 main 上某個 commit 拿過來
$ git cherry-pick a1b2c3d

# 一次 cherry-pick 多個
$ git cherry-pick a1b2c3d e4f5g6h

# 一個範圍 (含 e4f5g6h，不含 a1b2c3d)
$ git cherry-pick a1b2c3d..e4f5g6h

# 衝突解掉之後
$ git add <files>
$ git cherry-pick --continue
05 — reflog

reflog：救回任何「不見」的 commit

reflog 記錄你的 HEAD 移動過的所有位置。就算 commit 從分支上消失、被 rebase 改寫、被 reset --hard 蓋掉，只要還沒 GC (預設 90 天)，它都還在 reflog 裡。

bash 
# 看 HEAD 最近 30 個移動
$ git reflog

a1b2c3d HEAD@{0}: reset: moving to HEAD~3
e4f5g6h HEAD@{1}: commit: feat(profile): add avatar upload
i7j8k9l HEAD@{2}: commit: fix(profile): handle empty name
...

# 我想救回 e4f5g6h 那個 commit
$ git checkout -b rescue HEAD@{1}
# 或直接
$ git reset --hard HEAD@{1}
✓
記住這個指令，它救過很多人 只要你的 commit 曾經被你的 git 看過 (reachable from HEAD) — 即使誤 reset、誤 rebase、誤刪分支 — 它都還在 reflog 裡。但 untracked / 沒 commit 的檔案救不回來，那些連 reflog 也不會記錄。
06 — bisect

bisect：二分搜尋，找出哪個 commit 引入 bug

典型場景：「上週這個功能還能跑、今天壞了，但中間有 200 個 commit」 — 用 bisect 自動二分搜尋，最多 log₂(200) ≈ 8 次測試就找到兇手。

bash 
# 開始二分搜尋
$ git bisect start

# 告訴 git 哪個版本是壞的、哪個是好的
$ git bisect bad                  # 當前 (HEAD) 是壞的
$ git bisect good v1.4.0           # v1.4.0 是好的

# Git 會 checkout 中間的 commit，你跑測試後告訴它結果
$ npm test
$ git bisect bad                  # 或 git bisect good

# ... 反覆幾次後 git 會說：
# a1b2c3d is the first bad commit

# 結束
$ git bisect reset

自動化：bisect run

如果有測試指令能判斷好壞 (exit 0 = 好，非 0 = 壞)，可以全自動：

bash 
$ git bisect start HEAD v1.4.0
$ git bisect run npm test -- tests/regression.test.ts

# 喝口咖啡，回來時 git 已經告訴你哪個 commit 引入 bug
07 — worktree

worktree：同一個 repo 開多個工作目錄

你正在 feature 分支寫到一半，主管要你立刻 review 一個 PR。傳統做法是 git stash + git switch pr/123，但 stash 容易忘記、切回來還要解 IDE 重新索引。worktree 直接幫你開出第二個目錄，兩個分支同時存在不互相干擾。

bash 
# 在隔壁目錄開一個新工作區，checkout 別的分支
$ git worktree add ../myrepo-review pr/123

# 開 IDE 進去 ../myrepo-review 看 PR，原本目錄完全不動

# 看完了，刪掉這個 worktree
$ git worktree remove ../myrepo-review

# 列出全部 worktree
$ git worktree list
i
用 worktree 取代多 clone 以前需要看別的分支時，常常會 clone 兩份同樣的 repo。worktree 讓你共用 .git，磁碟空間省一半，fetch / pull 也只要做一次。
08 — reset vs revert

reset 跟 revert 的差別

兩個都能「取消 commit」，但機制完全不同：

⚠reset 改寫歷史
把指針往回搬，被略過的 commit 從分支上消失 (還在 reflog)。已 push 過的 commit 別 reset，會破壞別人的歷史。 適合：本地還沒 push 前把幾個 commit 撤掉重做。
✓revert 加新 commit
建立一個「反向操作」的新 commit，原本的 commit 還在歷史中。已 push 的公開分支該用 revert。 適合：production 上線後發現問題、要安全回退。

reset 的三種模式

模式HEAD 動?Staging 動?Working Tree 動?用在哪
--soft ✓ 動 ✗ 不動 ✗ 不動 把幾個 commit 退回成「即將要 commit」的狀態
--mixed (預設) ✓ 動 ✓ 動 (清空) ✗ 不動 退回 commit 又把 staging 清空，改動還在
--hard ✓ 動 ✓ 動 ✓ 動 (覆寫) 連檔案都還原 — 會吃掉沒 commit 的改動，務必確認
bash 
# reset：把最近 3 個 commit 撤掉但保留改動 (預設 mixed)
$ git reset HEAD~3

# reset --hard：連改動也丟掉 (危險，誤用要靠 reflog 救)
$ git reset --hard HEAD~3

# revert：產生一個反向 commit (安全)
$ git revert a1b2c3d
# 一次 revert 一個 merge commit (要指定父分支 -m 1)
$ git revert -m 1 <merge-commit>
09 — 安全的 force push

--force-with-lease：force push 的安全版

rebase 整理過自己的 PR 分支後，要把改寫過的歷史推上去，必須用 force push。但純 --force 有風險 — 如果同事在你 rebase 期間剛好 push 了東西到同一支分支，你會直接覆蓋掉他的工作。

✗危險
git push --force
無條件覆蓋，同事剛 push 的 commit 會被你蓋掉，且不會警告。
✓推薦
git push --force-with-lease
「除非遠端跟我上次 fetch 看到的一樣，否則拒絕推」 — 同事有新 push 時會擋下來提醒你。
i
建議設成預設 alias git config --global alias.fpush "push --force-with-lease" — 之後習慣性打 git fpush，徹底避開 --force。
10 — 加碼

還有兩個進階技巧

rerere：自動記住衝突解法 技巧

「reuse recorded resolution」— 當你在 long-running rebase 反覆解同樣的衝突，rerere 會記住你怎麼解的，下次自動套用。

$ git config --global rerere.enabled true

設好就忘了它存在，但它會默默幫你省下很多重複工作。

submodule 救命指令 深入

submodule 是 Git 給跨 repo 依賴用的，但很多新人 clone 完發現 submodule 是空的：

# clone 時就把 submodule 一起拉
$ git clone --recurse-submodules <repo>

# 已經 clone 了，補拉 submodule
$ git submodule update --init --recursive

# 把所有 submodule 更新到他們各自分支的最新
$ git submodule update --remote --merge

如果不是必要，現代專案多用 monorepo 或 package manager 取代 submodule，比較好維護。
