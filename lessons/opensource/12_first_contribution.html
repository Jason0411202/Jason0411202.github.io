01 — 找專案

哪裡找適合貢獻的專案

來源適合誰
firstcontributions.github.io 第一次練手 — 不是真實貢獻，但走完一次 fork → PR → merge 流程
goodfirstissue.dev 依語言聚合所有專案的 good first issue 標籤
github.com/topics/good-first-issue GitHub 內建 topic 搜尋
Hacktoberfest 每年 10 月活動，4 個合格 PR 拿紀念品
你日常用的工具 最佳起點 — 你天天用，知道 pain point；找其中你修得動的 issue
i
從你天天用的工具開始 你已經對它有領域知識、有實際 use case、知道哪裡好用哪裡爛。比挑陌生熱門專案來得快上手得多。
02 — 判斷專案

5 分鐘判斷一個專案值不值得貢獻

1
看 CONTRIBUTING.md 是否清楚
沒有 = 流程不明，貢獻過程容易卡住。寫得越詳細越友善。
2
看最近 30 天是否有活動
commit / PR / issue 留言頻率。死寂的專案 = 你 PR 送出去也沒人看。
3
看 PR 平均合併時間
點 Pull requests → 看最近合併的 PR 從開到合多久。一週內 = 好；一個月以上 = 你可能要等很久。
4
看 Issue 是否有人回應
最近 10 個 issue 是否有 maintainer 留過言？如果都是石沉大海 = 不友善。
5
看 CI 通過率
點 Actions tab。如果每個 build 都紅，代表專案維護混亂，你的 PR 也很難綠。
6
看社群氛圍
隨機看 5 個 issue / PR 的留言。Maintainer 是否禮貌？是否常拒絕新人貢獻？把氛圍嚴重不友善的專案排除。
03 — 授權

授權快速辨識 (5 個常見族群)

授權核心限制適合做什麼
MIT幾乎沒有 (僅需保留 copyright)個人 / 商業隨意用，最寬鬆
Apache 2.0類似 MIT 但加上專利條款大型專案 (k8s / Apache 系列)
BSD-2 / BSD-3類似 MITBSD / 學術圈傳統
GPL v2 / v3copyleft — 改作品也必須開源 GPLLinux Kernel、Bash 等
AGPL v3GPL 加強版 — 連網路服務也要開源MongoDB (舊版)、Grafana (舊版)
!
商業使用要小心 GPL / AGPL 公司專案使用 GPL 函式庫 = 整個專案可能被要求開源。AGPL 影響更大 — 連你內部 SaaS 服務都要開原始碼。商業案大多選 MIT / Apache 2.0 授權的依賴。但這不影響你貢獻 GPL 專案 — 只是要清楚授權含義。

不確定怎麼選授權給自己的開源專案：choosealicense.com 是 GitHub 官方做的決策樹。

04 — 必讀檔案

送 PR 前必須先讀的 5 個檔案

檔案看什麼
README.md 專案在做什麼、怎麼起手 dev、誰在用
CONTRIBUTING.md 貢獻流程、PR 命名規則、commit 規範、是否需要 DCO 簽署
CODE_OF_CONDUCT.md 社群行為準則。多數專案用 Contributor Covenant
SECURITY.md 發現漏洞怎麼通報 (Lesson 11)
LICENSE 授權條款 (上面學過)
i
DCO (Developer Certificate of Origin) 某些專案 (Linux Kernel、Docker、CNCF 系列) 要求 git commit -s 加上 Signed-off-by: footer，表示「我有權貢獻這段程式碼」。看 CONTRIBUTING 是否有提到。
05 — 認領 issue

怎麼認領 issue 不被搶

看到一個 good first issue，怎麼避免「我寫到一半，發現有人也在做」？

  1. 看 issue 留言區 — 確認沒有人說 "I'm working on this"
  2. 看 Linked PRs — 是否已有 draft PR 連結到這個 issue
  3. 留言說「我想做這個，預計 N 天內送 PR」 — 等 maintainer 確認後再開工
  4. 如果一兩天沒回應就直接做 — 大型專案 maintainer 可能很忙；Open Source 是行動派的世界，做出來通常比等著更有效
i
不要過度承諾 如果你只是「有興趣可能會做」，不要留 "I'll work on this" 然後消失。這會讓 issue 卡住沒人能做。實際開始動工才認領。
06 — 完整 Checklist

從零到 PR 被合併的完整 Checklist

把前 11 堂的所有東西收成一張可執行的清單。每一條都有對應 Lesson 編號。

① 起手準備 (Lesson 04) 10 分鐘
  • □ 在 GitHub 按 Fork 按鈕 fork 到自己帳號
  • □ git clone 自己 fork 過的版本
  • □ git remote add upstream <原始 repo>
  • □ git remote -v 確認 origin / upstream 都在
② 同步上游 (Lesson 04) 2 分鐘
  • □ git switch main
  • □ git fetch upstream
  • □ git rebase upstream/main
  • □ git push origin main (讓自己 GitHub 上的 fork 也是最新)
③ 開分支 + 設定本地環境 (Lesson 01, 09) ~30 分鐘
  • □ 從最新 main 開 feature 分支 git switch -c fix/issue-234
  • □ 看 README 安裝依賴 (npm ci / poetry install 等)
  • □ 跑現有測試確認環境 OK (npm test)
  • □ 看是否有 Husky / pre-commit 自動 install — 沒有就 pnpm prepare 或 pre-commit install
④ 動工 — TDD 流程 (Lesson 08) 主要時間
  • □ 先寫一個失敗的測試 (RED)
  • □ 寫最少實作讓測試通過 (GREEN)
  • □ 整理程式碼 (REFACTOR)
  • □ 重複，直到功能完整
  • □ 跑全套測試確認沒打到別的東西
⑤ 寫 commit (Lesson 06) 5 分鐘
  • □ git status 確認改了哪些檔案
  • □ git add <file> 一個一個加 (避免 debug 殘留)
  • □ git diff --staged 最後檢查
  • □ Conventional Commits 格式：fix(scope): subject
  • □ 如果是 breaking change 加 ! 或 footer 寫 BREAKING CHANGE
  • □ Body 寫清楚 why (不是 what)
  • □ Footer 加 Closes #234
⑥ Push + 開 PR (Lesson 04) 10 分鐘
  • □ git push -u origin fix/issue-234
  • □ GitHub 上點 "Compare & pull request"
  • □ 用標準 PR 描述格式：背景 / 改動 / 測試方式 / 截圖 / 是否 breaking
  • □ 連結 issue: Closes #234
  • □ 還沒準備好被 review → 開成 Draft PR
  • □ 確認 CI 全綠
⑦ 處理 Code Review (Lesson 05) ~數天
  • □ 收到留言不要急著回應，先全部讀完
  • □ 每條 comment 至少留 Done / Won't fix because... / Discussed offline
  • □ 不同意的有理由就提出討論，不是無條件接受
  • □ 改完 push 新 commit (不要改寫歷史，等 review 通過再整理)
  • □ 全部 comment 解掉後留 "Ready for re-review"
⑧ 處理衝突 (Lesson 02, 04) 看狀況
  • □ git fetch upstream
  • □ git rebase upstream/main
  • □ 在 IDE 解衝突
  • □ git add <files> + git rebase --continue
  • □ git push --force-with-lease (不要用 --force)
⑨ 合併 + 慶祝 5 分鐘
  • □ 等待 maintainer 合併 (依 CONTRIBUTING 的政策決定 squash / merge / rebase)
  • □ Issue 自動關閉 (因為 PR 描述有 Closes #234)
  • □ 把本地 fork 的 main 重新同步 upstream/main
  • □ 刪掉自己的 feature branch
  • □ 慶祝 — 把這個 PR 加進你的個人作品集 / LinkedIn
07 — 常見陷阱

新手最常踩的 8 個雷

  1. Force push 自己的 main — 嚴重災難，整個分支歷史毀掉。永遠在 feature branch 操作，不要碰 main。
  2. PR 太大 — 1000+ 行的 PR 沒人想 review。一個 PR 一個目的，盡量 < 400 行 diff。
  3. Commit history 太亂 — fix、asdf、finally working 這種 commit 滿天飛。送 PR 前用 rebase -i 整理。
  4. 沒回應 review comment — 改完就 push 不留言，reviewer 不知道你採納了哪些建議。每條至少回 Done。
  5. 忘記 sync upstream — 兩週沒同步，PR 衝突爆炸。每次開新分支前先 fetch + rebase upstream。
  6. 提交未跑測試的程式碼 — push 之前在本地跑 npm test + npm run lint。CI 紅燈讓 reviewer 不會看你 PR。
  7. 無視 CONTRIBUTING 的 commit 規範 — 有些專案要求 conventional commits、有些要 DCO sign-off。違反會被退件重做。
  8. PR 描述寫 "fix" — Reviewer 不知道你在做什麼。Lesson 04 的標準範本一定要填。
08 — 好/壞範例

第一次貢獻雙重對照

Issue 範例

✗糟糕的第一個 issue
不能用

無重現、無環境、無預期 vs 實際。Maintainer 只能回「請補資訊」。

✓好的第一個 issue
[Bug] DatePicker placeholder 在 Safari 17.4 顯示為亂碼

含完整重現步驟、環境資訊、截圖、可能的猜測。

PR 範例

✗糟糕的第一個 PR
title: update
desc: (空白)
commits: fix · WIP · finally · oops

Reviewer 完全不知道在做什麼，直接退件。

✓好的第一個 PR
title: fix(date-picker): handle Safari 17.4 placeholder render
desc: 完整 5 段描述
commits: 1 個乾淨的 conventional commit

Reviewer 5 秒進入狀況，可以直接看程式碼。

09 — 總回顧

12 堂課重點反向問答

Lesson 01：Working Tree、Staging、Local Repo、Remote 分別是什麼？
編輯中的檔案 → add 進 Staging → commit 進 Local → push 上 Remote。
Lesson 02：rebase 的黃金守則是什麼？
不要對「已經 push 到別人會用的公開分支」做 rebase。
Lesson 03：你 8 人 SaaS 團隊應該用什麼分支策略？
GitHub Flow — main + 短命 feature branch + PR。
Lesson 04：origin 跟 upstream 的差別？
origin = 你 fork 的版本；upstream = 原始專案。
Lesson 05：Reviewer 看到只剩 nit 級問題該按 Approve 還是 Request changes？
Approve + 留下 nit comment。Request changes 留給真正 blocking 的問題。
Lesson 06：版本 1.4.5 加一個新功能，下一版是？
1.5.0 (MINOR +1，PATCH 重設為 0)。
Lesson 07：good first issue 標籤代表什麼？
Maintainer 評估過範圍適合新手、有清楚說明、不需熟悉 codebase 的 issue。
Lesson 08：TDD 三步驟順序？
Red (寫失敗測試) → Green (最少實作通過) → Refactor (整理)。
Lesson 09：Continuous Delivery vs Continuous Deployment 差別？
Delivery = 自動到 staging 但 production 需手動按按鈕；Deployment = 合 main 直接上 production。
Lesson 10：ADR 的決策被推翻後怎麼處理？
把 Status 改成 Superseded by ADR-N 並保留所有內容，永不刪除。
Lesson 11：xz utils 後門的最大教訓是？
開源專案的單一維護者是攻擊者下手點，需要多人維護 + SBOM + artifact signing。
Lesson 12：開始動工前應該先讀哪 5 個檔案？
README.md / CONTRIBUTING.md / CODE_OF_CONDUCT.md / SECURITY.md / LICENSE。
10 — 畢業任務

三個畢業任務 (難度遞增)

Level 1 — 練習用 first-contributions

到 firstcontributions/first-contributions 走完一次 fork → clone → 修改 → push → PR 流程。雖然只是加自己名字到 list，但能跑通整套流程。30 分鐘搞定。

Level 2 — 改一個你常用工具的 docs typo

挑一個你天天用的工具 (你的編輯器外掛、你公司用的 npm 套件、Vue / React / Vite...)。在 docs/ 找錯字、過時範例、缺中文翻譯，送一個小 PR。關鍵是讓你「真的"擁有"一個被合併的開源 PR」。1-2 小時搞定。

Level 3 — 修一個 good first issue 的小 bug

到 goodfirstissue.dev 找一個你語言熟、看得懂的 small bug。完整走過 TDD / Conventional Commits / PR 流程，並回應 review。這是「真貢獻」的開始。視 bug 大小 1 天到 1 週。

✓
三題都過了，恭喜畢業！ 你現在已經具備加入任何大型開源專案、或進公司寫大型專案的協作能力。剩下的就是不斷實戰累積。繼續送 PR、繼續 review 別人 PR — 這是進步最快的方式。
