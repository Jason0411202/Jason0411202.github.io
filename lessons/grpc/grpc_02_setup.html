環境安裝與工具鏈

01 — 工具鏈總覽

四個工具,各司其職

很多人裝環境裝到放棄,是因為不知道哪個工具負責什麼。先把四件事釐清:

工具是什麼裝在哪本課版本
go Go 編譯器與套件管理 系統(brew/winget/官方安裝包) 1.24+ (官方建議「最新兩個主版本之一」)
protoc Protocol Buffers 編譯器(C++ 寫的) 系統(brew/apt/winget) 33+ (本課寫於 2026/04,當前 v34.1)
protoc-gen-go protoc 的 plugin,把 message 生成 Go struct Go bin($(go env GOPATH)/bin) @latest
protoc-gen-go-grpc protoc 的 plugin,把 service 生成 Go gRPC stub Go bin @latest

02 — 安裝 Go

確認 Go 版本

gRPC-go 官方文件寫:「any one of the two latest major releases of Go」 — 用最新或前一個主版本即可。寫這份教材時(2026/04)是 Go 1.24/1.25。

bash 
# 看現在裝的版本
$ go version
go version go1.24.2 darwin/arm64

# macOS 用 Homebrew
$ brew install go

# Windows 用 winget
$ winget install GoLang.Go

# Linux 直接抓官方 tarball 最穩(套件庫的版本通常落後)
$ curl -LO https://go.dev/dl/go1.24.2.linux-amd64.tar.gz
$ sudo tar -C /usr/local -xzf go1.24.2.linux-amd64.tar.gz
!
確認 GOPATH/bin 在 PATH 內 下一節要用 go install 裝 plugin,它會放到 $(go env GOPATH)/bin。如果 PATH 沒包到這個目錄,protoc 會找不到 plugin。 
export PATH="$PATH:$(go env GOPATH)/bin"
把這行加進你的 ~/.zshrc / ~/.bashrc / Windows 環境變數。
03 — 安裝 protoc

protoc — Protocol Buffers 編譯器

protoc 本體是一個 C++ 寫的命令列工具,負責把 .proto 解析成中間表示(descriptor),再交給 plugin 生成各語言的 code。三個平台的官方推薦裝法:

macOS 
$ brew install protobuf
$ protoc --version
libprotoc 34.1
Linux (Debian/Ubuntu) 
# 套件庫版本(通常較舊)
$ sudo apt install -y protobuf-compiler
$ protoc --version

# 想要最新版,從 GitHub Release 抓 pre-built binary
$ curl -LO https://github.com/protocolbuffers/protobuf/releases/download/v34.1/protoc-34.1-linux-x86_64.zip
$ unzip protoc-34.1-linux-x86_64.zip -d $HOME/.local
$ export PATH="$PATH:$HOME/.local/bin"
Windows 
> winget install protobuf
> protoc --version
libprotoc 34.1
i
確保版本 ≥ 33.0 官方 2026 文件示例使用 v33+,某些較新 proto3 行為(例如 well-known types 的某些 import path)在舊版會出錯。如果 apt 裝起來的版本太舊,改用 GitHub Release pre-built。
04 — 兩個 plugin

protoc-gen-go 與 protoc-gen-go-grpc

protoc 自己只懂 .proto 的語法,真正把它變成 Go code 的是兩個 plugin。它們是 Go 寫的,所以用 go install 裝:

bash 
# 1. message → Go struct(處理資料結構)
$ go install google.golang.org/protobuf/cmd/protoc-gen-go@latest

# 2. service → Go gRPC stub(處理 RPC)
$ go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest

# 確認兩個工具都在 PATH 上
$ which protoc-gen-go protoc-gen-go-grpc
/Users/jason/go/bin/protoc-gen-go
/Users/jason/go/bin/protoc-gen-go-grpc
i
為什麼是「兩個」 plugin 而不是一個? 因為 protobuf(資料序列化)和 gRPC(RPC 協定)是兩件獨立的事 — 你可以只用 protobuf 不用 gRPC。Go 官方把它們拆成兩個倉庫、兩個 plugin,這樣 protobuf 升版不會綁死 gRPC,反之亦然。--go_out 處理 message,--go-grpc_out 處理 service。
05 — 第一支 hello.proto

跑通完整流程:從 .proto 到生成 Go 檔

建一個最小專案來驗證環境。整個目標是讓你看到「.proto 寫進去 → 兩個 .pb.go 跑出來」。

1
建立專案
新開資料夾、初始化 Go module。
2
寫 hello.proto
定義一個最小的 service 與 message。
3
執行 protoc
同時呼叫兩個 plugin,生成 Go code。
4
確認生成檔
應該會多兩個 *.pb.go 檔。

Step 1 — 建立專案

bash 
$ mkdir grpc-hello && cd grpc-hello
$ go mod init example.com/hello
$ mkdir proto

Step 2 — 寫 proto/hello.proto

proto/hello.proto 
syntax = "proto3";

package hello;

// Go 生成檔的 import path,必填
option go_package = "example.com/hello/proto;hellopb";

service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest {
  string name = 1;
}

message HelloReply {
  string message = 1;
}
!
option go_package 必填 沒有它 protoc-gen-go 會報錯。格式是 "完整 import path;package 別名"。本例中生成檔會放到 proto/ 資料夾,Go package 名是 hellopb。第 3 課會深入講 proto 語法。

Step 3 — 執行 protoc

bash 
$ protoc \
    --go_out=. --go_opt=paths=source_relative \
    --go-grpc_out=. --go-grpc_opt=paths=source_relative \
    proto/hello.proto

這條指令的四個 flag 含意:

Flag作用
--go_out=.呼叫 protoc-gen-go plugin,輸出根目錄為當前目錄
--go_opt=paths=source_relative生成檔放到「跟 .proto 同一個資料夾」(否則會依 go_package 路徑深層展開)
--go-grpc_out=.呼叫 protoc-gen-go-grpc plugin,生成 gRPC 部分
--go-grpc_opt=paths=source_relative同上,放到 .proto 旁邊

Step 4 — 確認結果

bash 
$ tree proto/
proto/
├── hello.proto
├── hello.pb.go            # message 變成的 Go struct(由 protoc-gen-go 生成)
└── hello_grpc.pb.go       # service 變成的 gRPC stub(由 protoc-gen-go-grpc 生成)

這兩個檔案不要手動編輯(每次 protoc 都會覆蓋),也建議 commit 進 git(這樣 CI 不需要裝 protoc)— 後者實務上有兩派意見,本課傾向 commit 進去。

✓
環境驗證完成! 看到兩個 *.pb.go 檔出現,代表你的 gRPC 開發環境已經就緒。第 3 課會深入講 proto 語法,第 4 課會把這個 hello service 寫到能跑。
06 — 認識 buf

buf — 現代化的 protoc 替代方案

protoc 用久了會發現幾個痛點:

  • flag 一長串,每個專案都要寫一份 Makefile
  • 無法 import 跨專案的 proto(要自己處理 --proto_path)
  • 沒有 lint、沒有 breaking change 偵測
  • plugin 版本管理一團亂

buf(buf.build)是 Buf 這家公司開源的工具,把上述痛點全部解決:用一份 buf.yaml 配置取代散亂的 flag,還能 lint proto 風格、偵測破壞性改動、版本鎖定 plugin。

bash 
# macOS
$ brew install bufbuild/buf/buf

# 後續操作(等於上面那長串 protoc)
$ buf generate
$ buf lint
$ buf breaking --against '.git#branch=main'
i
本課還是用 protoc 教 因為 protoc 是 source of truth、官方範例都用它,先學會基礎再用 buf 加速。中大型專案幾乎都會切到 buf,你工作時可以再轉。
07 — 疑難排解

常見錯誤對照表

protoc-gen-go: program not found or is not executable 最常見

protoc 找不到 plugin。原因 99% 是 PATH 沒有包到 $(go env GOPATH)/bin。

$ go env GOPATH
/Users/jason/go

$ echo 'export PATH="$PATH:$(go env GOPATH)/bin"' >> ~/.zshrc
$ source ~/.zshrc
Missing go_package option in proto file 常見

.proto 沒寫 option go_package = "...";。每個 .proto 都必填這行,protoc-gen-go 用它決定 Go import path。

生成檔被放到很深的子資料夾,而不是 .proto 旁邊 常見

沒加 --go_opt=paths=source_relative(以及對應的 --go-grpc_opt=paths=source_relative)。預設行為會依 go_package 的 import path 展開出 example.com/hello/proto/hello.pb.go 這種深層結構,加上這個 flag 才會放到 .proto 旁邊。

生成檔內容怪怪的,跟 grpc.io 教學長得不一樣 進階

可能是 plugin 太舊或太新,生成 API 有變。protoc-gen-go-grpc v1.5 起改用 NewClient 介面,舊版生成的 stub 用 NewXxxServiceClient(*grpc.ClientConn)。重新 go install ...@latest 一次。
