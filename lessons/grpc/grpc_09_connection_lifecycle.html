01 — Channel 狀態機

ClientConn 不是單條 TCP — 是一個狀態機

第 3 課透過 conn := grpc.NewClient(...) 取得的 conn(*grpc.ClientConn),官方文件稱之為 channel。它對應「一個邏輯 connection」,內部可能是 0 條或多條 TCP(取決於 LB 策略)。

圖 9.1 — ClientConn 五種狀態
狀態意思
IDLE剛建立,尚未有 RPC 發出,無實際 connection
CONNECTING正在做 TCP + TLS handshake
READYHTTP/2 connection 已就緒,可送 RPC
TRANSIENT_FAILUREconnection 失敗,進指數退避 retry 中
SHUTDOWN呼叫了 conn.Close(),不會再連

取得當前狀態:

go 
import "google.golang.org/grpc/connectivity"

state := conn.GetState()                 // connectivity.State
if state == connectivity.Ready { ... }

conn.Connect()                           // 主動觸發 idle → connecting(平常不必)
conn.WaitForStateChange(ctx, currentState)
i
NewClient lazy 的延伸 第 3 課提過 NewClient 不立刻連線。具體而言,channel 會留在 IDLE,直到第一支 RPC 才轉 CONNECTING。因此「app 啟動成功」不代表「下游服務可達」 — 若需預先驗證,可呼叫 conn.Connect() 並等待 Ready。
02 — Keepalive

偵測「半開」死 connection

長 connection 最大的隱患為「半開」（half-open）狀態：雙方的 TCP 看似仍存在，但中間的 NAT 或 LB 已將 connection 狀態回收。若無 keepalive，將於下一次 RPC 才察覺 connection 已失效——此時 RPC 已經失敗。

Client 端

go (client) 
import "google.golang.org/grpc/keepalive"

conn, err := grpc.NewClient(addr,
    grpc.WithTransportCredentials(creds),
    grpc.WithKeepaliveParams(keepalive.ClientParameters{
        Time:                30 * time.Second,    // 多久 idle 後送 ping
        Timeout:             10 * time.Second,    // ping 後多久沒回視為死
        PermitWithoutStream: true,                 // 即使沒活躍 RPC 也送 ping
    }),
)

Server 端 — 注意 EnforcementPolicy

Server 預設會 enforce 「同一 client 不能太頻繁送 ping」(預設下限 5 分鐘)。若 client 設 30 秒 ping,會被 server 視為「bad ping」,累積到 MaxPingStrikes 後送出 GOAWAY 中斷 connection。兩端設定需對齊:

go (server) 
s := grpc.NewServer(
    grpc.KeepaliveEnforcementPolicy(keepalive.EnforcementPolicy{
        MinTime:             10 * time.Second,    // 允許 client ping 的最小間隔
        PermitWithoutStream: true,                 // 允許沒 stream 時 ping
    }),
    grpc.KeepaliveParams(keepalive.ServerParameters{
        Time:    30 * time.Second,
        Timeout: 10 * time.Second,
    }),
)
!
容易忽略的細節:client 設 keepalive 後遭 server 中斷 connection 症狀是 client log 持續出現 connection error: too many pings。原因在於 server EnforcementPolicy 的 MinTime 大於 client Time。調整 server,或將 client 的 Time 設為大於 server MinTime。
03 — Name Resolver

Target string 如何被解析

第 3 課提過 grpc.NewClient(target, opts...) 的 target 支援多種 scheme:

Target意思
foo.example.com:443預設 DNS 解析(NewClient 預設)
dns:///foo.example.com:443顯式 DNS
dns://8.8.8.8/foo.example.com指定 DNS server
unix:///var/run/x.sockUnix socket(本機 IPC)
passthrough:///localhost:50051不解析,直接連

DNS 回傳多個 A record 時,每一個都會變成一個 backend address — 這是 client LB 的基礎。

04 — Load Balancing

client round_robin 負載均衡

gRPC 的 LB 設計上「client LB」是 first-class — 不需要 NLB / HAProxy 居中。將解析取得的所有 backend address 逐項展開,各自建立 sub-channel,RPC 以 round-robin 分發。

啟用 round_robin

go (client) 
conn, err := grpc.NewClient(
    "dns:///api.example.com:443",            // 必須是會回多個 address 的 target
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultServiceConfig(`{"loadBalancingConfig":[{"round_robin":{}}]}`),
)

之後每支 RPC 會輪流送往不同 backend。當新 backend 加入(DNS 多出一筆 A record)或舊 backend 失效(connection 失敗),內部 sub-channel pool 會自動更新。

i
k8s headless service + DNS k8s 的 ClusterIP: None headless service 會讓 DNS 直接回傳所有 pod 的 IP。配合 dns:///my-svc.my-ns.svc.cluster.local:50051,client 會直接 round_robin 到所有 pod,不經過 kube-proxy NLB,是 gRPC + k8s 的推薦做法。
05 — Health Check

標準健康檢查協定:grpc.health.v1.Health

gRPC 定義了標準 health checking service(grpc.health.v1),兩個 method:

  • Check(HealthCheckRequest) — unary,問「你健康嗎?」
  • Watch(HealthCheckRequest) — server stream,持續推送健康狀態變化

Server 端啟用

go (server) 
import (
    "google.golang.org/grpc/health"
    healthpb "google.golang.org/grpc/health/grpc_health_v1"
)

s := grpc.NewServer()
hs := health.NewServer()
healthpb.RegisterHealthServer(s, hs)

// 設定整體狀態
hs.SetServingStatus("", healthpb.HealthCheckResponse_SERVING)
// 或 per-service 狀態
hs.SetServingStatus("greeter.v1.Greeter", healthpb.HealthCheckResponse_SERVING)

// 收到 SIGTERM 時切 NOT_SERVING、等 in-flight RPC 完成
<-shutdownCh
hs.Shutdown()  // 把所有狀態改 NOT_SERVING
s.GracefulStop()

Client 啟用「不健康自動跳過」

go (client) 
conn, err := grpc.NewClient("dns:///svc:50051",
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultServiceConfig(`{
        "loadBalancingConfig": [{"round_robin": {}}],
        "healthCheckConfig": {"serviceName": ""}
    }`),
)

啟用後 client 會對每個 sub-channel 自動 Watch server 健康狀態,NOT_SERVING 的 backend 自動被跳過(RPC 不會送往該 sub-channel)。健康狀態恢復後自動重新加入。

i
k8s liveness/readiness 也認得 k8s 1.24+ 內建 grpc-health-probe 工具(後又被 kubelet 原生支援)— Pod 只要實作 grpc.health.v1,readiness probe 直接設定 grpc: type 即可使用,不必額外 sidecar。
06 — Retry Policy

用 service config 自動 retry

第 6 課提過 UNAVAILABLE 適合 retry。grpc-go 內建支援自動 retry — 透過 service config 設定 retry policy。

json (service config) 
{
  "methodConfig": [{
    "name": [{ "service": "greeter.v1.Greeter" }],
    "retryPolicy": {
      "maxAttempts": 4,
      "initialBackoff": "0.1s",
      "maxBackoff": "2s",
      "backoffMultiplier": 2,
      "retryableStatusCodes": ["UNAVAILABLE"]
    }
  }]
}

套用至 client:

go (client) 
const svcCfg = `{
  "methodConfig": [{
    "name": [{"service": "greeter.v1.Greeter"}],
    "retryPolicy": {
      "maxAttempts": 4,
      "initialBackoff": "0.1s",
      "maxBackoff": "2s",
      "backoffMultiplier": 2,
      "retryableStatusCodes": ["UNAVAILABLE"]
    }
  }]
}`

conn, err := grpc.NewClient(addr,
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultServiceConfig(svcCfg),
)
!
retry 前確認 idempotency(再次強調) 自動 retry 對 read 操作(GetUser、ListItems)幾乎一定安全;對 write 操作(CreateUser、Charge)不一定 — 第 6 課提過,write 需要 idempotency-key 防重複。建議:retry policy per-method 設,write method 不要納入。
07 — 系列總結

9 堂課,從零到上線

整套 gRPC 課程到此完成。以下回顧各堂重點:

  1. L01 概念與環境 — RPC 心智模型、HTTP/2、四種 RPC 模式,以及 Go + protoc + 兩個 plugin 完整安裝
  2. L02 Protobuf — message、field number、scalar、enum、oneof、map、reserved
  3. L03 Unary — 第一支 server / client(用新版 NewClient)
  4. L04 Streaming — server / client / bidi 三種模式 + io.EOF
  5. L05 Context — metadata、deadline propagation、cancellation
  6. L06 Errors — 16 種 status codes + structured details
  7. L07 Interceptors — logging、recovery、auth、timing 四件組
  8. L08 Auth — TLS、mTLS、PerRPC token
  9. L09 connection(本堂)— channel 狀態、keepalive、LB、health、retry

後續進階方向

  • 觀測性 — 串接 OpenTelemetry(otelgrpc),一併處理 trace / metrics / logs
  • buf 工具鏈 — 大型專案的 proto 管理、lint、breaking detection
  • gRPC-Gateway — 自動把 gRPC 包成 REST(對外 API 場景)
  • xDS — 跨服務的動態配置與 LB(Envoy / Istio 用的)
  • Protobuf Editions — 等 gRPC 官方推薦升級時再切
✓
至此已具備將 gRPC 服務上線的能力 剩下的差距只在「工程實踐」:CI/CD、observability、SRE。本系列著重 gRPC 本身,將其銜接到既有的基礎設施(stockbot-long-backend-k8s-infra 之類)即可直接運用。
