Lesson 01 · gRPC 概念篇

gRPC 是什麼:HTTP/2 上的 RPC

01 — RPC 心智模型

RPC 是什麼:讓「呼叫遠端函式」像呼叫本地函式

RPC = Remote Procedure Call(遠端程序呼叫)。gRPC 官方文件第一段話直接點題:

"
gRPC 的核心理念 「a client application can directly call a method on a server application on a different machine as if it were a local object」 — 在不同機器上的方法,寫起來像呼叫本地物件。

聽起來很美,但這是個漏抽象(leaky abstraction)。本地函式呼叫不會失敗、不會超時、不會中途斷線、不會序列化錯誤;遠端呼叫每一項都會。所以實務上 RPC 框架要解決的不是「讓你忘記它在遠端」,而是把那些「遠端會發生的麻煩事」變成可控的工具。gRPC 之所以值得學,是因為它把這些麻煩事都處理得不錯:序列化、傳輸、錯誤、超時、取消、重試、認證、串流 — 都有官方的標準做法。

02 — vs REST

和 REST 的真正差別,不是「快」

很多介紹會說「gRPC 比 REST 快」。這只說對一半。真正的差別是抽象層次:

面向RESTgRPC
抽象單位 資源(URL) + HTTP 動詞 方法(method) + 訊息(message)
API 契約 OpenAPI / 口頭約定 / Markdown .proto 檔(編譯期就有型別)
序列化 JSON(文字) Protobuf(二進位)
傳輸 HTTP/1.1 為主 HTTP/2(必要)
串流 需另外用 SSE / WebSocket 四種模式天生內建
瀏覽器直接呼叫 原生支援 需 gRPC-Web 代理
i
用一句話分:REST 思考「資源」,gRPC 思考「動作」 REST 寫 GET /users/42(對 user 資源做讀取);gRPC 寫 UserService.GetUser(id=42)(呼叫 GetUser 方法)。前者適合對外 public API、CRUD;後者適合服務之間互相呼叫(microservice-to-microservice)、有複雜串流需求。

「快」是怎麼來的

真要比性能,gRPC 比 REST + JSON 快通常來自三件事:

  • Protobuf 序列化比 JSON 小且快(二進位、無 key 字串重複)
  • HTTP/2 多工避免 HTTP/1.1 的隊頭阻塞(head-of-line blocking)
  • 長連線重用省下反覆 TCP/TLS 建立成本

但「快多少」高度取決於 payload 大小、網路、語言實作。對短連線、小 payload,REST 不見得慢多少。選 gRPC 不該只為了快,應為了型別契約與串流模型。

03 — 為何 HTTP/2

HTTP/2 提供的三件 gRPC 必要的東西

gRPC 規格強制使用 HTTP/2,不是選擇題。原因是 HTTP/1.1 有三個能力做不到:

圖 13.2 — 隊頭阻塞 vs 多工 stream

三個 HTTP/2 必要特性

  1. 二進位 framing — HTTP/2 把訊息拆成 frame,不再是純文字。一個 frame 屬於一個 stream,多個 stream 可以在同一條 TCP 上交錯傳輸,這就是「多工」(multiplexing)。
  2. 全雙工 stream — 同一個 stream 上 client 和 server 可以同時收送訊息。這直接對應 gRPC 的 bidirectional streaming(第 5 課會看到)。
  3. HPACK header 壓縮 — Header 經常重複(例如 :authority、content-type),HPACK 用索引表壓縮,大量小請求時頻寬節省明顯。
!
HTTP/2 ≠ 自動快 HTTP/2 在「同一條連線多請求」時最有效。如果你的 client 每次呼叫都建一條新 connection、用完就關,HTTP/2 的好處幾乎全失效。這也是為什麼 gRPC 推薦長連線重用(第 10 課的 channel/keepalive)。
04 — 為何 Protobuf

Protobuf:契約即程式碼

Protocol Buffers(簡稱 protobuf)是 Google 2008 開源的序列化格式 + IDL(Interface Definition Language)。gRPC 預設用 protobuf,但兩者是分開的東西 — gRPC 也可以用 JSON,protobuf 也可以脫離 gRPC 單獨用。

protobuf 給你的三個東西

  1. 跨語言型別契約 — 寫一份 .proto,用 protoc 編譯出 Go / Java / Python / TypeScript 的型別定義。前後端、跨服務不會再為「這個欄位是 int 還是 string」吵架。
  2. 緊湊二進位 — 一個 int32 多數情況用 1–5 byte(varint 編碼),JSON 至少要 1 byte 的引號 + N byte 的數字字串 + key 字串。同樣訊息 protobuf 通常比 JSON 小 30–50%。
  3. 可演進的 schema — 透過欄位編號(field number)實現「新加欄位、舊版客戶端忽略」、「移除欄位、舊資料還能讀」。這是 protobuf 設計上最重要的取捨,第 3 課會詳講。
JSON寫起來
{"user_id": 42, "name": "Jason", "active": true}

63 bytes,key 重複佔位

PB傳輸時
08 2A 12 05 4A 61 73 6F 6E 18 01

11 bytes(欄位編號 + tag + 值),小約 80%

05 — 四種 RPC 模式

gRPC 支援的四種呼叫模式

本堂只做概念介紹,實作分別在第 16(Unary)、17(三種 streaming)講。

圖 13.3 — 四種 RPC 呼叫模式
模式Client → ServerServer → Client典型場景
Unary1 條訊息1 條訊息取單筆資料、寫入單筆
Server streaming1 條訊息N 條訊息訂閱、log tail、大查詢結果分批送
Client streamingN 條訊息1 條訊息大檔案分塊上傳、批次寫入
BidirectionalN 條訊息N 條訊息(可交錯)即時聊天、雙向同步、雙向協商
i
「Stream」≠「TCP 一直連著」 gRPC 的 stream 是 HTTP/2 stream 的概念,在邏輯上是一條訊息序列。底層連線(channel)是長存的,但 stream 本身有開始與結束。
06 — 何時不該用

不適合 gRPC 的三種情境

用對地方比學會它更重要。下列情況通常 REST 或其他方案更合適:

✗不要用 gRPC
  • 瀏覽器直連 backend — 瀏覽器無法直接發 HTTP/2 trailer,需 gRPC-Web 代理(Envoy / nginx),架構變複雜。直接 REST 或 GraphQL 通常較好。
  • 對外公開 API — 第三方開發者熟悉 REST + JSON,要他們 protoc 編譯 stub 阻力大。除非客戶是企業技術團隊。
  • 單純的 CRUD + 表單 — REST 對應 HTTP 動詞天然契合,gRPC 反而要為 GET/POST 各定一個 method。
  • 強烈依賴 HTTP 中介設施 — 某些舊 LB / WAF / debug proxy 只懂 HTTP/1.1,gRPC 流量會卡住或被誤判。
✓適合 gRPC
  • 內部微服務之間呼叫 — 高頻、低延遲、契約緊、不需要被瀏覽器直接看到。
  • 多語言團隊 — Go server、Python data pipeline、Java client 共用同一份 .proto。
  • 串流場景 — 即時報價、log tail、雙向同步,server streaming / bidi 比 SSE/WebSocket 更結構化。
  • 頻寬與序列化敏感 — 行動裝置、大量小訊息、IoT。
