01 — RPC 心智模型

RPC 是什麼:讓「呼叫遠端函式」像呼叫本地函式

RPC = Remote Procedure Call（遠端程序呼叫）。gRPC 官方文件第一段話直接點題：「a client application can directly call a method on a server application on a different machine as if it were a local object」——在不同機器上的 method，寫起來像呼叫本地物件。

但在這層「像本地」的表象之下，gRPC runtime 替開發者處理了一連串遠端通訊才會遇到的工作。一次看似單純的呼叫，背後實際承擔了多項處理：

圖 1.1 — gRPC runtime 統一封裝 serialization、傳輸與錯誤處理

runtime 區塊各項目的作用

上圖中 runtime 區塊列出的八個項目，逐一對應遠端呼叫必須處理的問題。下表各以一句話說明：

項目一句話說明
serialization將記憶體中的 request／response 物件編碼成可於網路傳輸的位元組（Protobuf 二進位），接收端再解碼還原。
HTTP/2 framing將 message 切成 HTTP/2 frame，使多個呼叫能在同一條 connection 上交錯並行傳輸（多工）。
deadline為單次呼叫設定最晚必須完成的時間點，timeout 即自動取消，避免無限等待。
metadata隨呼叫附帶的 key-value header（如 authentication token、trace id），作用類比 HTTP header。
retry呼叫遇到暫時性失敗時依策略自動重送，降低偶發網路抖動的影響。
load balance在多個 server 實例之間分配 request，分散負載並提高可用性。
tls以 TLS 加密 connection，確保傳輸內容不被竊聽或竄改。
auth驗證呼叫端身分（如 token、mTLS certificate），決定是否允許存取。

其中 serialization 預設採用 Protocol Buffers（protobuf）——以 .proto 定義跨語言 type 契約，傳輸時編碼為緊湊二進位（同樣 message 通常比 JSON 小 30–50%）。protobuf 的格式設計與 schema 演進規則於第 2 課詳述。

02 — 環境安裝

環境安裝:裝齊 gRPC 開發工具鏈

依以下三個步驟由上而下執行，每步選擇對應作業系統的指令於終端機執行。完成步驟三、兩個 plugin 皆能印出版本，即代表開發工具鏈安裝完成。

步驟一 · 安裝 Go 並設定 PATH

首先安裝 Go 編譯器：

bash 
# macOS:用 Homebrew 安裝 Go 編譯器
$ brew install go

# Windows:用 winget 安裝 Go 編譯器
> winget install GoLang.Go

# Linux:下載官方壓縮檔
$ curl -LO https://go.dev/dl/go1.24.2.linux-amd64.tar.gz
# Linux:解壓到 /usr/local
$ sudo tar -C /usr/local -xzf go1.24.2.linux-amd64.tar.gz

# 印出版本,確認安裝成功
$ go version

接著將 Go 的 bin 目錄加入 PATH（後續安裝的 plugin 會放置於此）：

bash 
# macOS(zsh):把 Go 的 bin 目錄寫進設定檔
$ echo 'export PATH="$PATH:$(go env GOPATH)/bin"' >> ~/.zshrc
# 重新載入設定檔,讓 PATH 立即生效
$ source ~/.zshrc

# Linux(bash):把 Go 的 bin 目錄寫進設定檔
$ echo 'export PATH="$PATH:$(go env GOPATH)/bin"' >> ~/.bashrc
# 重新載入設定檔,讓 PATH 立即生效
$ source ~/.bashrc

# Windows(PowerShell):把 Go 的 bin 目錄加進 PATH(需重開終端機)
> setx PATH "$env:PATH;$(go env GOPATH)\bin"
i
為什麼選用 Go？語言其實可自由選擇 gRPC 是跨語言框架，官方支援 Go、Java、Python、C++、Node.js、C# 等多種語言；同一份 .proto 契約可生成任一語言的程式碼，client 與 server 也能以不同語言互通。本課選用 Go，是因其編譯快速、部署簡單、官方工具鏈完整，適合作為學習範例。改用其他語言時，替換為對應的 plugin 即可，概念與流程完全相同。

步驟二 · 安裝 protoc

bash 
# macOS:安裝 protobuf 編譯器(含 protoc)
$ brew install protobuf

# Windows:安裝 protobuf 編譯器(含 protoc)
> winget install protobuf

# Linux:安裝 protobuf 編譯器(含 protoc)
$ sudo apt install -y protobuf-compiler

# 印出版本,確認安裝成功
$ protoc --version
i
protoc 是什麼？ protoc 是 Protocol Buffers 的編譯器。它讀取 .proto 契約檔，解析其中的 message 與 service 定義，再交由語言 plugin 生成對應的程式碼。其本身不直接產出 Go code，實際生成的工作由下一步安裝的兩個 plugin 完成。

步驟三 · 安裝兩個 plugin

bash 
# 安裝 plugin：將 message 生成為 Go struct
$ go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
# 安裝 plugin：將 service 生成為 gRPC stub
$ go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest

# 印出版本,確認兩個 plugin 都裝好且在 PATH 上
$ protoc-gen-go --version
$ protoc-gen-go-grpc --version
i
兩個 plugin 的對應關係 protoc-gen-go 負責「message → Go struct」（資料），protoc-gen-go-grpc 負責「service → gRPC stub」（method）。兩者皆由 protoc 在編譯時呼叫，分別輸出 *.pb.go 與 *_grpc.pb.go 兩個檔案，二者結合即構成可直接 import 使用的 gRPC 程式碼。
✓
開發工具鏈已就緒 本機現已具備 Go、protoc 與兩個 plugin，可開始開發 gRPC 服務。第 2 課深入 Protobuf 語法與 schema 演進規則；第 3 課則從建立專案開始，逐步寫出第一支可執行的 Unary RPC（proto → 生成 → server／client → 跑起來）。 → 前往第 3 課 · Unary RPC 實戰，動手建立第一支服務
