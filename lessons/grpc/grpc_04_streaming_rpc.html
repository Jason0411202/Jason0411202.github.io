Lesson 04 · Streaming 三模式

三種 Streaming RPC

從建立專案、撰寫 proto，一路到 server 與 client 完整實作並實際跑通三種 streaming：Server streaming、Client streaming 與 Bidirectional。完成本課後，即具備實作即時推送、批次上傳與雙向協商的能力，並理解 io.EOF 於 stream 收尾的角色。

01 — 流程總覽

從 0 到跑通三種 streaming

專案目錄

tree 
mychat/
├── go.mod
├── proto/
│   ├── mychat.proto
│   ├── mychat.pb.go        # 自動生成
│   └── mychat_grpc.pb.go   # 自動生成
├── server/
│   └── main.go
└── client/
    └── main.go

建立專案

先建立專案資料夾、初始化 Go module，並開出存放 proto、server、client 的目錄：

bash 
# 建立專案資料夾並進入
$ mkdir mychat && cd mychat
# 初始化 Go module(產生 go.mod)
$ go mod init example.com/mychat
# 建立放 proto、server、client 的資料夾
$ mkdir proto server client
02 — 定義 proto

Step 1 — 寫 proto/mychat.proto

proto/mychat.proto 
// 使用 proto3 語法
syntax = "proto3";

// package 名稱(以 .v1 標示版本,利於日後相容演進)
package mychat.v1;

// 生成 Go 檔的 import path(必填)
option go_package = "example.com/mychat/proto;mychatpb";

// 定義 MyChat service 及其三個 streaming method
service MyChat {
  // 1. Server streaming:訂閱頻道,server 持續推送 message
  rpc MySubscribe (MySubscribeRequest) returns (stream MyMessage);

  // 2. Client streaming:批次上傳,server 接收完畢後回傳總結
  rpc MyUploadLogs (stream MyLogLine) returns (MyUploadSummary);

  // 3. Bidirectional:雙向聊天,message 可交錯傳輸
  rpc MyChatRoom (stream MyMessage) returns (stream MyMessage);
}

// 訂閱 request:指定頻道
message MySubscribeRequest { string my_channel = 1; }
// 聊天 message:來源與內容
message MyMessage          { string my_from = 1; string my_text = 2; }
// 單行 log
message MyLogLine          { string my_line = 1; }
// 上傳總結:筆數與位元組數
message MyUploadSummary    { int32 my_count = 1; int64 my_bytes = 2; }
03 — 生成 stub

Step 2 — 執行 protoc 與安裝套件

bash 
# 在 mychat/ 根目錄,加入 grpc-go 與 protobuf runtime
$ go get google.golang.org/grpc@latest
$ go get google.golang.org/protobuf@latest

# 生成 stub
$ protoc \
    --go_out=. --go_opt=paths=source_relative \
    --go-grpc_out=. --go-grpc_opt=paths=source_relative \
    proto/mychat.proto

protoc 本身僅負責解析 .proto,實際生成程式碼則委派給各 --xxx_out 對應的 plugin — 此運作機制已於第 3 課圖解說明,此處不再重複。

執行後,proto 資料夾內應出現兩個生成檔:

bash 
$ ls proto/
mychat.proto
mychat.pb.go        # message 生成的 Go struct
mychat_grpc.pb.go   # service 生成的 gRPC stub

生成檔提供哪些內容

mychat_grpc.pb.go 會包含:

  • MyChatServer — server 端須實作的 interface(含 MySubscribe / MyUploadLogs / MyChatRoom 三個 method)
  • MyChatClient — client 呼叫用的 interface
  • RegisterMyChatServer(s *grpc.Server, srv MyChatServer) — server 註冊
  • NewMyChatClient(cc *grpc.ClientConn) MyChatClient — client 建構
  • MyChat_MySubscribeServer、MyChat_MyUploadLogsServer、MyChat_MyChatRoomServer 等 stream type — handler 收送 message 用
  • UnimplementedMyChatServer — 必須 embed 進 server struct(處理 forward compatibility)
i
每次改動 .proto 都需重新生成 *.pb.go 是 .proto 的衍生產物。日後若調整 message field 或 service method，須重新執行本步驟使生成碼同步，切勿手動修改生成檔（下次生成時會被覆寫）。實務上常將此指令寫入 Makefile 或腳本，以單一指令完成生成。
04 — 寫 Server

Step 3 — server/main.go

server 須 embed UnimplementedMyChatServer，實作三個 streaming method，再於 main() 啟動 listener。以下先看共用骨架，再逐一補上三個 handler。

go (server) · 骨架 
package main

import (
    "fmt"
    "io"
    "log"
    "net"
    "time"

    "google.golang.org/grpc"
    pb "example.com/mychat/proto"
)

// 定義 server type,作為 MyChat service 的實作載體。
// 內嵌 UnimplementedMyChatServer 取得 interface 所有 method 的
// 預設實作(固定寫法,提供向前相容)。
type server struct {
    pb.UnimplementedMyChatServer
}

// 三個 streaming handler(MySubscribe / MyUploadLogs / MyChatRoom)於下方逐一實作

func main() {
    // 呼叫 net 套件的 Listen() function,
    // 回傳一個 listener(lis),
    // 用來接受進 :50051 的 TCP connection
    lis, err := net.Listen("tcp", ":50051")
    if err != nil {
        log.Fatalf("failed to listen: %v", err)
    }

    // 呼叫 grpc framework 的 NewServer() method,
    // 回傳一個 gRPC 框架的 server 物件,
    // 該物件負責收 connection、解碼、把 request 分派給對應 method
    grpcServer := grpc.NewServer()

    // 呼叫生成的 RegisterMyChatServer(),
    // 把實作 &server{} 註冊進 grpcServer,
    // 框架才知道三個 method 由誰處理
    pb.RegisterMyChatServer(grpcServer, &server{})

    log.Printf("server listening on %v", lis.Addr())  // 印出實際監聽的位址

    // 呼叫 grpcServer 的 Serve() method 開始接收 request,
    // 此呼叫會一直 block 直到 server 被關閉
    if err := grpcServer.Serve(lis); err != nil {
        log.Fatalf("failed to serve: %v", err)
    }
}
!
務必 embed pb.UnimplementedMyChatServer 這是 protoc-gen-go-grpc 強制的(雖可透過 require_unimplemented_servers=false 停用,但不建議)。 其作用有二：(1) 為日後新增 method 提供「Unimplemented 預設回傳」，使 server 升級 proto 時不致編譯失敗；(2) 強制使 service 實作具備 forward-compatible 的特性。

Server streaming — MySubscribe

client 訂閱一個頻道,server 持續向其推送 message。

go (server) 
// 實作 .proto 中定義的 MySubscribe(server streaming):
//   - 接收者 (s *server):把 method 掛在 server type 上
//   - req:client 的 request(指定要訂閱的 my_channel)
//   - stream:server 端 stream 物件,以 stream.Send() 持續推送 message
//   - 回傳:error;return nil 表示正常結束 stream
func (s *server) MySubscribe(req *pb.MySubscribeRequest, stream pb.MyChat_MySubscribeServer) error {
    // stream 物件已帶呼叫 context,必要時可用 stream.Context() 取得
    for i := 1; i <= 5; i++ {
        // 組出一則 message(MyFrom = "server",MyText 帶上 my_channel 與序號)
        msg := &pb.MyMessage{
            MyFrom: "server",
            MyText: fmt.Sprintf("#%s msg %d", req.GetMyChannel(), i),
        }
        // 呼叫 stream.Send() 推送一則 message,client 斷線時此處會 return error
        if err := stream.Send(msg); err != nil {
            return err
        }
        time.Sleep(time.Second)
    }
    return nil  // return nil = 結束 stream(client 端會收到 io.EOF)
}

Client streaming — MyUploadLogs

client 將多筆 log line 上傳,server 統計後回傳一份總結。

go (server) 
// 實作 .proto 中定義的 MyUploadLogs(client streaming):
//   - 接收者 (s *server):把 method 掛在 server type 上
//   - stream:server 端 stream 物件;沒有獨立 req 參數,
//     request 改由 stream.Recv() 逐筆取出
//   - 回傳:error;以 stream.SendAndClose() 回傳總結並結束
func (s *server) MyUploadLogs(stream pb.MyChat_MyUploadLogsServer) error {
    var count int32
    var bytes int64
    for {
        // 呼叫 stream.Recv() 取出一筆 client 上傳的 log line
        line, err := stream.Recv()
        if err == io.EOF {
            // client 結束 stream → 用 SendAndClose() 回傳總結並關閉
            return stream.SendAndClose(&pb.MyUploadSummary{
                MyCount: count,
                MyBytes: bytes,
            })
        }
        if err != nil {
            return err  // 非 EOF 的真錯誤
        }
        // 累計筆數與位元組數
        count++
        bytes += int64(len(line.GetMyLine()))
    }
}

Bidirectional — MyChatRoom

雙方獨立決定何時 send、何時 close,message 交錯傳輸。

go (server) 
// 實作 .proto 中定義的 MyChatRoom(bidirectional):
//   - 接收者 (s *server):把 method 掛在 server type 上
//   - stream:server 端 stream 物件,可同時 Recv() 與 Send()
//   - 回傳:error;收到 io.EOF 後 return nil 結束
func (s *server) MyChatRoom(stream pb.MyChat_MyChatRoomServer) error {
    for {
        // 呼叫 stream.Recv() 取出 client 送來的一則 message
        in, err := stream.Recv()
        if err == io.EOF {
            return nil  // client 已不再傳送,正常結束
        }
        if err != nil {
            return err  // 非 EOF 的真錯誤
        }

        // 組出 echo response 並回送(實務上可從 broker 廣播給其他 client)
        out := &pb.MyMessage{
            MyFrom: "server",
            MyText: "echo: " + in.GetMyText(),
        }
        if err := stream.Send(out); err != nil {
            return err
        }
    }
}
!
關閉 stream 的兩個 helper
  • Server:stream.SendAndClose(resp) — 傳送 response 並結束 stream
  • Client:stream.CloseAndRecv() — 結束 send 方向並等待 server response
這是 client streaming 才有的 helper。Server streaming 與 Bidi 的 server 端是用 return nil 結束。
05 — 寫 Client

Step 4 — client/main.go

client 建立 connection、取得 stub 後,依序示範三種 RPC。先看 connection 骨架,再逐一呼叫。

go (client) · 骨架 
package main

import (
    "context"
    "io"
    "log"
    "time"

    "google.golang.org/grpc"
    "google.golang.org/grpc/credentials/insecure"
    pb "example.com/mychat/proto"
)

func main() {
    // 呼叫 grpc framework 的 NewClient() method,
    // 回傳一個 connection 物件 conn(lazy,此時還不會真的連線);
    // 第二個參數設定此 connection 不加密
    conn, err := grpc.NewClient(
        "localhost:50051",
        grpc.WithTransportCredentials(insecure.NewCredentials()),
    )
    if err != nil {
        log.Fatalf("failed to create client: %v", err)
    }
    defer conn.Close()  // main 結束前關閉 conn

    // 呼叫生成的 NewMyChatClient(),
    // 用 conn 包出一個 client stub,之後用它呼叫遠端 method
    client := pb.NewMyChatClient(conn)

    // 依序示範三種 streaming RPC
    ctx := context.Background()
    runSubscribe(ctx, client)   // Server streaming
    runUploadLogs(ctx, client)  // Client streaming
    runChatRoom(ctx, client)    // Bidirectional
}

Server streaming — 呼叫 MySubscribe

go (client) 
// Server streaming:呼叫一次,之後持續以 stream.Recv() 接收 server 推送
func runSubscribe(ctx context.Context, client pb.MyChatClient) {
    // 呼叫 MySubscribe() 取得 stream,request 指定要訂閱的 channel
    stream, err := client.MySubscribe(ctx, &pb.MySubscribeRequest{MyChannel: "general"})
    if err != nil {
        log.Fatal(err)
    }
    for {
        // 逐則接收 server 推送的 message
        msg, err := stream.Recv()
        if err == io.EOF {
            break  // server 正常結束 stream
        }
        if err != nil {
            log.Fatalf("recv error: %v", err)
        }
        log.Printf("got: %s", msg.GetMyText())
    }
}

Client streaming — 呼叫 MyUploadLogs

go (client) 
// Client streaming:連續 Send() 多筆,最後以 CloseAndRecv() 取回總結
func runUploadLogs(ctx context.Context, client pb.MyChatClient) {
    // 呼叫 MyUploadLogs() 取得 stream(client 端負責 Send)
    stream, err := client.MyUploadLogs(ctx)
    if err != nil {
        log.Fatal(err)
    }
    // 逐筆上傳 log line
    for _, l := range []string{"first", "second", "third"} {
        if err := stream.Send(&pb.MyLogLine{MyLine: l}); err != nil {
            log.Fatal(err)
        }
    }
    // 上傳完成後以 CloseAndRecv() 結束 send 方向並接收 server 總結
    summary, err := stream.CloseAndRecv()
    if err != nil {
        log.Fatal(err)
    }
    log.Printf("uploaded %d lines, %d bytes", summary.GetMyCount(), summary.GetMyBytes())
}

Bidirectional — 呼叫 MyChatRoom(goroutine 同時 send 與 recv)

go (client) 
// Bidirectional:一個 goroutine 收、主 goroutine 送,兩個方向同時進行
func runChatRoom(ctx context.Context, client pb.MyChatClient) {
    // 呼叫 MyChatRoom() 取得雙向 stream
    stream, err := client.MyChatRoom(ctx)
    if err != nil {
        log.Fatal(err)
    }

    // recv 與 send 分屬不同 goroutine — 不同方向 thread-safe,同方向則否
    done := make(chan struct{})
    go func() {
        defer close(done)
        for {
            in, err := stream.Recv()
            if err == io.EOF {
                return
            }
            if err != nil {
                log.Printf("recv error: %v", err)
                return
            }
            log.Printf("got: %s", in.GetMyText())
        }
    }()

    // 主 goroutine 負責送
    for _, text := range []string{"hi", "how are you", "bye"} {
        if err := stream.Send(&pb.MyMessage{MyFrom: "client", MyText: text}); err != nil {
            log.Fatal(err)
        }
        time.Sleep(500 * time.Millisecond)
    }

    // 通知 server「不再傳送」(server 端會收到 io.EOF),但仍會接收 server 的回覆
    stream.CloseSend()
    <-done
}
i
insecure.NewCredentials() 是什麼 用於告知 gRPC「此 connection 不加密」 — 純明文 HTTP/2(h2c)。僅適用於 localhost 開發、或 service mesh 內部已具備 mTLS 的情境。production 對外 connection 務必使用 TLS,第 8 課將詳述。
!
Bidi 的 thread-safety 規則 stream.Send 與 stream.Recv 在不同方向是 thread-safe(可一個 goroutine 收、一個 goroutine 送),但同方向多個 goroutine 同時 Send 不是 thread-safe(同方向若需多個 goroutine 寫入,須自行加上 mutex,或以 channel 匯流至單一 sender goroutine)。

07 — io.EOF 的角色

io.EOF 不是錯誤,是「正常結束」訊號

Go 的 io.EOF 在 streaming RPC 裡有明確語意:

場景Recv() 回傳
對方還會送更多(msg, nil)
對方正常結束 stream(return nil / SendAndClose / CloseSend)(nil, io.EOF)
對方異常(網路斷、server crash、deadline exceeded)(nil, <real error>) — 用 status.Code(err) 取代碼

所以慣用模式總是:

go 
for {
    msg, err := stream.Recv()
    if err == io.EOF {
        break      // 正常結束
    }
    if err != nil {
        return err  // 真錯誤
    }
    process(msg)
}
08 — 常見陷阱

Streaming 三個容易忽略的細節

1. stream.Send 沒檢查 error 高頻

Send 在「對方已斷線」、「buffer 已滿」、「context 已 cancel」等情況下會回傳 error。若忽略此 error 並繼續 send，將導致 message 遺失而無法察覺。每次 Send 後皆應檢查 err，發生錯誤即 break 並進行收尾。

2. Bidi 同一個 stream 用兩個 goroutine 同時 Send 資料毀損

Send 對同方向不是 thread-safe,兩個 goroutine 同時寫會混合 byte stream,造成 protobuf decode 失敗。解法是僅使用單一 sender goroutine,或以 channel 將待送 message 匯入單一 goroutine 依序傳送。

3. Server streaming handler 忘了關注 stream.Context().Done() 資源洩漏

長時間執行的 server streaming 應 select stream.Context().Done(),在 client 取消或斷線時及時退出。否則 server 會持續空轉迴圈,佔用 goroutine 與資源。

for {
    select {
    case <-stream.Context().Done():
        return stream.Context().Err()
    case msg := <-broker.NextMessage():
        stream.Send(msg)
    }
}
