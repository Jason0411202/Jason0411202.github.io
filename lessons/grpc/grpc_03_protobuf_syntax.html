Lesson 03 · Protobuf 語法

Protocol Buffers語法完整版

01 — message 基本

message 是 protobuf 的最小單位

每個 .proto 檔的頭三行幾乎都長這樣:

user.proto 
syntax = "proto3";          // 聲明使用 proto3 語法

package myapp.v1;             // 命名空間,避免跨檔衝突

option go_package = "example.com/myapp/v1;myappv1";

message User {
  string id    = 1;
  string name  = 2;
  int32  age   = 3;
  bool   active = 4;
}
  • syntax — 必填,目前都用 "proto3"
  • package — 邏輯命名空間。不同檔案可在同一 package,跨 package 引用要用 fully qualified name(如 auth.v1.Token)
  • option go_package — 告訴 protoc-gen-go 生成檔的 Go import path 與 package 名
  • message — 一個資料結構,由「欄位」組成,每個欄位有型別、名稱、編號三件事
i
命名慣例 message 與 service 用 UpperCamelCase,欄位用 snake_case(生成 Go 時會自動轉成 UpperCamelCase),enum 值用 UPPER_SNAKE_CASE。Google 官方 style guide 與所有主流公司都遵守。
02 — 欄位編號

欄位編號:protobuf 的命脈

每個欄位後面那個數字(= 1、= 2)叫欄位編號(field number)。它不是給人看的,是寫進二進位 wire format、用來識別欄位的關鍵。

!
欄位編號一旦在 production 用過,絕對不能改 改編號等同把舊欄位刪掉、加一個新欄位。所有舊版客戶端傳上來的資料會被當成「未知欄位」,你的 server 讀到的所有舊資料會變成預設值。這是 protobuf 工程師最常踩的雷。正確做法是:用 reserved 標記退役的編號,絕不重用。

編號範圍與成本

範圍說明實務建議
1 – 15wire format 上 tag 只佔 1 byte留給最常出現的欄位
16 – 2047tag 佔 2 bytes一般欄位
19000 – 19999protobuf 內部保留,不能用避開,protoc 會直接報錯
2048 – 536,870,911tag 佔 2+ bytes幾乎用不到
✗反模式
message User {
  string avatar_url = 1;     // 用很少
  string bio        = 2;     // 用很少
  string id         = 15;    // 每筆都有
  string name       = 16;    // 每筆都有
}

熱欄位放 16+,每筆都浪費 1 byte tag。

✓好寫法
message User {
  string id         = 1;
  string name       = 2;
  string avatar_url = 10;
  string bio        = 11;
}

高頻欄位佔 1–15。新增欄位往後加。

03 — scalar 型別

數字型別有 12 種,該怎麼挑

新手最常問:「int32 跟 sint32 跟 fixed32 差在哪?」答案是編碼方式不同,導致對不同數值範圍的效率不同。

型別編碼用在對應 Go
int32 / int64varint大多數時候用這個。負數會編成 10 byte(差),所以僅用於非負或極少負的場景int32 / int64
uint32 / uint64varint已知非負(計數、ID)uint32 / uint64
sint32 / sint64zigzag varint常出現負數時(差值、座標)int32 / int64
fixed32 / fixed64固定 4/8 byte數值通常很大(雜湊、時戳),varint 編碼反而更長uint32 / uint64
sfixed32 / sfixed64固定 4/8 byte同上但帶符號int32 / int64
float / doubleIEEE 754浮點數float32 / float64
bool1 byte布林bool
stringUTF-8必須是 UTF-8 文字string
bytes原始 byte非文字資料(圖片、檔案、加密過的內容)[]byte
i
新手選擇指南 90% 場景用 int32 / int64 / string / bool 就夠。sint* 只在「常常是負數」時值得;fixed* 只在「總是大數值」(例如 64-bit hash、Unix nanosec timestamp)時值得。
04 — repeated / optional

陣列與「該欄位有沒有設」

repeated — 0 個或多個

proto3 
message Order {
  string id = 1;
  repeated string item_ids = 2;       // 對應 Go []string
  repeated Item items = 3;             // 對應 Go []*Item
}

順序保留。scalar repeated 預設用 packed encoding(全部值連在一起共用一個 tag),省空間。

optional — 顯式表達「有沒有設」

proto3 有個歷史包袱:scalar 欄位沒辦法區分「沒設」和「設成預設值 0」。如果你需要區分(例如「年齡欄位是 0 歲嬰兒」 vs 「沒填」),就要加 optional:

proto3 
message User {
  string id = 1;
  int32 age = 2;             // 沒設 vs 0 無法區分
  optional int32 height_cm = 3;  // 可以區分(Go 生成 *int32)
}
!
scalar 預設值無法傳遞語意 proto3 中 int32 沒設等同於 0、string 沒設等同於 ""、bool 沒設等同於 false。需要區分「沒填」與「填預設值」時請加 optional。message 欄位天生就有「null/nil」概念,不需要 optional。
05 — enum

enum:零值必須是 UNSPECIFIED

protobuf enum 是有限的整數常數集。第一個值必須是 0(這是 proto3 的硬性規定),且按 Google 官方 style guide,慣例命名為 {ENUM}_UNSPECIFIED:

proto3 
enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;   // 必填,代表「沒設」
  ORDER_STATUS_PENDING     = 1;
  ORDER_STATUS_PAID        = 2;
  ORDER_STATUS_SHIPPED     = 3;
  ORDER_STATUS_DELIVERED   = 4;
  ORDER_STATUS_CANCELLED   = 5;
}

message Order {
  string id = 1;
  OrderStatus status = 2;
}

為什麼 0 要當 UNSPECIFIED?

因為 scalar 沒設等同預設值,而 enum 的預設值就是 0。如果你把 PENDING = 0,那「沒設 status」會被當成「PENDING」 — 這是隱含的錯誤,看 server 端永遠不會發現「客戶端忘了設 status」。把 0 留給 UNSPECIFIED,server 端就能 explicit 處理「客戶端沒設」這個情況。

未知值會保留

如果新版 server 加了 ORDER_STATUS_RETURNED = 6,舊版 client 收到時:

  • 不會 crash
  • 該欄位以「未知」形式保留(具體呈現方式依語言)
  • client 把訊息再轉送出去時,未知值會原封不動帶過去

這個「forward compatibility」是 protobuf 設計上的重要特性。

06 — oneof

oneof:多選一

當一組欄位最多只有一個會被設,用 oneof 表達語意比一堆 nullable 欄位清楚得多。

proto3 
message NotificationTarget {
  oneof target {
    string email   = 1;
    string phone   = 2;
    string webhook = 3;
  }
  string message = 10;
}

規則:

  • 同時設 email 與 phone → 後設的會清掉前面的
  • 不能用 repeated、不能用 map 在 oneof 裡
  • 欄位編號要在整個 message 範圍內唯一(不只 oneof 內)

生成的 Go code 提供 type-switch:

go 
switch t := target.GetTarget().(type) {
case *pb.NotificationTarget_Email:
    sendEmail(t.Email, msg)
case *pb.NotificationTarget_Phone:
    sendSMS(t.Phone, msg)
case *pb.NotificationTarget_Webhook:
    callHook(t.Webhook, msg)
case nil:
    // 沒設,error 處理
}
i
oneof 是 API 設計利器 很多 API 端點接受多種輸入(查詢可用 ID 或 Email 任一),用 oneof 表達遠比定義一堆 SearchByEmail / SearchByID 端點優雅。
07 — map

map:鍵值對

proto3 
message Resource {
  string id = 1;
  map<string, string> labels = 2;        // k8s style labels
  map<string, Tag> tags_by_key = 3;
}

限制:

  • key 只能是整數型或 string(不能是 float、bytes、enum、message)
  • value 任意,但不能是另一個 map
  • 不能加 repeated
  • 序列化順序、迭代順序未定義 — 不要依賴順序!
08 — reserved 與演進

schema 演進的安全網:reserved

當你刪掉一個欄位、或改名,正確做法是把它的編號和名稱都放進 reserved:

proto3 
message User {
  reserved 3, 5, 9 to 12;          // 這些編號絕不再用
  reserved "old_email", "legacy_id";    // 名稱也保留

  string id   = 1;
  string name = 2;
  // 編號 3 (legacy_id) 已退役,改放在 reserved 上方
  string email = 4;
}

這樣做的好處:

  • 有人手殘把新欄位編號設成 3,protoc 會直接報錯
  • 把 old_email 拿來重用當其他欄位名,protoc 也會報錯
  • line-of-sight 文件:讀 .proto 就知道哪些編號是退役的、不要碰

schema 演進的三條鐵律

  1. 不改編號 — 改編號等於刪舊增新,所有舊資料失效
  2. 不改型別(除非是 wire-format 相容的轉換,如 int32 ↔ int64 ↔ uint32;細節見官方 update guide)
  3. 退役欄位放 reserved,絕不重用編號或名稱
!
真實事故案例 某團隊把 int32 user_id = 5; 退役但沒寫 reserved。半年後新人用 5 編號加了 string user_email = 5;。Server 收到舊客戶端傳上來的整數 user_id,把那串 binary 解讀成 string,寫進 DB 的 email 欄位變成亂碼。reserved 就是用來防這種事。
09 — 巢狀與 import

跨檔引用與巢狀

巢狀 message

proto3 
message SearchResponse {
  message Result {           // 巢狀,僅在 SearchResponse 內可見
    string url = 1;
    string title = 2;
  }
  repeated Result results = 1;
}

外部要引用用 SearchResponse.Result。

跨檔 import

proto3 
syntax = "proto3";
package myapp.v1;

import "google/protobuf/timestamp.proto";   // well-known type
import "common/money.proto";                 // 自家共用

message Payment {
  string id = 1;
  google.protobuf.Timestamp paid_at = 2;
  common.Money amount = 3;
}

import 路徑由 protoc --proto_path=DIR(可重複)決定搜尋根目錄。Google 提供常用的 well-known types(Timestamp、Duration、Empty、Any、Struct…)在 google/protobuf/*.proto。

i
Protobuf Editions(2024 GA) 長期 protobuf 會用「Editions」取代 syntax = "proto3"(改成 edition = "2024"),把每個行為變成可調整的 feature。但 gRPC 官方目前文件仍以 proto3 為主流,本課程也用 proto3。等 Editions 在 gRPC 生態普及再學即可,現在學 proto3 不是白學 — 兩者 wire format 完全相容。
