Lesson 07 · 錯誤處理

錯誤處理與Status Codes

gRPC 有自己的 16 種錯誤碼,選對代碼比寫詳細 message 重要。學會這個,client 才能依錯誤類型決定:retry、prompt user、抑或記 log 就好。

01 — 為什麼有自己的 codes

HTTP status code 不夠用

REST 用 HTTP code(404、500、503…)表示錯誤。但 HTTP 本來就被瀏覽器、proxy、CDN 用走了 — 你的 backend 回 503 與 LB 回 503 client 區分不出來。所以 gRPC 自己定義一套跑在 HTTP body 上的錯誤碼,共 16 個,放在 trailer 的 grpc-status header。

i
選對 code,client 才能正確反應 每個 code 暗示retry 策略:UNAVAILABLE 可以馬上 retry、FAILED_PRECONDITION 要等系統狀態改變才該重試、NOT_FOUND retry 也沒用。Server 端隨便丟個 INTERNAL,client 永遠不知道該怎麼辦。
02 — 16 種 codes

完整 status codes 對照表

CodeID意義典型情境
OK0成功handler return nil
CANCELLED1呼叫方取消client cancel、ctx done
UNKNOWN2不明錯誤handler panic 被 recover、第三方錯不認得
INVALID_ARGUMENT3參數本身有問題email 格式錯、必填欄空
DEADLINE_EXCEEDED4超時處理時間超過 deadline
NOT_FOUND5資源不存在查不到 user_id=42
ALREADY_EXISTS6已存在email 重複註冊
PERMISSION_DENIED7沒權限已認證但無權限做這事
RESOURCE_EXHAUSTED8配額/資源耗盡rate limit、quota 用完
FAILED_PRECONDITION9系統狀態不滿足要刪非空目錄、未綁定卡片不能扣款
ABORTED10並發衝突樂觀鎖失敗、transaction 衝突
OUT_OF_RANGE11超出有效範圍seek 超過檔案結尾、page 號超過總頁數
UNIMPLEMENTED12未實作新版 client 呼叫舊版 server 的新 method
INTERNAL13內部錯誤不變式被破壞、無法分類的 server 內部 bug
UNAVAILABLE14暫時不可用server overload、正在重啟、connection 斷
DATA_LOSS15資料毀損checksum 對不上、不可逆的資料丟失
UNAUTHENTICATED16未認證無 token / token 失效

容易搞混的三組

AUNAUTHENTICATED
「你是誰?」沒問清楚

沒帶 token、token 過期、簽章無效。

BPERMISSION_DENIED
身份知道,但無權限

已登入,但這個 endpoint 你不能存取。

AINVALID_ARGUMENT
參數本身就錯

不論系統什麼狀態,這個輸入都會錯(格式、範圍)。

BFAILED_PRECONDITION
輸入本身合法,但系統狀態不對

「刪這個資料夾」合法,但它不是空的。

AUNAVAILABLE
暫時的,直接 retry 該成功

server overload、連線斷。client 該立刻 retry(指數退避)。

BFAILED_PRECONDITION
狀態問題,retry 也是錯

先去把系統狀態修對,才該重試。盲 retry 只是浪費。

03 — Server 回錯

用 status.Errorf 回傳

go (server) 
import (
    "google.golang.org/grpc/codes"
    "google.golang.org/grpc/status"
)

func (s *server) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.User, error) {
    if req.GetId() == "" {
        return nil, status.Error(codes.InvalidArgument, "id is required")
    }

    user, err := s.repo.FindByID(ctx, req.GetId())
    if errors.Is(err, ErrUserNotFound) {
        return nil, status.Errorf(codes.NotFound, "user %s not found", req.GetId())
    }
    if err != nil {
        // 不要把 DB error 直接吐給 client
        log.Printf("db error: %v", err)
        return nil, status.Error(codes.Internal, "internal error")
    }
    return user, nil
}
!
不要把 raw error 吐給 client return nil, err 直接傳原始 Go error 會被 gRPC 包成 Unknown,client 拿到也沒幫助。handler 邊界務必轉成 status.Error,並把細節 log 在 server 端(別洩漏到 client 形成資訊洩漏)。
04 — 帶結構化 details

用 status.WithDetails 帶 protobuf payload

單純的 message 字串不夠用 — client 想知道「哪一個欄位錯了」「retry 多久後再試」。WithDetails 可以塞 protobuf message 進錯誤裡。

Google 提供標準的 error details(google.rpc.*)放在 google.golang.org/genproto/googleapis/rpc/errdetails:

go (server) 
import "google.golang.org/genproto/googleapis/rpc/errdetails"

st := status.New(codes.InvalidArgument, "validation failed")
br := &errdetails.BadRequest{
    FieldViolations: []*errdetails.BadRequest_FieldViolation{
        {Field: "email", Description: "invalid format"},
        {Field: "age",   Description: "must be >= 0"},
    },
}
st, err := st.WithDetails(br)
if err != nil {
    return nil, status.Error(codes.Internal, "failed to attach details")
}
return nil, st.Err()

常用 standard error details:

類型用途
BadRequest逐欄位的驗證錯誤
RetryInfo建議 retry 延遲
QuotaFailure配額被耗盡的細節
ErrorInfo錯誤分類(reason / domain / metadata)
PreconditionFailure失敗前置條件的明細
05 — Client 解錯

Client:用 status.FromError 取出 code 與 details

go (client) 
resp, err := client.GetUser(ctx, &pb.GetUserRequest{Id: "42"})
if err != nil {
    st, ok := status.FromError(err)
    if !ok {
        // 非 gRPC 錯誤(網路 / parse 等)
        log.Fatalf("non-grpc error: %v", err)
    }

    switch st.Code() {
    case codes.NotFound:
        log.Println("user not found,提示 user 重新輸入")
    case codes.InvalidArgument:
        // 取結構化 details
        for _, d := range st.Details() {
            if br, ok := d.(*errdetails.BadRequest); ok {
                for _, v := range br.GetFieldViolations() {
                    log.Printf("  field %s: %s", v.GetField(), v.GetDescription())
                }
            }
        }
    case codes.Unavailable:
        log.Println("server 暫時不可用,稍後 retry")
    default:
        log.Printf("unhandled: %v - %s", st.Code(), st.Message())
    }
    return
}
i
快速取 code 用 status.Code(err) 只想知道 code、不取 details 時用這個 helper:if status.Code(err) == codes.NotFound { ... } — 對非 gRPC 錯誤會回 codes.Unknown,不需先 check ok。
06 — retry 與 idempotency

哪些 code 該 retry?

官方對 UNAVAILABLE / ABORTED / FAILED_PRECONDITION 的指引:

Code建議
UNAVAILABLE「client can retry just the failing call」 — 直接重試該 call
ABORTED「client should retry at a higher level」 — 在更外層重試(整個 transaction)
FAILED_PRECONDITION「client should not retry until system state has been explicitly fixed」 — 修了狀態才重試
DEADLINE_EXCEEDED / CANCELLED視情境,通常要重新建 ctx + 重試
INVALID_ARGUMENT / NOT_FOUND / ALREADY_EXISTS / PERMISSION_DENIED不要 retry(這次沒成功,下次也不會)
!
retry 前確認 idempotency 只有等冪(idempotent)的操作能安全 retry。「建立 user」可能 retry 後重複建;「扣款」可能扣兩次。寫 method 時就要思考是否 idempotent,server 可以靠 idempotency-key(metadata)去重。第 10 課會看 grpc-go 的 retry policy(透過 service config)。
