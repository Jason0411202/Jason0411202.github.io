Lesson 03 · Unary RPC 實戰

第一支 gRPC service:Unary RPC

從建立專案、撰寫 proto，一路到 server 與 client 完整實作並實際跑通。本課採用 grpc-go v1.80.0 的新 API：grpc.NewClient() 取代已 deprecated 的 grpc.Dial()。

01 — 流程總覽

從 0 到跑通一支 Unary(一來一回)

gRPC 共有四種呼叫模式,其示意圖與 service／rpc 定義語法已於第 2 課說明。本課實作其中最基礎的 Unary;三種 streaming 模式於第 4 課詳述。

專案目錄

tree 
mygreeter/
├── go.mod
├── proto/
│   ├── mygreeter.proto
│   ├── mygreeter.pb.go        # 自動生成
│   └── mygreeter_grpc.pb.go   # 自動生成
├── server/
│   └── main.go
└── client/
    └── main.go

建立專案

先建立專案資料夾、初始化 Go module，並開出存放 proto、server、client 的目錄：

bash 
# 建立專案資料夾並進入
$ mkdir mygreeter && cd mygreeter
# 初始化 Go module(產生 go.mod)
$ go mod init example.com/mygreeter
# 建立放 proto、server、client 的資料夾
$ mkdir proto server client
02 — 定義 proto

Step 1 — 寫 proto/mygreeter.proto

proto/mygreeter.proto 
// 使用 proto3 語法
syntax = "proto3";

// package 名稱(以 .v1 標示版本,利於日後相容演進)
package mygreeter.v1;

// 生成 Go 檔的 import path(必填)
option go_package = "example.com/mygreeter/proto;mygreeterpb";

// 定義 MyGreeter service 及其 method
service MyGreeter {
  // Unary RPC:1 req → 1 resp
  rpc MySayHello (MyHelloRequest) returns (MyHelloReply);
}

// request 格式
message MyHelloRequest {
  string my_name = 1;
}

// response 格式
message MyHelloReply {
  string my_message = 1;
}
03 — 生成 stub

Step 2 — 執行 protoc 與安裝套件

bash 
# 在 mygreeter/ 根目錄,加入 grpc-go 與 protobuf runtime
$ go get google.golang.org/grpc@latest
$ go get google.golang.org/protobuf@latest

# 生成 stub
$ protoc \
    --go_out=. --go_opt=paths=source_relative \
    --go-grpc_out=. --go-grpc_opt=paths=source_relative \
    proto/mygreeter.proto

這行指令背後的運作機制如下：protoc 本身僅負責解析 .proto，實際生成程式碼的工作則委派給對應的 plugin。每個 --xxx_out 參數即是告知 protoc「呼叫哪個 plugin、將結果輸出至何處」。

圖 3.1 — protoc 解析契約後,依 --xxx_out 參數分派給對應 plugin 生成

執行後,proto 資料夾內應出現兩個生成檔:

bash 
$ ls proto/
mygreeter.proto
mygreeter.pb.go        # message 生成的 Go struct
mygreeter_grpc.pb.go   # service 生成的 gRPC stub
i
每次改動 .proto 都需重新生成 *.pb.go 是 .proto 的衍生產物。日後若調整 message field 或 service method，須重新執行本步驟使生成碼同步，切勿手動修改生成檔（下次生成時會被覆寫）。實務上常將此指令寫入 Makefile 或腳本，以單一指令完成生成。
04 — 寫 Server

Step 3 — server/main.go

go 
package main

import (
    "context"
    "log"
    "net"

    "google.golang.org/grpc"
    pb "example.com/mygreeter/proto"
)

// 定義 server type,作為 MyGreeter service 的實作載體。
// 內嵌 UnimplementedMyGreeterServer 取得 interface 所有 method 的
// 預設實作(固定寫法,提供向前相容)。
type server struct {
    pb.UnimplementedMyGreeterServer
}

// 實作 .proto 中定義的 MySayHello(名稱與簽名都源自 proto):
//   - 接收者 (s *server):把 method 掛在 server type 上
//   - ctx:呼叫 context(deadline、取消訊號)
//   - req:client 的 request
//   - 回傳:*pb.MyHelloReply 與 error(成功回 nil)
func (s *server) MySayHello(ctx context.Context, req *pb.MyHelloRequest) (*pb.MyHelloReply, error) {
    // 呼叫 req.GetMyName() 取出 request 中的 my_name,用 log 印出
    log.Printf("received: %v", req.GetMyName())

    // 組出 response(MyMessage = "Hello, {name}"),
    // 第二個回傳值 error 給 nil 表示成功
    return &pb.MyHelloReply{
        MyMessage: "Hello, " + req.GetMyName(),
    }, nil
}

func main() {
    // 呼叫 net 套件的 Listen() function,
    // 回傳一個 listener(lis),
    // 用來接受進 :50051 的 TCP connection
    lis, err := net.Listen("tcp", ":50051")
    if err != nil {
        log.Fatalf("failed to listen: %v", err)
    }

    // 呼叫 grpc framework 的 NewServer() method,
    // 回傳一個 gRPC 框架的 server 物件,
    // 該物件負責收 connection、解碼、把 request 分派給對應 method
    grpcServer := grpc.NewServer()

    // 呼叫生成的 RegisterMyGreeterServer(),
    // 把我們的實作 &server{} 註冊進 grpcServer,
    // 框架才知道 MySayHello 由誰處理
    pb.RegisterMyGreeterServer(grpcServer, &server{})

    log.Printf("server listening on %v", lis.Addr())  // 印出實際監聽的位址

    // 呼叫 grpcServer 的 Serve() method 開始接收 request,
    // 此呼叫會一直 block 直到 server 被關閉
    if err := grpcServer.Serve(lis); err != nil {
        log.Fatalf("failed to serve: %v", err)
    }
}
!
務必 embed pb.UnimplementedMyGreeterServer 這是 protoc-gen-go-grpc 強制的(雖可透過 require_unimplemented_servers=false 停用,但不建議)。 其作用有二：(1) 為日後新增 method 提供「Unimplemented 預設回傳」，使 server 升級 proto 時不致編譯失敗；(2) 強制使 service 實作具備 forward-compatible 的特性。
05 — 寫 Client

Step 4 — client/main.go

go 
package main

import (
    "context"
    "log"
    "os"
    "time"

    "google.golang.org/grpc"
    "google.golang.org/grpc/credentials/insecure"
    pb "example.com/mygreeter/proto"
)

func main() {
    // 取名字:預設 "world",有帶參數(go run ./client Jason)則用第一個參數
    name := "world"
    if len(os.Args) > 1 {
        name = os.Args[1]
    }

    // 呼叫 grpc framework 的 NewClient() method,
    // 回傳一個 connection 物件 conn(lazy,此時還不會真的連線);
    // 第二個參數設定此 connection 不加密
    conn, err := grpc.NewClient(
        "localhost:50051",
        grpc.WithTransportCredentials(insecure.NewCredentials()),
    )
    if err != nil {
        log.Fatalf("failed to create client: %v", err)
    }
    defer conn.Close()  // main 結束前關閉 conn

    // 呼叫生成的 NewMyGreeterClient(),
    // 用 conn 包出一個 client stub,之後用它呼叫遠端 method
    client := pb.NewMyGreeterClient(conn)

    // 呼叫 context 的 WithTimeout(),
    // 建立一個 1 秒後自動取消的 ctx
    ctx, cancel := context.WithTimeout(context.Background(), time.Second)
    defer cancel()  // 釋放 ctx 資源

    // 呼叫 client stub 的 MySayHello() method,把 req 送到 server 並等 response
    //(connection 會在這次呼叫才真正建立)
    resp, err := client.MySayHello(ctx, &pb.MyHelloRequest{MyName: name})
    if err != nil {
        log.Fatalf("could not greet: %v", err)
    }

    // 呼叫 resp.GetMyMessage() 取出 response 內容並印出
    log.Printf("greeting: %s", resp.GetMyMessage())
}
i
insecure.NewCredentials() 是什麼 用於告知 gRPC「此 connection 不加密」 — 純明文 HTTP/2(h2c)。僅適用於 localhost 開發、或 service mesh 內部已具備 mTLS 的情境。production 對外 connection 務必使用 TLS,第 8 課將詳述。
06 — 跑起來

Step 5 — 開啟兩個 terminal 執行

啟動前先補齊相依,再開啟兩個 terminal 分別執行 server 與 client:

bash 
# 補齊相依(protobuf runtime、insecure credentials)
$ go mod tidy
terminal A 
$ go run ./server
2026/04/27 11:00:00 server listening on [::]:50051
terminal B 
$ go run ./client Jason
2026/04/27 11:00:05 greeting: Hello, Jason

同時 terminal A 會看到:

terminal A 
2026/04/27 11:00:05 received: Jason
✓
第一支 gRPC service 已成功運行! 這就是最小可行的 unary RPC。下一課介紹 streaming(三種模式),第 5 課說明 deadline、metadata、cancellation 的運作方式。
