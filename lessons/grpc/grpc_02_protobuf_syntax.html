Lesson 02 · Protobuf 語法

Protocol Buffers語法完整版

完整介紹 message、field number、scalar、repeated、enum、oneof、map、reserved，以及以 service／rpc 定義 RPC interface。field number 為本堂核心——其決定 API 後續是否得以演進。

01 — .proto 檔案結構

.proto 檔的三行宣告

每個 .proto 檔的開頭幾乎都有這三行宣告,接著才是 message 等 type 定義:

user.proto 
syntax = "proto3";          // 聲明使用 proto3 語法

package myapp.v1;             // 命名空間,避免跨檔衝突

option go_package = "example.com/myapp/v1;myappv1";
  • syntax — 必填,目前都用 "proto3"
  • package — 邏輯命名空間,避免不同檔案的 type 名稱衝突
  • option go_package — 指定生成的 .pb.go 對應的 Go import path 與套件名
02 — message 與 field 定義

message 與 field 定義

message 是 protobuf 描述資料的基本單位,對應一個由多個 field 組成的結構。每個 field 由 type、名稱、編號三者定義。以一個代表使用者的 message 為例:

proto3 
message User {
  string id   = 1;     // type string、名稱 id、編號 1
  string name = 2;     // type string、名稱 name、編號 2
  int32  age  = 3;     // type int32、名稱 age、編號 3
}

執行 protoc 生成後,這份 .proto 會轉成對應的 Go struct(此處隱去 protobuf runtime 的內部 field,聚焦於資料 field):

user.pb.go(自動生成) 
type User struct {
	Id   string `protobuf:"bytes,1,opt,name=id,proto3" json:"id,omitempty"`
	Name string `protobuf:"bytes,2,opt,name=name,proto3" json:"name,omitempty"`
	Age  int32  `protobuf:"varint,3,opt,name=age,proto3" json:"age,omitempty"`
}
i
補充:命名慣例 message 與 service 用 UpperCamelCase(如上方的 User)、field 用 snake_case(如 display_name,生成 Go 時自動轉為 DisplayName)。此為 Google 官方 style guide 的慣例。

編號:protobuf 的命脈

三者之中,編號最為關鍵。每個 field 後方的數字(= 1、= 2)稱為 field number,其作用並非供人閱讀,而是寫進二進位 wire format、作為識別 field 的依據。

編號範圍與成本

i
編號範圍的兩個重點 1 – 15 的 tag 在 wire format 上只佔 1 byte,應留給最常出現的 field;19000 – 19999 為 protobuf 內部保留,不可使用(protoc 會直接報錯)。

type:掌握常用幾種即可

field 最常用的 type 是 protobuf 內建的 scalar type,涵蓋整數、浮點、布林、字串與位元組。日常開發最常用到下列幾種:

type用途對應 Go
int32 / int64整數int32 / int64
float / double浮點數float32 / float64
bool布林值bool
stringUTF-8 文字string
bytes非文字的原始位元組(圖片、檔案、加密內容)[]byte
i
其餘變體 type 需要時再查 protobuf 另有 uint*、sint*、fixed*、sfixed* 等整數變體,差別僅在編碼方式針對特定數值範圍(經常為負數、或總是大數值)更省空間;多數情境用上表幾種即足以涵蓋,有特殊需求時再查官方文件。

field 的 type 不限於 scalar,也可以是自訂 type:另一個 message(形成巢狀結構)或 enum(固定選項集合)。換言之,一個 field 的 type 可為 scalar、message 或 enum 三者之一。message 即本節開頭介紹的基本單位,enum 詳見後續小節。

未設定的 scalar 會讀到「預設值」

proto3 有一個需先建立的觀念:scalar field 沒有「未設定」這個狀態。當一筆資料未填某個 scalar field,接收端讀到的不是 null,而是該 type 的預設值(零值):

  • int32 / int64 / float / double → 0
  • string → ""(空字串)
  • bool → false

因此接收端無法單憑值本身分辨「對方未填」與「對方填入預設值」。多數情境下這不成問題;若需要明確區分兩者(例如年齡 field 要分辨「0 歲嬰兒」與「未填寫」),在 field 前加上 optional 即可:

proto3 
message User {
  string id = 1;
  int32 age = 2;             // 未設定與 0 無法區分
  optional int32 height_cm = 3;  // 可區分「未設定」與 0(Go 生成 *int32)
}

對應生成的 Go(optional field 為指標 type):

user.pb.go(自動生成) 
type User struct {
	Id       string `protobuf:"bytes,1,opt,name=id,proto3" json:"id,omitempty"`
	Age      int32  `protobuf:"varint,2,opt,name=age,proto3" json:"age,omitempty"`
	HeightCm *int32 `protobuf:"varint,3,opt,name=height_cm,json=heightCm,proto3,oneof" json:"height_cm,omitempty"`
}

加上 optional 的 field,生成 Go 時為指標 type(如 *int32),未設定即為 nil,故能判斷是否曾被設定。message type 的 field 天生帶有 nil 概念,不需要 optional。

repeated:可放多個值

field 前可加修飾詞改變數量語意。proto3 預設每個 field 是「單一值」(最多帶一個值),repeated 則將 field 轉為可放多個值的清單,對應其他語言的陣列/slice:

proto3 
message Item {                          // 自訂 message,供下方引用
  string sku = 1;
  int32  qty = 2;
}

message Order {
  string id = 1;
  repeated string item_ids = 2;       // 對應 Go []string
  repeated Item items = 3;             // 對應 Go []*Item(Item 為上方定義的 message)
}

對應生成的 Go(repeated 轉為 slice):

order.pb.go(自動生成) 
type Item struct {
	Sku string `protobuf:"bytes,1,opt,name=sku,proto3" json:"sku,omitempty"`
	Qty int32  `protobuf:"varint,2,opt,name=qty,proto3" json:"qty,omitempty"`
}

type Order struct {
	Id      string   `protobuf:"bytes,1,opt,name=id,proto3" json:"id,omitempty"`
	ItemIds []string `protobuf:"bytes,2,rep,name=item_ids,json=itemIds,proto3" json:"item_ids,omitempty"`
	Items   []*Item  `protobuf:"bytes,3,rep,name=items,proto3" json:"items,omitempty"`
}

repeated 後可接 scalar type(如 item_ids),也可接另一個 message type(如 items,其中 Item 是同一個 .proto 檔內定義的 message)。順序保留。scalar repeated 預設用 packed encoding(全部值連在一起共用一個 tag),省空間。

oneof:多選一

當一組 field 最多只有一個會被設,以 oneof 表達語意,遠比並列多個 nullable field 清楚。

proto3 
message NotificationTarget {
  oneof target {
    string email   = 1;
    string phone   = 2;
    string webhook = 3;
  }
  string message = 10;
}

對應生成的 Go(oneof 生成一個 interface type 與各選項的 wrapper struct):

notification.pb.go(自動生成) 
type NotificationTarget struct {
	Target  isNotificationTarget_Target `protobuf_oneof:"target"`
	Message string `protobuf:"bytes,10,opt,name=message,proto3" json:"message,omitempty"`
}

type isNotificationTarget_Target interface {
	isNotificationTarget_Target()
}

type NotificationTarget_Email struct {
	Email string `protobuf:"bytes,1,opt,name=email,proto3,oneof"`
}

type NotificationTarget_Phone struct {
	Phone string `protobuf:"bytes,2,opt,name=phone,proto3,oneof"`
}

type NotificationTarget_Webhook struct {
	Webhook string `protobuf:"bytes,3,opt,name=webhook,proto3,oneof"`
}

規則:

  • 同時設 email 與 phone → 後設定者會覆寫先前的值
  • 不能用 repeated、不能用 map 在 oneof 裡
  • field number 要在整個 message 範圍內唯一(不只 oneof 內)

map:鍵值對

proto3 
message Resource {
  string id = 1;
  map<string, string> labels = 2;
}

對應生成的 Go(map 轉為 Go map):

resource.pb.go(自動生成) 
type Resource struct {
	Id     string            `protobuf:"bytes,1,opt,name=id,proto3" json:"id,omitempty"`
	Labels map[string]string `protobuf:"bytes,2,rep,name=labels,proto3" json:"labels,omitempty" protobuf_key:"bytes,1,opt,name=key,proto3" protobuf_val:"bytes,2,opt,name=value,proto3"`
}

限制:

  • key 只能是整數型或 string(不能是 float、bytes、enum、message)
  • value 任意,但不能是另一個 map
  • 不能加 repeated
  • serialization 順序、迭代順序未定義 — 不要依賴順序!

enum:零值必須是 UNSPECIFIED

enum(列舉)用來表達一個 field 只能是少數幾個固定選項之一的情況,例如訂單狀態(待付款／已付款／已出貨)、使用者角色(一般／管理員)。相較於用 string 或裸 int32 表示,enum 讓合法值一目了然、避免拼字錯誤,生成的程式碼也是具名常數,可讀性與 type 安全都更好。

用法上,protobuf enum 是有限的整數常數集:

proto3 
enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;   // 必填,代表「未設定」
  ORDER_STATUS_PENDING     = 1;
  ORDER_STATUS_PAID        = 2;
  ORDER_STATUS_SHIPPED     = 3;
  ORDER_STATUS_DELIVERED   = 4;
  ORDER_STATUS_CANCELLED   = 5;
}

message Order {
  string id = 1;
  OrderStatus status = 2;
}

對應生成的 Go(enum 轉為具名 int32 type 與一組常數):

order.pb.go(自動生成) 
type OrderStatus int32

const (
	OrderStatus_ORDER_STATUS_UNSPECIFIED OrderStatus = 0
	OrderStatus_ORDER_STATUS_PENDING     OrderStatus = 1
	OrderStatus_ORDER_STATUS_PAID        OrderStatus = 2
	OrderStatus_ORDER_STATUS_SHIPPED     OrderStatus = 3
	OrderStatus_ORDER_STATUS_DELIVERED   OrderStatus = 4
	OrderStatus_ORDER_STATUS_CANCELLED   OrderStatus = 5
)

type Order struct {
	Id     string      `protobuf:"bytes,1,opt,name=id,proto3" json:"id,omitempty"`
	Status OrderStatus `protobuf:"varint,2,opt,name=status,proto3,enum=OrderStatus" json:"status,omitempty"`
}

第一個值必須是 0(這是 proto3 的硬性規定);enum 值依 Google 官方 style guide 以 UPPER_SNAKE_CASE 命名,且第一個值慣例命名為 {ENUM}_UNSPECIFIED,作為代表「未設定」的零值。

03 — Schema 演進

Schema 演進:安全地修改 .proto

field 定義一旦在 production 使用,schema 便進入需要長期維護的階段:client 與 server 各自獨立部署,新舊版本必然同時存在於線上。protobuf 的相容性設計讓 schema 能在不破壞既有資料的前提下增刪 field,但前提是遵守以下規則。

!
field number 一旦在 production 用過,絕對不能改 改編號等同刪除舊 field、新增一個 field。所有舊版 client 傳上來的資料會被視為「未知 field」,server 讀到的舊資料會全部退化為預設值。這是 protobuf 工程師最常忽略的細節。正確做法是:以 reserved 標記退役的編號,絕不重用。

三條鐵律

  1. 不改編號 — 改編號等於刪舊增新,所有舊資料失效
  2. 不改 type(除非是 wire-format 相容的轉換,如 int32 ↔ int64 ↔ uint32;細節見官方 update guide)
  3. 退役 field 放 reserved,絕不重用編號或名稱

退役 field 的安全網:reserved

刪除一個 field 或為其改名時,正確做法是將它的編號和名稱都納入 reserved:

proto3 
message User {
  reserved 3, 5, 9 to 12;          // 這些編號絕不再用
  reserved "old_email", "legacy_id";    // 名稱也保留

  string id   = 1;
  string name = 2;
  // 編號 3 (legacy_id) 已退役,改放在 reserved 上方
  string email = 4;
}

對應生成的 Go(reserved 的編號與名稱皆不生成 field):

user.pb.go(自動生成) 
type User struct {
	Id    string `protobuf:"bytes,1,opt,name=id,proto3" json:"id,omitempty"`
	Name  string `protobuf:"bytes,2,opt,name=name,proto3" json:"name,omitempty"`
	Email string `protobuf:"bytes,4,opt,name=email,proto3" json:"email,omitempty"`
}

這樣做的好處:

  • 若誤將新 field number 設成 3,protoc 會直接報錯
  • 若將 old_email 重用為其他 field 名,protoc 也會報錯
  • 形同自我說明的文件:閱讀 .proto 即可得知哪些編號已退役、不應再使用
04 — 多檔案、import 與巢狀

跨檔引用與巢狀

大型專案會把 type 拆進多個 .proto 檔,再以 import 互相引用。以下用一個最小但完整的「訂單」範例,一次涵蓋多檔拆分、同 / 不同 package 的引用寫法、import 路徑、well-known types 與巢狀 message。先看檔案目錄:

目錄結構 
proto/
├── common/
│   └── money.proto   # package shop.common
└── order/
    ├── item.proto    # package shop.order
    └── order.proto   # package shop.order(主檔)

先看兩個被引用的 type 檔案。common/money.proto 是跨服務共用的 type,自成一個 package:

common/money.proto 
syntax = "proto3";
package shop.common;
option go_package = "example.com/shop/proto/common;commonpb";   // 生成 Go 的 import path

message Money {
  string currency = 1;     // 例:"TWD"
  int64  amount   = 2;     // 最小幣值單位(分)
}

order/item.proto 與主檔同屬 shop.order,並引用了跨 package 的 Money:

order/item.proto 
syntax = "proto3";
package shop.order;
option go_package = "example.com/shop/proto/order;orderpb";     // 生成 Go 的 import path

import "common/money.proto";        // 不同 package:shop.common

message Item {
  string sku = 1;
  int32  qty = 2;
  shop.common.Money price = 3;     // 跨 package 須寫完整名稱
}

主檔 order/order.proto 把全部引用方式集中示範:

order/order.proto 
syntax = "proto3";
package shop.order;
option go_package = "example.com/shop/proto/order;orderpb";     // 與 item.proto 同 package,故相同

import "google/protobuf/timestamp.proto";   // well-known type
import "order/item.proto";                   // 同 package、不同檔
import "common/money.proto";                  // 不同 package

message Order {
  message Shipping {                        // 巢狀:外部引用為 Order.Shipping
    string address = 1;
    string carrier = 2;
  }

  string id = 1;
  repeated Item items = 2;               // 同 package → 直接用 Item
  shop.common.Money total = 3;            // 不同 package → 完整名稱
  Shipping shipping = 4;                   // 巢狀 type,同檔直接引用
  google.protobuf.Timestamp created_at = 5;  // well-known type
}

對照範例中的四種引用方式:

  • 同 package、不同檔:order.proto import item.proto 後,直接寫 Item、無須前綴(兩者同屬 shop.order)。
  • 不同 package:引用 shop.common.Money 須寫完整名稱(fully qualified name)。
  • well-known types:Google 內建 type(Timestamp、Duration、Empty、Any…)位於 google/protobuf/*.proto,import 後以 google.protobuf.Timestamp 引用。
  • 巢狀 message:Shipping 定義在 Order 內、僅其內部可見;外部引用須寫 Order.Shipping。

import 的路徑(如 common/money.proto)相對於 protoc --proto_path=DIR 指定的搜尋根目錄(本例為 proto/);--proto_path 可重複指定多個根目錄。

05 — service 與 rpc 定義

用 service 定義 RPC interface

前面各節的 message 描述「資料長什麼樣子」,service 則描述「有哪些遠端 method 可呼叫」。service 是一組可遠端呼叫 method 的集合,等同於 client 與 server 之間的 interface 契約;區塊內每個 method 以 rpc 宣告,指定 method 名、request type 與 response type。request 與 response 必須是已定義(或已 import)的 message。基本骨架如下:

proto · 語法骨架 
service service名 {
  rpc method名 (request type) returns (response type);
}

以一個問候 service 為例,定義一個 method SayHello,收 HelloRequest、回 HelloReply:

proto3 
service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

message HelloRequest { string name = 1; }
message HelloReply   { string message = 1; }

對應生成的 Go(service 生成 server interface 與 client stub,此處示意其形狀):

greeter_grpc.pb.go(自動生成) 
// server 端須實作的 interface
type GreeterServer interface {
	SayHello(context.Context, *HelloRequest) (*HelloReply, error)
}

// client 端呼叫用的 interface
type GreeterClient interface {
	SayHello(ctx context.Context, in *HelloRequest, opts ...grpc.CallOption) (*HelloReply, error)
}

四種 method 型態:差別僅在 stream 關鍵字

method 依「哪一邊是 streaming」分為四種型態,如下圖所示:

圖 2.1 — 四種 RPC 呼叫模式

對應到 rpc 的寫法,差別僅在於是否在 request 或 response type 前加上 stream 關鍵字:

型態rpc 寫法
Unary(一來一回)rpc M (Req) returns (Resp);
Server streamingrpc M (Req) returns (stream Resp);
Client streamingrpc M (stream Req) returns (Resp);
Bidirectionalrpc M (stream Req) returns (stream Resp);
i
service 只定義 interface,不含實作 service 僅宣告「有哪些 method、收什麼、回什麼」,實際邏輯由 server 端填入、由 client 端呼叫。第 3 課會以上方的 Greeter(一個 Unary method)實際寫出 server 與 client;三種 streaming 型態則於第 4 課詳述。
