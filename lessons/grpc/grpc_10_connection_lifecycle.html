01 — Channel 狀態機

ClientConn 不是單條 TCP — 是一個狀態機

我們在第 4 課用 conn := grpc.NewClient(...) 拿到的 conn(*grpc.ClientConn),官方文件叫它 channel。它對應到「一個邏輯連線」,內部可能是 0 條或多條 TCP(取決於 LB 策略)。

圖 22.1 — ClientConn 五種狀態
狀態意思
IDLE剛建好,還沒有人發 RPC,沒實際連線
CONNECTING正在做 TCP + TLS handshake
READYHTTP/2 connection 已就緒,可送 RPC
TRANSIENT_FAILURE連線失敗,進指數退避重試中
SHUTDOWN呼叫了 conn.Close(),不會再連

取得當前狀態:

go 
import "google.golang.org/grpc/connectivity"

state := conn.GetState()                 // connectivity.State
if state == connectivity.Ready { ... }

conn.Connect()                           // 主動觸發 idle → connecting(平常不必)
conn.WaitForStateChange(ctx, currentState)
i
NewClient lazy 的延伸 第 4 課提過 NewClient 不立刻連線。具體就是把 channel 留在 IDLE,直到第一支 RPC 才轉 CONNECTING。所以「app 啟動成功」不代表「下游服務可達」 — 想預先驗證可呼叫 conn.Connect() 並等 Ready。
02 — Keepalive

偵測「半開」死連線

長連線最大的敵人是「半開」(half-open):雙方 TCP 看似還在,但中間 NAT/LB 已把連線狀態 GC 掉。沒 keepalive 你會在下一支 RPC 才發現「咦,這條死了」 — 而 RPC 已經失敗。

Client 端

go (client) 
import "google.golang.org/grpc/keepalive"

conn, err := grpc.NewClient(addr,
    grpc.WithTransportCredentials(creds),
    grpc.WithKeepaliveParams(keepalive.ClientParameters{
        Time:                30 * time.Second,    // 多久 idle 後送 ping
        Timeout:             10 * time.Second,    // ping 後多久沒回視為死
        PermitWithoutStream: true,                 // 即使沒活躍 RPC 也送 ping
    }),
)

Server 端 — 注意 EnforcementPolicy

Server 預設會 enforce 「同一 client 不能太頻繁送 ping」(預設下限 5 分鐘),你 client 設 30 秒 ping 會被 server 視為「bad ping」累積到 MaxPingStrikes 後送 GOAWAY 斷你連線。兩邊要對齊設定:

go (server) 
s := grpc.NewServer(
    grpc.KeepaliveEnforcementPolicy(keepalive.EnforcementPolicy{
        MinTime:             10 * time.Second,    // 允許 client ping 的最小間隔
        PermitWithoutStream: true,                 // 允許沒 stream 時 ping
    }),
    grpc.KeepaliveParams(keepalive.ServerParameters{
        Time:    30 * time.Second,
        Timeout: 10 * time.Second,
    }),
)
!
常見坑:client 設 keepalive 後 server 把你斷線 症狀是 client log 一直出現 connection error: too many pings。原因就是 server EnforcementPolicy 的 MinTime 比 client Time 還大。改 server,或調 client 的 Time 大於 server MinTime。
03 — Name Resolver

Target string 怎麼被解析

第 4 課提過 grpc.NewClient(target, opts...) 的 target 支援多種 scheme:

Target意思
foo.example.com:443預設 DNS 解析(NewClient 預設)
dns:///foo.example.com:443顯式 DNS
dns://8.8.8.8/foo.example.com指定 DNS server
unix:///var/run/x.sockUnix socket(本機 IPC)
passthrough:///localhost:50051不解析,直接連

DNS 回傳多個 A record 時,每一個都會變成一個 backend address — 這是客戶端 LB 的基礎。

04 — Load Balancing

客戶端 round_robin 負載均衡

gRPC 的 LB 設計上「客戶端 LB」是 first-class — 不需要 NLB / HAProxy 在中間。把 backend 拿到的所有 address 拆開,每個都建 sub-channel,RPC round-robin 分發。

啟用 round_robin

go (client) 
conn, err := grpc.NewClient(
    "dns:///api.example.com:443",            // 必須是會回多個 address 的 target
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultServiceConfig(`{"loadBalancingConfig":[{"round_robin":{}}]}`),
)

之後每支 RPC 會輪流送到不同 backend。新 backend 加入(DNS 多了一個 A record)、舊 backend 死掉(connection 失敗),內部 sub-channel pool 會自動更新。

i
k8s headless service + DNS k8s 的 ClusterIP: None headless service 會讓 DNS 直接回所有 pod 的 IP。配合 dns:///my-svc.my-ns.svc.cluster.local:50051,client 會直接 round_robin 到所有 pod,不經過 kube-proxy NLB。是 gRPC + k8s 推薦做法。
05 — Health Check

標準健康檢查協定:grpc.health.v1.Health

gRPC 定義了標準 health checking service(grpc.health.v1),兩個 method:

  • Check(HealthCheckRequest) — unary,問「你健康嗎?」
  • Watch(HealthCheckRequest) — server stream,持續推送健康狀態變化

Server 端啟用

go (server) 
import (
    "google.golang.org/grpc/health"
    healthpb "google.golang.org/grpc/health/grpc_health_v1"
)

s := grpc.NewServer()
hs := health.NewServer()
healthpb.RegisterHealthServer(s, hs)

// 設定整體狀態
hs.SetServingStatus("", healthpb.HealthCheckResponse_SERVING)
// 或 per-service 狀態
hs.SetServingStatus("greeter.v1.Greeter", healthpb.HealthCheckResponse_SERVING)

// 收到 SIGTERM 時切 NOT_SERVING、等 in-flight RPC 完成
<-shutdownCh
hs.Shutdown()  // 把所有狀態改 NOT_SERVING
s.GracefulStop()

Client 啟用「不健康自動跳過」

go (client) 
conn, err := grpc.NewClient("dns:///svc:50051",
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultServiceConfig(`{
        "loadBalancingConfig": [{"round_robin": {}}],
        "healthCheckConfig": {"serviceName": ""}
    }`),
)

啟用後 client 會對每個 sub-channel 自動 Watch server 健康狀態,NOT_SERVING 的 backend 自動被跳過(不會丟到那個 sub-channel)。健康狀態恢復後自動重新加入。

i
k8s liveness/readiness 也認得 k8s 1.24+ 內建 grpc-health-probe 工具(後又被 kubelet 原生支援)— 你的 Pod 只要實作 grpc.health.v1,readiness probe 直接設 grpc: type 就能用,不必額外 sidecar。
06 — Retry Policy

用 service config 自動 retry

還記得第 7 課說 UNAVAILABLE 該 retry 嗎?grpc-go 內建可自動做這件事 — 透過 service config 設定 retry policy。

json (service config) 
{
  "methodConfig": [{
    "name": [{ "service": "greeter.v1.Greeter" }],
    "retryPolicy": {
      "maxAttempts": 4,
      "initialBackoff": "0.1s",
      "maxBackoff": "2s",
      "backoffMultiplier": 2,
      "retryableStatusCodes": ["UNAVAILABLE"]
    }
  }]
}

放進 client:

go (client) 
const svcCfg = `{
  "methodConfig": [{
    "name": [{"service": "greeter.v1.Greeter"}],
    "retryPolicy": {
      "maxAttempts": 4,
      "initialBackoff": "0.1s",
      "maxBackoff": "2s",
      "backoffMultiplier": 2,
      "retryableStatusCodes": ["UNAVAILABLE"]
    }
  }]
}`

conn, err := grpc.NewClient(addr,
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultServiceConfig(svcCfg),
)
!
retry 前確認 idempotency(再次強調) 自動 retry 對 read 操作(GetUser、ListItems)幾乎一定安全;對 write 操作(CreateUser、Charge)不一定 — 第 7 課提過,write 需要 idempotency-key 防重複。建議:retry policy per-method 設,write method 不要納入。

10 堂課,從零到上線

恭喜跑完整套 gRPC 課程!回顧一下你學到的:

  1. L01 概念 — RPC 心智模型、HTTP/2、四種 RPC 模式、何時不該用

  6. L06 Context — metadata、deadline propagation、cancellation
  7. L07 Errors — 16 種 status codes + structured details
  8. L08 Interceptors — logging、recovery、auth、timing 四件組
