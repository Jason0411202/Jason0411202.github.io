01 — 兩層 credentials

Transport vs PerRPC — 兩種 credentials

gRPC 的安全分兩層,兩者可同時存在:

層次解決什麼典型實作
Transport credentials 「連線本身要不要加密、雙方身份驗不驗」 — 對應到 TLS 層 credentials.NewTLS(...)、insecure.NewCredentials()
PerRPC credentials 「每支 RPC 要附什麼 token」 — 對應到 metadata 層 OAuth、JWT bearer token、API key
i
兩層獨立、可組合 一個生產系統典型搭配:transport 用 TLS(對外)或 mTLS(內部),plus PerRPC 用 OAuth bearer token 帶 user identity。三件事有清楚分工,不要混用。
02 — 準備憑證

本機練習用 self-signed 憑證

Production 應該用 Let's Encrypt 或公司內部 CA。本機練習可以用 openssl 或更方便的 mkcert:

bash 
# 用 mkcert(推薦)— 自動裝信任根 + 簽 server 憑證
$ brew install mkcert
$ mkcert -install
$ mkcert localhost 127.0.0.1
# 產生:localhost+1.pem (cert) 與 localhost+1-key.pem (key)

# 或用原始 openssl(較囉嗦,但無依賴)
$ openssl req -x509 -newkey rsa:4096 -nodes \
    -keyout server.key -out server.crt -days 365 \
    -subj "/CN=localhost" -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"
03 — 單向 TLS

Server-only TLS:加密連線、驗 server

最常見的 production 設定 — 像瀏覽器連網站一樣,只驗 server 身份,連線加密。

Server 端

go (server) 
import "google.golang.org/grpc/credentials"

creds, err := credentials.NewServerTLSFromFile("server.crt", "server.key")
if err != nil {
    log.Fatalf("load tls: %v", err)
}

s := grpc.NewServer(grpc.Creds(creds))
pb.RegisterGreeterServer(s, &server{})
s.Serve(lis)

Client 端

go (client) 
// 簡單版:從檔案讀 CA 憑證,信任這個 CA 簽的 cert
creds, err := credentials.NewClientTLSFromFile("server.crt", "localhost")
if err != nil {
    log.Fatalf("load tls: %v", err)
}

conn, err := grpc.NewClient("localhost:50051", grpc.WithTransportCredentials(creds))

更彈性的版本:傳 tls.Config:

go (client) 
// 信任系統根憑證池(對 Let's Encrypt 等公開 CA 用這個)
pool, err := x509.SystemCertPool()
if err != nil { log.Fatal(err) }

creds := credentials.NewTLS(&tls.Config{
    RootCAs:    pool,
    ServerName: "api.example.com",    // 必須對應 cert 的 SAN
    MinVersion: tls.VersionTLS13,
})

conn, err := grpc.NewClient("api.example.com:443", grpc.WithTransportCredentials(creds))
04 — mTLS

mTLS:雙方都出示憑證

內部 service-to-service 常用 mTLS:不只 client 驗 server,server 也驗 client 的憑證。沒有對應 cert 連都連不進來。

Server 端 — 同時要求 client 出示 cert

go (server) 
cert, err := tls.LoadX509KeyPair("server.crt", "server.key")
if err != nil { log.Fatal(err) }

clientCAs := x509.NewCertPool()
caBytes, _ := os.ReadFile("client-ca.crt")
clientCAs.AppendCertsFromPEM(caBytes)

creds := credentials.NewTLS(&tls.Config{
    Certificates: []tls.Certificate{cert},
    ClientCAs:    clientCAs,
    ClientAuth:   tls.RequireAndVerifyClientCert,  // ← 強制驗 client cert
    MinVersion:   tls.VersionTLS13,
})

s := grpc.NewServer(grpc.Creds(creds))

Client 端 — 出示自己的 cert

go (client) 
cert, err := tls.LoadX509KeyPair("client.crt", "client.key")
if err != nil { log.Fatal(err) }

serverCAs := x509.NewCertPool()
caBytes, _ := os.ReadFile("server-ca.crt")
serverCAs.AppendCertsFromPEM(caBytes)

creds := credentials.NewTLS(&tls.Config{
    Certificates: []tls.Certificate{cert},   // ← client 自己的 cert
    RootCAs:      serverCAs,
    ServerName:   "api.example.com",
    MinVersion:   tls.VersionTLS13,
})

conn, err := grpc.NewClient("api.example.com:443", grpc.WithTransportCredentials(creds))
i
service mesh(Istio / Linkerd)會自動做 mTLS k8s + service mesh 的環境,sidecar proxy 會在你 application 不知情的情況下做 mTLS,你的 gRPC 還是用 insecure.NewCredentials() 即可 — 安全在 mesh 層處理。手寫 mTLS 仍適合理解原理,以及無 mesh 的環境。
05 — PerRPC token

PerRPCCredentials:把 token 自動附在每支 RPC

不想每次手動 metadata.AppendToOutgoingContext?實作 PerRPCCredentials interface 讓 grpc-go 自動帶。

go (client) 
// 自己實作:能動態取 token(例如從 oauth source refresh)
type tokenAuth struct {
    source func() (string, error)
}

func (t *tokenAuth) GetRequestMetadata(ctx context.Context, _ ...string) (map[string]string, error) {
    token, err := t.source()
    if err != nil { return nil, err }
    return map[string]string{
        "authorization": "Bearer " + token,
    }, nil
}

// RequireTransportSecurity = true 表示「沒 TLS 不准用」(token 不能裸傳)
func (t *tokenAuth) RequireTransportSecurity() bool { return true }

// 安裝
conn, err := grpc.NewClient(addr,
    grpc.WithTransportCredentials(tlsCreds),
    grpc.WithPerRPCCredentials(&tokenAuth{source: refreshToken}),
)

之後每支 RPC 自動帶 authorization header,handler 不用改。

!
token 在裸 HTTP/2 上是明文 PerRPCCredentials 把 token 放 metadata,沒 TLS 等於明文傳。實作 RequireTransportSecurity() bool 回 true,grpc-go 會在 connection 沒有 TLS 時直接拒絕掛上去 — 這是基礎安全保護。
06 — 怎麼選

實際 production 怎麼搭配

場景TransportPerRPC
對外公開 API(client 是其他公司 / 終端用戶) 單向 TLS(用公開 CA) OAuth bearer token / API key
內部 microservice(no mesh) mTLS(內部 CA 簽的) (mTLS cert 已帶身份,通常不必 PerRPC)
內部 microservice(有 service mesh) insecure(mesh sidecar 處理 mTLS) 視情況加 user-context token
本機開發 insecure (略)
!
對外服務絕對不要用 insecure insecure.NewCredentials() 等於明文 HTTP/2(h2c)。對網際網路曝露 = 流量可被中間人讀取/修改、token 一傳就洩漏。只在 localhost 開發、或 service mesh 已處理的情況下用。
