三種 Streaming RPC

01 — 三種模式對照

關鍵字:stream

protobuf 在 RPC 定義裡用 stream 關鍵字標記哪一邊是 streaming。三種 streaming 模式對應三種 stream 位置:

模式proto典型場景
Server streamingrpc M (Req) returns (stream Resp);log tail、訂閱、大查詢分批送
Client streamingrpc M (stream Req) returns (Resp);檔案分塊上傳、批次寫入、彙總
Bidirectionalrpc M (stream Req) returns (stream Resp);即時聊天、雙向同步、雙向協商
i
Streaming 共通模式 所有 streaming handler 都拿到一個 stream 物件,提供 Send(msg) 與 Recv() (msg, error)。對方關掉 stream 時 Recv 會回 io.EOF — 這是 Go 的慣例,代表「正常結束」(error 但不是 error)。
02 — proto 怎麼寫

三個 service method,一份 proto

proto/chat.proto 
syntax = "proto3";
package chat.v1;
option go_package = "example.com/chat/proto;chatpb";

service Chat {
  // 1. Server streaming:訂閱頻道,server 一直推訊息
  rpc Subscribe (SubscribeRequest) returns (stream Message);

  // 2. Client streaming:批次上傳,server 收完後給總結
  rpc UploadLogs (stream LogLine) returns (UploadSummary);

  // 3. Bidirectional:雙向聊天
  rpc ChatRoom (stream Message) returns (stream Message);
}

message SubscribeRequest { string channel = 1; }
message Message          { string from = 1; string text = 2; }
message LogLine          { string line = 1; }
message UploadSummary    { int32 count = 1; int64 bytes = 2; }
03 — Server streaming

Server 推、Client 收

場景:client 訂閱一個頻道,server 持續推訊息給它。

Server 端

go (server) 
func (s *server) Subscribe(req *pb.SubscribeRequest, stream pb.Chat_SubscribeServer) error {
    // handler 簽名:(req *Req, stream XxxServer) error
    // stream 物件已經帶了 ctx,可用 stream.Context()

    for i := 1; i <= 5; i++ {
        msg := &pb.Message{
            From: "server",
            Text: fmt.Sprintf("#%s msg %d", req.GetChannel(), i),
        }
        if err := stream.Send(msg); err != nil {
            return err  // client 斷線會在這邊 return error
        }
        time.Sleep(time.Second)
    }
    return nil  // return nil = 結束 stream(client 那邊會收到 io.EOF)
}

Client 端

go (client) 
stream, err := client.Subscribe(ctx, &pb.SubscribeRequest{Channel: "general"})
if err != nil {
    log.Fatal(err)
}

for {
    msg, err := stream.Recv()
    if err == io.EOF {
        break  // server 結束 stream
    }
    if err != nil {
        log.Fatalf("recv error: %v", err)
    }
    log.Printf("got: %s", msg.GetText())
}
i
真實場景:訂閱、log tail、cursor-based 分頁 server streaming 是「一次請求、長時間回應」 — 比起一直 polling REST endpoint,在「等通知」的場景上效率天差地遠。第 10 課會教 keepalive / health check 確保長連線穩。
04 — Client streaming

Client 推、Server 收完給總結

場景:client 把一堆 log line 上傳,server 統計後回一個總結。

Server 端

go (server) 
func (s *server) UploadLogs(stream pb.Chat_UploadLogsServer) error {
    // handler 簽名:(stream XxxServer) error
    // 沒有 req 參數,因為 req 是 stream 來的

    var count int32
    var bytes int64
    for {
        line, err := stream.Recv()
        if err == io.EOF {
            // client 結束 stream → 用 SendAndClose 回應 + 關閉
            return stream.SendAndClose(&pb.UploadSummary{
                Count: count,
                Bytes: bytes,
            })
        }
        if err != nil {
            return err
        }
        count++
        bytes += int64(len(line.GetLine()))
    }
}

Client 端

go (client) 
stream, err := client.UploadLogs(ctx)
if err != nil {
    log.Fatal(err)
}

lines := []string{"first", "second", "third"}
for _, l := range lines {
    if err := stream.Send(&pb.LogLine{Line: l}); err != nil {
        log.Fatal(err)
    }
}

// 上傳完用 CloseAndRecv 關 stream + 收 server 回應
summary, err := stream.CloseAndRecv()
if err != nil {
    log.Fatal(err)
}
log.Printf("uploaded %d lines, %d bytes", summary.GetCount(), summary.GetBytes())
!
關 stream 的兩個 helper
  • Server:stream.SendAndClose(resp) — 送回應 + 結束 stream
  • Client:stream.CloseAndRecv() — 結束 send 半邊 + 等 server 回應
這是 client streaming 才有的 helper。Server streaming 與 Bidi 的 server 端是用 return nil 結束。
05 — Bidirectional

雙向都串、可以交錯

場景:聊天室。雙方獨立決定何時 send、何時 close,訊息交錯傳。

Server 端

go (server) 
func (s *server) ChatRoom(stream pb.Chat_ChatRoomServer) error {
    for {
        in, err := stream.Recv()
        if err == io.EOF {
            return nil  // client 不再送了
        }
        if err != nil {
            return err
        }

        // 簡單 echo 回去(實務可以從 broker 廣播給其他 client)
        out := &pb.Message{
            From: "server",
            Text: "echo: " + in.GetText(),
        }
        if err := stream.Send(out); err != nil {
            return err
        }
    }
}

Client 端 — 用 goroutine 同時 send 與 recv

go (client) 
stream, err := client.ChatRoom(ctx)
if err != nil {
    log.Fatal(err)
}

// recv 與 send 必須在不同 goroutine — stream 的 Send/Recv 是 thread-safe 對另一方,
// 但同方向 (兩個 goroutine 同時 Send) 不是。
done := make(chan struct{})

go func() {
    defer close(done)
    for {
        in, err := stream.Recv()
        if err == io.EOF {
            return
        }
        if err != nil {
            log.Printf("recv error: %v", err)
            return
        }
        log.Printf("got: %s", in.GetText())
    }
}()

// 主 goroutine 負責送
for _, text := range []string{"hi", "how are you", "bye"} {
    if err := stream.Send(&pb.Message{From: "client", Text: text}); err != nil {
        log.Fatal(err)
    }
    time.Sleep(500 * time.Millisecond)
}

// 告訴 server「我不再送了」(server 那邊會收到 io.EOF),但仍會繼續收 server 的回覆
stream.CloseSend()
<-done
i
Bidi 的 thread-safety 規則 stream.Send 與 stream.Recv 在不同方向是 thread-safe(可一個 goroutine 收、一個 goroutine 送),但同方向多個 goroutine 同時 Send 不是 thread-safe(同方向想多 goroutine 寫請自己加 mutex 或用 channel 串到單一 sender goroutine)。
06 — io.EOF 的角色

io.EOF 不是錯誤,是「正常結束」訊號

Go 的 io.EOF 在 streaming RPC 裡有明確語意:

場景Recv() 回傳
對方還會送更多(msg, nil)
對方正常結束 stream(return nil / SendAndClose / CloseSend)(nil, io.EOF)
對方異常(網路斷、server crash、deadline exceeded)(nil, <real error>) — 用 status.Code(err) 取代碼

所以慣用模式總是:

go 
for {
    msg, err := stream.Recv()
    if err == io.EOF {
        break      // 正常結束
    }
    if err != nil {
        return err  // 真錯誤
    }
    process(msg)
}
07 — 常見坑

Streaming 三個易踩雷區

1. stream.Send 沒檢查 error 高頻

Send 在「對方已斷線」「buffer 滿」「context 已 cancel」時會回 error。如果忽略繼續 send,你會丟訊息也不會察覺。每個 Send 都要檢 err,出錯就 break 並收尾。

2. Bidi 同一個 stream 用兩個 goroutine 同時 Send 資料毀損

Send 對同方向不是 thread-safe,兩個 goroutine 寫會混合 byte stream 造成 protobuf decode 失敗。要嘛只用一個 sender goroutine,要嘛用 channel 把要 send 的訊息排入單一 goroutine。

3. Server streaming handler 忘了關注 stream.Context().Done() 資源洩漏

長時間 server streaming 應該 select stream.Context().Done(),client 取消或斷線時及時退出。否則 server 會一直跑迴圈、佔用 goroutine 與資源。

for {
    select {
    case <-stream.Context().Done():
        return stream.Context().Err()
    case msg := <-broker.NextMessage():
        stream.Send(msg)
    }
}
