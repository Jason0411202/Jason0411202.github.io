Context、Metadata、Deadline、Cancellation

ctx 是 gRPC 每支 RPC 的中樞神經 — 它帶 deadline、metadata、cancellation 訊號。學完這堂能正確控制超時、做認證 token 傳遞、處理 client 取消。

01 — ctx 的角色

context.Context 在 gRPC 裡裝了什麼

每支 gRPC handler 第一個參數都是 ctx context.Context。它帶三件你必須會用的東西:

圖 18.1 — ctx 三件事
i
不要把資料偷偷塞 ctx ctx.Value() 設計上只用來傳 request-scoped 控制資料(authn 結果、trace id)。業務資料應該走 request struct,不要塞 ctx。Go community 的共識:「ctx is not your bag」。
02 — 送 metadata

Client 端:把 metadata 塞進 outgoing ctx

Metadata 是 RPC 的 header,結構是 map[string][]string(同一 key 可以有多值)。最常見的用法:傳認證 token、trace id、locale。

go (client) 
import "google.golang.org/grpc/metadata"

// 方法 1:AppendToOutgoingContext(最常用)
ctx = metadata.AppendToOutgoingContext(ctx,
    "authorization", "Bearer "+token,
    "x-trace-id",    traceID,
)

// 方法 2:NewOutgoingContext + 完整 MD(會覆蓋之前 append 的)
md := metadata.New(map[string]string{
    "authorization": "Bearer " + token,
})
ctx = metadata.NewOutgoingContext(ctx, md)

// 然後正常呼叫 RPC,metadata 會自動隨 RPC 送出去
resp, err := client.SayHello(ctx, &pb.HelloRequest{Name: "Jason"})
!
Metadata 鍵的規則
  • 大小寫不敏感(server 收到都會變小寫)
  • 不能以 grpc- 開頭 — 那個前綴是 gRPC 自己保留
  • 傳二進位資料時 key 必須以 -bin 結尾(如 session-token-bin),value 是 byte slice
  • 非二進位 value 必須是 ASCII 可印字元
03 — 收 metadata

Server 端:從 incoming ctx 取出 metadata

go (server) 
func (s *server) SayHello(ctx context.Context, req *pb.HelloRequest) (*pb.HelloReply, error) {
    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Internal, "no metadata")
    }

    // 同 key 多值,所以 Get 回傳 []string
    auth := md.Get("authorization")
    if len(auth) == 0 {
        return nil, status.Error(codes.Unauthenticated, "missing authorization")
    }
    token := strings.TrimPrefix(auth[0], "Bearer ")

    // ... 驗 token ...
    return &pb.HelloReply{Message: "Hello, " + req.GetName()}, nil
}
04 — Header / Trailer

Server 也能送 metadata 回 client

Server → Client 方向有兩種:

類型送出時機典型用途
HeaderRPC 一開始(回應 body 之前)server 版本、quota 餘額、rate limit 提示
TrailerRPC 結束(回應 body 之後)處理時長、debug info

Server 送

go (server) 
// 設(可多次)
grpc.SetHeader(ctx, metadata.Pairs("x-server-version", "v1.2.3"))
grpc.SetTrailer(ctx, metadata.Pairs("x-elapsed-ms", fmt.Sprint(elapsed)))

// 主動立刻送 header(不等到 response,常用於 streaming)
grpc.SendHeader(ctx, metadata.Pairs("x-stream-id", sid))

Client 收

go (client) 
var header, trailer metadata.MD
resp, err := client.SayHello(
    ctx,
    &pb.HelloRequest{Name: "Jason"},
    grpc.Header(&header),    // CallOption,RPC 完成後 header 會被填值
    grpc.Trailer(&trailer),
)
if err == nil {
    log.Printf("server version: %v", header.Get("x-server-version"))
    log.Printf("elapsed: %v", trailer.Get("x-elapsed-ms"))
}
05 — Deadline

Deadline:gRPC 的「最遲完成時間」

gRPC 文件原話:

"
gRPC allows clients to specify how long they are willing to wait for an RPC to complete before the RPC is terminated with a DEADLINE_EXCEEDED error.

Client 設 deadline

go (client) 
// 從現在起 2 秒後 deadline
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()  // 一定要 defer cancel,否則資源洩漏

resp, err := client.SayHello(ctx, &pb.HelloRequest{Name: "Jason"})
if status.Code(err) == codes.DeadlineExceeded {
    log.Println("too slow")
}

Deadline 會自動 propagate

這是 gRPC 最重要的設計之一:deadline 跟 ctx 一起跨服務傳遞。

圖 18.2 — deadline 是「絕對時間」,跨服務不會 reset
go (Service A) 
func (a *serviceA) Handle(ctx context.Context, req *pb.Req) (*pb.Resp, error) {
    // ✓ 對:把 ctx 直接傳下去,deadline 自動 propagate
    res, err := a.bClient.Process(ctx, &pb.BReq{...})

    // ✗ 錯:context.Background() 會丟掉上游的 deadline
    // res, err := a.bClient.Process(context.Background(), ...)

    return wrap(res), err
}
!
不要用 context.Background() 呼叫下游 這會丟掉 deadline 與 cancellation,變成「上游已經放棄等了,下游還在傻傻 query」 — 浪費資源、難 debug、破壞 trace。handler 拿到的 ctx 應該原封不動傳給下游(除非你明確要縮短 deadline)。
06 — Cancellation

Cancellation:client 不等了,server 該停下

無論 client 主動 cancel()、deadline 到、或網路斷,server handler 拿到的 ctx 都會 cancel。handler 必須主動關注,否則只是繼續燒 CPU。

go (server) 
func (s *server) BigQuery(ctx context.Context, req *pb.Req) (*pb.Resp, error) {
    // 模式 1:每個迭代檢查
    for _, item := range hugeList {
        if err := ctx.Err(); err != nil {
            return nil, status.FromContextError(err).Err()
        }
        process(item)
    }

    // 模式 2:select(streaming / 等 channel)
    select {
    case result := <-resultCh:
        return result, nil
    case <-ctx.Done():
        return nil, status.FromContextError(ctx.Err()).Err()
    }
}

ctx.Err() 可能是兩種:

  • context.Canceled — client 主動 cancel
  • context.DeadlineExceeded — deadline 到了

status.FromContextError 會把它對應到正確的 gRPC status code(Canceled / DeadlineExceeded),server 端永遠用這個 helper,不要自己手寫。

i
關於「changes are not rolled back」 官方文件直白警告:「Changes made before a cancellation are not rolled back.」 Cancellation 只代表「不再做新事」,已經寫進 DB 的東西不會自動回滾。如果你的 handler 中途寫了 DB 才被 cancel,要嘛用 transaction、要嘛接受可能有部分變更。
