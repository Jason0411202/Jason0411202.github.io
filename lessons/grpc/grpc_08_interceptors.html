Interceptors 中介層

gRPC 的 middleware 機制 — 把橫切性邏輯(log、auth、recovery)從 handler 抽出來,變成可組裝的 interceptor。學完能寫一個生產級 server 的標準四件組。

01 — interceptor 是什麼

把橫切邏輯從 handler 抽出來

每支 handler 都要做的事(log 一行 access log、catch panic、檢查 token、量處理時間)如果寫進 handler,會把業務邏輯淹沒。Interceptor 就是 gRPC 的 middleware,讓你把這些事抽到外層。

圖 20.1 — chain 順序:第一個 = 最外層,最後一個 = 最貼近 handler
02 — 四種簽名

四個 interceptor 型別,Unary × Stream × Server × Client

型別簽名
UnaryServerInterceptor 
func(ctx, req any, info *UnaryServerInfo, handler UnaryHandler) (any, error)
StreamServerInterceptor 
func(srv any, ss ServerStream, info *StreamServerInfo, handler StreamHandler) error
UnaryClientInterceptor 
func(ctx, method string, req, reply any, cc *ClientConn, invoker UnaryInvoker, opts ...CallOption) error
StreamClientInterceptor 
func(ctx, desc *StreamDesc, cc *ClientConn, method string, streamer Streamer, opts ...CallOption) (ClientStream, error)

所有 interceptor 都必須呼叫 handler(server 端) 或 invoker/streamer(client 端)把控制權交給下一層,否則 RPC 會卡住。

03 — 鏈式執行

串多個 interceptor

Server 端

go (server) 
s := grpc.NewServer(
    grpc.ChainUnaryInterceptor(
        recoveryInterceptor,   // 最外層 — 第一個進、最後一個出
        loggingInterceptor,
        authInterceptor,
        timingInterceptor,     // 最內層 — 緊鄰 handler
    ),
    grpc.ChainStreamInterceptor(
        streamRecoveryInterceptor,
        streamLoggingInterceptor,
    ),
)

Client 端

go (client) 
conn, err := grpc.NewClient(
    addr,
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithChainUnaryInterceptor(
        clientLoggingInterceptor,
        clientRetryInterceptor,
    ),
)
i
順序很重要 recovery 必須是最外層(第一個),才能接住所有 inner interceptor 與 handler 的 panic。auth 通常放在 logging 之後 — 確保未授權 request 也有 access log。
04 — logging

1. Logging interceptor — 每支 RPC 一行 log

go 
func loggingInterceptor(
    ctx context.Context,
    req any,
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (any, error) {
    start := time.Now()
    resp, err := handler(ctx, req)  // 呼叫下一層
    log.Printf("%s | %s | %v",
        info.FullMethod,           // 例如 "/greeter.v1.Greeter/SayHello"
        status.Code(err),          // OK / NotFound / ...
        time.Since(start),
    )
    return resp, err
}
05 — recovery

2. Recovery interceptor — 接住 panic

handler 一旦 panic,gRPC 預設行為是整個 connection 被斷(server crash 雖然不會,但 client 連線會中斷)。recovery interceptor 把 panic 轉成 codes.Internal 回給 client。

go 
func recoveryInterceptor(
    ctx context.Context,
    req any,
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (resp any, err error) {
    defer func() {
        if r := recover(); r != nil {
            log.Printf("PANIC %s: %v\n%s",
                info.FullMethod, r, debug.Stack())
            err = status.Errorf(codes.Internal, "internal error")
        }
    }()
    return handler(ctx, req)
}
i
用社群現成套件 github.com/grpc-ecosystem/go-grpc-middleware(v2)提供 production-grade 的 logging、recovery、auth、retry、ratelimit 等 interceptor,實務上幾乎都直接用,不用自己手寫。本課手寫是為了讓你看懂內部結構。
06 — auth

3. Auth interceptor — 統一驗 token

go 
type userIDKey struct{}

func authInterceptor(
    ctx context.Context,
    req any,
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (any, error) {
    // 某些 method 不需要驗(健康檢查、登入)
    if info.FullMethod == "/auth.v1.AuthService/Login" {
        return handler(ctx, req)
    }

    md, ok := metadata.FromIncomingContext(ctx)
    if !ok {
        return nil, status.Error(codes.Unauthenticated, "missing metadata")
    }

    auth := md.Get("authorization")
    if len(auth) == 0 {
        return nil, status.Error(codes.Unauthenticated, "missing token")
    }
    token := strings.TrimPrefix(auth[0], "Bearer ")

    userID, err := verifyJWT(token)
    if err != nil {
        return nil, status.Error(codes.Unauthenticated, "invalid token")
    }

    // 把 user_id 塞進 ctx,handler 可以取出來用
    ctx = context.WithValue(ctx, userIDKey{}, userID)
    return handler(ctx, req)
}

// handler 取 user id
func userIDFromCtx(ctx context.Context) string {
    v, _ := ctx.Value(userIDKey{}).(string)
    return v
}
07 — timing

4. Timing interceptor — 量每支 RPC 的時長

go 
var rpcDuration = prometheus.NewHistogramVec(...)  // 略

func timingInterceptor(
    ctx context.Context,
    req any,
    info *grpc.UnaryServerInfo,
    handler grpc.UnaryHandler,
) (any, error) {
    start := time.Now()
    resp, err := handler(ctx, req)
    elapsed := time.Since(start).Seconds()

    rpcDuration.WithLabelValues(
        info.FullMethod,
        status.Code(err).String(),
    ).Observe(elapsed)

    return resp, err
}
i
真實 production 通常用 OpenTelemetry 手寫 prometheus histogram 適合學習,production 多用 go.opentelemetry.io/contrib/instrumentation/google.golang.org/grpc/otelgrpc — 一個 interceptor 包辦 metrics + traces + logs,且能跨服務 propagate trace ID。

Stream interceptor 的差異

Stream 版本拿到的是 ServerStream,不是 req。要在 stream 上加邏輯(例如 wrap stream 的 Send/Recv),需要自己包一個 wrapper struct:

go 
type wrappedStream struct {
    grpc.ServerStream
    ctx context.Context
}

func (w *wrappedStream) Context() context.Context { return w.ctx }

func streamAuthInterceptor(srv any, ss grpc.ServerStream,
    info *grpc.StreamServerInfo, handler grpc.StreamHandler) error {
    // 驗 token...
    newCtx := context.WithValue(ss.Context(), userIDKey{}, userID)
    return handler(srv, &wrappedStream{ServerStream: ss, ctx: newCtx})
}
