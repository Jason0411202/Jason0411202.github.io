01 — Token Ring

無 master 的環：Consistent Hashing 在工程上的實踐

Cassandra 沒有 primary 也沒有 leader。它把所有節點排成一個邏輯環，環上每一點都對應一個 64-bit 的 token。每筆資料的 partition key 經過 hash 也得到一個 token，然後從那個 token 出發順時針往下找第一個節點 — 那個節點就是這筆資料的「主節點（primary replica）」。

圖 C2.1 — partition key 經過 hash 落到環上某點，順時針取到的第一個 node 即主 replica

Vnode（virtual node）：為什麼真實環上有幾百個點

實作上每台實體機不是只占環上一個位置。預設每台 server 有 16 個 vnode（虛擬節點），每個 vnode 對應一個獨立 token。這帶來兩個好處：

  • 負載更平均：實體機不再對應「環上一段大區間」，而是被切成 16 段散落各處。即便 4 台機器，環上也有 64 個切點。
  • 加減節點時資料搬移更平均：新節點加入時 16 個 vnode 各從不同既有 vnode 偷一塊資料；不會像舊版（單 token）只跟相鄰兩台搶。
i
num_tokens 別亂改 5.0 預設每台 16 個 vnode（過去 4.x 預設 256）。16 是社群實測對 streaming repair 較友善的值。新建叢集就用預設，不要因為網路文章說「設高一點」就改。
02 — Partitioner

Murmur3Partitioner：把 partition key 變 token

Cassandra 5.0 預設使用 Murmur3Partitioner。它的工作很單純：

概念 
token = MurmurHash3(partition_key_bytes)   // 64-bit signed int
       // 範圍：-2^63 ~ +2^63 - 1

Murmur3 是一個非加密 hash，速度極快、分布均勻。早期版本另有 RandomPartitioner（用 MD5），現在已不推薦，主要因為 Murmur3 速度快很多。

請注意：partitioner 是叢集級設定，叢集起來後幾乎無法改。所有 keyspace 都共用同一個 partitioner。

03 — Replication

RF、replica placement、NetworkTopologyStrategy

每個 keyspace 建立時要指定 Replication Factor（RF） — 一筆資料要存幾份。RF=3 是業界最常見：

cql 
-- 多 DC、production-grade 設法
CREATE KEYSPACE stockbot
  WITH replication = {
    'class': 'NetworkTopologyStrategy',
    'dc1': 3,    -- dc1 放 3 份
    'dc2': 3     -- dc2 也放 3 份
  };

規則：找到 partition 的主 replica 後，沿著環繼續順時針取下一個，但不能在同一個 rack（架），也不能在同一個 DC 超過該 DC 的 RF 數量。這個放置邏輯由 NetworkTopologyStrategy + snitch 共同決定。

Snitch：告訴 Cassandra 誰跟誰是同 rack / 同 DC

  • GossipingPropertyFileSnitch（最常用）：每台機器讀 cassandra-rackdc.properties 設定自己的 rack/DC，再透過 gossip 告訴大家。
  • Ec2Snitch / GoogleCloudSnitch：自動讀 cloud metadata。
  • SimpleSnitch：所有機器都假裝同一個 rack（demo 用）。
!
SimpleStrategy 別在 production 用 上一堂為了 demo 用 SimpleStrategy + RF=1。它不分 DC、不分 rack，只能單 DC 玩玩。production 一律 NetworkTopologyStrategy，每個 DC 明確指定 RF。
04 — Gossip

Gossip 協定：節點怎麼互相認識

Cassandra 沒有 master，那怎麼知道叢集現在有幾台機器、誰活著、token 範圍在哪？答案是 gossip 協定：

  • 每秒，每台節點挑 1~3 台 隨機節點交換狀態（包含「我認識的所有其他節點目前在哪、token 是什麼、上次心跳幾秒前」）。
  • 狀態用 version vector（version + heartbeat）合併，新訊息覆蓋舊訊息。
  • 幾秒內，整個叢集就會收斂。

當你 nodetool status 看到節點狀態（UN / DN / ?），都是 gossip 收上來的。

Hinted Handoff：暫時掛掉的節點怎麼補資料

假設要寫到 N1, N2, N3 三個 replica，但 N3 此刻網路斷線：

  1. Coordinator 把寫入給 N1、N2 並回 client「成功」（依 consistency level）。
  2. Coordinator 同時為 N3 留一份 hint（暫存記錄）。
  3. 等到 gossip 偵測到 N3 復活，coordinator 自動把 hint 推給 N3 補上。

預設 hint 最多保留 3 小時。超過就放棄、改用 repair 補資料（C-04 會講）。

05 — Coordinator

沒有 master，那 client 連誰？

Client 可以連叢集中任何一台節點。被連到的那台節點就成為這次請求的 coordinator。它做的事：

  1. 用 partitioner 算出資料該到哪些 replica。
  2. 把寫入 同步 廣播給該寫的 replica（依 RF 與 consistency level）。
  3. 等到收齊「足夠多」的成功回覆，就回 client。
圖 C2.2 — coordinator 等到「達成 consistency level」就回，剩下慢的 replica 在背後補上
06 — Consistency Level

可調的一致性：CL 是每筆讀寫的旋鈕

CL寫入意義讀取意義常見用途
ONE1 個 replica 寫成功即回1 個 replica 回應即回最快、容錯最差
QUORUM多數派（> RF/2）寫成功多數派回應強一致的關鍵
LOCAL_QUORUM本 DC 多數派本 DC 多數派多 DC 標配
EACH_QUORUM每個 DC 都要多數派(寫入用)跨 DC 寫入強一致
ALL所有 replica 都成功所有 replica 回應很少用，太脆弱
ANY連 hint 也算（最弱）(無)幾乎不要用

強一致的公式

!
R + W > RF 即強一致 假設 RF=3，W=QUORUM=2 並 R=QUORUM=2。最差情況下，寫入的 2 個 replica 跟讀取的 2 個 replica必有交集，所以一定能讀到最新值。
→ 但代價：寫慢一點、讀慢一點、容忍更少節點掛掉。

多 DC：用 LOCAL_QUORUM 而不是 QUORUM

跨 DC 網路 RTT 動輒 50ms 以上。如果用 QUORUM，每次寫入都要等遠端 DC 回應，延遲爆炸。實務上幾乎一律用 LOCAL_QUORUM：本地強一致、跨 DC 用非同步複寫達成最終一致。

07 — PACELC

CAP 之外：正常時的取捨叫 PACELC

CAP 只描述「分區發生時」要選 C 或 A。但分散式系統大多時間是正常運作的 — 此時還是有取捨：Latency 還是 Consistency？這就是 PACELC（Daniel Abadi 2010 提出）：

概念 
If Partition happens → choose Availability or Consistency
Else            → choose Latency or Consistency

Cassandra: PA / EL   // 分區時 A、平時 L → 預設追求快
MongoDB:   PC / EC   // 分區時 C、平時 C → 追求對
DynamoDB:  PA / EL   // 跟 Cassandra 一樣血統
HBase:     PC / EC

這也是為什麼很多人說「Cassandra 預設下不是強一致」 — 不是因為它不能，而是它預設選了延遲。你要強一致，就調 CL。

08 — 動手玩

用 nodetool 看自己的環

bash 
# 1. 看叢集所有節點狀態
$ docker exec -it cass5 nodetool status

Datacenter: datacenter1
=======================
Status=Up/Down
|/ State=Normal/Leaving/Joining/Moving
--  Address     Load        Tokens  Owns (effective)  Host ID
UN  127.0.0.1   183.5 KiB   16      100.0%            7e62...

# 2. 看 token 範圍（每個 vnode 一行）
$ docker exec -it cass5 nodetool ring

# 3. 看一筆 partition key 對應到哪台機器
$ docker exec -it cass5 nodetool getendpoints stockbot price_by_symbol 'TSLA'
127.0.0.1   # 單節點所以只有自己

# 4. 在 cqlsh 裡切換 consistency level 觀察
$ docker exec -it cass5 cqlsh
cqlsh> CONSISTENCY QUORUM;
Consistency level set to QUORUM.

cqlsh> USE stockbot;
cqlsh:stockbot> SELECT * FROM price_by_symbol WHERE symbol='TSLA';

# 5. 看一張表「每個 partition 在哪些 replica 上」
cqlsh> DESCRIBE KEYSPACE stockbot;
i
單節點看不到太多東西 要看真實的多節點 ring，最簡單方式是用 ccm（Cassandra Cluster Manager）在本機起 3 節點：ccm create test -v 5.0.8 -n 3 -s。或者用 docker-compose。本系列為了上手簡單只跑單節點，但 nodetool 指令在多節點環境會有趣很多。
