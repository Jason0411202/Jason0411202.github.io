Cassandra 04 · 引擎內部

寫入快、讀取慢的祕密

01 — 寫入

寫入路徑：為什麼 Cassandra 寫得快

圖 C4.1 — Cassandra 是 LSM-tree，純 sequential 寫入，極快
  1. Commit Log：append 一行到磁碟（durability 保險）。
  2. Memtable：寫到記憶體裡的有序結構（5.0 起預設 Trie Memtable，更省 RAM）。
  3. 回 Client：上面兩步完成就回成功。整個寫入路徑沒有「找舊資料 + 修改」的步驟，全部 append-only。
  4. 背景 flush：memtable 滿了（或定期）會 flush 成新的 SSTable 到磁碟。SSTable 是 immutable（永不修改）。
i
「更新」其實是「再寫一筆新的」 UPDATE x SET name='new' WHERE id=1; 在 Cassandra 內部就是寫一筆新的「name 欄位 = 'new'」record，舊的還在某個 SSTable 裡。讀取時取 timestamp 最新的那一筆勝出。這就是「寫入快」的代價：讀取要做合併。
02 — 讀取

讀取路徑：把多份散落的資料合併

同一筆資料可能散落在多個 SSTable + memtable 裡。讀取流程：

  1. 查 row cache（如果開）：直接命中。
  2. 查 memtable：找最新版本。
  3. 查 SSTable 之前先用 Bloom filter：每個 SSTable 都有 bloom filter（記憶體中），用 partition key 快速判斷「這個 SSTable 可能包含這個 partition」。命中才去開檔。
  4. 查 Partition Index / Summary：定位 partition 在 SSTable 內的 offset。
  5. 讀出多份結果，依 timestamp 合併：保留最新的、丟掉舊的、處理 tombstone。
  6. Read Repair（如果有設）：發現 replica 之間不一致時，背景修補。
!
SSTable 越多，讀越慢 每多一個 SSTable，讀取就多查一次 bloom filter（命中時還要 disk I/O）。compaction 的本質就是不斷把多個 SSTable 合併成更少、更大的 SSTable。
03 — Tombstone

刪除其實是「寫一個墓碑」

Cassandra 的 SSTable 是 immutable，所以 DELETE 不能就地把資料移除 — 它會寫一個特殊標記叫 tombstone 進去，記錄「這個 column 在這個 timestamp 被刪了」。

Tombstone 必須保留一段時間（叫 gc_grace_seconds，預設 10 天）才能真的清除。為什麼？

!
tombstone 是分散式刪除的「保險絲」 想像 N3 節點離線時 client 在 N1/N2 刪了一筆資料。當 N3 回來，N1/N2 必須記得「這筆已被刪」並推給 N3 — 不然 N3 那份舊資料會在下次讀取時「復活」。tombstone 就是這個記號。gc_grace_seconds 必須 > 你能容忍節點離線的最長時間 + repair 週期。

Tombstone 太多會發生什麼

  • 讀取 partition 時要把有效資料 + tombstone都掃過才能算出最終結果。
  • 如果一個 partition 有上萬個 tombstone，讀取超慢，且會在 log 看到 Read X live and Y tombstone cells in... 警告。
  • 反模式：用 Cassandra 當 queue（不斷 enqueue + dequeue）→ tombstone 暴增、讀取變慢。Cassandra 不適合這類 high-churn 場景。
04 — Compaction

Compaction 策略：合併 SSTable 的方式

Compaction 在背景跑，把多個 SSTable 合併成一個大的，順便清理過期 tombstone。Cassandra 提供 5 種策略：

策略適合場景取捨
STCS (SizeTieredCompactionStrategy) 寫入為主、讀取較少 寫便宜；讀可能要查很多 SSTable
LCS (LeveledCompactionStrategy) 讀寫平均、需要可預測讀延遲 每筆資料保證在每個 level 最多 1 個 SSTable；I/O 高
TWCS (TimeWindowCompactionStrategy) 時序資料 + TTL 相同時間窗的資料同檔，整檔過期可一次刪
UCS (UnifiedCompactionStrategy) 5.0 預設 幾乎所有場景 用一個策略動態調，類似 STCS+LCS+TWCS 的綜合體
DTCS (已 deprecated) 由 TWCS 取代
i
5.0 起的建議：先用 UCS UCS 是 Cassandra 5.0 引入的「統一 compaction」，社群目標是讓你不用為了不同 workload 切換策略。預設值對 90% 場景夠好。除非你有實測證明 LCS / TWCS 更好，就先用 UCS。

怎麼觀察

bash 
$ docker exec -it cass5 nodetool compactionstats
pending tasks: 2
- ks.tbl: 2

$ docker exec -it cass5 nodetool tablestats stockbot.price_by_symbol
SSTable count: 4
SSTable Compression Ratio: 0.39
Number of partitions (estimate): 1024
Memtable cell count: 12305
Memtable data size: 824 KiB
05 — Repair

Repair：定期讓 replica 達成一致

Cassandra 是 AP 系統，加上有 hinted handoff 上限、CL 不見得都用 ALL，所以 replica 之間會逐漸偏差。nodetool repair 就是用來強制比對所有 replica、補齊差異的指令。

  • full repair：比較 replica 上所有資料的 Merkle tree，找差異補齊。代價高。
  • incremental repair：只比較「上次 repair 後的變更」。預設方法，便宜。
  • subrange repair：只 repair 某一段 token range，可分批跑。
!
必須在 gc_grace_seconds 內 repair 過一次 否則被刪的 tombstone 過期清除後，舊資料可能在某個 replica 上「復活」。所以多數 production 設定每 7~10 天跑一次 incremental repair（可用 Reaper 工具排程）。
06 — 動手玩

觀察 SSTable + tombstone

bash + cqlsh 
# 1. 寫進去 → 看 memtable 大小
$ docker exec -it cass5 cqlsh
cqlsh> USE stockbot;
cqlsh> INSERT INTO price_by_symbol (symbol, ts, price, volume)
       VALUES ('NVDA', toTimeStamp(now()), 700.5, 5000);
cqlsh> EXIT;

$ docker exec -it cass5 nodetool tablestats stockbot.price_by_symbol \
    | grep -E '(SSTable count|Memtable)'

# 2. 強制 flush memtable → 變成新 SSTable
$ docker exec -it cass5 nodetool flush stockbot price_by_symbol

$ docker exec -it cass5 ls /var/lib/cassandra/data/stockbot/price_by_symbol-*

# 3. 用 sstabledump 看 SSTable 內容
$ docker exec -it cass5 \
    sstabledump /var/lib/cassandra/data/stockbot/price_by_symbol-*/nb-*-big-Data.db | head -40

# 4. 製造一些 tombstone
$ docker exec -it cass5 cqlsh -e \
    "DELETE FROM stockbot.price_by_symbol WHERE symbol='NVDA' AND ts > '2026-01-01';"

$ docker exec -it cass5 nodetool tablestats stockbot.price_by_symbol \
    | grep -i tombstone

# 5. 強制 compact (production 不要這樣做!) → 觀察清理
$ docker exec -it cass5 nodetool compact stockbot price_by_symbol
!
nodetool compact（major compaction）在 production 不要用 Major compaction 把所有 SSTable 合併成一個巨大檔案。短期內讀取變快，但之後新進來的資料無法跟這個大檔合併，磁碟使用率失衡。除非你用 STCS 且很清楚自己在做什麼。
