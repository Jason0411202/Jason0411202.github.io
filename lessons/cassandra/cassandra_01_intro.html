Cassandra 是什麼、何時用

01 — 起點

為什麼會出現 Cassandra：關聯式 DB 的水平擴展痛苦

關聯式資料庫（MySQL、PostgreSQL）為單台或主從架構而生。一旦資料量、寫入量超過一台機器的負荷，常見三條路：

  • 讀寫分離：擴讀，但寫入仍卡在 primary。
  • 應用層分片（sharding）：要自己處理路由、跨片 join、再平衡，幾乎所有大公司都踩過這個雷。
  • 升級單機（vertical scaling）：換更大的機器，總有一天還是會撞牆。

Cassandra 的設計目標是直接拿掉這個痛點：把分片、複寫、failover 全部內建，加機器即擴容。它出生於 Facebook（2008，給 Inbox Search 用），後來捐給 Apache，最初的論文受 Amazon Dynamo + Google Bigtable 兩篇影響。

i
一句話定位 Cassandra = 線性擴展的分散式寬欄式資料庫，犧牲一部分一致性（AP 型），換到「每加一台機器，吞吐就接近線性增加」。
02 — 整體定位

三個關鍵字：寬欄式、AP、無 master

圖 C1.1 — Cassandra 一次記住三個關鍵字：寬欄式、AP、無 master

這三個身分互相依賴：因為無 master，所以每個寫入要在多個節點被同時接受；因為要在多節點同時接受，所以選了 AP（網路分區時優先可用）；因為要快速 append-only 寫入大量欄位，所以選了 寬欄式 儲存模型。

03 — 資料模型

什麼是「寬欄式」

關聯式資料庫的 row 結構是固定的：你 CREATE TABLE 定義 5 欄，每一筆資料都有這 5 欄。寬欄式不一樣 — 同一張 table 裡，每筆資料（叫 partition 或 row）可以有不同數量的 column，而且一個 partition 內可放數百萬個 column 值。

更精準地說，Cassandra 的儲存模型是一個多層 map：

概念 
map<
  PartitionKey,         // 決定資料放哪台機器
  map<
    ClusteringKey,      // partition 內排序
    map<ColumnName, Value>
  >
>

舉個 stockbot 的例子：「依股票代碼查歷史報價」，partition key 就是 symbol（讓同一支股票的所有報價都在同一個節點），clustering key 是 ts DESC（時間倒序排），每個 timestamp 下放 price/volume/bid/ask 等欄位。下一堂課（C-02 / C-03）會深入。

!
不是 column-oriented，是 wide-column 很多人誤把 Cassandra 跟 ClickHouse、Redshift 這類「欄式儲存」混為一談。欄式（columnar）是把同一欄的所有值連續儲存來壓縮、加速 OLAP 掃描；寬欄式（wide-column）是 row 內部 column 數量可動態增長。Cassandra 是後者，內部其實仍以 partition 為單位連續寫入。
04 — CAP 簡介

CAP 定理：分散式系統的「鐵三角」

CAP 是 Eric Brewer 2000 年提出、Seth Gilbert & Nancy Lynch 2002 證明的定理：在一個分散式系統中，當網路分區（Partition）發生時，你只能在一致性（Consistency）與可用性（Availability）之間二選一。

類型選擇代表白話解釋
CP 一致性 + 分區容忍 HBase、ZooKeeper、etcd、MongoDB（預設） 分區時拒絕服務，保證資料對齊
AP 可用性 + 分區容忍 Cassandra、DynamoDB、Riak 分區時繼續讀寫，事後修補（最終一致）
CA 一致性 + 可用性 單機 DB（不算分散式） 實務上不可達；網路分區一定會發生
!
「Cassandra 是 AP」其實要打折 Cassandra 提供 可調的一致性（tunable consistency）：寫入 / 讀取時可指定 CONSISTENCY ONE / QUORUM / ALL ...。當你選 QUORUM，公式 R + W > RF 成立時，就能拿到強一致。所以 Cassandra 是「預設 AP、可選 CP」。下一堂 C-02 會展開講。

06 — 適用判斷

什麼時候該用 Cassandra？什麼時候別用？

✓適合
  • 時序資料：股價、IoT sensor、log、metric。partition by entity，clustering by time。
  • 寫入量遠大於讀取：寫入是 append-only，所以 Cassandra 寫起來特別快。
  • 跨地理多 DC：原生支援多 datacenter 複寫，且每個 DC 可獨立讀寫。
  • 需要線性擴展：上線後預期資料量、吞吐持續成長，不希望未來改架構。
  • 單一 entity 為主的查詢：例如「給我這個用戶的所有訂單」、「給我這個 device 的所有事件」。
✗不適合
  • 需要複雜 join / 動態查詢：CQL 不支援 join，schema 要為查詢設計。
  • 強交易（ACID）：Cassandra 有 LWT（Lightweight Transactions）但代價昂貴，不要當主交易系統。
  • OLAP / ad-hoc 分析：請用 ClickHouse、BigQuery、Snowflake；Cassandra 不是分析引擎。
  • 資料量很小（單機塞得下）：殺雞用牛刀，PostgreSQL 更省事。
  • 頻繁更新同一筆 row：每次更新都會留 tombstone，後面 C-04 會講。
07 — 動手玩

5 分鐘起一個單節點 Cassandra

Cassandra 5.0 起官方 Docker image 直接可用。先在本機開兩個終端視窗：

終端 ① 起 Cassandra 節點

bash 
$ docker run --rm --name cass5 \
    -p 9042:9042 \
    -e CASSANDRA_CLUSTER_NAME="DemoCluster" \
    cassandra:5.0

# 等到 log 出現 "Startup complete" 再開下一個視窗
# 第一次啟動需要 30~60 秒（Java 進程暖機 + schema 初始化）
i
記憶體建議至少 4GB Cassandra 是 JVM 進程，預設 heap 約 1/4 系統記憶體。Docker Desktop 的 VM 若只給 2GB 會頻繁 OOM。建議至少 4GB。

終端 ② 用 cqlsh 連進去

bash 
$ docker exec -it cass5 cqlsh

Connected to DemoCluster at 127.0.0.1:9042
[cqlsh 6.2.0 | Cassandra 5.0.8 | CQL spec 3.4.7 | Native protocol v5]
Use HELP for help.
cqlsh> DESCRIBE KEYSPACES;

system_auth     system_traces       system        system_distributed
system_schema   system_virtual_schema

看到這幾個 system_* keyspace 就代表叢集已就緒。「keyspace」是 Cassandra 的最上層命名空間，跟 PostgreSQL 的「database」類似。

建立第一個 keyspace + table

cql 
-- 建立 keyspace（單節點先用 SimpleStrategy / RF=1）
CREATE KEYSPACE stockbot
  WITH replication = {
    'class': 'SimpleStrategy',
    'replication_factor': 1
  };

USE stockbot;

-- 一張時序表：依 symbol 分 partition、依 ts 倒序排
CREATE TABLE price_by_symbol (
  symbol text,
  ts     timestamp,
  price  decimal,
  volume bigint,
  PRIMARY KEY ((symbol), ts)
) WITH CLUSTERING ORDER BY (ts DESC);

-- 灌幾筆假資料
INSERT INTO price_by_symbol (symbol, ts, price, volume)
  VALUES ('TSLA', '2026-04-27 09:30', 285.40, 12500);
INSERT INTO price_by_symbol (symbol, ts, price, volume)
  VALUES ('TSLA', '2026-04-27 09:31', 285.62, 9800);

-- 查最新兩筆
SELECT ts, price, volume
  FROM price_by_symbol
  WHERE symbol = 'TSLA'
  LIMIT 2;

看到資料就成功了！這個 schema 之後在 C-03 會深入解釋為什麼這樣設計。

!
SimpleStrategy 只能用在單節點 demo 正式環境一定要用 NetworkTopologyStrategy（指定每個 DC 的 RF），這個下一堂會說明。SimpleStrategy 在多 DC 部署會出怪事。
