Cassandra 05 · 5.0 新特性

Cassandra 5.0 的六個重點升級

SAI 把索引玩法翻新；Vector Search 讓 Cassandra 直接能跑 RAG / 語意搜尋；Trie 結構大幅省 RAM/磁碟；Dynamic Data Masking 處理 PII；UCS 統一 compaction；單節點容量 4TB → 10TB+。

01 — SAI

Storage-Attached Indexes (SAI)：索引終於好用了

C-03 提過：傳統 secondary index (2i) 在大叢集要 scatter-gather，效能差。SAI 是 5.0 最受矚目的新索引機制：

  • 與儲存引擎深度整合：每個 SSTable 自帶對應的 SAI 檔案，跟資料一起 flush / compact / 刪除。
  • 支援多種型態：text（含 case-insensitive）、numeric range、boolean、geo、UUID、collection（list/set/map）— 一個機制全包。
  • 查詢時可組合多 SAI：WHERE a=? AND b>? AND c IN (...) 可同時用多個 SAI。
  • 記憶體效率高：每個 SAI 約 1% 表 size 的開銷，遠優於 2i。
cql 
-- 建立 SAI (注意 USING 'sai')
CREATE INDEX price_idx
  ON stockbot.price_by_symbol (price)
  USING 'sai';

-- 現在這個 query 可行了 (沒 SAI 之前要 ALLOW FILTERING)
SELECT * FROM stockbot.price_by_symbol
  WHERE symbol = 'TSLA'
    AND price > 280
    AND ts >= '2026-04-01';
-- 要在 partition 內過濾 price 範圍 → SAI 會幫忙

-- 跨 partition 的 SAI 查詢也可用 (不過是 scatter-gather)
SELECT * FROM stockbot.price_by_symbol
  WHERE price > 500;
i
SAI 是 5.0 的「預設 secondary index」 官方文件已經把 SAI 列為主要索引。新 schema 一律用 USING 'sai'；舊的 2i 可逐步遷移（先建 SAI，等驗證後 drop 舊的）。
02 — Vector Search

Vector Search：Cassandra 直接做 RAG

5.0 引入 vector<float, N> CQL 型態 + ANN 查詢。底層用 Lucene 的 HNSW（Hierarchical Navigable Small World）演算法 — 跟 Pinecone / Weaviate / pgvector 用的是相似的家族。

cql 
-- 建一張帶 embedding 的表 (例如 OpenAI text-embedding-3-small 是 1536 維)
CREATE TABLE stockbot.docs (
  doc_id    uuid PRIMARY KEY,
  title     text,
  body      text,
  embedding vector<float, 1536>
);

-- 建 SAI on the vector column (HNSW index)
CREATE INDEX docs_emb_idx
  ON stockbot.docs (embedding)
  USING 'sai'
  WITH OPTIONS = {'similarity_function': 'COSINE'};

-- 灌幾筆 (應用層算好 embedding 後寫進來)
INSERT INTO stockbot.docs (doc_id, title, body, embedding)
VALUES (uuid(), 'Q1 earnings', '...', [0.012, -0.034, ...]);

-- 查詢:取「最像 query embedding」的前 5 筆
SELECT doc_id, title
  FROM stockbot.docs
  ORDER BY embedding ANN OF [0.022, -0.018, ...]
  LIMIT 5;

關鍵語法重點

  • ORDER BY ... ANN OF [...]：注意，是 ORDER BY 不是 WHERE。因為這是「給我最相似的」、不是「給我等於某值的」。
  • 必須加 LIMIT N：ANN 是「取前 N 個近似最近鄰」，不是「找全部相似的」。
  • similarity_function：COSINE（語意相似最常用）、DOT_PRODUCT（已 normalize 的向量）、EUCLIDEAN。
  • 結果是近似，不是精確。HNSW 用犧牲一點召回率換速度。
i
Cassandra Vector vs 專業向量庫 專業 vector DB（Pinecone、Weaviate）有更多索引調參、混合搜尋、metadata 過濾的進階功能。但 Cassandra Vector 的優勢是已經跟你既有的 transactional 資料同庫 — 不用再多一個元件、不用 ETL。中等規模的 RAG / 推薦系統很適合。
03 — Trie Memtable

Trie Memtable / SSTable：省 RAM 與磁碟

5.0 把記憶體中 memtable 與部分 SSTable 結構從 B-tree 改成 Trie（前綴樹）。對 Cassandra 常見的 key 模式（partition key + clustering key 字串）有顯著好處：

  • 記憶體用量降 1/3 ~ 1/2（看 key 共同前綴比例）。
  • 讀寫效能持平或略升（CPU cache 更友善）。
  • 對「同一 partition 內大量 clustering key」場景特別有效（因為它們共用同一 partition key 前綴）。

使用者通常不用做什麼 — Cassandra 5.0 預設啟用。可在 cassandra.yaml 用 memtable.configurations 進階調，但預設值已合理。

04 — Dynamic Data Masking

Dynamic Data Masking：欄位級 PII 保護

5.0 新增的功能：在 schema 上指定某欄位「給沒有解密權限的角色看時，自動遮罩」。原始資料不變，只是查詢時依據角色決定回什麼。

cql 
-- 建表時對欄位指定 mask
CREATE TABLE users (
  id    uuid PRIMARY KEY,
  name  text,
  email text MASKED WITH mask_inner(2, 2),
  ssn   text MASKED WITH mask_default()
);

-- 賦予 admin 解碼權限
GRANT UNMASK ON users TO admin_role;

-- 沒 UNMASK 權限的 role 看到:
-- email: ja****on@gmail.com
-- ssn:   ****

-- 有 UNMASK 權限的 role 看到原始值

內建 mask 函式

  • mask_default()：全部變星號
  • mask_inner(prefix, suffix)：保留前/後幾字
  • mask_outer(start, end)：保留中間
  • mask_hash(algo)：hash 後輸出
  • mask_replace(value)：替換成固定字串

這個功能本來只能在 application 層做、容易漏，現在能在 schema 層強制 — GDPR / HIPAA 場景非常有用。

05 — UCS

Unified Compaction Strategy：一個策略統一全部

C-04 提過：以前要在 STCS / LCS / TWCS 之間挑，每種有自己甜蜜點，運維痛點是「workload 一變就要換」。

UCS（5.0 新增）的目標是一個策略動態適應：

  • 內部用 scaling parameter (T) 控制行為 — 高 T → 接近 STCS；低 T → 接近 LCS。
  • 支援 sharded compaction：把一個 SSTable 切成多 shard 並行 compact，加速大檔處理。
  • 原生支援 TTL hints（取代 TWCS 的功能）。
cql 
CREATE TABLE events (...)
  WITH compaction = {
    'class': 'UnifiedCompactionStrategy',
    'scaling_parameters': 'T4'   -- 預設,中庸
  };

如果你升級舊叢集到 5.0，舊表沿用原 strategy，新建表才預設 UCS。可以用 ALTER TABLE ... WITH compaction = {...} 切換。

06 — 10TB

單節點容量：4TB → 10TB+

5.0 之前社群多半建議單節點資料量 <= 4TB（超過後 streaming repair 與 compaction 變慢、節點重建太久）。5.0 透過：

  • UCS sharded compaction → 大表 compact 變並行
  • Trie memtable → 同記憶體可裝更多 key
  • Streaming 改進（zerocopy）

讓單節點推上 10TB+ 仍可健康運作。對叢集規模規劃影響很大 — 同樣 100TB 資料以前可能要 25+ 節點，現在 10 節點就夠。

!
不過更大不見得更好 雖然 10TB 可行，但節點越大失敗時的「重建時間」越長、blast radius 越大。建議實測自家 workload 並採折衷（例如 4~6TB / node）。「推上 10TB」是 5.0 提供的選項，不是強制的目標。
07 — 動手玩

跑一次 SAI + 假 vector

cql 
$ docker exec -it cass5 cqlsh
cqlsh> USE stockbot;

-- 1. 在現有 price_by_symbol 加 SAI
cqlsh:stockbot> CREATE INDEX price_sai
  ON price_by_symbol (price) USING 'sai';

cqlsh:stockbot> SELECT * FROM price_by_symbol
  WHERE symbol='TSLA' AND price > 285;

-- 2. 建 vector 表 + 填假資料
cqlsh:stockbot> CREATE TABLE news (
  id uuid PRIMARY KEY,
  title text,
  emb vector<float, 3>   -- 3 維方便手寫
);

cqlsh:stockbot> CREATE INDEX ann_idx ON news (emb)
  USING 'sai'
  WITH OPTIONS = {'similarity_function': 'COSINE'};

cqlsh:stockbot> INSERT INTO news VALUES (uuid(), 'TSLA up', [0.9, 0.1, 0.0]);
cqlsh:stockbot> INSERT INTO news VALUES (uuid(), 'AAPL up', [0.85, 0.05, 0.1]);
cqlsh:stockbot> INSERT INTO news VALUES (uuid(), 'Macro inflation', [0.1, 0.9, 0.0]);

-- 3. ANN 查詢:接近「股票漲」(0.88, 0.08, 0.05)
cqlsh:stockbot> SELECT title FROM news
  ORDER BY emb ANN OF [0.88, 0.08, 0.05]
  LIMIT 2;
-- 預期:TSLA up, AAPL up 排前面

-- 4. 試 Dynamic Data Masking (5.0 預設要打開 mask_inner、mask_default 等函式)
cqlsh:stockbot> CREATE TABLE users_pii (
  id    uuid PRIMARY KEY,
  email text MASKED WITH mask_inner(2, 2)
);
cqlsh:stockbot> INSERT INTO users_pii (id, email)
  VALUES (uuid(), 'jason@example.com');
cqlsh:stockbot> SELECT email FROM users_pii;
-- ja*************om   (除非 GRANT UNMASK)
