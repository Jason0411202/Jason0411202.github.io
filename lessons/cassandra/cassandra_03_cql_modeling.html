01 — CQL vs SQL

看起來像 SQL，骨子裡是 KV

CQL（Cassandra Query Language）的語法是刻意設計成像 SQL 的，目的是降低學習門檻。但骨子裡 Cassandra 是個分散式 KV 系統，CQL 把它包裝成 row/column 介面。最常見的誤會：

SQL 能做但 CQL 不能做原因
JOIN 兩張表 跨 partition 跨節點 join 會變分散式 query，違反 Cassandra 設計
沒指定 partition key 的 WHERE 等於要掃整個叢集；強行用會被 ALLOW FILTERING 警告
GROUP BY + 聚合（除非同 partition） aggregation 跨 partition 不可行；要做請用 Spark
ORDER BY 任意欄位 只能依 clustering key 排序；其他欄位不行
外鍵 (FK) / 真正的 transaction 有 LWT 但代價高，不是設計目標
!
看到 ALLOW FILTERING 就要警覺 這是 Cassandra 給你「我知道你要做掃全表的危險查詢」的明示開關。除了管理工具，幾乎不該出現在生產 query 裡。
02 — Primary Key

核心心智模型：PK = Partition Key + Clustering Key

cql 
CREATE TABLE price_by_symbol (
  symbol     text,
  ts         timestamp,
  price      decimal,
  volume     bigint,
  PRIMARY KEY ((symbol), ts)
  --           ↑          ↑
  --    partition key   clustering key
);
  • Partition key：決定這筆資料放在哪台機器（用上一堂提的 Murmur3 hash）。同一個 partition key 的所有資料一定在同一台節點。
  • Clustering key：partition 內部的排序依據。同一 partition 內的資料按 clustering key 連續儲存。

複合 partition key

有時單欄當 partition key 容易產生 hot partition（例如全部資料 partition key 都是 "prod"）。可以用 複合 partition key 把流量分散：

cql 
CREATE TABLE events_by_day (
  day        date,
  bucket     int,        -- 0~9 隨機分桶
  ts         timestamp,
  payload    text,
  PRIMARY KEY ((day, bucket), ts)
  --           ↑               ↑
  --   複合 partition key   clustering key
);

同一天的資料原本只會在一個 partition、可能塞爆。加上 bucket 後，同一天的資料分到 10 個 partition、平均到不同節點。但 query 時要記得「day + bucket」一起給。

03 — Wide Row

同一 partition 內的資料是怎麼排列的

圖 C3.1 — 「TSLA」的所有報價在同一個 partition 內依 ts 連續儲存

因為 partition 內連續儲存，範圍查詢非常便宜：

cql 
-- 只到 1 台節點，partition 內 sequential scan,飛快
SELECT * FROM price_by_symbol
  WHERE symbol = 'TSLA'
    AND ts >= '2026-04-27 09:30'
    AND ts <  '2026-04-27 16:00';

-- 但這個爆炸 — 沒指定 partition key
SELECT * FROM price_by_symbol
  WHERE ts >= '2026-04-27 09:30';
-- ↑ 沒給 symbol → Cassandra 要掃所有 partition,拒絕
04 — Query-First

不是先設計 schema，是先列查詢

關聯式資料庫的工作流：先 normalize 出 schema → 之後查不到再加 index。Cassandra 反過來：

1
列出所有 query
「給我這支股票最近 100 筆報價」「給我某用戶的所有訂單」「給我這個 device 今天的事件」 — 把所有讀取需求清單列出。
2
為每個 query 設計一張表
每張表的 partition key 對應 query 裡固定的那一欄，clustering key 對應「要排序或範圍取的那一欄」。
3
同一份資料寫多份表
一筆訂單同時寫進 orders_by_user、orders_by_status、orders_by_day ...，這叫「反正規化」，不是壞味道。
4
驗證 partition size 不會炸
每個 partition 估算大小，目標 < 100 MB、< 100 萬個 cell。超過就要切 bucket。
05 — 反正規化

同一筆資料寫多份是常態

場景：訂單系統，常見三個查詢需求：

  1. 給一個用戶看他的所有訂單（最新優先）。
  2. 給後台看「待出貨」狀態的所有訂單。
  3. 給財務看某一天的所有訂單。
cql 
-- 1. 用戶視角
CREATE TABLE orders_by_user (
  user_id    uuid,
  created_at timestamp,
  order_id   uuid,
  status     text,
  amount     decimal,
  PRIMARY KEY ((user_id), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);

-- 2. 後台 status 視角 (用 day + status 當複合 partition key 避免 hot)
CREATE TABLE orders_by_status (
  day        date,
  status     text,
  order_id   uuid,
  user_id    uuid,
  amount     decimal,
  PRIMARY KEY ((day, status), order_id)
);

-- 3. 財務日視角
CREATE TABLE orders_by_day (
  day        date,
  bucket     int,    -- 0~9 分桶
  created_at timestamp,
  order_id   uuid,
  user_id    uuid,
  amount     decimal,
  PRIMARY KEY ((day, bucket), created_at, order_id)
);

建立訂單時，應用層（或一個輕量的 batch / event handler）負責把同一筆資料寫到三張表。是的，這會增加寫入量，但 Cassandra 寫入便宜（C-04 會說明），而且這樣每個 query 都能 O(1) 直達 partition。

i
用 BATCH 把多張表寫入綁在一起 Cassandra 提供 BEGIN BATCH ... APPLY BATCH;（logged batch）保證「全部成功或全部失敗」。但 batch 跨多個 partition 時有效能成本，只在「不能容忍部分寫入」時用。
06 — 索引選擇

除了 PK 還能怎麼查：2i / SAI / Materialized View

機制機制說明建議
Secondary Index (2i) 每個節點各自維護自己 partition 的 local index，查詢時要 scatter-gather 不建議在大叢集用，低基數欄位才勉強
SASI SSTable-Attached Secondary Index 4.x 已 deprecated，不要新用
SAI（5.0 預設選項） Storage-Attached Index，與儲存引擎深度整合，效能與彈性接近 ES 5.0 起的首選（C-05 詳細）
Materialized View Cassandra 自動為你維護「同樣資料但 PK 不同」的另一張表 3.0 就有但實驗性且有 race condition；社群多建議「應用層自己反正規化」

實務上 5.0 的建議路徑：應用層反正規化（一筆寫多表）為主，SAI 為輔。Materialized View 除非你能接受其限制，否則別用。

07 — 反模式

四個保證踩坑的反模式

✗Hot Partition
所有資料用同一 partition key（例如 "global"）。
→ 一台節點扛全部流量，其他節點閒著。
解法：加 bucket、用更分散的 key。
✗Unbounded Partition
partition 持續長大不切（例如「所有事件用同一個 device 當 PK」幾年累積千萬筆）。
→ 單 partition > 100 MB 後讀取明顯變慢。
解法：把時間切進 PK（年/月/週）。
✗無界 IN
WHERE symbol IN (?, ?, ..., 1000 個)。
→ coordinator 要對 1000 個 partition fan-out，多半超時。
解法：應用層拆成 N 條 query 並行。
✗ALLOW FILTERING
SELECT ... WHERE non_pk_col = ? ALLOW FILTERING。
→ 等於告訴 Cassandra「我知道要全表掃」。
解法：新增一張對應 query 的表（反正規化）。
08 — 動手玩

建一個 stockbot 風格的 schema

cql 
-- 接續 C-01 的 stockbot keyspace
USE stockbot;

-- 1. 主表：依股票代碼查歷史報價(已存在)

-- 2. 每天每支股票的最新一筆 (給 dashboard 用)
CREATE TABLE latest_price_by_day (
  day      date,
  symbol   text,
  ts       timestamp,
  price    decimal,
  PRIMARY KEY ((day), symbol)
);

-- 3. 「告警」事件流：依 user 查近期告警
CREATE TABLE alerts_by_user (
  user_id    uuid,
  created_at timestamp,
  alert_id   uuid,
  symbol     text,
  rule       text,
  PRIMARY KEY ((user_id), created_at, alert_id)
) WITH CLUSTERING ORDER BY (created_at DESC, alert_id ASC);

-- 試查
INSERT INTO latest_price_by_day (day, symbol, ts, price)
  VALUES ('2026-04-27', 'TSLA', '2026-04-27 15:59', 288.20);
INSERT INTO latest_price_by_day (day, symbol, ts, price)
  VALUES ('2026-04-27', 'AAPL', '2026-04-27 15:59', 175.80);

SELECT * FROM latest_price_by_day WHERE day='2026-04-27';

練習：嘗試這些 query，預測哪個會成功、哪個會被拒：

  • SELECT * FROM price_by_symbol WHERE symbol='TSLA'; ← 主鍵命中
  • SELECT * FROM price_by_symbol WHERE ts > '2026-04-27'; ← 沒 partition key，Cassandra 拒絕
  • SELECT * FROM price_by_symbol WHERE ts > '2026-04-27' ALLOW FILTERING; ← 加上才允許，但是反模式
  • SELECT * FROM price_by_symbol WHERE symbol='TSLA' AND price > 280; ← 沒 SAI 不行（5.0 可加 SAI 索引讓它變可行，C-05 講）
