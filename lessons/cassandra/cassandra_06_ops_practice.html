01 — nodetool

nodetool 必背 12 個指令

指令看什麼什麼時候用
nodetool status [keyspace]叢集所有節點 UN/DN 狀態、ownership每天看、scale 後
nodetool info本節點 heap、cache、uptime、token 數排查單節點
nodetool ring [keyspace]所有 vnode token 的對應scale / 重平衡前
nodetool tablestats <ks.tbl>表的 SSTable 數、size、tombstone、bloom filter慢查排查
nodetool tpstats所有 thread pool 狀態(active/pending/blocked)看是否有 backup
nodetool compactionstats進行中與排隊的 compaction觀察 compaction debt
nodetool gossipinfo每個節點的 gossip 狀態與版本叢集分裂排查
nodetool getendpoints <ks> <tbl> <pk>某 partition key 在哪些 replicadebug 路由
nodetool flush <ks>把 memtable flush 成 SSTable備份前/觀察
nodetool repair -pr <ks>修復本節點負責的 token range排程每週跑
nodetool drain停止接受新請求,flush memtable關機前
nodetool snapshot <ks>建立 SSTable 硬連結快照備份
i
「健康檢查腳本」三條黃金指令
  1. nodetool status — 看有沒有 DN 節點
  2. nodetool tpstats | grep -E '(Pending|Blocked)' — 看 thread pool 是否塞車
  3. nodetool compactionstats — 看 compaction debt 是否爆炸
排成 cron 每分鐘跑、結果丟 Prometheus / Slack。
02 — Metrics

關鍵 Metrics（JMX / Prometheus）

Cassandra 透過 JMX 暴露幾百個 metrics。這幾個是「無論什麼場景都該監控」：

類別Metric警戒值
讀延遲ClientRequest.Read.Latency p99> 100ms 警告
寫延遲ClientRequest.Write.Latency p99> 50ms 警告
逾時ClientRequest.{Read|Write}.Timeouts任何非零都要查
compactionCompaction.PendingTasks> 100 持續飆 → I/O 跟不上
droppedDroppedMessage.{MUTATION|READ}任何非零都要查
JVMjvm_gc_pause_seconds p99> 1s 警告
JVMjvm_memory_heap_used / max> 75% 警告
磁碟per-node data dir 使用率> 70% 警告 (compact 需要空間)
repair上次成功 repair 距今> gc_grace_seconds 紅燈

用 cassandra_exporter 把 JMX 轉成 Prometheus、配 Grafana 儀表板（社群有現成的 dashboard 1287）。

03 — 備份

Snapshot + sstableloader 是基本盤

Cassandra 的「備份」其實是建立 SSTable 的硬連結快照（因為 SSTable 是 immutable，建立硬連結不佔額外空間，直到壓縮後檔案被刪）。

bash 
# 1. 建快照
$ docker exec -it cass5 nodetool snapshot stockbot -t backup-$(date +%F)
Requested creating snapshot(s) for [stockbot] with snapshot name [backup-2026-04-27]

# 快照位置:每個 SSTable 目錄下的 snapshots/ 子目錄
$ docker exec -it cass5 ls /var/lib/cassandra/data/stockbot/price_by_symbol-*/snapshots/

# 2. 把快照 rsync / s3 sync 到別處
$ docker exec -it cass5 tar czf /tmp/backup.tar.gz \
    /var/lib/cassandra/data/stockbot/*/snapshots/backup-2026-04-27

# 3. 還原:把備份回的 SSTable 用 sstableloader 灌進來
$ docker exec -it cass5 \
    sstableloader -d 127.0.0.1 \
    /var/lib/cassandra/data/stockbot/price_by_symbol-xxx
i
Production 用 Medusa 自動化 Medusa 是 The Last Pickle 開發的備份工具，做了 snapshot + S3/GCS 上傳 + 還原 schema 重建一條龍。production 別自己手刻 backup。
04 — 升級

Rolling Upgrade：一台一台來

Cassandra 同時支援「混合版本叢集」(N 與 N+1 minor version 共存) 給你 rolling upgrade 的機會。SOP：

  1. 看升級路徑：跨 major version 不能直接升（4.x → 5.x 可，3.x → 5.x 要先過 4.x）。看官方 upgrading 文件。
  2. 備份：先 snapshot，最壞情況可還原。
  3. 跑一次完整 repair：升級期間混版本、出 bug 的話資料才不會更亂。
  4. 挑一台：nodetool drain → 關 cassandra → 升級包 → 啟動。
  5. 驗證：nodetool status 確認 UN，nodetool tpstats 看正常運作後再做下一台。
  6. 不要在升級期間做 schema 變更：混版本期間 schema migration 可能出怪事。
  7. 升完所有節點再做 nodetool upgradesstables：把舊版 SSTable 改寫成新格式。
!
不要硬跨版本 看到「我從 3.11 直接升 5.0」這種需求，正確做法是 3.11 → 4.1 → 5.0。每次 minor 都該完整測過再做下一跳。Cassandra 不像 Redis 升級那麼寬容。
05 — 容量

容量規劃 4 條黃金法則

  1. per-node 資料量：5.0 之前 ≤ 4 TB；5.0 起可推到 10 TB+，但實務折衷 4~6 TB。
  2. partition size：< 100 MB / partition、< 100 萬 cell / partition。超過要切 bucket。
  3. 磁碟空間：disk used < 70%，compaction 需要空間（worst case 寫一份新檔再刪舊檔，需要 50%+ 空檔）。
  4. JVM heap：4~16 GB（heap > 32 GB 反而低效因為 compressed oops）。物理機剩下的 RAM 給 page cache。

怎麼估「我需要幾個節點」

假設你估自己每天進 100 GB raw 資料、保留 30 天 → 3 TB raw。

  • RF=3 → 9 TB total。
  • + tombstone / overhead × 1.3 → ~12 TB。
  • 單節點 4 TB → 至少 3 節點，但這樣 disk 會 100% 滿。
  • 實務開 6 節點，每台占 50% disk，留 50% 給 compaction 與成長。
06 — 地雷

讓你變生產事故的 5 個地雷

✗LWT 過量
Lightweight Transactions（IF NOT EXISTS）依靠 Paxos，每筆寫入要 4 次 round-trip。偶爾用沒問題，當主流程用會死。
✗SELECT *
把整個 partition 拉回來,當 partition 變 wide 後變慢。明確指定欄位。
✗跨 partition IN
WHERE pk IN (?, ?, ..., 1000) 等於 fan-out。應用層自己拆並行 query。
✗把 Cassandra 當 Queue
不斷 enqueue + delete → tombstone 暴增,讀取超慢。請用 Kafka / Redis Streams。

第五個：沒設 prepared statement。每筆 query 用字串拼接 → server 每次都要 re-parse。一律用 driver 的 prepared statement API。

07 — Production

實際 production 都用什麼工具

本系列為了上手簡單只跑單節點、純 nodetool。實際上線的話，沒人手動 SSH 管 100 個 Cassandra 節點。常見組合：

  • k8ssandra-operator v1.31+（2026/03 出）：在 Kubernetes 上以 CRD 描述「我要 5 節點 RF=3 的 Cassandra」，operator 處理部署、升級、scale、repair 排程。底層包 cass-operator。
  • Cassandra Reaper：自動排程 incremental repair，避免你忘記跑。
  • Medusa：自動化備份 / 還原。
  • cassandra_exporter + Prometheus + Grafana：監控標配。
  • DataStax Astra DB：完全託管 SaaS（serverless Cassandra）— 不想管基礎設施直接用。
  • AxonOps：Cassandra 商業監控與運維平台。

推薦進階學習

  • 《Cassandra: The Definitive Guide》3rd ed（Carpenter / Hewitt，O'Reilly）— 仍是最完整的 5.0 對應書籍。
  • DataStax Academy — 免費線上課，含資料建模、運維、driver。
  • The Last Pickle 部落格（已被 DataStax 收購但內容仍在）— 大量運維細節、compaction 深入、tombstone 案例。
  • Apache Cassandra Slack / mailing list — 直接問。
i
