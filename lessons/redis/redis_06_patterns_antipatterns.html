01 — Cache 模式

四種 Cache 模式怎麼選

模式讀寫取捨
Cache-Aside (look-aside) app 先查 cache,miss 再查 DB,回填 app 寫 DB → 主動 invalidate cache 最常見;app 邏輯包,簡單可控
Read-Through app 只查 cache,cache miss 時自動回 DB 取 同 Cache-Aside cache 客戶端要支援(如 caffeine);邏輯收斂但綁框架
Write-Through 同 Read-Through app 寫 cache → cache 同步寫 DB 強一致但寫慢
Write-Behind 同 Read-Through app 寫 cache → cache 非同步刷到 DB 寫快;但若 cache 掛資料丟

業界 90% 場景是 Cache-Aside。Read-Through / Write-Through 多見於 application server 內的本地 cache（Caffeine、Guava cache）。Write-Behind 出現於高吞吐但能容忍少量資料丟失的場景（如計數器）。

go — cache-aside 
func GetUser(id string) (*User, error) {
    // 1. 先查 cache
    if raw, err := rdb.Get(ctx, "user:"+id).Bytes(); err == nil {
        return deserialize(raw), nil
    }

    // 2. miss → 查 DB
    user, err := db.QueryUser(id)
    if err != nil { return nil, err }

    // 3. 回填 cache (TTL 加隨機抖動,後面講)
    rdb.Set(ctx, "user:"+id, serialize(user), 300*time.Second + jitter())

    return user, nil
}

func UpdateUser(u *User) error {
    db.UpdateUser(u)              // 寫 DB
    rdb.Del(ctx, "user:"+u.ID)  // 刪 cache,讓下次讀重新回填
    return nil
}
02 — 三大災難

Cache 三大經典事故與解法

Cache Penetration（快取穿透）

問題：query 一個「DB 也沒有」的 key（例如惡意爬蟲試 user:9999999999），cache 永遠 miss → 每次都打 DB。

解法：

  • 把「不存在」這件事也 cache — set 一個短 TTL 的「null marker」（SET user:99 "" EX 60）。
  • 或在 DB 前用 Bloom filter：先判斷「這個 key 是否可能存在」，明顯不存在的就直接拒。

Cache Breakdown（快取擊穿）

問題：一個熱 key 的 TTL 到期了，瞬間幾千個 request 同時 miss、同時打 DB。

解法：

  • 用 distributed lock（後面講）— 第一個 miss 的拿到 lock 才打 DB，其他 wait。
  • 或用 logical expiration：cache 裡寫一個「邏輯過期時間」，到期後仍回舊資料但非同步更新。

Cache Avalanche（快取雪崩）

問題：大量 cache 在同一秒同時過期（例如服務重啟後同時寫入），瞬間 DB 壓力倍增。

解法：

  • TTL 加隨機抖動：原本 5 分鐘 TTL，改成 5 分鐘 ± 30 秒 random。
  • 多級 cache：local (in-process) + Redis 兩層。
  • Redis 自己 HA（Sentinel / Cluster）— 避免 Redis 整個掛。
i
三招組合拳
  1. 所有 cache TTL 帶 ±10% 隨機抖動 — 防雪崩
  2. 未命中時用 SET NX 做 mutex - 防擊穿
  3. 不存在也 cache 短期 null - 防穿透
03 — 分散式 Lock

分散式 Lock 的真相（Redlock 爭議）

單機簡單版：SET NX PX 即可

redis-cli 
# 拿 lock — NX 表示「只有不存在時才寫」,PX 30000 表示 30 秒過期
SET lock:order:1 "random-uuid-abc" NX PX 30000
OK   # 拿到
(nil) # 沒拿到

# 釋放 lock - 一定要用 Lua 確認 value 才刪
# (避免別人 lock 過期重新拿,你刪到別人的)
EVAL "if redis.call('GET', KEYS[1]) == ARGV[1] then
        return redis.call('DEL', KEYS[1])
      else return 0 end" 1 lock:order:1 "random-uuid-abc"

這個方案對 99% 場景夠用。重點：

  • value 必須隨機 — 釋放時驗證才不會誤刪別人的 lock。
  • 必須有 PX 過期 — 沒過期 + client 崩潰 = lock 永久卡住。
  • 釋放用 Lua 原子比對 + DEL — 兩步分開有 race。

多機 Redlock：Antirez vs Kleppmann

Antirez（Redis 創造者）為了「單機 Redis 掛掉時仍能拿 lock」設計了 Redlock：用 N（建議 5）台獨立的 Redis instance，client 在多數派（3/5）拿到 lock 才算成功。

Martin Kleppmann（《DDIA》作者）2016 寫了著名的 「How to do distributed locking」，論點：

  1. Redlock 依賴各機器時鐘同步，但時鐘漂移在實務上不可控。
  2. 長時間 GC pause / 網路分區可能讓 client A 以為自己還持有 lock，B 已經拿到 — 違反互斥。
  3. 解法是用 fencing token（單調遞增 ID），下游服務驗證 token 大小決定是否接受。

Antirez 寫了 反駁文：時鐘的擔憂被誇大、實務上 Redlock + 合理的 lock 持有時間夠用。

!
實務怎麼做
  • 大部分場景：單機 SET NX PX 就夠。Redis 配 Sentinel/Cluster 已經有 HA。
  • 真的需要強互斥（金融轉帳、絕不能重複扣款）：用 fencing token + 後端驗證，而不是依賴 Redlock。
  • 選用 Redlock 前先看：你的下游服務有沒有辦法驗 token？沒辦法驗的話，多 Redis 也救不了。
04 — Rate Limiter

三種 Rate Limiter 與 Sliding Window 實作

策略原理取捨
固定窗 (Fixed Window) INCR rate:user:1:202604271500 實作簡單;邊界突刺(15:00:01 與 15:00:59 各 100 = 100 內 200)
Token Bucket 每秒補 N 個 token,超量拒 支援突發,計算稍複雜
Sliding Window(用 Sorted Set) 記每筆 request 時間戳到 ZSET,查窗口內筆數 準確、無邊界突刺、用 ZSET

Sliding Window 實作（最推薦）

Lua — 限制 user 1 分鐘內 100 次 
-- KEYS[1]: 該 user 的限流 key
-- ARGV[1]: 當下時間 (ms)
-- ARGV[2]: 窗口大小 (ms,如 60000)
-- ARGV[3]: 限制次數 (如 100)
-- ARGV[4]: 本次的 unique id

local now    = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit  = tonumber(ARGV[3])

-- 1. 移除窗口外的舊紀錄
redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, now - window)

-- 2. 算窗口內幾筆
local count = redis.call('ZCARD', KEYS[1])
if count >= limit then
  return 0   -- 拒
end

-- 3. 加入這次紀錄
redis.call('ZADD', KEYS[1], now, ARGV[4])
redis.call('PEXPIRE', KEYS[1], window)
return 1

這段 Lua 在 Redis 裡是單緒原子，沒有 race。一條 EVAL 完成「清舊 + 計算 + 寫入」三件事。

05 — 反模式

四個生產上保證踩坑的反模式

✗KEYS *
掃整庫、O(N) 阻塞單緒。1 千萬 key 跑 10 秒 → 所有 client 卡 10 秒。
替代: SCAN 0 MATCH pattern COUNT 100 分批游標掃。
✗Big Key
單一 hash 上 GB,LIST 上百萬筆。讀寫 / 複寫一次都吃光網路與 CPU。
檢測: redis-cli --bigkeys
替代: 拆成多 key (例如 hash by hash tag),或用 sorted set + 分頁。
✗Hot Key
單一 key (如 config:global) 被一台 client 每秒查 100 萬次。
檢測: redis-cli --hotkeys
替代: 應用層 in-process cache、或拆 N 個 replica 加 read scaling、或 key shard 成 N 份隨機讀。
✗當主資料庫
把 Redis 當 PostgreSQL 替代。
記憶體成本貴、強一致弱、ad-hoc 查能力差。Redis 是 cache + 資料結構服務,不是 source of truth。

第五個常見地雷：用 Redis 當訊息佇列但用 Pub/Sub。Pub/Sub 沒持久化、subscriber 離線即丟訊息。需要可靠佇列請用 Streams（R-02）。

必看的 9 個指標

類別Metric警戒值
命中率keyspace_hits / (hits + misses)< 80% 要查（cache 設計問題）
記憶體used_memory / maxmemory> 80% 警告
記憶體碎片used_memory_rss / used_memory> 1.5 → 碎片高,考慮 ACTIVE DEFRAG

阻塞blocked_clients持續 > 0 → BLPOP/WAIT 在等

07 — Production

實際 production 都用什麼工具

本系列為了上手簡單只跑單機 / docker-compose。實際上線常見組合：

  • Redis Operator (OT-CONTAINER-KIT)：開源、活躍維護的 K8s operator，CRD 描述 standalone / cluster / sentinel / replication 模式，含 redis-exporter for Prometheus。
  • Redis Enterprise for Kubernetes：Redis 公司商業版 K8s operator，多 DC + active-active + 模組管理。
  • redis_exporter + Prometheus + Grafana：監控標配。社群 dashboard 763 / 11835 都好用。
  • RedisInsight：官方 GUI，看資料、查 slowlog、profiling。
  • Cloud managed：AWS ElastiCache、GCP Memorystore、Azure Cache for Redis、Upstash、Redis Cloud — 不想管基礎設施直接用。
  • Valkey：Linux Foundation fork，BSD 授權。雲端 OSS 替代選項。

推薦進階學習

  • 《Redis in Action》（Josiah L. Carlson，Manning）— 雖為 Redis 2.x 寫，但模式部分仍經典。
  • Antirez 部落格（antirez.com）— Redis 創造者親寫的設計理由。
  • 《Designing Data-Intensive Applications》Kleppmann — 分散式系統聖經，第 8、9 章與 Redlock 爭議直接相關。
  • Redis 官方 docs — 持續更新、品質高。
