Redis 03 · 持久化與複製

記憶體 ≠ 沒持久化

RDB / AOF / Hybrid 三種持久化模式各自的取捨；主從複製怎麼追、為什麼非同步複製在 failover 可能掉資料。本堂結束你能正確設定一台「可重啟、有 replica」的 Redis。

01 — 為何

記憶體型 ≠ 不能持久化

很多人對 Redis 的第一印象是「資料放記憶體，重啟就沒了」。這只說對一半 — Redis 從第一天就支援把記憶體 snapshot 到磁碟，重啟後可從磁碟還原。但持久化策略要怎麼選，會直接影響「重啟可能掉幾秒資料」。

三個方案：

  • RDB（snapshot）：定期把整個 dataset dump 成二進位檔。
  • AOF（append-only file）：記錄每一條寫入指令到 log。
  • Hybrid（8.x 預設）：AOF 但 rewrite 時把現存資料寫成 RDB 格式 + 之後的指令繼續 append。
02 — RDB

RDB：定期拍照

RDB 在指定條件下用 fork() 開子行程，子行程把整個 dataset 寫到 dump.rdb。父行程繼續服務。

redis.conf 
# 「N 秒內超過 M 次寫入就觸發 dump」
save 3600 1       # 1 小時內有任何寫
save 300 100      # 5 分鐘內 >= 100 次寫
save 60 10000     # 1 分鐘內 >= 10000 次寫

dbfilename dump.rdb
dir /data

優點

  • 檔案小：是 binary snapshot，比 AOF log 小很多。
  • 還原快：重啟時直接 mmap 整檔。
  • fork 子行程做事，不阻塞主行程指令處理。

缺點

  • 掉資料風險高：兩次 snapshot 之間的資料若崩潰會全失。預設配置最差可能掉幾分鐘。
  • fork() 在大 dataset 時很慢：fork 要複製記憶體頁表（COW，copy-on-write）。Redis 用 50GB 時 fork 可能花 1~3 秒，在這期間 server 仍能服務但延遲毛刺。
!
用 Linux Transparent Huge Pages (THP) 會放大 fork 痛點 THP 會把記憶體頁變成 2 MB 大頁，fork COW 觸發時複製成本暴增。Redis 啟動會警告 — production 一定要關 THP（echo never > /sys/kernel/mm/transparent_hugepage/enabled）。
03 — AOF

AOF：記下每條寫指令

AOF（Append-Only File）在每條寫入指令執行後，把它附加到一個 log 檔。重啟時 replay 整個 log 就還原了。

redis.conf 
appendonly yes
appendfilename "appendonly.aof"

# fsync 三選一
appendfsync always      # 每條寫都 fsync ← 最安全 / 慢
appendfsync everysec    # 每秒 fsync ← 預設 / 最差掉 1 秒
appendfsync no          # 由 OS 決定 ← 最快 / 不安全

auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

fsync 三檔位的真實意義

策略寫入吞吐崩潰最多掉建議
always低（每條 syscall）0 條金融類交易
everysec高~ 1 秒業界 99% 場景
no最高幾十秒（看 OS）純 cache 不在意

AOF Rewrite

AOF log 越積越大會變慢。Redis 會在 log 達到一定 size 時自動觸發 rewrite：fork 子行程，把當下記憶體狀態用「最少指令」重新寫成新的 AOF。例如 INCR x 100 次原本 100 行，rewrite 後變成 SET x 100 一行。

04 — Hybrid

Hybrid（RDB + AOF）：8.x 預設組合

4.0 引入、5.0+ 預設打開：aof-use-rdb-preamble yes。意思是：

  • AOF rewrite 時，把當下記憶體 dump 成 RDB 格式寫到 AOF 開頭。
  • 之後新進的寫入指令仍以 AOF 文字格式 append 在後面。

還原時，Redis 先讀 RDB 部分（快），再 replay 後段 AOF（覆蓋最近幾秒）。結合 RDB 的「載入快」和 AOF 的「掉資料少」。

i
業界推薦組合 appendonly yes + appendfsync everysec + aof-use-rdb-preamble yes（8.x 預設）。另外仍建議保留 RDB save 策略當二級備份（每天自動 snapshot 並備份到 S3 / NFS）。
05 — 複製

主從複製：replicaof & psync

Redis 是主從非同步複製架構。replica 連到 master 後：

1
REPLICAOF master:port
replica 主動連到 master，告知自己的 replication ID 和 offset。
2
Full resync 或 Partial resync
如果 master 還記得 replica 之前的 offset（在 repl_backlog 環形 buffer 裡）→ partial 補差量。否則 master fork 出 RDB 整份送過去（耗網路）。
3
持續串流寫入
之後 master 每寫一條，就 push 到所有 replica。replica 不必 ACK 即可繼續 — 非同步。
4
replica 可被讀
設 replica-read-only yes（預設）後，replica 可服務讀取，提供 read scaling。但讀的可能是舊資料（複製延遲）。

repl_backlog：partial resync 的關鍵

master 維護一個環形 buffer（repl-backlog-size 預設 1 MB），記錄最近的寫入。replica 短暫斷線重連，只要它的 offset 還在這個 buffer 內，master 就只補差量、不 full resync。大流量場景請把這個值調大（例如 256 MB），免得每次 replica 重連都觸發 full resync。

06 — 陷阱

「非同步複製」= failover 可能掉資料

因為複製是非同步的，這個情境完全可能：

圖 R3.1 — async 複製的根本特性：master 已回應 client 但還沒同步到 replica

所以「主從複製 ≠ 強一致」。Redis 沒提供同步複製選項，原因之一就是同步複製會把延遲拖到不可接受的程度。實務做法：

  • 用 WAIT n timeout 指令：等到「至少 n 個 replica 確認接收」再回 client。可大幅降低（不能完全消除）掉資料風險。
  • 讓 replica 也開 AOF + everysec：即使 replica 重啟也能還原。
  • 關鍵資料雙寫：超關鍵的資料同時寫 PostgreSQL，Redis 只當 cache。
!
「強一致 + 高可用 + 高吞吐」三者選兩個 這就是 CAP / PACELC 的回音：Redis 預設組合是 (高可用 + 高吞吐)，犧牲了強一致。如果你的場景受不了任何資料丟失，請別把 Redis 當主資料庫，配合可靠的後端（PostgreSQL / Cassandra）做雙寫或 cache-aside。
07 — 動手玩

用 docker-compose 起 master + 2 replica

yaml — docker-compose.yml 
services:
  master:
    image: redis:8.6
    command: redis-server --appendonly yes --aof-use-rdb-preamble yes
    ports: ["6379:6379"]

  replica1:
    image: redis:8.6
    command: redis-server --appendonly yes --replicaof master 6379
    depends_on: [master]
    ports: ["6380:6379"]

  replica2:
    image: redis:8.6
    command: redis-server --appendonly yes --replicaof master 6379
    depends_on: [master]
    ports: ["6381:6379"]
bash 
$ docker compose up -d

# 看 master 狀態
$ redis-cli -p 6379 INFO replication
# role:master
# connected_slaves:2
# slave0:ip=...,port=6379,state=online,offset=...
# slave1:ip=...,port=6379,state=online,offset=...
# repl_backlog_size:1048576

$ redis-cli -p 6380 INFO replication
# role:slave
# master_link_status:up
# master_repl_offset:42
# master_last_io_seconds_ago:1

# 寫到 master
$ redis-cli -p 6379 SET hello world

# 從 replica 讀（read scaling）
$ redis-cli -p 6380 GET hello
"world"

# 試「同步保證」: 寫完等至少 1 個 replica 收到，最多等 100ms
$ redis-cli -p 6379 SET key1 val1
$ redis-cli -p 6379 WAIT 1 100
(integer) 1   # 1 個 replica 已確認

挑戰：殺掉 master 之前，先在 client 跑一個迴圈寫 key1, key2, ... 然後在某時間點 docker stop master。觀察 replica 上少了哪些 key。這就是非同步複製的本質。
