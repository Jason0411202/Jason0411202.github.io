Sentinel vs Cluster:解決什麼問題

兩個常被混為一談的東西：Sentinel 解決可用性(自動 failover)；Cluster 解決容量/吞吐(水平分片)。本堂讓你能正確選擇。

01 — 切清楚

兩個完全不同的問題

新手最常搞混的：

  • 「我的 master 掛了想要自動切到 replica」 → Sentinel 的問題（HA / 高可用）
  • 「我的資料量超過一台機器的記憶體」 → Cluster 的問題（水平擴展）
面向SentinelCluster
解決HA / 自動 failover水平擴展
資料份數1 份（master + 多個 replica 都是同一份資料）分片（每個 shard 持有 1/N 資料）
擴展方式垂直（更大機器）水平（加 shard）
客戶端一般 client（透過 Sentinel 找 master）cluster-aware client（懂 MOVED/ASK）
運維複雜度中高
適合資料量< 一台機器記憶體超過、或要分流
i
兩者其實可以組合 Cluster 模式內，每個 shard 自己就是「1 master + N replica」+ 內建 failover（自動 sentinel-like 行為）。所以實際上很少人「Sentinel + Cluster 一起用」 — 直接用 Cluster 就涵蓋了 HA。
02 — Sentinel

Sentinel：監控 + 自動 failover

Sentinel 是另一個獨立進程（不是 redis-server 模式），專門做三件事：

  • 監控：每秒 ping master/replica，判斷是否 down。
  • 通知：發訊息給管理員（webhook / pub/sub）。
  • 自動 failover：master 掛了，從 replica 中選一個升 master，並通知 client 與其他 replica。

三個關鍵概念

  1. Quorum：「至少幾個 Sentinel 同意 master 已死，才真的 failover」。例如 3 個 Sentinel + quorum=2 = 半數以上才動手。
  2. Majority：除了 quorum，failover 還需要「Sentinel 群體中超過半數同意誰當新 master」 — 這是為了避免腦裂（split brain）。
  3. 至少 3 個 Sentinel：2 個會在網路分區時無法達成 majority；1 個是單點。業界標準是 3 個 Sentinel + 2 個 replica = 5 進程。
圖 R4.1 — 3 個 Sentinel + 1 master + 2 replica 是業界經典配置

Client 怎麼用

Client 不直接連 Redis master，而是先問 Sentinel「現在 master 是誰」，再連那個 IP。Master 換掉時 Sentinel 會 push 新 IP 給已連線的 client。

03 — Cluster

Cluster：16384 個 hash slot 分片

Redis Cluster 把 keyspace 切成 16384 個 hash slot，每個 master 負責其中一段。一個 key 用 CRC16(key) mod 16384 算出它屬於哪個 slot，再查表知道在哪個 master。

為什麼是 16384？

不是 16K 整數。原因是 Redis 用 bitmap 表示「我這個節點負責哪些 slot」 — 16384 bits = 2048 bytes。在叢集 gossip message 裡帶這個 bitmap 成本很低。如果用 65536 個 slot，bitmap 會變 8KB，gossip 流量翻倍且大多 slot bit 是 0 浪費。

Resharding（重新分片）

新增節點要把資料從現有節點移過去：

  1. redis-cli --cluster reshard 指定要搬幾個 slot、來源、目的。
  2. 來源節點把那段 slot 的 key 一個個 MIGRATE 到目標。
  3. 過程中 client 收到 ASK redirect（「這個 key 暫時去問另一節點」）。

客戶端必須 cluster-aware

Cluster 模式下，client 連任何一個節點，發指令時若 key 不在這個節點，會收到：

  • MOVED redirect：永久性 — 「這個 slot 在另一節點，以後 key 直接連那邊」。client 收到後更新自己的 slot map。
  • ASK redirect：臨時性（resharding 中）— 「這個 key 正在搬，這次去別處問」。一次性，下次仍連原節點。

所以 client lib 必須懂這兩個 redirect 並維護自己的 slot 映射表。常見 lib：go-redis, jedis, ioredis, lettuce 都支援。

04 — Hash Tag

Hash Tag {} ：multi-key 操作的鑰匙

Cluster 的硬規則：multi-key 指令的所有 key 必須在同一個 slot。例如：

  • MSET k1 v1 k2 v2
  • SUNION s1 s2 s3
  • EVAL "..." 2 k1 k2 ...（Lua 帶多 key）

因為它們可能跨節點，違反了「指令在單一節點原子執行」的前提。預設不允許。

但有時你就是要把同一個用戶的多個 key 放一起做原子操作。Hash tag 解決這個：

redis-cli 
# key 內第一對 {} 中的內容單獨拿來算 slot

SET {user:123}.profile "..."     # slot = CRC16("user:123") % 16384
SET {user:123}.cart    "..."     # 同上,所以兩 key 同 slot
SET {user:123}.session "..."     # 同上

# 現在這個可以了
MSET {user:123}.profile "j" {user:123}.cart "[]"
!
Hash tag 用過頭會破壞分片 如果所有 key 都用 {global}.xxx，所有資料就跑到同一個 slot、同一個 master — 等於沒分片。Hash tag 是「精準綁定一小撮相關 key」的工具，不是無腦套。
05 — 決策

該選哪個？

場景建議
資料量 < 32 GB（一台機器塞得下）+ 要 HA Master + 2 replica + 3 Sentinel
資料量 > 32 GB / 寫入 QPS > 5 萬 / 要水平擴展 Cluster（每 shard 自帶 replica，免另設 Sentinel）
純 cache，可接受重啟資料丟失 單機就好（或 client-side sharding）
使用雲端 managed Redis 看雲廠商提供的選項（多半 Sentinel 與 Cluster 都有）
i
「先 Sentinel，未來再 Cluster」可行嗎？ 可行但要規劃 — 從 master/replica 模式改 cluster 需要資料重灌或用 redis-cli --cluster import。建議：能預估到「未來 1~2 年資料量會超過單機」就直接上 cluster，避免日後痛苦遷移。
06 — 動手玩

用 docker-compose 起 3-master + 3-replica cluster

yaml — docker-compose.yml 
x-redis: &redis-base
  image: redis:8.6
  command: [
    redis-server,
    --cluster-enabled, yes,
    --cluster-config-file, nodes.conf,
    --cluster-node-timeout, "5000",
    --appendonly, yes,
    --port, "6379"
  ]

services:
  r1: { <<: *redis-base, ports: ["7001:6379"] }
  r2: { <<: *redis-base, ports: ["7002:6379"] }
  r3: { <<: *redis-base, ports: ["7003:6379"] }
  r4: { <<: *redis-base, ports: ["7004:6379"] }
  r5: { <<: *redis-base, ports: ["7005:6379"] }
  r6: { <<: *redis-base, ports: ["7006:6379"] }
bash 
$ docker compose up -d

# 把 6 個節點組成 cluster (3 master + 1 replica each)
$ docker exec -it $(docker compose ps -q r1) \
    redis-cli --cluster create \
      r1:6379 r2:6379 r3:6379 r4:6379 r5:6379 r6:6379 \
      --cluster-replicas 1 --cluster-yes

# 觀察狀態
$ redis-cli -p 7001 -c CLUSTER INFO
# cluster_state:ok
# cluster_slots_assigned:16384
# cluster_slots_ok:16384

# -c 是 cluster mode flag,自動 follow MOVED redirect
$ redis-cli -p 7001 -c SET hello world
-> Redirected to slot [866] located at r2:6379
OK

# 看一個 key 屬於哪個 slot/node
$ redis-cli -p 7001 -c CLUSTER KEYSLOT user:123
5798

# 用 hash tag 把多個 key 綁同 slot
$ redis-cli -p 7001 -c MSET {u:123}:a 1 {u:123}:b 2
