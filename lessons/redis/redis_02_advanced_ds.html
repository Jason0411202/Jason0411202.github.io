01 — Streams

Streams：認真的訊息佇列（5.0 引入）

Stream 是一個 append-only 的有序日誌，每筆 entry 有自動產生的 ID（ms-seq），支援 consumer group（多消費者協作 + 已讀追蹤），是 Redis 最像 Kafka 的結構。

redis-cli 
# 寫入訊息
XADD orders * user_id 100 amount 299 sku "AAPL"
"1745912345678-0"

XADD orders * user_id 101 amount 158 sku "TSLA"
"1745912345892-0"

# 建一個 consumer group（從頭開始消費）
XGROUP CREATE orders billing 0 MKSTREAM
OK

# consumer "worker-A" 拿一筆
XREADGROUP GROUP billing worker-A COUNT 1 STREAMS orders >
# 收到一筆訊息

# 處理完，ACK 掉
XACK orders billing 1745912345678-0
(integer) 1

# 看「還沒 ACK」的（pending entries list）
XPENDING orders billing
  • 多 consumer 協作：consumer group 確保一筆訊息只被組內一個 worker 收到。
  • 已讀追蹤：未 ACK 的訊息會留在 pending entries list，worker 重啟後可重拿。
  • Cap by length：用 XADD ... MAXLEN ~ 10000 限制長度，避免 memory 爆炸。
i
什麼時候用 Streams 而不是 Kafka？ 訊息量小（< 10 萬 msg/s）、要求低延遲、不需要長期保存（< 1 day）、團隊已經有 Redis — 用 Streams 省一個元件。訊息量大、要長期保存、有跨團隊接上下游的對接需求 — 用 Kafka。
02 — Pub/Sub

Pub/Sub：fire-and-forget 廣播（不可靠）

Redis 早期的訊息機制。publisher 發出去，沒有 subscriber 在線就丟掉，沒有訂閱歷史、沒有 ACK、沒有重試。

redis-cli (兩個視窗) 
# 視窗 ① 訂閱
127.0.0.1:6379> SUBSCRIBE news.tech
Reading messages...

# 視窗 ② 發布
127.0.0.1:6379> PUBLISH news.tech "Redis 8.6 GA released"
(integer) 1   # 1 個 subscriber 收到
!
Pub/Sub 不適合做訂單佇列 沒有持久化、沒有重試、subscriber 暫時離線就漏訊息。有「至少一次」需求請改用 Streams。Pub/Sub 適合：即時通知 (chat)、cache invalidation 廣播、metric streaming。

Redis 7.0 起新增 Sharded Pub/Sub（SPUBLISH / SSUBSCRIBE），訊息只會跨 cluster 中持有對應 hash slot 的節點，避免在大叢集內 fan-out 造成效能問題。

03 — Bitmap

Bitmap：用 bit 算「特徵」

Bitmap 不是獨立 type，而是 SETBIT / GETBIT / BITCOUNT / BITOP 一組指令操作 String 的二進位位元。1KB 字串就能放 8000 個 bit。

redis-cli 
# 場景：每天記錄哪些 user 上線（user_id 當 offset）
SETBIT active:2026-04-27 100 1   # user 100 來了
SETBIT active:2026-04-27 501 1   # user 501 來了

BITCOUNT active:2026-04-27          # 今天有幾個人上線
(integer) 2

# 算「昨天和今天都來」的人
BITOP AND retained active:2026-04-26 active:2026-04-27
BITCOUNT retained

1 億用戶用 1 億 bit = 12.5 MB，10 元美金的記憶體就能算出每天 DAU、留存率、A/B 分群。傳統 SQL 做這事要多少資源？

04 — HyperLogLog

HyperLogLog：12 KB 算 UV，誤差 0.81%

要算「不重複訪問人數」（UV / Unique Visitors），最直覺是用 Set 把每個訪客 ID 加進去再算 size。但 1 億用戶就要 1 億個字串，記憶體吃不下。

HyperLogLog 用機率近似演算法，無論基數多大都只用 12 KB，標準誤差 0.81%。

redis-cli 
PFADD uv:home user_a user_b user_c user_a   # 重複的 user_a 算一次
(integer) 1

PFCOUNT uv:home
(integer) 3

# 合併兩個 HLL（聯集）
PFMERGE uv:total uv:home uv:product
PFCOUNT uv:total
(integer) 4521
i
HLL 的限制 只能算個數，無法列出所有元素。如果你需要拿到「上線的用戶 ID 清單」，請改用 Set 或 Bitmap。
05 — Geo

Geo：附近 1 公里有什麼店

Geo 內部其實是 Sorted Set + GeoHash。經緯度經過 GeoHash 編成 52-bit int 當 score。所以你能用 ZSET 的所有指令操作 Geo。

redis-cli 
GEOADD shops 121.5654 25.0330 "taipei-101"
GEOADD shops 121.5170 25.0478 "ximen"
GEOADD shops 121.5436 25.0856 "shilin-night"

# 從台北車站(121.5170, 25.0478) 找半徑 3km 內的點
GEOSEARCH shops FROMLONLAT 121.5170 25.0478 \
  BYRADIUS 3 km ASC WITHCOORD WITHDIST

GEOSEARCH 是 6.2 後推薦的指令（取代舊的 GEORADIUS，後者已 deprecated）。

06 — Pipelining

Pipelining：把 100 條指令的 RTT 砍掉

每次 redis-cli 發指令到 server 都有一次網路 round-trip（RTT）。如果你要連送 1000 條指令，1ms RTT × 1000 = 1 秒。Pipelining 讓 client 把多條指令一次塞進 TCP buffer 一起送，server 也一次回 1000 條 reply。

圖 R2.1 — pipeline 把 N 次 RTT 壓縮成 1 次
bash 
# 用 redis-cli --pipe 灌 10 萬筆
$ seq 100000 | awk '{print "SET key:"$1" val:"$1}' \
    | docker exec -i redis8 redis-cli --pipe

All data transferred. Waiting for the last reply...
Last reply received from server.
errors: 0, replies: 100000
!
Pipeline 不是 Transaction Pipeline 只是「把多條指令一次送出」 — 中間還是可能被別的 client 指令穿插（但因為 single-thread，每條仍是原子）。要保證「這幾條指令連續執行不被穿插」，要用 MULTI/EXEC 或 Lua。
07 — Transactions

MULTI / EXEC + WATCH 樂觀鎖

Redis 的「transaction」跟 RDBMS 的 ACID 不太一樣，更像「指令打包後一次執行」：

redis-cli 
# 簡單 MULTI/EXEC：包成一個原子單位
MULTI
INCR account:1:balance
DECR account:2:balance
EXEC
1) (integer) 101
2) (integer) 49

# WATCH：樂觀鎖（CAS） 「如果中間有人改了 watched key，就 abort」
WATCH account:1:balance
GET account:1:balance      # app 邏輯：看餘額決定要不要扣
MULTI
DECR account:1:balance
EXEC
# 若中間有別人改 account:1:balance, EXEC 回 (nil) 表示 abort

注意：Redis transaction 不能 rollback。如果 MULTI 內某條指令執行失敗（例如 type error），其他指令照樣執行 — 跟 SQL 的 ROLLBACK 完全不同概念。

08 — Lua Scripting

Lua：把多條指令做成一條原子操作

Lua script 在 Redis server 端執行，整個 script 是一個原子操作。比 MULTI/EXEC 強的地方是 — 你可以基於前面指令的結果做分支判斷。

redis-cli 
# 場景：原子地「扣庫存,但不能扣到負」
EVAL "local cur = tonumber(redis.call('GET', KEYS[1]) or 0)
       if cur > 0 then
         redis.call('DECR', KEYS[1])
         return 1
       else
         return 0
       end" 1 stock:AAPL

# 8.0+ 可以用 FUNCTION（更現代的方式,可命名、可版本化）
FUNCTION LOAD "#!lua name=mylib
  redis.register_function('decr_safe', function(keys, args)
    local cur = tonumber(redis.call('GET', keys[1]) or 0)
    if cur > 0 then redis.call('DECR', keys[1]); return 1
    else return 0 end
  end)"

FCALL decr_safe 1 stock:AAPL
!
Lua 別寫太久 Lua 在單緒內跑 — script 執行多久，所有 client 就卡多久。生產環境的 script 應該 < 1 ms。複雜邏輯該往 application 層放。
