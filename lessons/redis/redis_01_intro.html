01 — 起點

為什麼會出現記憶體型資料庫

在 Redis 出現以前（2009 年），網站要做以下事情都很痛：

  • cache 一個熱資料：要嘛自己包 Memcached（純 KV、沒結構），要嘛去 hit 資料庫，每次 5–50ms。
  • 排行榜：MySQL 排序 1000 萬筆每秒幾百次？硬碟跟 index 都會冒煙。
  • 輕量級訊息佇列：用 RabbitMQ 太重、用 MySQL polling 太慢。
  • 分散式 lock：自己用 DB 的 row lock 寫，慘不忍睹。

Redis（REmote DIctionary Server）由 Salvatore Sanfilippo 創造、Antirez 為其代名。它的「驚人」之處不是「把資料放在記憶體」（這個 Memcached 早就做了），而是 —

i
一句話定位 Redis = 把資料結構當 server 開出來。它不只給你 KV，還給你 List、Hash、Set、Sorted Set、Stream、HLL、Bitmap、Geo、Vector Set，全部以原生指令操作，亞毫秒延遲。它是「在網路另一端的 std::map / std::set」。

圖 R1.1 — Redis 把記憶體中的資料結構搬到網路另一端，多進程共享

所以「cache 一個 user object」只是 Redis 能做的最簡單一件事。看到「排行榜」就用 Sorted Set；看到「最近 100 筆事件」就用 Stream；看到「附近 1km 的店家」就用 Geo；看到「使用者連續 30 天簽到」就用 Bitmap — 這些都是原生指令直接搞定，不用拉資料回來自己算。

03 — 為何快

單緒事件迴圈：簡單卻快的祕密

Redis 處理使用者指令的部分是 single-threaded（單一執行緒）。這是 Redis 從第一天就決定的設計，至今沒變（雖然 6.0+ 已加 I/O threads 處理網路收送、8.x 持續優化，但命令的執行仍是單緒）。為什麼選這個反直覺的設計？

  • 沒有 lock 競爭：所有指令依序執行，每個指令都是原子的。你要做一個原子計數器？INCR counter。要原子地拿出最舊的訊息？LPOP list。完全不需要自己處理 race condition。
  • 記憶體比 CPU 慢得多：對 KV 操作來說，瓶頸是 RAM access、不是 CPU。多緒能加多少？實測加 4 倍 thread 只快 1.5 倍，但複雜度爆炸。
  • I/O multiplexing：用 epoll / kqueue 在單一 thread 處理數萬條連線。網路是非同步的，CPU 永遠不會閒著。
!
單緒的代價：慢指令會卡住所有人 因為命令是依序執行，一個 KEYS * 跑 10 秒，就是所有 client 卡 10 秒。Lua script 跑太久也一樣。這是 Redis 最常見的生產事故，R-06 會專門講。先記住一句：「Redis 的指令必須是 sub-ms。」
04 — 五大資料結構

必背的五個資料結構（基礎）

八成的 Redis 用法都繞著這五個。R-02 會展開到 Streams、HLL、Geo、Bitmap、Vector Set。

結構類比常用指令典型場景
String str / int / bytes SET, GET, INCR, EXPIRE cache、計數器、SESSION token
List deque LPUSH, RPUSH, LPOP, BLPOP 輕量訊息佇列、最新 N 筆
Hash map<str, str> HSET, HGET, HGETALL, HINCRBY 儲存物件（user / product）
Set set<str> SADD, SISMEMBER, SUNION, SINTER tag、好友、IP 黑名單
Sorted Set map<str, score> 排序 ZADD, ZRANGE, ZINCRBY, ZRANGEBYSCORE 排行榜、延遲佇列、滑動窗口 rate limiter
i
Sorted Set 是 Redis 最被低估的結構 多數初學者只用前四個。實際上 Sorted Set 解決了 排行榜、滑動視窗 rate limiter、延遲佇列、有序事件流 等一拖拉庫進階場景。R-02 + R-06 會展開。
05 — 跟同類比較

Redis 旁邊有哪些選項

專案定位Redis 比較取捨
Memcached 純 KV cache 沒有資料結構、沒有持久化、沒有複寫 更簡單，記憶體用得稍少；其餘輸 Redis
Valkey Redis 7.2 fork（Linux Foundation） API 100% 相容到 7.2，由 AWS / Google / Oracle 主導 Redis 7.4 改 license 後的「OSS 版本」；很多雲商正在移轉
DragonflyDB 多緒重寫的 Redis 替代 API 大致相容；單機吞吐宣稱 25× Redis 授權商業友好；少數高階指令仍未支援
KeyDB 多緒 Redis fork（已被 Snap 收購） API 相容，多 thread 社群活躍度走低，慎選
i
關於 license 與 Valkey Redis 7.4 開始改成 RSALv2 / SSPL（限制雲廠商重打包）。Linux Foundation 因此 fork 了 Valkey，採 BSD license。目前 Redis 仍是主流且 Redis 8.x 的新模組（Search / Vector Sets）功能領先；本系列以 Redis 8.6 為準，但 API 大致也適用於 Valkey。
06 — 適用判斷

什麼時候該用 Redis？什麼時候別用？

✓適合
  • 快取：DB 查詢結果、HTML 片段、API response。
  • session / token store：分散式 web app 的 session。
  • 排行榜：Sorted Set 一條指令。
  • rate limiter / 限流：滑動窗、token bucket。
  • 輕量訊息佇列：Stream（reliable）或 Pub/Sub（fire-and-forget）。
  • 分散式 lock：`SET NX PX`。
  • 計數 / 統計：INCR、HyperLogLog 算 UV。
  • 向量相似度搜尋（小規模）：8.x 的 Vector Sets。
✗不適合
  • 當主要資料庫：記憶體成本貴。雖能持久化但複雜場景仍不如 PostgreSQL。
  • 儲存大物件（單一 value > 100KB）：阻塞單緒、複寫變慢。
  • 需要強一致 + 高可用：Redis 主從是非同步複製，failover 可能掉資料（R-03 會講）。
  • 需要複雜查詢 / 全文搜尋（大規模）：可用 RediSearch，但 ElasticSearch 仍是更好的選項。
  • 非常大的 dataset（> 100GB）：記憶體成本爆炸。考慮 KV on-disk 解（如 RocksDB）或加 Cluster。
07 — 動手玩

3 分鐘起一個 Redis 並敲指令

終端 ① 起 Redis 8.6

bash 
$ docker run --rm --name redis8 \
    -p 6379:6379 \
    redis:8.6

# 看到 "Ready to accept connections tcp" 即可
# Redis 開機 1 秒以內，幾乎沒有等待

終端 ② 用 redis-cli

bash 
$ docker exec -it redis8 redis-cli

127.0.0.1:6379> PING
PONG

# 五大結構各一範例
127.0.0.1:6379> SET user:1:name "Jason"
OK
127.0.0.1:6379> GET user:1:name
"Jason"

127.0.0.1:6379> INCR page:home:views
(integer) 1
127.0.0.1:6379> INCR page:home:views
(integer) 2

127.0.0.1:6379> HSET user:1 name "Jason" age 28 city "Taipei"
(integer) 3
127.0.0.1:6379> HGETALL user:1
1) "name"
2) "Jason"
3) "age"
4) "28"

127.0.0.1:6379> LPUSH events "login" "view-product" "add-cart"
(integer) 3
127.0.0.1:6379> LRANGE events 0 -1
1) "add-cart"
2) "view-product"
3) "login"

127.0.0.1:6379> ZADD leaderboard 1500 "alice" 2300 "bob" 1800 "carol"
(integer) 3
127.0.0.1:6379> ZREVRANGE leaderboard 0 -1 WITHSCORES
1) "bob"     2) "2300"
3) "carol"   4) "1800"
5) "alice"   6) "1500"

# EXPIRE：5 秒後自動消失
127.0.0.1:6379> SET short:msg "hi" EX 5
OK
127.0.0.1:6379> TTL short:msg
(integer) 4

# 觀察延遲
127.0.0.1:6379> DEBUG SLEEP 0
OK

看一筆指令的真實延遲

bash 
$ docker exec -it redis8 redis-cli --latency
min: 0, max: 1, avg: 0.18 (1245 samples)  # 0.18 ms 平均延遲

$ docker exec -it redis8 redis-cli MONITOR
# 看到所有指令即時印出 — 但 production 千萬不要開,會慢死
i
所有指令都是同步阻塞的 Redis 沒有「async 指令」這種東西（除了 client 端的 pipeline）。你發一個指令、收到回應，這之間 server 一定處理完。簡單到不可思議。
